# AI x Web3 School - AI 在 Web3 的应用 总结

## 核心观点

2026.05.21这场分享主要讨论 AI 与 Web3 如何结合。讲者认为，AI 当前高度中心化，算力、模型和数据集中在少数组织或服务商手中；Web3 则强调去中心化账本、开放协作和链上验证。两者结合的价值不只是“发币”或包装概念，而是让 AI 的算力、数据、身份、支付、验证和执行能力可以通过更开放的基础设施协同起来。

## 主要内容

1.  AI 的中心化问题  
    讲者提到，OpenAI 等 AI 服务会受到地区、政策、公司规则和账号风控影响，用户访问和稳定使用并不总是顺畅。AI 如果长期依赖少数中心化服务商，就会带来访问限制、封号、算力集中和数据集中等问题。
    
2.  Web3 对 AI 的可能赋能  
    Web3 可以尝试把 AI 的算力、存储、模型服务分散到全球节点，由个人或组织运行节点并提供模型/API 能力。分享中提到类似 Bittensor 的方向：节点不只是质押代币，而是提供算力或模型服务，由网络或网关把用户请求分发到不同节点。
    
3.  当前分布式 AI 的限制  
    讲者也指出，分布式推理和分布式任务处理目前还没有非常成熟。复杂任务拆分后，效果往往不如把完整上下文直接交给一个强模型处理。短期内，许多方案还处于探索阶段，商业价值和用户体验未必已经跑通。
    
4.  AI Agent 与链上能力结合  
    讨论中提到，可以让 AI 理解交易意图、分析风险、辅助钱包交互、帮助收集信息和生成文档。钱包安全助手、交易风险评分、链上操作辅助、信息检索和自动化执行，是比较自然的结合方向。但真正可用的产品还需要解决安全、可控性、执行权限和用户信任问题。
    
5.  产品与创业建议  
    讲者强调不要只为了“AI + Web3”而做项目，关键还是找到真实问题和清晰场景。AI 降低了开发门槛，但一个项目仍然需要产品设计、运营、推广和团队协作。一个人可以更快做原型，但产品要真正跑起来，通常仍需要团队。
    
6.  学习路径与求职建议  
    技术同学可以从 EVM、Solidity、合约结构和知名 DeFi 合约入手，理解区块链账本和合约执行逻辑，再通过黑客松或项目积累经验。非技术同学可以从运营、产品、社区和项目研究切入，先理解 Web3 产品如何运转，再逐步补足技术基础。交易所等机构可能有英文面试要求，但不同组织差异较大。
    
7.  面对 AI 焦虑  
    讲者提到大家不必过度焦虑。AI 概念更新很快，不可能每个新词都马上掌握。更重要的是持续学习、持续尝试，把 AI 当作日常工具，用它做信息整理、代码辅助、文档生成和项目验证。
    

## 一句话结论

AI 与 Web3 的结合目前仍在早期，真正有价值的方向不是简单贴标签，而是用 Web3 解决 AI 的中心化、身份、支付、验证和协作问题，同时用 AI 降低 Web3 产品的使用和开发门槛。

\# AI 时代，Web3 开发者需要具备的基础知识和架构能力 > 主讲：TC（安全行业出身，香港 Web3 公司技术专家） > 时间：2026年05月19日 --- ## 一、常见问题解答 ### 有了 AI 还需要学基础知识吗？ \*\*结论：更需要。\*\* - 基础知识不足，无法判断 AI 给出的方案是否正确 - 人与 AI 的协作边界： - \*\*人\*\*：设计方案、评估合理性、验收代码、分析问题 - \*\*AI\*\*：协助细化方案、梳理数据流/请求流、按 plan 执行编码 - AI 编码高质量的关键：\*\*AI 自我验证\*\*（不断写代码 + 不断验证） ### Web3 和 Web2 是完全不同的体系吗？ \*\*结论：不是。\*\* - 大部分 Web3 项目中，Web2 能力占比很高（如 CEX、Polymarket、DeFi 前端） - Web3 ≈ Web2 + 链上逻辑，二者深度融合 ### 安全应该如何对待？ - \*\*安全源于设计\*\*，而非上线后补救 - Web3 资金交互特性决定：出错即归零，不可逆 - 安全无法证伪，做得越多越安全，应贯穿项目全生命周期 --- ## 二、系统设计：以支付系统为例 ### Web2 支付流程 \`\`\` 用户 → 电商平台 → 支付服务 → Bank Institution → 回调 → 电商平台 \`\`\` \*\*资金流\*\*：用户银行卡 → 支付服务账户 → 定时清结算给商户 \*\*安全考量\*\*： - 服务间 API Key / Secret Key 身份验证 - 用户侧支付密码、MFA、风控 - 信任问题：各方需互相信任 ### Web3 支付流程 \`\`\` 用户 → 电商平台 → 生成地址 → 用户转账 → 链上记录 → 链上监听服务 → 支付服务回调 \`\`\` \*\*核心变化\*\*：Bank Institution → Blockchain（本质定位相同） \*\*Web3 如何解决信任问题\*\*： - 数据不可篡改（共识机制：PoW、PoS 等） - 逻辑代码公开透明（智能合约） - 私钥签名替代 API Key 验证 --- ## 三、钱包核心概念 ### 钱包的本质 > \*\*输入\*\*：Private Key（随机数）→ \*\*签名算法\*\*（ECDSA / EdDSA）→ \*\*输出\*\*：Signature 钱包在 Web3 中扮演\*\*身份 + 资产\*\*的双重认证角色： - \*\*消息签名\*\*：证明身份 - \*\*交易签名\*\*：证明意图、确认转账 ### 钱包分类 | 维度 | 类型 | 说明 | |------|------|------| | 账号类型 | EOA 钱包 | 私钥钱包，最常见 | | | 合约钱包 | 链上程序实现（如 Safe 多签） | | | AA 钱包 | 融合 EOA + 合约，未来方向 | | 控制方式 | 单签 | 一个私钥控制 | | | 多签 | 多人持有，N-of-M 同意才可操作 | | 托管方式 | 自托管 | 私钥自己保管，最安全 | | | 全托管 | 交易所模式，私钥在平台 | | | 混合托管 | MPC 方案，私钥切片分持 | | 其他 | 硬件钱包 | Ledger 等，抗软件攻击 | | | 隐私钱包 | Railgun 等，基于 ZK | | | 分层钱包 | HD Wallet，BTC 常用 | ### 钱包安全风险 1. \*\*私钥泄露\*\*：私钥丢失 = 资产全失，不可逆 2. \*\*签名欺骗（钓鱼）\*\*：防护方案 → EIP-712 签名可视化 3. \*\*权限滥用\*\*：谨慎授权合约操作 --- ## 四、交易生命周期 ### 关键参数 - \*\*Gas Fee\*\*：\`Gas Count × Gas Price\`（EIP-1559：Base Fee + Tip Fee） - \*\*Nonce\*\*：防止重放攻击，EVM 要求严格递增 - \*\*Call Data\*\*：链上应用交互的灵魂，赋予链生态多样性 ### 各链 Nonce 实现对比 | 链 | Nonce 机制 | |----|-----------| | EVM（以太坊） | 严格递增 Nonce | | Solana | 使用最近 Block Hash，无 Nonce | | BTC | UTXO 模型，天然防重放 | ### 交易监听 1. 监听每个 Block 中的交易（轮询或 WebSocket） 2. 等待足够的区块确认（防止链重组 reorg） 3. 拿到 Confirmation 后进行合规筛查（KYA / KYT） 4. 回调支付服务 ### 交易模拟 - \*\*必须在签名前模拟\*\*，签名后模拟无意义 - 作用：预览交易执行结果，防止误签恶意交易 --- ## 五、服务端钱包安全 - \*\*权限拆分\*\*：资金量分级 + 多签方案，避免单点风险 - \*\*可视化审核\*\*：大额交易人工审核 - \*\*TEE 环境签名\*\*：私钥在可信执行环境（Intel SGX / AWS Nitro）中解密签名，明文不出环境 --- ## 六、AI 时代的变与不变 ### AI 改变了什么 - 编码速度大幅提升（多 Agent 编码框架） - 学习门槛降低，AI 是超级专家 ### AI 没有改变什么 - \*\*系统复杂度\*\*：判断系统架构、复杂度仍靠人脑 - \*\*安全责任\*\*：AI 生成代码出了安全问题，开发者来背锅 - \*\*最终审核权\*\*：方案思考、架构设计、功能判断仍需人来决策 > 核心理念：\*\*驾驭 AI，而不是被 AI 驱动\*\* --- ## 七、AI 时代架构师的核心能力模型 \`\`\` 基础知识能力（地基） ↓ 架构能力（系统设计、分层拆解） ↓ Debug 能力（深入底层协议，从下到上梳理逻辑） \`\`\` \*\*三条建议\*\*： 1. 基础知识一定要扎实，这是驾驭 AI 的前提 2. 根据目标岗位倒推需要的能力，有的放矢 3. 找到真正感兴趣的方向，坚持做下去 --- ## 八、推荐资源 - \*\*密码学\*\*：B站 Lindell（密码学博士，椭圆曲线 / MPC 算法讲解极佳） - \*\*多链收款参考\*\*：开源项目 \`EP-USDT\`（GMO 团队） - \*\*链上数据分析\*\*：DeBank、The Graph 等 - \*\*Web3 安全\*\*：Claude Code 的 MissOS（AI 辅助安全审计）
