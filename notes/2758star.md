## 推理服务（Inference）学习笔记

## 一、学习主题概述

本次学习的主题是 **推理服务（Inference）**。

在人工智能系统中，训练和推理是两个非常重要但作用不同的阶段。**训练决定模型学到了什么，推理决定模型如何在真实产品中响应用户。** 如果说训练阶段更关注模型能力的形成，那么推理阶段更关注模型能力如何被稳定、快速、低成本、安全地提供给用户。

很多人在刚开始接触 AI 应用开发时，会认为只要调用一个大模型 API，就已经完成了 AI 产品的核心功能。但实际上，当一个 AI 应用从 Demo 进入真实产品环境后，推理层的问题会迅速暴露出来。例如，用户等待时间太长、接口调用费用过高、模型输出不稳定、服务超时、隐私数据是否可以发送给第三方模型、模型版本变化是否会影响产品结果等。

因此，Inference 并不是一个简单的“调用模型”动作，而是一个综合性的工程系统。它涉及模型选择、部署方式、成本控制、延迟优化、上下文管理、服务稳定性、监控、日志、降级方案以及安全合规等多个方面。

我对这一部分的核心理解是：

> 推理服务不是让模型“跑出答案”，而是在真实约束条件下，让模型持续、稳定、可控地交付可用答案。

* * *

## 二、为什么要学习 Inference

学习 Inference 的原因在于，AI 产品最终面对的是用户，而用户体验很大程度上取决于推理层。

一个模型在实验环境中表现很好，并不意味着它可以直接进入产品系统。真实产品通常会面对以下问题。

### 1\. 响应速度问题

用户在使用 AI 产品时，通常希望系统能尽快返回结果。如果一个 AI 助手需要等待几十秒才能回答问题，即使答案质量较高，用户体验也会受到影响。

例如，在一个 AI × Web3 产品中，如果用户正在查看一笔链上交易，希望 AI 帮助判断风险，那么推理服务必须在较短时间内完成分析。如果等待时间过长，用户可能已经错过操作时机，或者直接放弃使用产品。

因此，推理服务需要关注延迟，包括：

-   首 token 延迟，即用户多久能看到模型开始输出；
    
-   总响应时间，即完整答案生成需要多久；
    
-   多轮 Agent 调用的整体耗时；
    
-   外部工具调用、RAG 检索和模型生成之间的协调时间。
    

### 2\. 成本控制问题

大模型 API 通常按照 token 数量计费。一个 Demo 阶段的小应用可能成本很低，但当用户数量增加后，推理成本会迅速放大。

例如，一个用户每天调用 20 次模型，每次请求消耗几千 token。如果有上千名用户同时使用，整体成本就会变得非常高。尤其在 Agent 场景中，一次用户请求可能不是一次模型调用，而是多次模型调用、工具调用和结果整理的组合。因此，推理成本往往会被进一步放大。

这说明 AI 产品不能只考虑“能不能回答”，还要考虑“每次回答值不值得”。

### 3\. 稳定性问题

推理服务还需要面对各种不稳定情况，例如：

-   模型 API 超时；
    
-   第三方服务限流；
    
-   网络连接失败；
    
-   GPU 服务崩溃；
    
-   模型输出格式不符合预期；
    
-   Agent 调用工具失败；
    
-   长上下文请求导致响应变慢。
    

如果没有稳定性设计，AI 产品在真实流量下很容易出现用户体验下降。尤其是面向金融、Web3、企业服务等场景时，系统不能依赖“模型大概率能回答”这种不确定状态，而必须具备明确的失败处理机制。

### 4\. 隐私与合规问题

在一些场景中，用户输入可能包含敏感信息，例如：

-   钱包地址；
    
-   交易记录；
    
-   企业内部文档；
    
-   用户身份信息；
    
-   财务数据；
    
-   合约代码；
    
-   私有业务逻辑。
    

如果直接将这些内容发送给第三方模型 API，就可能涉及隐私和合规风险。因此，推理服务需要根据业务场景判断哪些数据可以发送到云端模型，哪些数据必须在本地或私有环境中处理。

### 5\. 产品可持续性问题

AI 产品不能只依赖某一个模型或某一种调用方式。因为模型可能会更新，价格可能变化，API 可能出现故障，输出效果也可能发生改变。如果产品架构和某个模型深度绑定，未来切换模型会非常困难。

因此，推理服务需要具备可替换性和可扩展性。比较成熟的方式是将模型调用封装成统一接口，让产品能够根据任务类型、成本要求和失败情况选择不同模型。

## 三、Inference 的第一性原理

推理服务的第一性原理可以概括为：

> 在质量、延迟、成本、隐私和运维复杂度之间做平衡。

模型推理不是孤立存在的。它需要在具体产品约束中运行。

同一个模型，在不同部署环境下可能会有完全不同的表现。例如：

-   通过云端 API 调用，部署简单，但成本和数据外发问题较明显；
    
-   在自己的 GPU 服务器上部署，控制权更高，但运维压力更大；
    
-   在本地设备上运行量化模型，隐私更好，但模型能力可能下降；
    
-   在边缘设备上运行小模型，响应可能更快，但能力边界更明显。
    

因此，选择推理方式不是单纯选择“最强模型”，而是选择最适合当前产品目标的方案。

### 1\. 质量有代价

更强的模型通常意味着更高的成本、更长的延迟，或者更复杂的部署方式。

例如，一个复杂的合约风险分析任务可能需要能力较强的大模型来完成，因为它涉及代码理解、金融风险判断和上下文推理。但如果只是做简单的文本分类、标签识别或字段抽取，使用大模型就可能造成资源浪费。

所以在产品中，不同任务应该匹配不同模型，而不是所有任务都使用最贵、最大的模型。

### 2\. 部署方式决定控制边界

如果使用 API Model，开发者可以减少基础设施管理，把主要精力放在业务逻辑上。但同时，开发者对模型运行环境、版本变化、数据流向和底层性能的控制较少。

如果使用 Local Model，开发者可以获得更高控制权，例如可以决定模型版本、推理框架、日志策略和数据保存方式。但代价是需要承担更多硬件、部署、监控和维护工作。

因此，API 和本地部署并不是简单的优劣关系，而是控制权和复杂度之间的取舍。

### 3\. 服务要可替换

一个成熟的 AI 产品不应该把业务逻辑完全写死在某一个模型上。更合理的做法是设计清晰的模型调用层，例如：

```text
应用层
 ↓
模型调用接口
 ↓
模型 A / 模型 B / 本地模型 / Fallback 模型
```

这样，当某个模型出现问题时，系统可以切换到备用模型；当新模型上线时，也可以进行灰度测试；当成本过高时，可以将部分简单任务迁移到更便宜的模型。

## 四、API Model：托管模型 API

### 1\. API Model 的含义

API Model 指的是通过云端服务调用模型。开发者将输入内容发送给模型服务提供方，模型返回文本、结构化输出、工具调用结果或多模态结果。

常见的托管模型 API 包括 OpenAI、Anthropic、Google、DeepSeek 等提供的模型服务。

它的基本流程可以理解为：

```text
用户输入
 ↓
应用后端
 ↓
模型 API
 ↓
模型生成结果
 ↓
返回给用户
```

### 2\. API Model 的优势

API Model 最大的优势是上手快。开发者不需要购买 GPU，也不需要自己部署模型，只要通过接口调用就可以使用先进模型能力。

此外，API Model 通常由模型厂商负责底层基础设施，包括模型加载、推理优化、系统扩容、安全更新等。这对中小团队和早期产品非常重要，因为它降低了 AI 应用开发门槛。

API Model 也适合快速验证产品想法。例如，一个团队想做 AI 投研助手、AI 客服、AI 合约解释工具，最开始可以直接使用 API 快速做出 Demo，再根据用户反馈逐步优化架构。

### 3\. API Model 的局限

但是，API Model 并不意味着没有工程问题。

首先是成本问题。模型 API 通常按照输入和输出 token 收费。用户数量越多、上下文越长、Agent 调用越复杂，成本越高。

其次是速率限制问题。模型服务商通常会限制单位时间内的请求数量。如果产品突然出现大量用户访问，可能会触发限流，导致请求失败。

再次是版本变化问题。模型厂商可能会更新模型版本。模型能力提升是好事，但输出风格、格式稳定性和任务表现也可能发生变化。如果没有评估机制，模型升级可能破坏原有产品功能。

最后是隐私问题。如果用户输入包含敏感数据，直接发送到第三方 API 可能不适合。

### 4\. 对 API Model 的理解

我过去容易把“调用 API”理解成 AI 应用的全部。但学习后发现，API 只是推理服务的一种形式。真正的产品还需要处理：

-   请求超时；
    
-   失败重试；
    
-   日志脱敏；
    
-   成本统计；
    
-   输出格式校验；
    
-   模型版本管理；
    
-   安全过滤；
    
-   fallback 机制。
    

也就是说，API Model 降低了模型使用门槛，但并没有消除产品工程问题。

## 五、Local Model：本地模型

### 1\. Local Model 的含义

Local Model 指在自己的设备、服务器或私有环境中运行模型，而不是通过第三方 API 调用模型。

常见的本地模型包括 Llama、Qwen、Mistral、Gemma 等开源或开放权重模型。

本地模型可以部署在：

-   个人电脑；
    
-   公司服务器；
    
-   私有云；
    
-   GPU 集群；
    
-   边缘设备。
    

### 2\. Local Model 的优势

本地模型最大的优势是数据可控。模型运行在自己的环境中，数据不需要发送给外部服务。这对于隐私敏感场景非常重要。

例如，企业内部知识库问答、用户资产分析、链上交易风险识别、合约审计辅助等场景，都可能涉及敏感信息。如果这些数据不能外发，那么本地模型就是一个重要选择。

本地模型的第二个优势是长期成本可能更低。虽然初始部署需要硬件和技术投入，但当请求量足够大时，自己部署模型可能比长期调用 API 更划算。

第三个优势是可定制性更强。团队可以对模型进行微调、LoRA 训练、领域数据适配，或者根据业务需要调整推理框架。

### 3\. Local Model 的限制

本地模型的限制也很明显。

首先是硬件要求。模型越大，需要的显存和计算资源越多。对于大参数模型，如果没有高性能 GPU，很难保证推理速度。

其次是部署复杂。开发者需要处理模型权重加载、推理框架选择、显存占用、并发请求、服务监控、日志管理等问题。

再次是模型能力边界。很多本地模型在简单任务上表现不错，但在复杂推理、长上下文、多语言、代码理解和工具调用方面，可能不如顶级云端模型稳定。

### 4\. 对 Local Model 的理解

我认为本地模型不是简单地替代 API Model，而是应该在系统中承担合适的任务。

例如：

```text
复杂推理、综合分析、长文本理解
→ 使用云端大模型

简单分类、字段抽取、隐私预处理
→ 使用本地模型

云端 API 不可用
→ 使用本地模型作为 fallback
```

这种混合架构比单一依赖某个模型更加灵活。

## 六、Quantization：量化

### 1\. Quantization 的含义

Quantization，即量化，是指降低模型权重或计算精度，从而减少模型占用的显存和计算资源。

例如，模型原本使用 FP16 精度，量化后可以使用 INT8 或 INT4 精度。

简单理解就是：

```text
更高精度
→ 更大体积、更高显存、更强质量

更低精度
→ 更小体积、更低显存、更快运行
```

### 2\. 为什么需要量化

大模型通常体积很大。参数越多，占用的显存越高。普通消费级显卡或个人电脑很难直接运行大型模型。

量化可以让模型变小，使它能够在更低硬件条件下运行。例如，一些 7B、13B 甚至更大的模型经过量化后，可以在个人电脑上运行。

这对开发者非常重要，因为它降低了本地模型实验和部署门槛。

### 3\. 量化的优势

量化主要有三个优势。

第一，节省显存。模型体积变小后，可以部署在更便宜的硬件上。

第二，提高推理速度。低精度计算通常更快，因此模型响应可能更快。

第三，降低部署成本。原本需要高端 GPU 才能运行的模型，量化后可能可以在较低成本设备上运行。

### 4\. 量化的风险

量化不是没有代价。降低精度可能导致模型输出质量下降。

这种下降在简单聊天中不一定明显，但在以下场景中可能比较明显：

-   数学推理；
    
-   代码生成；
    
-   多语言任务；
    
-   长上下文理解；
    
-   工具调用；
    
-   严格格式输出；
    
-   合约分析；
    
-   风险判断。
    

例如，在 AI × Web3 场景中，如果模型需要分析合约函数风险，量化模型可能因为细节理解能力下降而漏掉关键问题。这种错误可能带来较大风险。

### 5\. 对量化的理解

我认为量化的关键不是“模型能不能跑起来”，而是“模型能不能在目标任务中稳定可用”。

因此，判断量化模型是否适合产品，不能只通过主观聊天体验，而应该使用真实业务样本进行评估。例如，可以准备一批合约分析样本、交易风险样本、ABI 抽取样本，比较不同量化版本的准确率、稳定性和输出格式。

## 七、Serving：模型服务化

### 1\. Serving 的含义

Serving 是指将模型部署成稳定服务，让应用可以通过接口调用模型。

如果说模型本身提供能力，那么 Serving 负责让这种能力能够被产品系统稳定消费。

Serving 通常包括：

-   模型加载；
    
-   请求接收；
    
-   队列管理；
    
-   批处理；
    
-   并发控制；
    
-   流式输出；
    
-   缓存；
    
-   日志；
    
-   监控；
    
-   健康检查；
    
-   扩容缩容；
    
-   版本管理。
    

### 2\. Serving 为什么重要

模型能力再强，如果 Serving 做不好，用户仍然无法获得良好体验。

例如，一个模型本身回答很好，但在线上经常出现：

-   请求排队太久；
    
-   服务超时；
    
-   GPU 利用率低；
    
-   并发一高就崩溃；
    
-   输出过程无法监控；
    
-   日志里保存了敏感数据；
    
-   模型升级后结果变差。
    

这些问题都属于 Serving 层需要解决的问题。

### 3\. Serving 中的关键能力

第一，并发处理

真实产品可能同时有很多用户请求模型。Serving 层需要决定如何处理并发请求，是直接执行、排队，还是做批处理。

第二，请求队列

当请求量超过系统承载能力时，请求不能无限制地直接进入模型，否则系统可能崩溃。队列可以帮助系统平稳处理高峰流量。

第三，流式输出

流式输出是提升用户体验的重要方式。即使完整答案需要较长时间生成，只要用户能先看到模型开始回答，就会感觉响应更快。

ChatGPT 就是典型的流式输出体验。

第四，缓存

有些请求可能是重复的，例如同一份文档摘要、同一笔交易解释、同一个合约 ABI 的方法抽取。如果每次都重新调用模型，会浪费成本。缓存可以减少重复推理。

第五，监控

Serving 层需要监控关键指标，例如：

-   平均延迟；
    
-   P95 延迟；
    
-   错误率；
    
-   超时率；
    
-   token 消耗；
    
-   GPU 利用率；
    
-   单次请求成本；
    
-   fallback 触发次数。
    

这些指标可以帮助团队判断系统是否健康。

第六，版本管理

模型不是固定不变的。产品可能需要从旧模型切换到新模型。Serving 层要支持模型版本管理和灰度发布。

例如，可以先让 5% 的用户使用新模型，观察质量、成本和错误率，再决定是否全量切换。

### 4\. Serving 中必须回答的问题

一个成熟的推理服务至少要回答以下问题：

请求失败时怎么办？

可以设计重试机制。如果第一次调用失败，可以自动重试一次。如果仍然失败，可以切换备用模型，或者返回较简单的降级答案。

模型版本如何灰度？

不能直接把所有用户切到新模型。应该先做小流量测试，并配合评估指标观察效果。

输入输出日志如何脱敏？

日志对排查问题非常重要，但不能直接保存敏感数据。比如钱包地址、身份信息、私有密钥、企业内部资料等，都需要脱敏处理。

长请求是否进入队列？

对于长文档总结、复杂合约分析、多轮 Agent 任务，系统可以将任务放入队列，避免阻塞普通请求。

成本和错误率如何监控？

没有监控就无法优化。推理服务必须知道成本花在哪里、失败发生在哪里、延迟高在哪里。

### 5\. 对 Serving 的理解

Serving 是我认为最容易被初学者忽视但最接近真实工程的部分。

很多 AI Demo 看起来效果不错，是因为用户少、任务简单、失败成本低。但进入产品阶段后，Serving 决定了模型能力能否真正被用户稳定使用。

所以，AI 产品不是“模型越强越好”，而是“模型能力必须被服务系统稳定承载”。

## 八、Inference 与 Prompt、RAG、Agent 的关系

Inference 并不是孤立的一层，它连接了很多 AI 应用中的关键模块。

### 1\. 与 Prompt 的关系

Prompt 是模型输入的一部分。Prompt 设计不好，模型输出就会不稳定。

例如，如果要求模型输出 JSON，但 Prompt 没有明确格式，模型可能生成自然语言说明，导致下游系统解析失败。

因此，推理服务中需要对 Prompt 进行版本管理和测试。

### 2\. 与 RAG 的关系

RAG 会在模型推理前检索外部知识。推理层需要处理检索结果的拼接、上下文长度控制、引用来源管理和无关信息过滤。

如果 RAG 检索内容质量不好，即使模型很强，也可能生成错误答案。

### 3\. 与 Agent 的关系

Agent 场景下，模型可能多次推理，并调用多个工具。一次用户请求可能包括：

```text
理解任务
 ↓
选择工具
 ↓
调用工具
 ↓
阅读工具结果
 ↓
继续推理
 ↓
生成最终回答
```

这会显著增加成本和延迟。因此，Agent 对 Inference 的要求更高，需要更严格的监控、超时控制和失败处理。

* * *

## 九、Inference 在 AI × Web3 中的位置

AI × Web3 场景对推理服务有更高要求，因为 Web3 涉及链上数据、资产风险和不可逆操作。

在普通聊天场景中，模型回答错了，用户可能只是重新提问。但在 Web3 场景中，如果 AI 参与交易解释、合约交互、策略建议或 Agent 执行，错误可能导致资产损失。

### 1\. 链上数据复杂

链上数据通常包括：

-   钱包地址；
    
-   交易哈希；
    
-   合约地址；
    
-   ABI；
    
-   事件日志；
    
-   token 转账记录；
    
-   DeFi 协议交互。
    

这些信息需要模型理解，也需要工具和数据源支持。Inference 层要负责把这些数据合理输入给模型，并控制上下文长度。

### 2\. 用户意图需要准确识别

例如，用户说：

> 帮我看看这笔交易有没有风险。

模型需要判断用户真正需要的是风险解释，而不是直接执行交易。

如果 AI Agent 具备操作能力，那么推理服务必须区分：

-   解释型任务；
    
-   建议型任务；
    
-   执行型任务。
    

不同任务需要不同安全边界。

### 3\. 推理过程需要可审计

由于链上行为不可逆，AI × Web3 产品中的推理过程应该留下可审计记录。

例如，需要记录：

-   使用了哪个模型；
    
-   输入数据来自哪里；
    
-   是否使用 RAG；
    
-   是否调用链上工具；
    
-   模型输出是什么；
    
-   是否触发风险提示；
    
-   是否发生 fallback；
    
-   最终是否建议用户确认操作。
    

这样，当用户对结果产生疑问时，系统可以追踪推理过程，而不是只能说“模型当时就是这样回答的”。

### 4\. 风险提示必须稳定

在 Web3 场景中，AI 不能只追求回答自然，还要追求风险提示稳定。

例如，如果合约存在授权风险、钓鱼风险、恶意转账风险，模型必须能够稳定识别并提醒用户。推理服务需要通过评估集不断测试这些风险场景，避免模型版本更新后出现退化。

* * *

## 十、最小实践设计

为了更好理解 Inference，可以做一个最小推理对比实验。

### 实验任务

选择一个具体任务，例如：

> 总结一笔链上交易的潜在风险。

输入内容可以包括：

-   交易哈希；
    
-   交易调用方法；
    
-   token 转账变化；
    
-   合约地址；
    
-   用户钱包变化；
    
-   是否涉及授权；
    
-   是否调用未知合约。
    

### 方案一：使用托管 API 模型

使用云端大模型进行分析。

优点

-   理解能力强；
    
-   输出更自然；
    
-   对复杂上下文处理更好；
    
-   不需要自己部署模型。
    

缺点

-   成本较高；
    
-   数据需要发送到第三方；
    
-   请求可能受到限流；
    
-   模型版本变化可能影响输出。
    

### 方案二：使用本地模型或小模型

使用本地部署模型进行分析。

优点

-   数据更可控；
    
-   长期成本可能更低；
    
-   可以离线运行；
    
-   适合隐私敏感任务。
    

缺点

-   模型能力可能不足；
    
-   部署复杂；
    
-   推理速度取决于硬件；
    
-   对复杂风险判断可能不稳定。
    

### 对比指标

实验中可以记录以下指标：

| 指标 | API Model | Local Model |
| --- | --- | --- |
| 输出质量 | 是否准确识别风险 | 是否漏掉关键风险 |
| 延迟 | 响应时间 | 本地生成时间 |
| 成本 | API token 成本 | GPU/设备成本 |
| 隐私 | 数据是否外发 | 数据是否本地处理 |
| 稳定性 | 是否超时或限流 | 是否崩溃或卡顿 |
| 可维护性 | 接口是否简单 | 部署是否复杂 |

### 我的产品选择

如果我设计一个 AI × Web3 产品，我不会只选择单一模型，而是采用混合推理方案：

```text
复杂交易风险分析
→ 使用能力较强的 API Model

简单字段抽取、标签分类
→ 使用本地模型

涉及敏感数据的预处理
→ 本地完成

API 失败或超时
→ 本地模型提供 fallback

高风险操作
→ 必须加入人工确认或二次确认
```

这样的方案可以更好地平衡质量、成本、隐私和稳定性。

* * *

## 十一、学习后的个人理解

通过学习 Inference，我意识到 AI 产品开发并不是简单地把大模型接入应用。真正困难的部分在于如何让模型能力进入一个稳定、可控、可观测的产品系统。

模型训练决定了模型的能力上限，但推理服务决定了用户实际体验到的能力。如果推理层设计不好，即使模型很强，产品也可能表现不稳定。相反，如果推理层设计合理，即使模型不是最强，也可以通过缓存、fallback、任务拆分、本地模型和评估机制，构建出更可靠的产品体验。

我认为 Inference 的核心不是技术炫技，而是工程取舍。开发者需要不断回答以下问题：

-   这个任务是否必须使用最强模型？
    
-   用户能接受多长延迟？
    
-   每次请求成本是否合理？
    
-   数据能否发送到第三方？
    
-   模型失败时系统怎么办？
    
-   模型升级后如何保证输出不退化？
    
-   是否有监控指标证明系统稳定？
    
-   是否能追踪每次模型输出的来源和过程？
    

这些问题都说明，推理服务是 AI 产品从 Demo 走向真实应用的关键环节。

* * *

## 十二、总结

本次学习让我对 Inference 有了更系统的理解。

Inference 不是简单的模型调用，而是连接模型能力和产品体验的关键层。它同时影响响应速度、成本、隐私、安全、稳定性和可扩展性。

API Model 适合快速开发和复杂任务，但需要关注成本、限流、隐私和版本变化。Local Model 提供更高控制权和隐私保护，但需要承担硬件和运维复杂度。Quantization 可以降低本地部署门槛，但必须通过真实任务测试质量损失。Serving 则负责把模型变成稳定服务，是 AI 产品工程化的核心。

在 AI × Web3 场景中，Inference 更加重要。因为链上行为具有不可逆性，模型输出不能只是自然流畅，还必须可审计、可追踪、可降级、可验证。只有这样，AI 才能安全地参与交易解释、合约分析、风险判断和 Agent 执行等任务。

最终，我对 Inference 的理解可以总结为一句话：

> 训练让模型获得能力，推理让模型能力变成可靠的产品服务。

# Fine-tuning 学习笔记

## 一、核心理解

Fine-tuning，中文通常译为“微调”，指在已有基础模型之上，用特定任务的数据继续训练模型，使模型更稳定地掌握某类输出格式、语言风格、领域任务或行为模式。

但是，fine-tuning 不是“模型效果不好就训练一下”。在实际 AI 工程中，微调往往不是第一步。很多模型表现不佳的问题，可能并不是模型本身不会，而是 prompt 不清楚、上下文缺失、检索结果不准确、输出格式没有约束，或者缺少系统性的 evaluation。

因此，fine-tuning 的真正价值，不是让模型凭空知道更多事实，而是让模型在一类明确任务上表现得更一致、更稳定、更符合预期。

* * *

## 二、为什么要学习 Fine-tuning

很多人在遇到模型回答不理想时，会第一时间想到“是不是需要微调”。但在真实开发中，微调通常应该排在更后面。

在考虑 fine-tuning 之前，应该先问几个问题：

1.  是不是 prompt 写得不够清楚？
    
2.  是不是上下文信息没有提供完整？
    
3.  是不是 RAG 没有检索到正确资料？
    
4.  是不是输出格式没有 schema 限制？
    
5.  是不是基础模型能力本身不够？
    
6.  是否已经通过 eval 证明这个问题稳定存在？
    

这些问题很重要，因为 fine-tuning 会增加系统复杂度。如果没有明确目标、没有高质量数据、没有 eval 验证，微调很可能只是让问题变得更难排查。

例如，一个模型回答最新政策、实时价格、链上状态时出错，这通常不应该靠 fine-tuning 解决，而应该靠检索、工具调用或数据库更新解决。因为微调后的模型并不会自动保持知识实时更新。

* * *

## 三、第一性原理

Fine-tuning 的第一性原理可以概括为：

**Fine-tuning 改的是模型行为分布，不是产品系统边界。**

也就是说，微调可以让模型更像训练数据中的表现方式，但它不会自动解决事实正确性、权限安全、引用可靠性、工具调用安全等问题。

如果训练数据中存在偏差、错误或不稳定格式，模型不仅会学到这些问题，而且可能把这些问题更稳定地复现出来。

因此，fine-tuning 需要遵循几个基本原则：

### 1\. 先有 eval，再谈 fine-tuning

如果没有 eval，就无法判断微调后到底是真的变好了，还是只是少数样本看起来更顺了。  
微调前后应该使用同一套评估样本和指标进行对比。

### 2\. 先修数据，再修模型

数据质量决定微调效果。  
如果训练样本中存在错误答案、不一致格式、模糊边界或不良表达，模型会把这些问题一起学进去。

### 3\. 不要用微调存实时知识

实时状态、价格、文档更新、链上信息和用户数据，不适合通过微调写进模型。  
这类信息更适合放在 RAG、数据库、API 或工具调用中。

* * *

## 四、重要概念

## 1\. SFT

SFT 是 Supervised Fine-Tuning，即监督微调。  
它使用“输入 + 期望输出”的样本，让模型学习某类任务的回答方式。

SFT 比较适合以下场景：

-   固定格式输出
    
-   特定语气或表达风格
    
-   特定任务流程
    
-   领域术语和回答习惯
    
-   工具调用样式
    

例如，如果希望模型始终按照固定 JSON 字段输出摘要、风险点和开放问题，SFT 可能有帮助。  
如果希望模型用某种固定语气写客服回复、治理摘要或合约注释，SFT 也可能适用。

但 SFT 对数据质量非常敏感。如果样本中的格式不一致、答案有误、边界不清，模型会把这些问题学进去。

* * *

## 2\. LoRA

LoRA 是 Low-Rank Adaptation，即低秩适配。  
它是一种参数高效微调方法，不直接更新模型的全部参数，而是训练一小部分适配参数。

LoRA 的优势是训练成本更低、显存需求更小，因此常用于开源模型微调。对于资源有限、想快速测试特定任务效果的团队来说，LoRA 是一种常见选择。

但 LoRA 不是魔法。它只是降低了实验成本，并不会自动解决任务定义不清、数据质量差、评估缺失等问题。最终效果仍然取决于数据、任务设计和 eval。

* * *

## 3\. PEFT

PEFT 是 Parameter-Efficient Fine-Tuning，即参数高效微调方法的统称。  
LoRA 就是 PEFT 的一种。

PEFT 的核心意义是：不需要重新训练整个大模型，也可以通过较小规模的参数调整，让模型适配某个任务或领域。

PEFT 通常适合以下情况：

-   模型较大，完整微调成本太高
    
-   任务范围比较明确
    
-   数据量中等
    
-   希望并行试验多个 adapter 版本
    

对于团队来说，PEFT 的价值在于降低微调实验门槛，让不同任务、不同版本的适配器可以更灵活地测试和切换。

* * *

## 4\. Dataset

Dataset 是 fine-tuning 的核心资产。  
一个可用的数据集，不只是很多问答对的集合，而应该包含清晰的任务定义、输入来源、输出标准、质量检查和数据切分方式。

通常至少要区分四类数据：

1.  **训练集**：用于训练模型。
    
2.  **验证集**：用于调参和选择版本。
    
3.  **测试集**：用于最终评估模型效果。
    
4.  **回归集**：用于防止历史问题复发。
    

其中一个重要原则是：不要把测试集拿去训练。  
如果测试集被用于训练，评估结果就会失去意义，因为模型可能只是记住了测试样本，而不是具备了真正的泛化能力。

* * *

## 5\. Overfitting

Overfitting 指过拟合，也就是模型把训练数据记得太死，导致面对新样本时表现变差。

微调时很容易出现过拟合，常见原因包括：

-   训练样本太少
    
-   样本风格太单一
    
-   训练轮数太多
    
-   输出格式过度僵化
    
-   验证集和训练集太相似
    

过拟合的表现是：模型在训练样本或类似样本上看起来很好，但真实用户稍微换一种问法，就容易失败。

因此，微调不能只看训练集效果，还必须看验证集、测试集和真实场景中的表现。

* * *

## 五、Fine-tuning 在 AI x Web3 中的位置

在 AI x Web3 场景中，fine-tuning 可以用于一些明确、稳定的任务，例如：

-   交易解释风格
    
-   治理提案摘要格式
    
-   风险标签输出
    
-   合约注释风格
    
-   工具调用样式
    
-   固定结构的安全提示
    

例如，一个 Web3 助手需要把交易信息解释成固定结构：交易目的、涉及资产、权限变化、潜在风险和建议操作。这样的输出格式和表达习惯，可以考虑通过微调提高一致性。

但 fine-tuning 不能替代以下能力：

-   链上实时状态查询
    
-   合约安全审计
    
-   交易模拟
    
-   钱包权限控制
    
-   用户确认机制
    
-   外部事实引用
    

换句话说，fine-tuning 可以让模型更懂某类任务格式，但不能让模型直接成为可信执行层。  
在涉及资产、权限和链上执行的系统中，微调最多是辅助层，不能替代安全边界和验证机制。

* * *

## 六、最小实践方案

一个比较合理的入门实践，是先做“结构化摘要格式”的微调前评估，而不是立刻开始训练。

可以先准备 50 条样本：

-   输入：一段技术文档、治理提案或产品说明。
    
-   输出：固定 JSON 字段，例如：
    
    -   summary
        
    -   risks
        
    -   open\_questions
        
    -   sources
        

然后比较三种方案：

1.  只改 prompt。
    
2.  Prompt + few-shot 示例。
    
3.  小规模 fine-tuning 或 adapter 方案。
    

每种方案都使用同一套 eval 进行检查：

-   字段是否完整
    
-   是否编造来源
    
-   风险点是否漏掉
    
-   输出是否稳定
    
-   是否符合 schema
    
-   是否能处理边界问题
    

只有当前两种方案无法稳定解决问题时，再考虑 fine-tuning。  
这样可以避免过早训练，也能更清楚地判断微调是否真的带来收益。

* * *

## 七、学习重点总结

Fine-tuning 的重点，不是“训练一个更聪明的模型”，而是让模型在明确任务上更稳定地表现出某种行为模式。

可以用一句话总结：

**Fine-tuning 适合提高稳定性和一致性，不适合替代知识更新、权限控制和系统评估。**

学习 fine-tuning 时，需要特别记住三个判断标准：

1.  有没有明确任务？
    
2.  有没有高质量数据？
    
3.  有没有 eval 证明微调真的有效？
    

如果这三个条件不满足，fine-tuning 很可能不是当前最优解。

* * *

## 八、关键词速记

-   **Fine-tuning**：在基础模型上继续训练，使模型适配特定任务。
    
-   **SFT**：监督微调，用输入和期望输出样本训练模型。
    
-   **LoRA**：低秩适配，降低微调成本的参数高效方法。
    
-   **PEFT**：参数高效微调方法的统称。
    
-   **Dataset**：微调的核心资产，需要清晰定义和切分。
    
-   **Overfitting**：过拟合，指模型过度记忆训练数据，泛化能力下降。
    
-   **Eval**：微调前后判断效果是否提升的必要机制。
    

* * *

## 九、个人理解

Fine-tuning 应该被理解为 AI 系统优化中的一个工程手段，而不是解决所有问题的万能方法。

如果模型的问题来自信息缺失，就应该补上下文或检索；如果问题来自实时数据，就应该接工具或数据库；如果问题来自格式不稳定，可以先尝试 schema、prompt 和 few-shot；如果这些方法仍然不能稳定解决，并且 eval 已经证明问题持续存在，才更适合考虑 fine-tuning。

因此，fine-tuning 的正确使用顺序应该是：先明确任务，再准备数据，然后建立 eval，最后才决定是否训练。真正成熟的 AI 应用，不是靠“多训练一下”变可靠，而是靠数据、评估、系统边界和工程流程共同保证可靠性。

## Evaluation 学习笔记

## 一、核心理解

Evaluation，简称 eval，是 AI 应用开发中用来判断系统表现是否真正变好的方法。它的作用不是简单地评价“这个回答看起来好不好”，而是把主观感觉转化为可重复、可比较、可追踪的改进机制。

在 AI 应用中，prompt、模型、RAG、Agent、工具调用等环节都会影响最终输出。很多时候，一个改动会让部分问题表现更好，但也可能让另一些场景变差。如果没有 eval，团队只能依靠人工试用和主观判断，很难发现隐藏的回归问题。

因此，eval 的核心价值是帮助团队回答一个关键问题：这次系统改动，是否真的让关键任务更可靠？有没有带来新的失败模式？

* * *

## 二、为什么需要 Evaluation

AI 应用的最大特点之一是输出不稳定。即使输入相似，模型也可能给出不同质量的回答；即使整体效果提升，关键场景仍然可能出错。

常见问题包括：

1.  改了一句 prompt 后，部分问题变好，但部分问题变差。
    
2.  更换模型后，平均能力提升，但关键任务仍然失败。
    
3.  接入 RAG 后，答案变长，但引用来源可能不准确。
    
4.  Agent 增加工具调用后，功能更强，但也更容易出现参数错误或越权调用。
    

Evaluation 要解决的就是这些问题。它通过固定样本、明确指标、统一评分方式和回归测试，让系统改进从“凭感觉”变成“有证据”。

* * *

## 三、第一性原理

Eval 的第一性原理可以概括为：

**不能被重复测量的 AI 行为，就不能被稳定改进。**

AI 系统具有概率性，用户输入也非常开放。如果没有固定的测试样本和评估标准，就很难判断一次效果变化到底来自真实改进、随机波动，还是测试样本太少。

因此，eval 应该遵循几个基本原则：

### 1\. 先测任务，不只测模型

用户真正关心的是整个 AI 应用是否完成任务，而不是模型在榜单上的分数。  
例如，一个文档问答系统的效果，不只取决于模型能力，还取决于检索质量、引用准确性、上下文拼接方式和最终回答格式。

### 2\. 先保住关键失败场景

高风险错误、常见问题、边界条件和历史 bug，都应该进入 regression set。  
这些场景不一定数量很多，但对系统稳定性非常关键。

### 3\. 评估要贴近真实产品

Eval 样本越接近真实用户输入，评估结果越有参考价值。  
如果测试样本过于理想化，eval 很容易变成自我安慰，无法反映真实使用中的问题。

* * *

## 四、重要概念

## 1\. Harness

Harness 是运行 eval 的框架。它负责把测试样本输入系统，收集系统输出，调用评分器，并生成结果报告。

一个最小可用的 harness 至少包括：

-   输入样本
    
-   期望输出或评分规则
    
-   被测系统版本
    
-   模型与参数配置
    
-   运行日志
    
-   结果报告
    

Harness 的关键价值是“可重复”。只有 eval 可以重复运行，团队才能比较不同 prompt、不同模型、不同 RAG 策略之间的真实差异。

* * *

## 2\. Golden Set

Golden Set 是一组经过认真挑选和标注的测试样本。

它不一定要很大。早期阶段，30 到 100 条高质量样本通常已经很有价值。相比大量随便收集的问题，少量但覆盖关键场景的样本更适合发现真实问题。

Golden Set 通常应该包含：

-   常见正常问题
    
-   边界问题
    
-   容易误判的问题
    
-   高风险问题
    
-   历史 bug
    
-   用户真实反馈样本
    

一个重要实践是：每修复一个重要 bug，都应该考虑把它加入 regression 样本，防止以后再次出现。

* * *

## 3\. LLM-as-Judge

LLM-as-Judge 指用大模型来评价另一个模型的输出。它适合评估开放式任务，例如：

-   摘要是否准确
    
-   回答是否完整
    
-   是否遵循格式
    
-   是否引用了来源
    
-   是否遗漏了重要信息
    

但 LLM-as-Judge 不能被当作绝对真相。Judge 模型本身也可能有偏差，可能被表达风格影响，也可能漏掉关键错误。

更稳妥的做法是：

-   能用规则判断的字段，优先用规则评分。
    
-   开放式质量问题，可以使用 LLM judge。
    
-   高风险样本保留人工抽检。
    
-   定期比较 judge 评分和人工评分的一致性。
    

也就是说，LLM-as-Judge 是评估工具，不是最终裁判。

* * *

## 4\. Regression

Regression 指旧问题在系统改动后重新出现。  
在 AI 应用中，“修 A 坏 B”非常常见。一次 prompt 修改、模型升级、retriever 调整，都可能影响原本正常的场景。

Regression set 的作用就是把历史问题固定下来，在每次发布前重新测试。

一个实用流程是：

1.  用户反馈一个错误。
    
2.  团队复现并记录输入。
    
3.  标注期望输出或拒答条件。
    
4.  加入 regression set。
    
5.  之后每次发布前重新运行。
    

这样可以避免同样的问题反复出现。

* * *

## 5\. Observability

Observability 是线上观察系统真实行为的能力。  
Eval 多数发生在发布前，而 observability 发生在真实使用中。

AI 应用至少应该记录：

-   输入类型和来源
    
-   检索结果
    
-   工具调用过程
    
-   模型输出
    
-   错误和重试
    
-   用户反馈
    
-   成本和延迟
    

Observability 的价值在于帮助团队发现真实用户在哪些场景失败，并把这些失败样本补充进 Golden Set 或 Regression Set。

* * *

## 五、Evaluation 在 AI x Web3 中的重要性

在 AI x Web3 场景中，eval 尤其重要。因为系统错误可能不仅影响用户体验，还可能影响资产安全、权限判断、治理决策和链上执行。

需要重点评估的内容包括：

1.  交易解释是否准确。
    
2.  风险提示是否漏报。
    
3.  工具调用参数是否越界。
    
4.  系统是否能拒绝不确定请求。
    
5.  是否能识别 Prompt Injection。
    
6.  引用和来源是否可追溯。
    
7.  高风险操作是否要求 human check。
    

需要注意的是，eval 不能替代交易模拟、权限控制和安全机制。它的作用是持续发现系统在哪些场景下不可靠，从而帮助团队改进系统。

* * *

## 六、最小实践方案

可以选择一个具体原型进行最小 eval，例如：

-   交易解释系统
    
-   文档问答系统
    
-   Agent 工具调用系统
    

先准备 30 条样本：

-   10 条正常问题
    
-   10 条边界或容易混淆的问题
    
-   5 条历史 bug 或预期失败样本
    
-   5 条恶意或注入样本
    

然后为每条样本定义：

-   输入
    
-   期望行为
    
-   必须包含的信息
    
-   必须拒绝或提醒的情况
    
-   是否需要引用来源
    

之后，每次修改 prompt、模型或检索策略前后，都运行一次 eval，并记录结果变化。

* * *

## 七、学习重点总结

Evaluation 的重点不是做一份好看的测试报告，而是建立一个可持续改进 AI 系统的机制。

可以用一句话总结：

**Eval 是 AI 应用从“能用”走向“可靠”的关键工程方法。**

它让团队能够持续回答三个问题：

1.  系统现在在哪些任务上表现好？
    
2.  系统在哪些场景下容易失败？
    
3.  这次改动到底是提升了系统，还是引入了新问题？
    

* * *

## 八、关键词速记

-   **Evaluation**：评估 AI 系统表现的方法。
    
-   **Harness**：运行 eval 的框架。
    
-   **Golden Set**：高质量标注测试样本集。
    
-   **LLM-as-Judge**：用模型评估模型输出。
    
-   **Regression**：防止旧问题复发的测试机制。
    
-   **Observability**：线上观察真实系统行为的能力。
    
-   **AI x Web3 Eval**：重点关注资产安全、权限、链上执行和风险提示。
    

* * *

## 九、个人理解

学习 eval 的关键，是把 AI 应用看成一个需要持续测试和迭代的工程系统，而不是一次性写好 prompt 就结束的产品。

在传统软件中，测试可以检查函数是否返回正确结果；而在 AI 系统中，输出更开放、更不稳定，所以 eval 需要同时关注任务完成度、事实准确性、安全性、格式遵循、引用可靠性和用户真实反馈。

因此，eval 不是 AI 开发的附加步骤，而是 AI 产品迭代的基础设施。没有 eval，系统改进就很容易停留在主观体验层面；有了 eval，团队才能更清楚地知道什么变好了，什么变差了，以及下一步应该优化哪里。

今天主要学习了模型上下文协议，也就是 MCP。通过今天的学习，我对 MCP 的理解不再停留在“给模型接工具”这个层面，而是更清楚地意识到，它真正解决的是模型、外部工具、数据源和应用上下文之间如何标准化连接的问题。

在实际的 AI 应用中，模型不可能只依靠 prompt 中的文本完成复杂任务。它往往需要读取文件、访问数据库、调用 API、使用外部工具，甚至结合用户当前的应用上下文来完成工作。如果每一个应用都用自己的方式连接工具，那么整个 Agent 生态就会变得非常碎片化：同一个工具可能需要为不同客户端重复开发适配层，权限控制、参数格式、错误处理和日志记录也很难统一。

MCP 的价值就在于把这层连接抽象成一种协议。它类似于 AI 应用中的“工具接口标准”，让客户端、模型和工具服务端之间形成更清晰的分工。Client 负责和模型交互，发现 server 暴露的能力，并把模型的调用请求转化为协议消息；Server 则负责提供 resources、tools、prompts 等能力，让模型能够在被授权的范围内读取信息或执行动作。

今天让我印象最深的是 MCP 的一个核心原则：模型不应该直接拥有世界，而应该通过明确协议访问被授权的上下文和工具。也就是说，真正重要的不是让模型“能接更多工具”，而是让工具接入变得可描述、可发现、可限制。一个工具应该有清晰的 schema，说明它叫什么、什么时候使用、需要哪些参数、会返回什么结果，以及是否会产生外部副作用。如果没有 schema，模型调用工具就容易变成自然语言式的猜测，参数填写和执行结果都会变得不可控。

同时，我也意识到权限设计是 MCP 真正进入产品时非常关键的一环。读取文档、查询 issue、创建 PR、删除文件、发起支付，这些动作的风险完全不同。MCP 可以标准化工具入口，但不能替代真正的授权、审计和隔离机制。尤其是在 AI x Web3 场景中，MCP 可以作为 Agent 连接链上工具、合约文档、RPC、区块浏览器和开发工具的接口层，但它本身并不是钱包安全方案。最终的账户权限、签名确认、交易模拟、session key 和审计日志，仍然需要由 Web3 账户系统和安全机制来完成。

今天的学习主要围绕 AI Agent 工作流、安全威胁建模、DAO 治理协作流程，以及隐私文化对于 Builder 的意义 展开。整体来看，今天的核心收获是：在设计 AI + Web3 产品时，不能只关注 Agent 能不能自动完成任务，更要提前定义它能访问什么、能执行什么、什么时候必须停下来让人确认，以及哪些行为必须进入治理或链上记录。

## 一、Agent Workflow Threat Model 与确认策略

今天首先围绕 Adaptive DAO Communication Strategy Agent 做了 Agent Workflow Threat Model 的梳理。相比单纯描述 Agent 的功能，这部分更像是在回答一个问题：当 Agent 被放进真实 DAO 协作环境中，它可能会误用哪些权限、泄露哪些数据、被怎样攻击，以及哪些关键节点必须由人类确认。

我从四个方面进行了整理：

第一是 资产清单。需要明确 Agent 在运行过程中会接触哪些资产，例如 DAO 成员画像、治理提案内容、会议纪要、投票数据、沟通渠道、钱包权限、外部 API、策略建议结果等。不同资产的敏感程度不同，因此不能把所有数据都默认交给 Agent 自由处理。

第二是 权限边界。Agent 可以自动完成信息整理、任务拆解、提醒生成、草案撰写等低风险工作，但涉及发出正式治理建议、代表用户行动、调用钱包、发布公开消息、触发链上交易等行为时，必须有清晰边界。尤其是与钱包、投票、资金、身份声誉相关的操作，不能让 Agent 默认拥有无限权限。

第三是 数据流威胁。Agent 在“输入—处理—调用工具—输出”的过程中，可能出现 prompt injection、敏感信息泄露、错误上下文引用、外部工具返回污染、成员身份画像被滥用等风险。因此，不能只在最终输出时检查结果，而应该在数据进入、工具调用、跨系统传递和最终发布几个环节都设置防护。

第四是 模拟攻击场景。我梳理了四类典型风险：

1.  恶意输入诱导 Agent 忽略原有规则；
    
2.  外部数据源污染导致 Agent 生成错误治理建议；
    
3.  Agent 误把私人讨论内容发布到公开治理渠道；
    
4.  钱包或治理权限被滥用，导致未经授权的投票、签名或链上操作。
    

基于这些威胁，人工确认策略就变得非常重要。我的理解是，Human-in-the-loop 不应该只是“最后看一眼”，而是要嵌入到关键节点中。例如：公开发布前确认、调用钱包前确认、涉及成员隐私时确认、策略影响治理方向时确认、超过额度或权限范围时确认。这样 Agent 才能既提高效率，又不会替代人的责任判断。

## 二、DAO Meeting-to-Action Pipeline 治理协作流程草图

今天还整理了一个 DAO Meeting-to-Action Pipeline，把 DAO 会议从讨论到行动的过程拆成 11 个步骤，并用四种边界标记不同责任：

-   🔵 AI 自动
    
-   🟡 人工确认
    
-   🔴 治理投票
    
-   🟢 链上记录
    

这个流程让我更清楚地看到，DAO 协作不是简单的“开会—总结—执行”，而是一个多阶段的治理流水线。AI 可以在其中承担大量辅助工作，比如会议转录、摘要生成、任务提取、责任人匹配、行动项提醒、历史提案关联、风险提示等。但并不是所有步骤都适合自动化。

比如，会议内容整理、行动项初步拆解可以由 AI 自动完成；但是否把某个讨论升级为正式提案，需要人工确认；涉及资源分配、预算、规则变更时，需要进入治理投票；而投票结果、资金流转、关键承诺等，则适合链上记录。

这让我意识到，设计 DAO Agent 的关键不是“让 AI 做完所有事”，而是 明确每个阶段的责任归属。AI 负责提高信息处理效率，人类负责判断和确认，治理机制负责集体决策，链上记录负责保证可追溯性。

## 三、GovReach 项目初步 Proposal

今天也继续推进了项目方向深挖，初步形成了 GovReach 的 Proposal。这个项目的想法是：基于 Snapshot 等链上/治理数据，构建一个帮助 DAO 优化治理参与率的 Agent。

GovReach 关注的问题是，很多 DAO 并不是没有治理机制，而是成员参与率低、提案触达弱、投票动机不足、不同类型成员的信息需求不一样。Agent 可以通过分析历史投票数据、成员参与行为、提案类型和社区沟通节奏，帮助 DAO 发现哪些成员容易流失、哪些提案参与率较低、什么时间发布更有效、哪些提醒方式更适合不同成员。

不过，今天的隐私学习也提醒我，这类项目必须特别注意边界。治理参与率优化不能变成对成员的过度画像或操控。GovReach 应该优先采用聚合数据、匿名化分析和可解释建议，而不是直接给成员贴标签或做过度个性化干预。比较理想的方向是：让 DAO 更好地理解治理参与状况，同时让成员保有选择权和隐私空间。

## 四、Handbook 学习收获

今天早上的 Handbook 学习主要包括三个部分：Agent Workflow、Agent Wallet 和几个新概念。

在 Agent Workflow 部分，我学习到 Agent 的运行可以理解为一个五步循环：任务分解、工具选择、执行、观察、纠错。这个框架帮助我把 Agent 从“一个会回答问题的 AI”理解为“一个可以持续执行任务的系统”。它不是一次性输出答案，而是在目标、工具、反馈和修正之间不断循环。

在 Agent Wallet 部分，我重点学习了最小权限、时间锁、额度上限和 session key。这里最大的收获是：Agent 拥有钱包能力之后，风险会明显上升，所以必须通过权限设计限制它的行动范围。比如，Agent 不应该长期持有高权限私钥，而应该使用临时 session key；不应该拥有无限额度，而应该有额度上限；不应该随时执行高风险操作，而应该通过时间锁和人工确认降低风险。

今天也理解了几个重要概念：

Workflow 是 Agent 执行任务的结构化流程，不只是简单对话。  
Human-in-the-loop 是指在关键决策或高风险操作中保留人工确认。  
Session Key 是一种临时授权机制，可以让 Agent 在有限时间和有限权限内执行任务，而不是直接暴露主钱包权限。

这些概念和上午的 Threat Model 结合起来之后，我更清楚地理解了 Agent 安全设计的基本原则：不是完全禁止 Agent 行动，而是让它在被限制、可观察、可撤销、可确认的环境中行动。

## 五、晚间会议学习：Co-learning 与 Neo-Cypherpunk

晚上参加了两场会议，其中最有启发的是 Neo-Cypherpunk & the Cultural Layers of Privacy: Why Privacy Matters for Builders。

这场分享让我重新理解了隐私。隐私不是少数技术专家才关心的极端议题，也不是只有在遇到风险之后才需要补救的功能。对于 AI 和 Web3 Builder 来说，隐私应该是产品设计早期就要考虑的基础能力。

Peter 在分享中提到，过去的 Cypherpunk 更强调技术专家、加密工具和个人防御，而 Neo-Cypherpunk 更强调开放、协作、社区、关怀和可访问性。也就是说，隐私不应该只是“全副武装的技术精英”才能做到的事情，而应该变成普通用户也能逐步采用的实践。

我印象最深的一点是：不必追求一开始就做到 100% 隐私。很多时候，用户可以先从一个工具、一个习惯、一个选择开始，例如使用更隐私友好的浏览器、通信工具或钱包策略。隐私不是非黑即白，而是一条从低到高的连续光谱。

这对我今天的 Agent 项目设计很有启发。比如在 GovReach 中，不应该默认把所有治理数据都用于精细化画像，而应该给 DAO 和成员提供不同层级的隐私选项。某些数据可以公开聚合展示，某些分析只能在本地或匿名环境中完成，某些建议则必须经过人工确认后才能进入治理流程。

## 六、今日总结

今天最大的收获是：AI Agent 的价值不只是自动化，更在于能否被安全地嵌入真实协作流程中。

一个好的 Agent 系统需要同时考虑四件事：  
一是任务流程是否清晰；  
二是权限边界是否明确；  
三是隐私和安全风险是否被提前建模；  
四是人类、治理机制和链上记录分别在什么时候介入。

对于 AI + Web3 Builder 来说，未来的产品设计不能只追求“更快、更自动、更智能”，还要追求“更可控、更透明、更尊重隐私、更适合社区协作”。这也是今天从 Agent Workflow、Agent Wallet、DAO Pipeline 到 Neo-Cypherpunk 分享中贯穿起来的一条主线。

今天主要围绕 Agent Wallet 的安全执行机制 和 Agent Profile 与能力声明设计 做了学习与梳理。整体感受是：Agent 未来真正有价值的地方，不只是“能自动完成任务”，而是能够在清晰边界内，被用户信任地执行任务。

## 1\. Agent Wallet：核心不是自动转账，而是受控执行

今天会议中最重要的启发是：Agent 操作链上资产时，问题不只是“AI 能不能帮用户转账”，而是 AI 是否能在用户明确授权的范围内安全地动用资金。

如果只是给 Agent 一个钱包，让它自己执行任务，风险会非常大。因为自然语言 prompt 只能表达意图，但不能形成强制约束。比如用户说“最多花 100 美金”，Agent 仍然可能因为幻觉、错误理解或执行路径异常，实际发起超过预算的交易。链上资金一旦转出，通常不可撤回，所以必须在执行层加入硬性限制。

会议里提到的 Pact 思路很有启发：用户不是一次性把钱包完全交给 Agent，而是通过一份授权协议，规定 Agent 能做什么、不能做什么、最多花多少钱、可以和哪些地址或合约交互、什么时候自动停止。这样 Agent 的行为就不再只是依赖模型“听话”，而是被系统规则约束。

我理解这对 Agent 产品非常重要：  
未来 Agent 的可信度，不只来自模型能力，而来自 权限边界、审计能力、撤销机制和异常拦截能力。

## 2\. 对黑客松方向的启发

结合会议内容，我觉得黑客松可以选择一个更小、更清晰、更容易落地的 Agent 支付场景，而不是一上来做很复杂的 DeFi Agent 或全自动交易 Agent。

一个比较合适的方向是：Agent 小额自动充值 / 自动续费工具。

比如 Agent 在使用 OpenAI、Claude、Zapier、Notion、数据库 API 或其他 SaaS 服务时，可能会遇到额度不足、token 不够、订阅即将到期等问题。这个时候，可以让 Agent 在用户预先授权的范围内自动完成小额支付。

这个场景的优势是：

用户需求明确：余额不足就充值。  
风险相对可控：金额小、对象固定、频次有限。  
产品逻辑清楚：预设规则，触发条件，自动支付，超限提醒。  
适合黑客松展示：可以做出完整 demo，包括授权、执行、风控、通知。

关键不是展示“Agent 会付款”，而是展示“Agent 只能在被允许的范围内付款”。例如用户可以设置：只允许给指定服务商充值；单次不超过 50 或 100 美金；每天最多充值一次；总预算不超过 1000 美金；超过预算必须人工确认。

这个方向本质上是在做一个 Agent Auto Top-up with Spending Guardrails，也就是带有资金护栏的 Agent 自动充值工具。

## 3\. Agent Profile 与能力声明草图

今天还做了 Agent Profile 与能力声明草图，案例是 Zapier Sales Lead Qualification Agent。

这个部分让我意识到，Agent 不能只是一个“黑盒自动化工具”，而应该像一个可以被识别、被理解、被授权的数字角色。Agent Profile 的作用，就是让用户、系统或其他 Agent 在交互前就知道：这个 Agent 是谁、能做什么、不能做什么、需要什么权限、输出什么结果、风险在哪里。

以 Zapier Sales Lead Qualification Agent 为例，它的核心任务是帮助销售团队自动判断线索质量。它可能会读取 CRM、表单、邮件、公司官网信息，然后根据用户设定的标准判断这个 lead 是否值得跟进。

这个 Agent 的能力声明可以包括：

它可以读取 lead 信息、公司名称、职位、邮箱、行业、预算线索等数据；  
它可以根据预设规则给 lead 打分；  
它可以把线索分成 high priority、medium priority、low priority；  
它可以把结果写回 CRM，或者触发 Zapier workflow；  
但它不能擅自发送销售邮件，不能删除 CRM 数据，也不能修改关键客户信息，除非用户额外授权。

这个设计和今天会议里的 Agent Wallet 很像：都强调 Agent 必须有明确边界。区别是，Agent Wallet 管的是资金权限，而 Agent Profile 管的是身份、能力和任务权限。

## 4\. 今日最大的收获

今天最大的收获是：  
Agent 的未来不是简单地让 AI 代替人操作，而是要让 AI 在可声明、可授权、可审计、可撤销的框架内操作。

无论是 Agent Wallet，还是 Zapier Sales Lead Qualification Agent，本质上都需要回答几个问题：

这个 Agent 是谁？  
它能做什么？  
它不能做什么？  
它需要哪些权限？  
它执行任务时如何被监督？  
如果出错，用户如何撤销或接管？

这也让我对黑客松项目有了更清晰的判断：与其做一个“看起来很智能但边界模糊”的 Agent，不如做一个能力范围很小、但权限设计清楚、用户能放心使用的 Agent。这样的产品更容易讲清楚，也更容易获得信任。

今天参加了 AI and Web3 School 的 Bonus Session，主题是 Open Agentic Economy。本次分享主要围绕 AI agents 与 Ethereum 的关系展开，帮助我理解了当 AI 不再只是聊天工具，而是逐渐成为经济参与者时，为什么需要开放、中立、可验证的基础设施。

一、本次分享的主要收获

Sophia 提到，过去的大语言模型更多像是一个可以回答问题的 chatbot，但现在 AI agent 已经可以写代码、调用工具、做决策、执行支付，甚至和其他 agent 协作。这意味着 AI 正在从工具变成一种新的经济参与者。

当 AI agent 开始参与经济活动时，就会出现很多问题：它们如何付款？如何证明身份？如何建立声誉？如何被限制权限？传统互联网基础设施主要是为人类设计的，并不完全适合机器之间的大规模协作。

因此，Ethereum 的作用就变得重要。它可以提供一个开放、中立、可编程的协调层，让 agent 的身份、支付、规则和行为可以被验证，而不是完全依赖某个中心化平台。

二、关于 Ethereum 和 CROPS 的理解

分享中 Sophia 提到，Ethereum for AI actually means Ethereum for humans。我理解这句话的意思是，Ethereum 不是单纯为 AI 服务，而是在 AI agent 越来越强大的时代，帮助人类继续保留控制权、所有权和选择权。

她还介绍了 Ethereum 的 CROPS 原则，包括抗审查、开源自由、隐私和安全。抗审查意味着没有单一主体可以随意阻止有效交易；开源自由意味着开发者可以查看、复制和继续构建；隐私保护可以帮助用户在不暴露底层数据的情况下完成验证；安全则保证系统和智能合约按照规则运行。

这些原则共同构成了 AI agent 所需要的基础设施，例如身份、声誉、支付和可执行协议。

三、关于 ERC-8004 / ERC-8183 和信任机制的理解

本次分享中，我对 ERC-8004 印象比较深。它主要关注 AI agent 的身份和声誉问题，也就是让用户或其他 agent 能够判断一个 agent 是否可信。

在 Q&A 中，有人问 reputation 代表 agent 本身还是 agent owner 的声誉。Sophia 回答说，它更偏向 agent 本身，也就是某个具体数字软件实体的声誉，而不是背后开发者个人的声誉。

这让我意识到，未来如果用户要使用 AI agent，不能只看它是否“智能”，还要看它是否有可验证的身份、历史记录和安全边界。

四、关于支付场景的收获

Sophia 提到，payment 可能是 AI agent 最早落地的重要场景之一。她用买披萨的例子解释了 programmable money 的意义：如果直接把信用卡交给 agent，风险很高；但如果通过智能合约限制预算和支付对象，就可以让 agent 在可控范围内完成任务。

这个例子让我更直观地理解了 Web3 支付和 AI agent 的结合。它不是简单地让 AI 去花钱，而是通过规则控制 AI 能花多少钱、能向谁付款、能完成什么任务。

五、我的思考

今天最大的收获是，我意识到 AI agent 和 Ethereum 的结合，不只是技术上的“AI 上链”，而是在解决一个更重要的问题：当 AI 开始代表人类执行任务和参与经济活动时，人类如何继续保持控制权？

从市场营销角度看，我也觉得未来这类项目不能只强调智能合约、链上声誉、去中心化这些技术词，而要把它们转化成普通用户能理解的价值。比如告诉用户：你可以限制 AI 的预算，可以查看 agent 的历史记录，也可以通过规则降低风险。

六、下一步行动

接下来，我会继续了解 ERC-8004 / ERC-8183 中关于 agent identity、reputation 和 verification 的内容。同时，我也会关注 AI agent 在支付、钱包和任务执行中的真实应用案例，思考这些技术如何被普通用户理解和接受。

总的来说，今天这场分享让我对 AI agent 和 Ethereum 的关系有了更清晰的认识，也让我看到未来 AI/Web3 项目在技术、信任和用户教育方面都有很大的探索空间。
