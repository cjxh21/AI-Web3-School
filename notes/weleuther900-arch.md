### **1.1 AI Agent 入门**

AI Agent 可以先理解成一个能围绕目标做事的 AI 系统。

普通聊天型 AI 更像是用户提出问题，模型给出回答。  
Agent 则更进一步，它不只是回答问题，还会根据任务目标判断下一步应该做什么。

比如让 Agent 整理一个项目，它可能不是直接生成一段话，而是先读取文件，再分析内容，最后整理结果。

主要特点

-   理解任务目标
    
-   拆解任务步骤
    
-   调用外部工具
    
-   读取工具返回结果
    
-   根据结果继续判断
    
-   必要时等待人工确认
    

核心理解

Agent 不是“更会聊天的 AI”，而是更接近一个能执行任务的 AI 助手。

### **1.2 Agent 和聊天型 AI 的区别**

聊天型 AI 更偏向回答问题。

常见场景

-   解释一个概念
    
-   总结一篇文章
    
-   写一段文案
    
-   回答一个问题
    
-   分析一段代码
    

Agent 更偏向完成任务。

常见场景

-   搜索资料
    
-   读取文件
    
-   调用工具
    
-   修改代码
    
-   整理项目
    
-   连续处理一个任务
    

简单理解

聊天型 AI 更像“用户提问，它给建议”。  
Agent 更像“用户给目标，它推进任务”。

区别不在于回答长不长，而在于是否真正接入工具、流程和外部环境。

### **1.3 Agent 大概是怎么工作的**

Agent 做事一般不是一步完成，而是一个连续过程。

大致流程

-   接收任务
    
-   理解目标
    
-   拆分步骤
    
-   调用工具
    
-   查看结果
    
-   继续判断
    
-   输出结果
    

比如让 Agent 查资料，它可能先搜索，再筛选，再总结，而不是直接凭空回答。

关键点

Agent 不是只靠大模型本身完成所有事情。  
大模型主要负责理解、判断和生成，真正执行任务还需要工具和流程配合。

### **2.2 OpenAI Agents SDK Intro**

OpenAI Agents SDK 可以理解成一个帮助开发者搭建 Agent 的框架。

它主要负责把模型、工具、指令和执行流程组织起来。

重点内容

-   模型负责理解和生成
    
-   工具负责执行具体操作
    
-   指令负责规定 Agent 的行为
    
-   SDK 负责组织整体流程
    

如果没有 SDK，很多 Agent 相关能力可能需要从零开始搭建。  
有了 SDK，可以用更清晰的结构构建 Agent 应用。

核心理解

Agents SDK 不是普通聊天接口，而是用来组织 Agent 的工具框架。

### **2.3 LangGraph Overview**

普通聊天更像一条直线：

LangGraph 更像是把 Agent 的执行过程组织成一张图。

主要理解

-   不同步骤之间可以连接
    
-   执行过程可以根据结果跳转
    
-   适合处理复杂、多步骤、有循环的 Agent 流程

## 1.1 LLM API 调用入门

LLM API，可以理解为让程序调用大模型能力的接口。  
聊天窗口是人直接和模型对话，API 则是程序把请求发给模型，再接收模型返回的内容。

主要作用

-   把大模型接入网站或应用
    
-   实现智能客服、自动总结、写作辅助
    
-   为 AI 工具和 Agent 提供模型能力
    
-   让大模型从聊天框走向真实产品
    

一次基础 API 调用通常包括

-   API Key，验证调用身份
    
-   模型名称，决定调用哪个模型
    
-   输入内容，告诉模型要做什么
    
-   参数设置，影响输出结果
    
-   返回内容，供程序继续处理
    

今天的理解

API 本身不是模型，而是模型能力对外开放的方式。  
想把 AI 真正做进产品和工具里，API 是很关键的一步。

## 3.1 钱包、私钥和个人主权

Web3 里的钱包，并不是真正装资产的地方，而是用户进入链上世界的身份入口。  
资产记录在区块链上，钱包主要负责管理地址、发起交易和完成签名。

核心概念

-   钱包，管理链上身份和操作入口
    
-   地址，用于接收资产
    
-   私钥，控制资产的核心凭证
    
-   助记词，私钥的备份形式
    

今天的理解

谁掌握私钥，谁就掌握资产控制权。  
Web3 强调个人主权，也意味着用户要自己承担更高的安全责任。

* * *

## 3.2 交易与签名

在 Web3 中，转账、授权、调用智能合约，本质上都属于交易。

一笔交易通常包含

-   发起地址
    
-   接收地址
    
-   操作内容
    
-   手续费
    
-   交易顺序信息
    

签名的作用

-   证明交易由私钥持有者授权
    
-   防止别人冒充自己操作资产
    
-   是交易真正被网络接受的重要前提
    

简单理解

私钥像印章，签名像盖章确认。

* * *

## 3.3 区块链网络运行

区块链不是由单一服务器维护，而是由多个节点共同运行。

交易上链过程

-   用户发起交易
    
-   交易在网络中传播
    
-   节点验证交易是否合法
    
-   合法交易被打包进区块
    
-   新区块加入链上
    

核心概念

-   节点，共同维护网络的参与者
    
-   区块，存放一批交易记录
    
-   共识机制，让大家认可同一份账本
    

今天的理解

区块链最重要的不只是“分布式存储”，而是让不同参与者在不完全互相信任的情况下，仍然能维护同一套规则和账本。

* * *

## 3.4 智能合约

智能合约可以理解为部署在区块链上的自动执行程序。  
当条件满足时，它会按照预先写好的规则运行。

常见用途

-   发行代币
    
-   自动转账
    
-   借贷
    
-   交易
    
-   质押
    
-   NFT 和链游资产管理
    

优势

-   规则公开
    
-   自动执行
    
-   结果可验证
    

风险

-   代码漏洞可能导致资产损失
    
-   链上执行后往往难以撤回
    

* * *

## 3.5 区块链协议如何升级

区块链协议并不是永远不变，也会不断升级。  
升级通常是为了提升性能、修复问题或增加新功能。

升级常见目的

-   提高运行效率
    
-   降低交易成本
    
-   修复安全问题
    
-   增加新能力
    

和普通软件更新的区别

-   普通 App，可以由公司直接推送新版本
    
-   区块链协议，需要开发者、社区、节点和验证者逐步形成共识
    

可能出现的问题

-   如果分歧过大，可能发生分叉
    

* * *

## 3.6 Web3 关键特性与边界

Web3 关键特性

-   用户掌握资产和身份
    
-   规则公开透明
    
-   数据和操作可验证
    
-   智能合约自动执行
    
-   不同协议之间可以组合
    

Web3 的边界

-   使用门槛较高
    
-   私钥管理责任重
    
-   交易不可逆
    
-   犯错成本高
    
-   去中心化程度因项目而异
    
-   一些场景体验仍不如 Web2
    

今天的理解

Web3 的价值，在于把部分信任交给密码学、共识和公开规则。  
但相应地，也把更多责任重新交还给用户自己。

### 2.1 LLM 不只是聊天机器人

### 2.2 Token、上下文和 Prompt

今天也补了一些之前常听到、但理解得不够清楚的概念。

-   **Token**：可以先粗略理解成模型处理文字时拆出来的“小片段”。
    
-   **上下文**：模型当前能看到、能参考的内容，比如你的问题、前面的对话、上传的材料等。
    
-   **Prompt**：就是你给模型下任务的方式。
    

我现在的理解是：  
**Prompt 写得越清楚，模型越知道你到底想让它干什么。**

比如单纯说“介绍一下人工智能”，和说“我是完全小白，请用生活例子介绍人工智能，并控制在 300 字以内”，得到的结果肯定会差很多。

### 2.3 LLM API 的初步理解

今天也开始接触 **LLM API** 这个概念。

可以简单理解成：

> **聊天产品是人直接和模型说话，API 是程序去调用模型能力。**

有了 API，大模型就不只是待在聊天框里，而是可以被接进：

-   智能客服
    
-   笔记总结工具
    
-   教育批改产品

## 今日开始学习week1的内容，首先学习了AI相关基础知识，即模块A，LLM相关知识

### 1.1 大语言模型 LLM

**LLM（Large Language Model，大语言模型）**，指在海量文本数据上训练出来、能够理解和生成自然语言的 AI 模型。  
ChatGPT、Claude、Llama 等都属于大语言模型。

主要能力

-   回答问题
    
-   总结文章
    
-   翻译内容
    
-   写文案、邮件、报告
    
-   提取关键信息
    
-   续写文本
    
-   辅助写代码和解释代码
    

基本工作直觉

LLM 会结合已有上下文，判断接下来最可能出现的内容，并生成符合语境的回答。  
训练数据越多、模型规模越大，语言理解、内容组织和任务完成能力通常越强。

和搜索引擎的区别

-   **搜索引擎**：从已有网页中查找信息
    
-   **LLM**：基于训练内容与当前上下文生成回答
    

因此，LLM 更擅长**解释、总结和重组信息**，但并不天然保证事实百分百准确。

常见局限

-   可能生成错误事实
    
-   可能出现“幻觉”
    
-   对最新信息掌握有限
    
-   长文本理解能力受上下文窗口限制
    
-   回答流畅不等于内容一定正确
    

* * *

### 1.2 NLP 与 LLM 的关系

**NLP（Natural Language Processing，自然语言处理）**，指让计算机理解、分析和生成自然语言。

常见 NLP 任务

-   文本分类
    
-   情感分析
    
-   命名实体识别
    
-   翻译
    
-   摘要
    
-   问答
    
-   文本生成
    

过去，很多 NLP 任务需要分别训练专门模型；  
现在，大语言模型可以依靠统一的语言能力完成大量不同任务，因此成为现代 NLP 的核心方向之一。

* * *

### 1.3 Transformer 架构

**Transformer** 是现代大语言模型最重要的基础架构。  
GPT、BERT、Llama 等模型都与 Transformer 架构密切相关。

Transformer 的意义

-   提升长文本理解能力
    
-   更好捕捉词与词之间的依赖关系
    
-   支持大规模并行训练
    
-   提升生成与理解能力
    

* * *

### 1.4 Attention 注意力机制

**Attention（注意力机制）**，指模型在处理信息时，对不同词语或片段分配不同关注程度。

主要作用

-   抓住一句话中的重点信息
    
-   理解上下文中的指代关系
    
-   判断词语之间的联系
    
-   提升翻译、问答、摘要等任务的准确性
    

简单理解：  
阅读一句长句时，不同词的重要性并不相同。Attention 的作用，就是让模型把更多“注意力”放在更关键的信息上。

* * *

### 1.5 Transformer 的三种常见结构

Encoder

偏重**理解输入内容**。  
常用于：

-   文本分类
    
-   情感分析
    
-   语义理解
    

Decoder

偏重**生成内容**。  
常用于：

-   文本续写
    
-   聊天模型
    
-   内容创作
    

GPT 类模型属于这一方向。

Encoder-Decoder

先理解，再生成。  
常用于：

-   机器翻译
    
-   文本摘要
    
-   信息转换
    

* * *

### 1.6 Transformer 能处理的任务类型

-   **文本任务**：分类、问答、翻译、摘要、生成
    
-   **图像任务**：图像分类、图片理解、看图描述
    
-   **音频任务**：语音识别、音频分类、文本转语音
    
-   **多模态任务**：同时处理文字、图像、音频等多种信息形式
    

* * *

### 1.7 今日 AI 基础笔记整理

-   **LLM**：在海量文本上训练出来的大语言模型，能够理解和生成语言
    
-   **NLP**：让计算机处理人类语言的技术领域
    
-   **Transformer**：现代大模型的重要基础架构
    
-   **Attention**：帮助模型关注关键信息的机制
    
-   **Encoder**：偏理解
    
-   **Decoder**：偏生成
    
-   **Encoder-Decoder**：先理解再生成
    
-   **LLM 常见能力**：问答、总结、翻译、写作、提取信息、代码辅助
    
-   **LLM 常见局限**：可能幻觉、可能过时、回答流畅但不一定正确

通过本次分享，初步了解了 Web3 安全与链上交互的一些基础知识。

### 3.1 Web3 安全提醒

**保护好自己的密钥，不要让第二个人知道。**  
私钥一旦泄露，资产可能被直接转走，而且链上资产转移通常不可逆。  
同时要警惕钓鱼网站、恶意签名和伪装交易。

### 3.2 基础知识记录

-   **钱包的分类**：主要分为托管钱包与非托管钱包、热钱包与冷钱包，本质区别在于私钥由谁掌握、是否长期联网。
    
-   **Web3 交易关键参数**：包括发起地址、接收地址、金额、手续费、Nonce、链 ID 等，共同决定交易给谁、做什么、花多少钱、能否正确执行。
    
-   **Web3 交易监听逻辑**：持续追踪交易从发起、广播、打包到成功或失败的全过程，并监听链上事件变化。
    
-   **交易模拟**：在真正签名前，先预演这笔交易可能带来的结果，用来提前发现资产转移、危险授权或失败风险。
    
-   **Web3 Server 测钱包安全**：通过安全服务分析钱包交互、签名和交易请求，判断是否存在钓鱼、恶意合约或资产被盗风险。
    

### 3.3 今日最重要的安全认知

**安全理解力，等于资产保护力。**  
在 Web3 世界里，不懂私钥、不看签名内容、不理解交易风险，很可能直接导致资产损失。
