**1.1 Agent 从工具调用走向场景应用**

前面学习 Tool Use 和 MCP 时，重点是理解 Agent 如何连接外部工具和数据源。

今天进一步整理到，工具调用本身不是最终目标。  
真正重要的是，Agent 能不能进入具体场景，解决实际问题。

在 Web3 场景中，Agent 可以连接：

区块浏览器。  
链上数据 API。  
项目文档。  
钱包地址记录。  
交易哈希。  
合约交互信息。  
安全风险数据库。

重点记录：

Agent 的价值不是单纯“会调用工具”，而是能把工具返回的信息整理成用户能理解的内容。

**1.2 Web3 Agent 的实际价值**

Web3 里的信息很多都是公开的，但公开不等于容易理解。

比如一笔链上交易，里面可能包含：

发送地址。  
接收地址。  
交易哈希。  
Gas 费用。  
合约调用。  
Token 转移。  
授权信息。  
交易状态。

这些内容普通用户直接看很吃力。

Web3 Agent 可以做的事情：

解释一笔交易发生了什么。  
总结钱包地址的行为。  
提醒可疑授权。  
分析合约交互的大概含义。  
整理项目资料。  
辅助判断操作风险。

重点记录：

Web3 Agent 更适合做解释、整理、提醒和辅助分析。  
它的价值不是替用户操作资产，而是帮助用户看懂链上信息。

**1.3 Web3 Agent 的安全边界**

Web3 Agent 越接近真实场景，越需要明确边界。

可以交给 Agent 辅助完成的内容：

查询公开信息。  
整理链上记录。  
解释交易内容。  
总结项目文档。  
提醒潜在风险。  
生成安全检查清单。

不能直接交给 Agent 自动完成的内容：

输入私钥。  
输入助记词。  
自动签名。  
自动转账。  
自动授权。  
自动连接陌生网站。  
自动调用未知合约。

重点记录：

Web3 场景里的错误成本比普通应用更高。  
因为很多链上操作不可逆，所以涉及资产控制权的步骤必须由用户自己确认。

**1.4 Agent 不能替代用户决策**

Agent 可以辅助判断，但不能完全替代用户决策。

尤其是 Web3 场景中，Agent 的判断也可能受到这些限制：

工具数据不完整。  
合约信息不透明。  
风险识别不准确。  
模型可能理解错误。  
项目本身可能存在隐藏风险。  
链上操作一旦执行很难撤回。

所以 Agent 更适合作为辅助工具，而不是最终决策者。

重点记录：

Agent 可以提供信息和建议，但最终是否签名、授权、转账，必须由用户判断。  
AI 的作用是降低理解门槛，不是替代资产决策。

**2.1 从 LLM 到 Agent**

Week1 前半部分主要学习了 LLM。

LLM 负责：

理解语言。  
生成内容。  
回答问题。  
总结文本。  
辅助写作。  
解释代码。

后来进一步学习到，LLM 本身并不等于 Agent。  
LLM 更像是智能能力底座，Agent 是在 LLM 基础上加入工具、记忆、流程和执行能力之后形成的系统。

重点记录：

LLM 是基础能力。  
Agent 是围绕任务执行组织起来的系统。

**2.2 从 API 到 Tool Use**

API 让大模型可以被程序调用。

通过 API，大模型可以进入：

网站。  
应用。  
自动化脚本。  
AI Coding 工具。  
Agent 系统。

Tool Use 则更进一步，让 Agent 不只是调用模型生成文字，而是可以调用外部工具完成任务。

重点记录：

API 解决的是模型如何接入应用。  
Tool Use 解决的是模型如何连接工具并执行操作。

**2.3 从 MCP 到 Web3 数据连接**

MCP 可以理解成一种让模型和外部工具、数据源连接得更标准的方式。

在 Web3 场景中，Agent 需要连接的数据源更多，比如：

链上交易数据。  
区块浏览器。  
钱包地址。  
智能合约。  
项目文档。  
安全数据库。

如果每个工具都单独接入，维护会比较复杂。  
MCP 这类连接方式的意义，就是让 Agent 接入工具和数据源时更加统一。

重点记录：

MCP 的重点不是让模型更聪明，而是让模型更稳定地连接外部工具和数据。

**2.4 从概念学习到应用理解**

前几天的学习更多是在理解概念。

包括：

LLM 是什么。  
Prompt 怎么影响输出。  
API 如何调用模型。  
Agent 和聊天型 AI 有什么区别。  
RAG 如何补充外部知识。  
Tool Use 如何让 AI 使用工具。  
MCP 如何连接外部数据源。  
Web3 Tool Use 如何读取链上信息。

今天开始更清楚地看到，这些概念不是分开的。

可以串成一条线：

LLM 提供理解和生成能力。  
API 把模型接入应用。  
RAG 让模型能参考外部知识。  
Tool Use 让模型能调用工具。  
MCP 让工具连接更标准。  
Agent 把这些能力组织成任务流程。  
Web3 提供真实的链上数据和应用场景。

重点记录：

AI × Web3 的学习重点，不只是分别学 AI 和 Web3，而是理解 AI 如何连接 Web3 数据，并在安全边界内辅助用户完成分析和判断。

**3.1 AI × Web3 的整体理解**

目前对 AI × Web3 的理解可以先概括成：

AI 提供理解、总结、推理和生成能力。  
Web3 提供公开、可验证、可追踪的链上数据。  
Agent 负责连接工具、读取数据、组织流程。  
安全边界决定 Agent 能做什么、不能做什么。

AI × Web3 的价值主要体现在：

降低链上信息理解门槛。  
帮助用户解释交易。  
辅助识别风险。  
整理项目和链上数据。  
提高研究和操作前的判断效率。

重点记录：

AI × Web3 不是让 AI 自动控制钱包，而是让 AI 帮助用户更好地理解链上世界。

Week1 的学习主线基本从 AI 基础走到了 AI × Web3 结合。

主要内容可以分成几层：

理解 RAG、Tool Use、MCP，以及 Agent 如何连接外部知识和工具。

第五层，Web3 场景。  
理解钱包、私钥、交易、签名、智能合约、链上数据和 Web3 Agent 的安全边界。

今日主要收获：

这段时间的学习不是单纯学工具，而是在建立一条完整路线。  
从大模型到 Agent，再到工具调用和 Web3 数据连接，逐步理解 AI 如何从“回答问题”走向“辅助真实场景”。  
现阶段最重要的认识是：Agent 可以帮助用户理解和处理复杂信息，但在 Web3 场景中，涉及资产、签名和授权的部分，必须始终保留人工确认。

**今天继续推进 AI × Web3 School 的学习内容，重点从“Agent 可以做什么”进一步推进到“一个 Web3 Agent 应用应该怎么设计”**

前面已经学习了 LLM、API、Agent、RAG、Tool Use、MCP、Web3 Tool Use，以及 Web3 Agent 的应用场景。  
今天主要开始把这些内容整理成一个更具体的应用结构，重点放在链上交易解释、钱包风险提醒和 Agent 工作流设计上。

**1.1 Web3 Agent 的场景选择**

Web3 Agent 可以做很多事情，但并不是所有场景都适合一开始就做。

比较适合作为入门方向的场景：

解释一笔链上交易。  
分析钱包地址的交互记录。  
提醒可能存在的授权风险。  
整理项目公开资料。  
辅助理解智能合约交互。

不适合作为早期方向的场景：

自动转账。  
自动签名。  
自动授权。  
自动连接钱包。  
自动执行高风险合约操作。

核心理解：

Web3 Agent 早期更适合做信息解释和风险提醒，不适合直接控制用户资产。

**1.2 链上交易解释 Agent 的定位**

链上交易解释 Agent 的目标，是帮助用户看懂一笔交易到底发生了什么。

它不直接操作钱包，也不替用户签名，而是把复杂的链上记录整理成普通人能理解的说明。

主要解决的问题：

这笔交易是否成功。  
交易从哪个地址发出。  
交易发送到了哪个地址。  
是否调用了智能合约。  
是否发生 Token 转移。  
是否产生授权行为。  
手续费大概是多少。  
有没有明显风险点。

核心理解：

链上数据本来是公开的，但不代表容易理解。  
Agent 的价值，是把复杂的链上数据翻译成清楚的自然语言。

**1.3 用户输入与结果输出**

一个简单的 Web3 Agent，需要先明确用户输入什么，系统输出什么。

用户可能输入：

交易哈希。  
钱包地址。  
合约地址。  
项目名称。  
一段看不懂的交易记录。

Agent 可以输出：

交易状态说明。  
地址关系说明。  
Token 变化说明。  
合约交互解释。  
风险点提醒。  
下一步查看建议。

比较理想的输出格式：

先说明交易是否成功。  
再说明交易双方是谁。  
然后解释发生了什么操作。  
最后提醒是否存在风险。

核心理解：

好的 Agent 输出，不应该只是堆数据，而是要按用户能理解的顺序解释清楚。

**1.4 Agent 工作流设计**

Web3 Agent 不能直接凭空回答，需要按照流程处理任务。

一个基础工作流可以分成：

接收用户输入。  
判断输入类型。  
调用工具查询数据。  
整理交易或地址信息。  
分析是否存在风险。  
生成自然语言解释。  
涉及高风险操作时停止并提醒人工确认。

比如输入一笔交易哈希时，流程可以是：

识别交易哈希。  
查询交易详情。  
判断交易状态。  
查看发送方和接收方。  
检查是否调用合约。  
整理 Token 转移情况。  
生成解释和风险提醒。

核心理解：

Agent 的工作流越清楚，输出越稳定。  
如果流程不清楚，模型很容易直接猜测或跳过关键步骤。

**2.1 数据获取层**

数据获取层负责拿到真实信息。

在 Web3 场景中，数据可以来自：

区块浏览器。  
链上节点。  
第三方 API。  
项目官方文档。  
合约公开信息。  
钱包交互记录。

这一层的重点不是解释，而是尽量拿到准确的数据。

核心理解：

Agent 不能只靠模型记忆回答链上问题。  
链上数据需要通过工具或 API 实时获取。

**2.2 分析解释层**

分析解释层负责把数据转成人能看懂的内容。

它需要处理：

交易是否成功。  
交易类型是什么。  
是否只是普通转账。  
是否调用了智能合约。  
是否涉及 Token 授权。  
是否存在异常交互。  
数据之间的关系是否清楚。

这一层的重点，是把原始数据整理成有逻辑的说明。

核心理解：

模型的作用不是替代链上数据，而是解释链上数据。  
数据提供事实，模型负责组织语言和降低理解门槛。

**2.3 风险提醒层**

Web3 Agent 需要单独考虑风险提醒。

常见风险包括：

未知合约交互。  
过大的 Token 授权。  
可疑地址交互。  
交易失败但仍消耗手续费。  
钓鱼链接诱导签名。  
授权范围不清楚。

风险提醒不等于直接下结论。  
更适合的方式是提示用户哪些地方需要注意。

例如：

这笔交易涉及合约调用，需要确认合约来源。  
这次操作可能包含授权行为，需要检查授权额度。  
该地址交互记录较复杂，不建议直接进行下一步操作。

核心理解：

Web3 Agent 的风险提醒应该谨慎，不能夸大，也不能忽略关键风险。

**2.4 人工确认与安全边界**

Web3 场景必须保留人工确认。

Agent 可以帮助用户理解信息，但不能直接替用户完成资产操作。

必须人工确认的环节：

连接钱包。  
签名消息。  
发起交易。  
Token 授权。  
调用合约。  
转账。  
跨链。  
撤销授权。

Agent 可以做的部分：

解释交易内容。  
分析链上数据。  
提醒风险点。  
整理操作步骤。  
说明可能后果。

核心理解：

Web3 Agent 的安全边界非常重要。  
越接近资产操作，越不能自动化，必须让用户自己确认。

**3.1 从应用设计到 MVP 结构**

今天的内容开始从概念学习进入到 MVP 结构设计。

一个最小版本的 Web3 Agent 可以先包括：

输入层：接收交易哈希或钱包地址。  
工具层：查询链上数据。  
分析层：整理交易状态、地址关系和合约交互。  
输出层：用自然语言解释结果。  
安全层：遇到签名、授权、转账等内容时提醒人工确认。

这个结构不复杂，但已经能把前面学到的 LLM、Tool Use、MCP、Agent 工作流和 Web3 数据结合起来。

核心理解：

MVP 不需要一开始功能很多。  
先把一个具体问题讲清楚、跑通流程，比堆很多功能更重要。

**今日主要收获**

今天的重点，是把 Web3 Agent 从“应用想法”进一步整理成“基本结构”。

主要收获：

Web3 Agent 早期更适合做解释、分析和风险提醒。  
链上交易解释 Agent 是比较适合入门的应用方向。  
Agent 需要明确输入、工具、分析、输出和安全边界。  
链上数据必须通过工具获取，不能只靠模型直接回答。  
涉及签名、授权和转账的操作，必须保留人工确认。  
MVP 的关键不是功能复杂，而是流程清楚、边界清楚、结果能看懂。

**1.1 Agent 从工具调用走向完整工作流**

前面学习 Tool Use 时，重点是理解 AI 可以调用外部工具。  
但真正的 Agent 应用，不只是调用一次工具，而是要把多个步骤组织成完整流程。

一个完整 Agent 工作流通常包括：

任务输入。  
理解用户目标。  
判断需要哪些信息。  
调用工具获取数据。  
分析工具返回结果。  
生成解释或建议。  
必要时等待人工确认。  
输出最终结果。

核心理解：

工具调用只是 Agent 能力的一部分。  
真正有用的 Agent，需要把工具调用、判断过程和最终输出组织成一个稳定流程。

**1.2 Agent 的任务拆解和执行顺序**

Agent 面对复杂任务时，不能直接一步生成结果，而是要先拆解任务。

比如用户想了解一笔链上交易，Agent 不能直接凭空解释，而是应该按照顺序处理：

先识别交易哈希。  
再查询交易状态。  
然后查看发送方和接收方。  
继续分析是否调用合约。  
再查看 Token 转移或授权情况。  
最后用普通人能理解的话解释这笔交易。

任务拆解的意义：

让复杂任务变得更清楚。  
减少模型胡乱回答。  
方便检查每一步是否正确。  
方便在高风险步骤加入人工确认。

核心理解：

Agent 做任务时，顺序很重要。  
好的 Agent 不是一次性给答案，而是按步骤推进。

**1.3 Agent 的结果检查与失败处理**

Agent 调用工具后，不代表结果一定能直接使用。

可能出现的问题：

工具返回失败。  
查询不到数据。  
数据格式太复杂。  
结果和用户问题不匹配。  
模型误解了工具返回内容。  
同一问题可能需要多次查询。

所以 Agent 需要有结果检查能力。

可以检查：

工具是否成功返回。  
返回内容是否为空。  
数据是否和任务相关。  
是否需要继续调用其他工具。  
是否需要提醒用户信息不足。

核心理解：

Agent 不只是调用工具，还要判断工具结果能不能用。  
失败处理和结果检查，是让 Agent 更稳定的重要部分。

**1.4 Human in the loop 与安全边界**

Agent 越接近真实任务，越需要保留人工确认。

尤其是 Web3 场景，很多操作可能直接影响资产安全。

必须人工确认的操作：

连接钱包。  
发起交易。  
授权 Token。  
签名消息。  
调用未知合约。  
转账。  
跨链。  
修改资产权限。

Agent 更适合自动完成的部分：

查询信息。  
解释交易。  
整理资料。  
提醒风险。  
生成操作建议。  
对比多个项目。  
分析公开数据。

核心理解：

Web3 Agent 的边界必须很清楚。  
可以让 Agent 帮忙看懂信息，但不能让 Agent 直接替用户控制资产。

**2.1 链上数据解释 Agent**

链上数据解释 Agent，可以理解为帮助用户看懂区块链记录的工具。

它可以处理：

交易哈希。  
钱包地址。  
合约地址。  
区块信息。  
Token 转账记录。  
合约交互记录。

基本流程：

用户输入交易哈希或地址。  
Agent 调用区块浏览器或链上数据工具。  
获取交易详情。  
分析交易状态、金额、手续费和合约调用。  
用自然语言解释发生了什么。

适合解决的问题：

这笔交易成功了吗。  
手续费花了多少。  
资产从哪里转到哪里。  
是否调用了智能合约。  
这次交互有没有明显风险。

核心理解：

链上数据本来是公开的，但不一定容易看懂。  
Agent 的价值，是把复杂数据翻译成普通人能理解的解释。

**2.2 钱包风险提醒 Agent**

钱包风险提醒 Agent，重点不是替用户操作钱包，而是帮助用户识别风险。

可以关注：

是否有高风险授权。  
是否和可疑合约交互。  
是否频繁出现异常转出。  
是否存在未知 Token 授权。  
是否访问过钓鱼风险地址。  
是否有异常交易模式。

适合输出的内容：

风险提醒。  
授权检查建议。  
交易解释。  
下一步排查方向。  
安全操作建议。

不适合输出的内容：

直接要求用户签名。  
直接执行撤销授权。  
直接转移资产。  
直接连接钱包并操作。

核心理解：

钱包风险提醒 Agent 的价值，是帮助用户提前看懂风险，而不是代替用户做资产操作。

**2.3 交易分析与合约交互解释**

Web3 里很多交易不是简单转账，而是和智能合约交互。

普通用户看到合约调用时，可能不知道自己到底做了什么。

Agent 可以辅助解释：

调用了哪个合约。  
执行了什么方法。  
是否发生 Token 转移。  
是否产生授权。  
交易是否失败。  
失败可能是什么原因。  
这次操作大概属于交易、质押、领取、授权还是交换。

核心理解：

合约交互对新手来说很不直观。  
Agent 可以把“看不懂的链上记录”变成“可以理解的操作说明”。

**2.4 Web3 Agent 不适合自动执行的操作**

Web3 Agent 不能因为能调用工具，就什么都自动做。

不适合自动执行：

自动签名。  
自动转账。  
自动授权。  
自动连接陌生网站。  
自动调用未知合约。  
自动导入私钥或助记词。  
自动处理资产迁移。

原因：

链上交易通常不可逆。  
私钥和助记词一旦泄露，资产可能直接丢失。  
授权错误可能导致资产被盗。  
合约风险不一定能完全识别。  
AI 判断也可能出错。

核心理解：

Web3 Agent 的第一原则应该是安全。  
任何涉及资产控制权的动作，都应该由用户自己确认。

**1.1 MCP 的进一步理解**

MCP 可以先理解为一种连接模型和外部工具、数据源的标准方式。

普通模型本身主要负责理解和生成内容。  
但真实任务经常需要外部信息，比如文件、数据库、网页、API、GitHub、链上数据等。

MCP 的作用

-   让模型更方便地连接外部工具
    
-   让不同工具接入方式更统一
    
-   降低 Agent 使用工具的复杂度
    
-   帮助模型访问外部数据源
    
-   让 Agent 更容易进入真实工作流
    

核心记录

MCP 不是新的大模型，而是帮助模型连接外部世界的一种方式。  
它解决的重点不是“模型会不会回答”，而是“模型能不能稳定地使用工具和数据”。

**1.2 MCP 和 Tool Use 的关系**

Tool Use 指的是 AI 调用工具完成任务。  
MCP 更偏向为这些工具连接提供一套统一方式。

可以简单区分为：

Tool Use：AI 使用工具这件事。  
MCP：让 AI 更标准地连接和使用工具的一种协议或方式。

常见工具类型

-   文件读取
    
-   网页搜索
    
-   数据库查询
    
-   API 调用
    
-   GitHub 仓库读取
    
-   区块链数据查询
    
-   本地命令执行
    

核心记录

Tool Use 让 Agent 能做事。  
MCP 让 Agent 接工具这件事变得更标准、更容易扩展。

**1.3 Agent 为什么需要统一的工具连接方式**

如果每一个工具都用完全不同的方式接入，Agent 的开发和维护会变得很复杂。

比如一个 Agent 同时需要：

-   读本地文件
    
-   查网页资料
    
-   访问 GitHub
    
-   查询数据库
    
-   读取链上交易
    
-   调用外部 API
    

如果没有统一的连接方式，每接一个工具都要单独处理。  
MCP 的价值就在于，让模型和工具之间的连接更有规则。

核心记录

Agent 真正有用的地方，不只是模型本身聪明，而是能稳定连接多个工具。  
工具越多，越需要统一的接入方式和清晰的边界。

**1.1 RAG 基础理解**

RAG 可以简单理解为，让大模型在回答前先去查资料，再根据查到的内容生成回答。

普通 LLM 主要依赖训练时学到的知识和当前上下文。  
但很多时候，模型本身不知道最新信息，也不知道某个具体项目、文档或数据库里的内容。

RAG 的作用

-   给模型补充外部资料
    
-   减少模型凭空编答案
    
-   让回答更贴近指定资料
    
-   适合处理文档问答、知识库问答、项目资料查询等任务
    

基本流程

-   用户提出问题
    
-   系统先去检索相关资料
    
-   把相关资料放进上下文
    
-   模型根据资料生成回答
    

核心记录

RAG 不是让模型变成搜索引擎，而是让模型在回答前多一步“查资料”的过程。  
这样可以让模型回答得更有依据。

**1.2 Agent 为什么需要外部知识**

Agent 如果只依赖模型本身，很容易受限于已有知识和当前对话。

真实任务通常需要更多外部信息，比如：

-   项目文件
    
-   数据库
    
-   网页内容
    
-   API 返回结果
    
-   GitHub 仓库
    
-   链上交易数据
    
-   用户长期记录
    

所以 Agent 要真正做事，不能只会生成文字，还要能连接外部知识和工具。

核心记录

LLM 负责理解和生成，外部知识负责提供事实依据。  
Agent 要完成真实任务，就需要把模型能力和外部信息连接起来。

**1.3 Tool Use 和 MCP 的初步理解**

Tool Use 指的是让 AI 调用外部工具完成任务。

Agent 框架可以理解为帮助开发者组织 Agent 的工具。

单独的大模型主要负责理解和生成内容，但真正要完成任务，还需要把模型、工具、指令、记忆和执行流程连接起来。

-   组织模型调用
    
-   管理工具调用
    
-   控制任务流程
    
-   保存任务状态
    
-   处理多步骤任务

Agent 框架不是新的大模型，而是把模型能力组织成任务系统的一种方式。

**1.2 Tool Calling 工具调用**

Tool Calling 是 Agent 能真正执行任务的重要能力。

普通聊天 AI 只能根据已有内容生成回答。  
有了工具调用之后，Agent 可以调用外部工具，完成搜索、读文件、运行代码、查数据等操作。

常见工具调用场景

-   搜索网页
    
-   读取文件
    
-   调用 API
    
-   执行命令
    
-   查询数据库
    
-   修改代码或生成文件
    

核心理解

工具调用让 AI 从“只会说”进一步变成“可以做”。  
没有工具调用，Agent 很容易停留在回答层；有了工具调用，Agent 才能进入执行层。

**1.3 Agent 的执行流程**

Agent 完成任务通常不是一步完成，而是一个持续推进的过程。

基本流程

-   接收任务目标
    
-   理解任务要求
    
-   拆分执行步骤
    
-   判断需要调用什么工具
    
-   调用工具并读取结果
    
-   根据结果继续判断
    
-   输出最终结果
    

例如整理一个项目时，Agent 可能需要先读取文件，再分析结构，再总结内容，最后生成报告。

核心理解

Agent 的重点不是一次性给出答案，而是围绕目标不断推进任务。  
这个过程更像一个循环，而不是简单的一问一答。

**1.4 Human in the loop 人工确认**

-   修改重要代码
    
-   提交 GitHub
    
-   发送邮件或消息
    
-   执行可能带来风险的命令
    
-   涉及钱包、资产或权限的操作
    

核心理解

Agent 越能执行真实任务，越需要控制边界。

**2.3 节点、边和状态流转**

在 LangGraph 里，可以先用三个词理解整体结构。

节点

代表一个具体步骤。  
比如理解任务、调用工具、总结结果、人工确认等。

边

代表步骤之间的连接关系。  
比如完成 A 之后进入 B，或者根据结果选择进入 C。

状态

代表任务过程中的信息记录。  
比如用户目标、工具返回结果、当前进度和中间输出。

核心理解

节点、边和状态共同决定 Agent 如何一步一步完成任务。  
这让 Agent 不再只是“生成回答”，而是可以被组织成一个可追踪的工作流。

-   重复做同一件事
    
-   忽略关键确认
    
-   输出结果不稳定
    

因此，Agent 框架和流程控制很重要。

核心理解

好的 Agent 不是单纯让模型自由发挥，而是要把任务拆清楚，把流程设计清楚，把风险边界控制清楚。

## 1.1 LLM API 调用入门

LLM API，可以理解为让程序调用大模型能力的接口。

主要作用

-   把大模型接入网站或应用
    
-   实现智能客服、自动总结、写作辅助
    
-   为 AI 工具和 Agent 提供模型能力

-   模型名称，决定调用哪个模型
    
-   输入内容，告诉模型要做什么
    
-   参数设置，影响输出结果
    
-   返回内容，供程序继续处理
    

今天的理解

-   是交易真正被网络接受的重要前提

## 3.3 区块链网络运行

区块链不是由单一服务器维护，而是由多个节点共同运行。

交易上链过程

-   用户发起交易
    
-   交易在网络中传播
    
-   节点验证交易是否合法
    
-   合法交易被打包进区块
    
-   新区块加入链上
    

核心概念

-   节点，共同维护网络的参与者
    
-   区块，存放一批交易记录
    
-   共识机制，让大家认可同一份账本
    

今天的理解

区块链最重要的不只是“分布式存储”，而是让不同参与者在不完全互相信任的情况下，仍然能维护同一套规则和账本。

智能合约可以理解为部署在区块链上的自动执行程序。

常见用途

-   发行代币
    
-   自动转账
    
-   借贷
    
-   交易
    
-   质押
    
-   NFT 和链游资产管理
    

优势

-   规则公开
    
-   自动执行
    
-   结果可验证
    

风险

-   代码漏洞可能导致资产损失
    
-   链上执行后往往难以撤回

区块链协议并不是永远不变，也会不断升级。  
升级通常是为了提升性能、修复问题或增加新功能。

升级常见目的

-   提高运行效率
    
-   降低交易成本
    
-   修复安全问题
    
-   增加新能力
    

和普通软件更新的区别

-   普通 App，可以由公司直接推送新版本
    
-   区块链协议，需要开发者、社区、节点和验证者逐步形成共识
    

可能出现的问题

-   如果分歧过大，可能发生分叉

## 3.6 Web3 关键特性与边界

Web3 关键特性

-   用户掌握资产和身份
    
-   规则公开透明
    
-   数据和操作可验证
    
-   智能合约自动执行
    
-   不同协议之间可以组合
    

Web3 的边界

-   使用门槛较高
    
-   私钥管理责任重
    
-   交易不可逆
    
-   犯错成本高
    
-   去中心化程度因项目而异
    
-   一些场景体验仍不如 Web2
    

今天的理解

Web3 的价值，在于把部分信任交给密码学、共识和公开规则。  
但相应地，也把更多责任重新交还给用户自己。

### 1.1 大语言模型 LLM

**LLM（Large Language Model，大语言模型）**，指在海量文本数据上训练出来、能够理解和生成自然语言的 AI 模型。  
ChatGPT、Claude、Llama 等都属于大语言模型。

主要能力

-   回答问题
    
-   总结文章
    
-   翻译内容
    
-   写文案、邮件、报告
    
-   提取关键信息
    
-   续写文本
    
-   辅助写代码和解释代码
    

基本工作直觉

LLM 会结合已有上下文，判断接下来最可能出现的内容，并生成符合语境的回答。  
训练数据越多、模型规模越大，语言理解、内容组织和任务完成能力通常越强。

和搜索引擎的区别

-   **搜索引擎**：从已有网页中查找信息
    
-   **LLM**：基于训练内容与当前上下文生成回答
    

因此，LLM 更擅长**解释、总结和重组信息**，但并不天然保证事实百分百准确。

常见局限

-   可能生成错误事实
    
-   可能出现“幻觉”
    
-   对最新信息掌握有限
    
-   长文本理解能力受上下文窗口限制
    
-   回答流畅不等于内容一定正确
    

* * *

### 1.2 NLP 与 LLM 的关系

**NLP（Natural Language Processing，自然语言处理）**，指让计算机理解、分析和生成自然语言。

常见 NLP 任务

-   文本分类
    
-   情感分析
    
-   命名实体识别
    
-   翻译
    
-   摘要
    
-   问答
    
-   文本生成
    

过去，很多 NLP 任务需要分别训练专门模型；  
现在，大语言模型可以依靠统一的语言能力完成大量不同任务，因此成为现代 NLP 的核心方向之一。

* * *

### 1.3 Transformer 架构

**Transformer** 是现代大语言模型最重要的基础架构。  
GPT、BERT、Llama 等模型都与 Transformer 架构密切相关。

Transformer 的意义

-   提升长文本理解能力
    
-   更好捕捉词与词之间的依赖关系
    
-   支持大规模并行训练
    
-   提升生成与理解能力

### 1.4 Attention 注意力机制

**Attention（注意力机制）**，指模型在处理信息时，对不同词语或片段分配不同关注程度。

主要作用

-   抓住一句话中的重点信息
    
-   理解上下文中的指代关系
    
-   判断词语之间的联系
    
-   提升翻译、问答、摘要等任务的准确性
    

简单理解：  
阅读一句长句时，不同词的重要性并不相同。Attention 的作用，就是让模型把更多“注意力”放在更关键的信息上。

### 1.5 Transformer 的三种常见结构

Encoder

偏重**理解输入内容**。  
常用于：

-   文本分类
    
-   情感分析
    
-   语义理解
    

Decoder

偏重**生成内容**。  
常用于：

-   文本续写
    
-   聊天模型
    
-   内容创作
    

GPT 类模型属于这一方向。

Encoder-Decoder

先理解，再生成。  
常用于：

-   机器翻译
    
-   文本摘要
    
-   信息转换

## 3\. 参加《AI 时代的 Web3 架构能力》分享

通过本次分享，初步了解了 Web3 安全与链上交互的一些基础知识。

### 3.1 Web3 安全提醒

**保护好自己的密钥，不要让第二个人知道。**  
私钥一旦泄露，资产可能被直接转走，而且链上资产转移通常不可逆。  
同时要警惕钓鱼网站、恶意签名和伪装交易。

### 3.2 基础知识记录

-   **钱包的分类**：主要分为托管钱包与非托管钱包、热钱包与冷钱包，本质区别在于私钥由谁掌握、是否长期联网。
    
-   **Web3 交易关键参数**：包括发起地址、接收地址、金额、手续费、Nonce、链 ID 等，共同决定交易给谁、做什么、花多少钱、能否正确执行。
    
-   **Web3 交易监听逻辑**：持续追踪交易从发起、广播、打包到成功或失败的全过程，并监听链上事件变化。
    
-   **交易模拟**：在真正签名前，先预演这笔交易可能带来的结果，用来提前发现资产转移、危险授权或失败风险。
    
-   **Web3 Server 测钱包安全**：通过安全服务分析钱包交互、签名和交易请求，判断是否存在钓鱼、恶意合约或资产被盗风险。
    

### 3.3 今日最重要的安全认知

**安全理解力，等于资产保护力。**  
在 Web3 世界里，不懂私钥、不看签名内容、不理解交易风险，很可能直接导致资产损失。
