# 2026-05-29
<!-- DAILY_CHECKIN_2026-05-29_START -->
这份文件是关于 **“5.25 Co-learning 区块链隐私与应用实践”** 的会议内容学习笔记。它深刻剖析了区块链因“公开透明”而衍生出的用户行为暴露风险，重点探讨了 **RPC（远程过程调用）节点** 与 **钱包** 作为数据中转站带来的隐私黑洞，并对比了不同底层密码学模型（账户模型 vs UTXO 模型）在隐私保护上的优劣。最后，文件探讨了隐私与合规的平衡，并为黑客松参赛者指明了如何从用户体验（UX）和有趣应用（如 AI + Web3、预测市场）切入的创新方向。

## 核心内容总结

该文件围绕“隐私痛点 、入口风险 、技术解法 、现实挑战、 实践创新”的逻辑展开：

1.  **隐私的本质**：链上透明性虽然解决了信任问题，但也会让用户的协议交互、持仓、转账长期处于可被关联分析的状态。隐私保护的核心并不是为了“隐藏坏事”，而是为了让普通用户在数字世界拥有合理的信息边界。
    
2.  **RPC 节点的风险**：钱包无法凭空直连区块链，必须通过 RPC 节点进行交互。中心化 RPC 服务商可以轻易收集用户的钱包地址、IP、地理位置并做活动关联，容易形成严重的数据集中与可用性单点故障。
    
3.  **模型选择偏好**：相比于以太坊“易受状态更新干扰、之前生成的隐私证明易失效”的账户模型，隐私协议更偏爱 **UTXO（未花费交易输出）模型（钞票模型）**，因为其每笔资产独立、互不干扰且并发性更好。
    
4.  **现实制约与黑客松破局**：隐私协议目前面临成本高、速度慢、数据同步困难（久不登录需同步半小时以上）等体验瓶颈。未来创新的关键在于“把复杂性留在系统内部，把简单性呈现给用户”，通过游戏化交互或隐私投票、AI + Web3 等具体产品方向打开大规模采用的通道。
    

## 有效笔记

### 1\. RPC 信任依赖与轻客户端的自主验证能力

-   **代理隐患**：钱包通过 RPC 节点广播交易和读取链上数据。如果用户完全信任 RPC 返回的信息，一旦 RPC 误报“交易失败”，用户可能会误以为真进而重复操作，造成严重的资产损失。
    
-   **理想钱包的改进方向**：钱包应集成 **轻客户端（Light Client）** 能力。轻客户端介于完全依赖第三方 RPC 和自己运行完整节点之间，可以让用户在不运行全节点的情况下本地验证链上状态，是钱包走向去中心化和隐私保护的重要技术基础。
    

### 2\. 资金池模型的网络效应

-   **混币机制**：隐私协议通常采用“资金池”模式，用户 A 将资金存入隐私池获得凭证，随后用该凭证通过另一个全新地址 B 提出资产，从而切断外部观察者对资金流向的直接追踪。
    
-   **网络效应决定效果**：隐私协议的安全性存在强烈的网络效应。池子里的使用人数越多、资金量越大、交易越活跃，匿名集合（Anonymity Set）就越大，外部就更难猜测和判断具体资金的归属，整体隐私性通常越强。
    

### 3\. 证明失效痛点：为什么账户模型不适合做隐私？

-   **账户模型的并发缺陷**：在以太坊等账户模型中，资产是一个整体的状态余额。当用户想要基于“当前账户拥有 5 元”的状态去生成零知识证明并提取资产时，如果在此期间有人刚好向该账户转入了 10 元，账户状态就会被迫更新为 15 元。这会导致原先基于 5 元状态生成的证明直接失效，严重影响系统的并发性和稳定性。
    
-   **UTXO 钞票模型的优势**：UTXO 将每笔资产视为独立的“钞票”对象。花掉其中一张 5 元钞票完全不会干扰另一张 10 元钞票的状态，因此状态变化对证明生成的影响极小，更契合零知识证明系统的构建。
    

### 4\. 既要隐私又要合规：合规证明机制

-   隐私协议的发展经常面临监管、交易所与防范洗钱/黑客攻击之间的天然矛盾。目前的平衡思路是引入“合规证明”或“不良交易预防系统”。系统允许用户在**不公开完整交易历史和细节**的前提下，通过某种密码学机制向中心化交易所证实，该笔资金在进入隐私系统前并未与任何已知的攻击地址或高风险黑客地址发生过交互，从而在风控与隐私间寻找平衡。

### 1\. 2-of-2 MPC 门限签名机制：拒绝单点托管风险

-   **原理与分片持有**：Cobo 拒绝采用向 AI 临时发放 Session Token 或 API Key 的脆弱做法，而是基于 Self-custody（自托管）理念，让 Cobo、Agent 和 Human 三方各自持有私钥分片，任何单方都无权独立挪用资金。
    
-   **双轨 2-of-2 签名模式**：
    
    1.  **Agent + Cobo 共管**：当人类在手机端 App 批准某项 Pact 授权后，Agent 发起自动化交易，Cobo 策略引擎校验符合规则后配合协同签名，实现安全无感的自动化执行。
        
    2.  **Human + Cobo 共管**：保留传统 Web3 钱包功能，当人类需要手动转移大额资产时，由 Human 发起签名，Cobo 配合完成验证。
        
-   **冗余兜底方案**：即使 Agent 服务完全挂掉或掉线，移动端设备本地和云端仍有 re-share 后的私钥分片备份，用户可通过移动端 App 随时安全导出或转出所有资金。
    

### 2\. Pact 授权协议：用密码学与策略引擎构建“硬护栏”

由于自然语言（Prompt）本身具有幻觉且不具备强制约束力，Cobo 将人类意图转化为包含以下四要素的 **Pact 刚性协议**，并由底层的钱包策略引擎在每次交易时进行 ABI 参数级的重复校验：

-   **Intent（意图目标）**：用户定义的交易愿景（如“低于 2000 美元时买入 ETH”）。
    
-   **Execution Plan（执行计划）**：AI 将意图拆解并转译出的透明、可审计的合约调用步骤。
    
-   **Policy（风控策略）**：**整个协议的核心**。严格限制单次/总预算上限、充值频次、白名单交互地址、链与特定代币。
    
-   **Completion Condition（完结条件）**：预防“僵尸授权”。一旦达到时间节点或交易额满（如买满 1000 美元），该 Pact 自动作废（Revoke），Agent 重新回到锁死受限状态。
    

### 3\. Recipe 技能层：消除大模型幻觉的“知识胶囊”

-   **大模型的链上硬伤**：在大量开发调试中发现，主流大模型普遍缺乏自主正确操作资金的能力，如果依赖其现有知识临时随意构造复杂合约参数或 API 规范，交易失败率极高且不可控。
    
-   **路径验证与规范化**：Recipe 作为一个预加载的知识库文件，提前打包了 Aave、Uniswap、Polymarket、Hyperliquid 等主流 DeFi 协议所需的合约地址、ABI 参数与风控边界。它通过给 Agent 赋予“技能包”的形式，强迫大模型必须基于已验证、确定性的路径去执行交易，从源头上遏制了 AI 的胡思乱想。
    

### 4\. 落地场景：高频小额免密支付与多 Agent 资产隔离

-   **细颗粒度隔离**：支持多 Agent 架构。一个 Human 用户可以创建多个独立的子钱包分别 delegate（委托）给 Trading Agent 或 DeFi Agent，不同 Agent 的 Pact 边界与资金额度彼此独立、互不干扰。
    
-   **小额免密自动化**：在 Pact 体系下，完美支持小额高频免密场景（如 AI 发现自身 API Token 用量不足时自动向服务商充值购买）。通过 Policy 锁死单次上限（如 100 美元）、单日频次（1次）和白名单收款地址，若遇超大额需求则触发 Human-in-the-loop（人在回路）的 App 端单次单笔授权，兼顾高频自动化与资产安全。

**隐私的本质**：区块链的公开透明解决了信任问题，但也让用户的交易频次、资产持仓和行为关联暴露无遗 。隐私保护不是为了“隐藏坏事”，而是为了给普通用户建立合理的数字边界 。

-   **UTXO 的“钞票独立性”**：UTXO 模型下，每笔资产如同相互独立的“钞票” 。花掉其中一张 5 元钞票，完全不会影响另一张 10 元钞票的状态 ，这种流水式的独立性对零知识证明和多指令并发极其友好 。

### 1\. 自动化工具与文件操作约束

-   **高效搜索**：在寻找文件或文本内容时，必须优先使用速度更快的 `rg`（Ripgrep）或 `rg --files`，仅在未安装时使用 `grep` 等替代方案。
    
-   **并行化调用**：处理 `cat`, `rg`, `sed`, `ls` 等文件读取指令时，应使用 `multi_tool_use.parallel` 工具进行**并行化处理**，严禁使用分号（如 `echo "===";`）拼接 Bash 命令。
    
-   **无损修改**：修改代码时必须使用 `apply_patch`，严禁使用破坏性命令（如 `git reset --hard` 或 `git checkout -`），且不能擅自还原用户已有的未提交改动。
    

### 2\. 证据优先级与沙盒认知原则

-   **动态行为优先**：当分析中出现证据冲突时，Codex 必须遵循以下信任链：**实时运行行为 $\\rightarrow$ 捕获的网络流量 $\\rightarrow$ 动态托管的资产 $\\rightarrow$ 当前进程配置 $\\rightarrow$ 源码/死代码/注释**（即相信运行时的真实表现，而非静态代码的表面描述）。
    
-   **沙盒视界界定**：将所有挑战目标视为靶场 fixture，不浪费时间进行 WHOIS 或 traceroute 等旨在“证明目标是否为公网”的无效检查。
    

### 3\. CTF 专项分析方法论

针对不同类型的题目，文件明确了分析的最高优先级入口：

-   **Web / API**：优先检查入口 HTML、路由注册、鉴权/会话流、隐藏端点和真实的请求顺序。
    
-   **Native / Pwn**：优先映射二进制格式、保护机制（Mitigations）、加载器/libc/运行时环境，寻找可控字节及漏洞原语。
    
-   **Crypto / 隐写**：重点在于完整恢复加密/解密链条的顺序，并记录确切的配置参数。

### 1\. **会议主旨**

Sophia探讨了**Ethereum在AI未来中的角色**，核心论点是：**“Ethereum for AI 实际上就是 Ethereum for Humans”**。Ethereum作为**公共、可编程、去中心化的基础设施**，能够为未来的“代理经济”（Agentic Economy）提供价值结算、身份、声誉和可执行协议的协调层，确保其运作是开放、中立且由人类掌控的。

### 2\. **核心背景与问题**

-   **AI Agent的进化**：AI（特别是大语言模型）已从工具演变为能**自主进行经济活动**（如写代码、转移资金、决策、雇佣其他代理）的新参与者。
    
-   **基础设施的缺失**：当前为人类设计的互联网基础设施（如密码、银行账户、法律合同）**不适合AI代理**。AI代理需要新的基础设施来处理**身份、声誉、支付和可执行承诺**。
    
-   **关键抉择**：AI代理运行所依赖的基础设施将决定未来经济的形态。是建立一个**开放、中立、由公共协议**支撑的经济，还是由**少数公司控制**？这是当下构建者面临的关键选择。
    

### 3\. **Ethereum的独特价值：CROPS原则**

Ethereum的基石是其设计原则，概括为**CROPS**：

-   **C**ensorship Resistance (抗审查)：无单一行为体能阻止网络上的有效交易或活动。
    
-   **R**esilience, **O**pen source and **F**ree (弹性、开源且自由)：代码开源，可审计、复制、分叉和在其上构建，创造了可组合的“乐高”式创新环境。
    
-   **P**rivacy (隐私)：通过零知识证明等技术，可在不泄露数据的情况下证明声明。
    
-   **S**ecurity (安全)：代码和区块链能如其承诺般安全、持续地运行。
    

这些原则使Ethereum能够为AI代理提供可信的**身份、声誉、支付和协议执行**基础。

### 4\. **关键项目与案例**

-   **EIP/ERC-8004**：为AI代理提供**可验证身份和声誉**的标准。包含验证、注册表和声誉系统，使其他方在与代理交互时可核查其信任状态。
    
-   **x402**：一种用于**机器对机器（M2M）支付**的“需支付”状态码协议，旨在简化代理间的支付流程，可使用稳定币，无需传统支付设置。
    
-   **真实案例**：以太坊基金会试点了一个项目，让**AI代理协助分配公共物品资金**。人类设定规则和价值观，AI代理规模化地执行人类决策，展示了人类如何通过设定护栏来引导AI的资本分配。
    

### 5\. **未来展望与机遇**

-   **主要应用领域**：**支付（Programmable Money）**​ 被认为是AI代理经济的关键应用场景。智能合约能对资金使用设置精确规则（如限额、用途），为代理操作提供安全性和可控性。
    
-   **突破点**：可能来自大型企业（如Stripe, Visa）对AI驱动支付和交易的需求。
    
-   **对开发者的建议**：Sophia鼓励开发者基于自身**对特定市场（如中国市场）的深入理解**，在应用层进行创新和定制，因为协议层目前无法覆盖所有场景。她认为构建实际应用比单纯开发新协议更重要。

**笔记一：AI Agent 要"做事"，就必须有经济账户**

AI 从"聊天助手"升级为"执行者"的那一刻，就会碰到现实问题——谁来付钱、谁来授权、出了事谁负责。Web3 在这里的角色不是给 AI 套一个 token，而是提供一整套经济基础设施：钱包给 agent 身份和支付能力（Coinbase AgentKit），链上记录让行为可追溯、可审计。没有这层"经济账户"，agent 永远只能建议，不能行动。

**笔记二：五层能力栈缺一不可，每个项目都在回答一个具体问题**

整个 AI × Web3 生态可以用五层来理解：资源层（算力/存储，如 Bittensor、Render）→ 行动层（agent 钱包与执行，如 AgentKit）→ 认知层（链上数据分析，如 Arkham、Nansen）→ 用户保护层（交易模拟与风险拦截，如 Blockaid）→ 机构风控层（KYT 与合规监控，如 Chainalysis）。判断任何项目时，先定位它解决的是哪一层的什么问题，就不会被叙事带偏。

**笔记三：区分真实基础设施和投机包装，关键看四个信号**

AI + Web3 容易被炒概念，文中给出了清晰的风险边界：模型输出不等于执行权（需要授权、限额、撤销机制）；链上标签是概率判断而非法律事实（需要人工复核）；去中心化资源要验证质量而非只看 token；钱包体验简化不能牺牲安全。真正值得关注的项目应该同时具备真实用户、工程接口、安全约束和合规意识。

-   Web3 里没有“注册账号”这回事。随机生成一组密钥，你就有了链上身份。助记词是私钥的人类可读备份，私钥控制账户操作权，地址是公开的收款标识。链不认人，只认签名——谁能用私钥签名，谁就是账户的主人。
    
-   资产不在钱包里。钱包只是管钥匙、构造交易、发起签名的工具，真正的余额、NFT、合约状态全部记在链上账本中。所以丢钱包和丢私钥是两回事，钱包换一个就行，私钥丢了才是真没了。
    
-   一笔交易从你点 Confirm 到最终确认，大致走这么一条路：本地签名 → 节点传播 → Mempool 排队 → Builder 排序 → Validator 出块证明 → 写入区块可查询。私钥全程不离开本地，广播出去的是签好名的交易数据。以太坊 PoS 下正常大约 12.8 分钟达到最终确认。
    
-   智能合约没有“智能”，就是部署在链上、被交易触发后按规则跑的代码。它的意义在于：规则公开可查、执行靠代码而不是某个人临时拍板、信任对象从“相信机构”变成“检查代码加共识加审计”。
    
-   最后把 Web3 归结为三门学科的交叉——密码学保障所有权可验证，经济学（Gas、质押、罚没）用成本和激励协调行为，社会学回答“谁说了算”。同时也点出了现实边界：RPC 入口层可能是中心化的，公链默认公开而非隐私，可组合性在带来乐高式灵活的同时也放大了级联风险。

### 1\. AI 的进化：从“问答交互”到“任务执行”

-   **传统大模型（问答交互）：**核心模式是“问答”，主要应用于解析代码、总结文章、写文章或回答特定问题，本质上像一个非常聪明的聊天助手。
    
-   **AI Agent（任务执行）：**核心模式演变为“任务执行”，能够主动去创建文件、运行代码、修改 Bug、读取项目等，真正登录系统并生成实际的行动。
    

### 2\. AI Agent 的核心运行机制

AI Agent 能够独立完成复杂任务，依赖于其内部的闭环运行机制：

-   **理解目标：**接收并拆解复杂的任务。
    
-   **调用工具：**执行外部能力（如运行代码、搜索等）。
    
-   **观察结果：**检查执行状态与输出。
    
-   **修正执行：**出错时能够自动调整并进行自我修复。
    

### 3\. 聊天机器人（Chatbot）与 AI Agent 的本质区别

-   **聊天机器人：**主要由用户驱动条件。用户输入“帮我配置这个环境？”，AI 反馈一段说明文档，告诉你怎么去下载、安装和配置，用户需要自己动手。
    
-   **AI Agent：**调动主体由智能体自身驱动。用户输入“帮我检查并配置好当前的运行环境”，AI 会自动分析并调用工具，在后台直接帮你进行自动下载、安装和环境配置。
    

### 4\. 认识 Hermes Agent 框架与实战安装

-   **定位：**Hermes 是一个用于全自动生产、部署和运行 AI 智能体的开源平台，它不仅仅是一个简单的聊天工具或代码编辑器，更像是一个可以通过本地工具、内存和消息平台（如微信、飞书、Telegram）连接大模型的“AI 底座”。
    
-   **实战演示：**视频在 Linux/WSL 环境下演示了通过 GitHub 开源仓库利用 `curl` 一键脚本快速安装的完整流程，系统会自动下载依赖（如 Python、Node.js、Git、uv 包管理器等），并引导用户配置 API 提供商（如 OpenRouter、Anthropic 等）及消息网关通道。视频演示中指出，Hermes 的底层运行依赖于一套现代化的技术栈。其一键安装脚本会自动配置：
    
    -   **uv：**Python 包及环境管理器（替代传统 pip/venv）。
        
    -   **Node.js & Git：**用于处理前端组件、网关连接及版本控制。
