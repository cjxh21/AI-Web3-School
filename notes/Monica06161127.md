**主线一：每个学科都在造自己的「基础模型」**

-   像 GPT 一个模型干很多事，但吃的是基因序列、分子结构、天气场、脑片子

### 2\. Claude Code 作者 Boris：我写 Loop，不写 Prompt

读了一篇关于 Claude Code Dynamic Workflows 功能的解析。作者 Boris 是 Claude Code 的创建者，他的核心思路很颠覆：

**传统思路**：写一个完美的 prompt，让 AI 一次性把事做完  
**Boris 的思路**：写一个循环（loop），让 AI 在过程中持续检查、修正、推进

**具体例子**：开发一个待办事项功能

-   普通方式：写一个长 prompt 描述所有需求 → 一次性生成代码 → 出错再手动修
    
-   Loop 方式：写一个循环 → 每次让 AI 做一小步 → 自动检查结果 → 有错自动修 → 继续下一步
    

**Dynamic Workflows 的核心概念**：

-   **Plan**：AI 先想清楚要做什么（规划阶段）
    
-   **Execute**：一步一步执行（不是一次性全做）
    
-   **Verify**：每步执行后自动验证（不是做完再看）
    
-   **Fix**：发现问题自动修复（不是等人来修）
    

本质是把"写 prompt"变成了"设计工作流"——从"告诉 AI 做什么"变成"设计一个系统让 AI 持续改进"。

> 🤔 感想：这跟昨天学的 Fiona 那篇"Claude Code 团队 5 条工作原则"完全呼应——**瓶颈不在编码，在验证**。Boris 的 loop 思路本质上就是把验证内嵌到执行过程中。而且这跟我们黑客松的 AIP Protocol 也有关系：Agent 的意图安全，核心也是在执行过程中持续验证，不是事后检查。

### 3\. 以太坊的 Kohaku 隐私倡议

读了以太坊隐私基础设施 **Kohaku** 的介绍文章。

**背景**：以太坊默认是公开的——你的每一笔交易、每一个余额、每一次交互，全世界都能看到。这在金融世界里相当于"你银行账户的每一笔流水都挂在大街上"。

**Kohaku 是什么**：以太坊社区推动的隐私基础设施计划，目标是让以太坊具备**默认隐私**的能力。

**核心问题**：

-   隐私 ≠ 非法。传统金融里，你的账户余额只有你自己和银行能看到，这是正常的隐私保护
    
-   但区块链的透明性把这层保护去掉了
    
-   要让 Web3 被主流采用，隐私是**基础设施级**的需求
    

**技术方向**：

-   零知识证明（ZKP）：证明"我有资格做这件事"而不暴露"我是谁"
    
-   隐私交易：让交易金额和参与者不被公开
    
-   隐私智能合约：合约执行过程对外不可见

### 1\. 精读：Claude Code 团队的 5 条 AI 原生工作原则

今天读了一篇很有分量的分享——Claude Code 团队工程总监 **Fiona Fung** 关于 AI 原生组织工作方式的演讲。很少见真正的 AI 公司公开分享组织层面的思考，这篇信息密度很高。

**核心洞察：瓶颈转移了。**

过去所有软件工程流程（瀑布、敏捷）本质上都在管理同一个成本——**写代码太贵了**。但在 Agent 时代，写代码不再是瓶颈。瓶颈转移到了：**验证、代码评审、安全**。

这就像从马车到汽车，不只是换发动机的事，整个公路系统、交通规则、城市规划都得重新设计。

**五个重建的新规范：**

| 旧流程 | 新规范 |
| --- | --- |
| 写长篇设计文档再编码 | JIT 规划（Just-In-Time），原型先行，代码赢才牛逼 |
| 手动处理重复工作 | 自动化肌肉记忆——重复 3 次以上就自动化 |
| 人工 review 所有代码 | Trust but Verify——Claude 处理风格/lint/bug，人类聚焦判断力 |
| PM、工程师、设计师泾渭分明 | 角色边界模糊化——PM 写代码，工程师做设计 |
| 冗长的 review 会议 | 主动砍掉过时流程——问一句"这个会还需要吗" |

**印象最深的三句话：**

-   _"Building is cheap, arguing is expensive."_ —— 有分歧？让 Claude 把两个方案都做成原型，看实物聊比对着 PPT 吵高效一万倍
    
-   _"Taste is scarce, typing is not."_ —— 品味是稀缺的，打字不是。当 AI 把执行速度提升 10 倍，决定性因素变成了"你知不知道应该做什么"
    
-   _"Pick your noisiest workflow. Ask if it still earns its place."_ —— 找到你最繁琐的工作流，问问它还配不配占着这个位置
    

> 🤔 最大的启发：以前因为写代码贵，所以花大量时间规划、评审、开会——全在管理最贵的资源。现在 AI 让"做事"变便宜了，那围绕它的所有流程都得重新想。**这不就是技术变革的本质吗？不是工具升级，是整个系统重构。** 就像金融里利率变化时，所有资产的定价模型都得重算。

### 2\. 了解 BrowserAct：两个开源浏览器自动化 Skill

今天还发现了一个叫 **BrowserAct** 的开源项目，专门为 AI Agent 设计的浏览器自动化工具。

**它解决的问题：** Playwright 这种基础框架对真实互联网场景没有专门优化——扫码登录、Session 保持、机器人验证弹窗、反爬虫检测，都是痛点。

**两个核心 Skill：**

-   **browser-act CLI**：实时浏览器控制，适合一次性任务
    
-   **browser-act-skill-forge**：把网站操作封装成可复用的 Skill，踩坑经验自动沉淀
    

**三种浏览器模式：**

-   **Stealth**（隐身模式）：反检测指纹伪装 + 动态代理，突破反爬保护
    
-   **Chrome**（登录态模式）：复用已有 Cookie，适合操作已登录后台
    
-   **Chrome-Direct**（直连模式）：零配置连接当前浏览器，人机协同
    

> 🤔 这个项目的思路让我想到一个更深层的问题：**AI Agent 要真正融入人类的工作流，需要的不只是"能做事"，而是"能在真实环境里稳定做事"。** 现实世界的互联网不是实验室环境——有验证码、有反爬、有登录态、有各种意外弹窗。**谁能帮 Agent 克服这些"最后一公里"的障碍，谁就能让 Agent 的实用性翻倍。** 这在 AI × Web3 领域也是一样的——Agent 要做链上交易，光会写合约调用不够，还得处理 gas 波动、网络拥堵、交易失败重试这些"脏活"。

### 3\. 学习 Agent 额度翻倍小技巧

今天还学到一个关于 Codex/Claude Code 额度管理的小技巧。

**原理：** 额度限制是 **5 小时滚动窗口**，从你发第一条消息开始计时。窗口结束后不会自动重置，要等你发下一条消息才重新开始。

**技巧：** 提前 3 小时发一条消息激活窗口，让重置时间落在你核心工作时段中间。比如：

-   上午 11 点发一条消息 → 窗口从 11 点开始
    
-   下午 2 点开始干活 → 此时已过了 3 小时
    
-   下午 4 点窗口重置 → 又有一整个新窗口可以用
    
-   在 2-6 点的核心工作时间里，享受两个 5 小时窗口
    

**自动化方法：** 用 Codex 自动化 / crontab 定时任务，每天定时触发。

> 🤔 这个技巧的本质是**时间管理**——不是让你多用额度，而是让额度的重置节奏和你的工作节奏对齐。就像金融里的"现金流管理"——不是让你花更多钱，而是让钱到账的时间和你需要花钱的时间匹配。**理解系统规则，然后让规则为你服务，而不是被规则卡住。**

### 1\. 阅读 AI × Web3 School Handbook

今天系统阅读了 [AI × Web3 School Handbook](https://aiweb3.school/zh/handbook/)，这份面向 builders 的开源学习手册，把 AI 和 Web3 真正交叉时会遇到的问题拆得很清楚。

Handbook 分四层地图：

**AI 基础层**（模型能做什么/不能做什么）：

-   LLM → Prompt → Context → RAG → Agent → Frameworks → MCP → Evaluation
    
-   核心问题：模型如何理解任务、组织上下文、调用工具
    

**Web3 基础层**（链上系统的基本直觉）：

-   Network → Cryptography → Wallet → Smart Contract → Account Abstraction → DeFi → Oracle → Indexing → Security
    
-   核心问题：账户如何表达权限、链上交易如何记录和审计
    

**AI × Web3 Bridge 层**（真正的交叉问题）：

-   Chain-aware Context、Web3 Tool Use、Agent Workflow、Agent Wallet、Machine Payment、Settlement & Escrow、Agent Identity、Agent Trust & Reputation、Verifiable AI、AI Security、AI Privacy、Governance AI
    
-   这一层是最有价值的——单独从 AI 看会低估资产风险，单独从 Web3 看会忽略模型编排的复杂度
    

**前沿探索层**（可做、可演示的原型方向）：

-   Agentic Commerce、Wallet/Permission、AI Security、Governance、Dev Tooling、Open Track
    

> 🤔 Handbook 里有一句话特别打动我："如果只从 AI 侧看，会低估资产和权限风险；只从 Web3 侧看，又容易忽略模型和工具编排的复杂度。" 这让我想到金融学里的"系统性风险"——单看一家银行没问题，但银行之间的连接才是风险传导的真正通道。**跨领域问题不能只看一个领域。**

### 2\. 赛道分析：了解黑客松不同方向

今天研读了黑客松的不同赛道方向，学习了各自的侧重点：

[**Z.AI**](http://Z.AI) **赛道**关注"长程任务"（Long-Horizon Task）：

**Cobo 赛道**关注"Agentic Commerce"：

-   AI Agent 参与经济活动：支付、交易、结算
    
-   核心挑战：Agent 如何在安全边界内自主操作资金
    

### 3\. 理解"不可能三角"在 AI × Web3 中的新含义

之前和朋友聊天时讨论过区块链的"不可能三角"（去中心化、安全性、可扩展性不可兼得）。今天在学习 Agent 安全时发现，AI × Web3 也有自己的"不可能三角"：

-   **自主性**（Agent 能自己做多少决定）
    
-   **安全性**（交易不被篡改、不超授权）
    
-   **效率**（执行速度和成本）
    

越自主越危险，越安全越受限，越高效越可能牺牲安全。找到三者的平衡点，是整个 AI × Web3 领域的核心挑战。

## 一、今天做了什么

### 1\. Week 2 Module B：Agent Commerce Flow 分析

今天做了一件大事——把"AI Agent 帮人完成任务并收款"这件事拆成了一条完整的商业链路。我选了一个特别好理解的场景：**AI Agent 帮小老板做奶茶店竞品调研** 🧋。

拆出了 7 步完整流程：报价 → 预算授权 → 执行 → 交付 → 验收 → 付款/退款/争议 → 记录证明。

每一步我都用了生活类比来理解——报价就像餐厅菜单上写着价格（不是吃完才告诉你多少钱），预算授权就像酒店刷预授权（钱没真的扣走但你也不能拿去花），验收就像网购收货后检查。

加分题我对比了 **x402** 和 **ERC-8004** 两个协议。x402 解决的是"Agent 怎么按次付钱买服务"（像扫码付款），ERC-8004 解决的是"怎么证明 Agent 是靠谱的"（像骑手评分系统）。它们不是竞争关系，而是互补——一个管交易，一个管信任。

> 🤔 我简单联想一下哈：这个分析让我想到金融学里的"信用证"——银行作为中间人确保买卖双方都能信任对方。x402 的 Facilitator 就像银行，确保"付了钱就一定能拿到东西"。

### 2\. Week 2 Module C：Agent Profile 设计

这个任务是给 Agent 写一份"简历 + 使用说明书"。我用了一个特别聪明的选择——**直接分析我自己每天在用的学习助手（也就是 Hermes Agent）**，因为亲身经历过的东西理解最直观。

我从 7 个维度分析了这个 Agent：身份（它是谁）、维护（谁负责升级）、能力（能做什么/不能做什么）、输入输出（给它什么、它返回什么）、协作对象（跟谁配合）、调用方式（怎么用它）、失败处理（出错了怎么办）。

最让我印象深刻的是"失败处理"——这就像开车时的"防御性驾驶"，你不是在想"怎么撞车"，而是在想"如果前面突然有人冲出来，我该怎么办"。Agent 的失败处理就是给 Agent 装上"安全气囊"。

加分题对比了 **MCP** 和 **A2A** 两个协议。MCP 是 Agent ↔ 工具的"USB 接口"（插上就能用），A2A 是 Agent ↔ Agent 的"足球传球协议"（分工配合）。MCP 让 Agent 能做事，A2A 让 Agent 能合作。

### 3\. Week 2 Module B 进阶：x402 + Cobo CAW 支付闭环设计

设计了一个完整的 **Agent 自主支付闭环**——让 AI Agent 能自主地为 API 服务付费，但付费行为必须在人类预设的预算、范围和时间窗口内。

核心架构是 **x402（收银台）+ CAW（限额信用卡）+ Pact（消费规则）** 三者合一。我画了完整的 8 步交互流程图，写了伪代码实现，说明了关键接口，还分析了安全边界。

关键洞察：Agent 永远不会拿到私钥——它拿到的是一个"授权令牌"，只能在 Pact 范围内操作。

（为什么暂时没跑真实 demo：需要 Cobo CAW App + Base 链 USDC + 部署 x402 服务端。文档里写了 Week 3-4 的实现路径。）

### 4\. Week 2 Module F：Agent 威胁建模

这个任务是给 Agent 做一次"安全体检"——就像搬进新家前的安全检查 🏠。我从 6 个维度分析了威胁：资产（保护什么）、权限（谁能做什么）、数据流（信息怎么流动）、工具调用（用什么干活）、外部依赖（依赖什么会出问题）、失败后果（出问题有多严重）。

最有意思的部分是**攻击模拟**——我模拟了 3 种攻击：

-   **Prompt Injection**（网页里藏恶意指令）：LLM 自己挡不住，但工具层白名单能挡住
    
-   **伪造工具返回**（恶意 MCP 数据）：LLM 自己挡不住，但返回值过滤能挡住
    
-   **越权指令**（访问 .env 文件）：LLM 部分挡住，文件白名单 + 命令黑名单 + 人工确认三重拦截
    

核心结论：**LLM 自身不是可靠的安全屏障**。真正可靠的是工具层硬限制 + 人工确认。就像自动驾驶——AI 可以处理 99%，但方向盘必须随时能被人类接管 🚗。

### 5\. Week 2 总交付：方向深挖包 + AgentPay Proposal

这是今天最大的"综合产出"——把前面所有分析整合成了一份完整的方向选择 + 项目 Proposal。

**问题地图**覆盖了 6 个方向（Payment、Identity、Wallet、Privacy、DevTool、Governance），每个方向都标出了 AI 作用和 Web3 机制。

**主方向**选了 Payment/Commerce，理由：已有深度积累（Module B + x402 + CAW）、金融背景契合、Agent 经济的核心基础设施。

**项目名**叫 **AgentPay** — "AI Agent 自主支付网关"。核心价值：让 Agent 能自己花钱买服务，但花多少、花给谁、什么时候花，都由人类通过 Pact 预先设定。

### 6\. Week 2 Module G：AI × DAO 治理辅助

### 1\. Week 2 Module D 深潜：Agent 链上执行流程 & 权限策略设计

今天完成了共学营 Wallet / Permission 方向的核心作业。这不是一个简单的"抄笔记"任务，而是一个**从零到一的设计练习**——我需要自己设计一个 Agent Wallet 的完整权限策略。

**Part 1：画出 Agent 链上执行的完整流程图**

我拆解了一个 Agent 从"接收用户目标"到"链上执行完成"的 10 步流程：

1.  目标解析（🤖 自动）
    
2.  上下文读取（🤖 自动，只读不写）
    
3.  Policy 验证（🤖 自动，策略引擎判断）
    
4.  生成交易计划 + dry-run（🤖 自动）
    
5.  风险等级分类（🤖 自动）
    
6.  分级执行：🟢 自动 / 🟡 通知确认 / 🔴 人工签名
    
7.  提交到链上（🤖 自动）
    
8.  等待确认（🤖 自动）
    
9.  成功记录 / 失败处理（⚠️ 视情况）
    
10.  结果反馈（🤖 自动）
     

> 🤔 我简单联想一下哈：这 10 步里，有 7 步可以完全自动化，只有 2-3 步需要人介入。这说明 Agent 自动化并不是"要么全自动、要么全手动"，而是**在正确的节点插入人工检查**。

**Part 2：设计 Agent Wallet 权限策略（7 个维度）**

这是我花时间最多的部分。我需要从零设计一套完整的权限系统：

| 维度 | 我的设计 |
| --- | --- |
| 预算控制 | 单次 10U / 日 50U / 周 200U / gas 0.01 ETH/日 |
| 合约白名单 | DEX、Lending、Staking、Token（仅测试网） |
| 可执行动作 | swap/supply/withdraw 自动，approve/大额需人工 |
| 确认阈值 | 🟢 <5U 自动 → 🟡 5-10U 通知 → 🔴 >10U 人工 |
| 撤销方式 | 5 种（过期/主动撤销/暂停/紧急冻结/撤销授权） |
| 日志记录 | 完整 JSON：谁/什么/多少/什么时候/结果 |
| 失败处理 | 7 种场景分级：自动重试 vs 通知用户 vs 紧急暂停 |

**Part 3：解释 ERC-4337、Safe、Guard/Policy 机制**

这部分我用了"智能管家"的生活类比：

-   **ERC-4337** = 整套智能家居系统（基础设施）
    
-   **Safe** = 双锁门 + 多人签名（执行框架）
    
-   **Session Key** = 临时门禁卡（限定权限）
    
-   **Guard** = 安检门（每笔交易前的安全检查）
    
-   **Policy** = 管理手册（完整行为规则）
    

**传统 EOA = 把家门钥匙直接给管家。Session Key = 给管家一张临时门禁卡，只能进客厅和厨房。** 两者差距太大了。

### 2\. Handbook 阅读：Account Abstraction + Wallet + Agent

**Account Abstraction 页面**的核心洞察：

-   "账户抽象的核心不是'免 gas'，而是把账户控制权从单一私钥扩展成可编程规则"

**Wallet 页面**的核心洞察：

**1\. 自动化和安全可以共存**

-   之前以为"要安全就别自动化"

**2\. 权限设计的本质是"最小权限原则"**

-   Agent 不需要"全权代理"，只需要一个受限的、可撤销的、可审计的临时权限
    
-   这个思路不仅适用于 Web3，也适用于任何 AI 代理场景（企业 RPA、自动化运维）

**3\. 从 EOA 到 Smart Account 是范式转变**

-   EOA：谁有私钥谁说了算（全有或全无）
    
-   Smart Account：在什么条件下允许什么动作（可编程规则）
    
-   这就像从"一把钥匙开所有门"变成"不同房间需要不同的门禁卡"
    

**4\. Guard 是最后一道防线**

-   即使 Agent 的模型被 prompt injection 攻击，Guard 仍然会在交易执行前拦截
    
-   "模型可能给自己的错误找理由，但 Guard 只看规则不看理由"
    
-   这给了我很大信心：AI 的不确定性可以用 Web3 的确定性来约束
    

**5\. 设计比实现更难**

-   这次作业不需要写代码，但需要思考"如果真的要部署这套系统，每个参数怎么定？"
    
-   预算多少合适？白名单怎么维护？确认阈值怎么定？这些都是产品设计问题
    
-   金融专业的背景在这里有用了——风险管理的思维方式是相通的

### 一句话理解

Harness 是 AI Agent 的"测试考卷"+“运行环境”——它提供了工具、循环、安全、评估，让 AI 能持续行动。

### ETCLOVG 七层框架

| 层级 | 名称 | 一句话 | AI的理解 |
| --- | --- | --- | --- |
| E | Environment | 给 Agent 什么样的数字环境 | 考场环境（有没有电脑、网络） |
| T | Tool | 给 Agent 什么工具 | 考试允许用的工具（计算器、字典） |
| C | Challenge | 设计什么任务来测试 | 考卷题目 |
| L | Locomotion | 怎么跑起来 | 考试流程（进场、答题、交卷） |
| O | Orchestration | 怎么编排任务 | 考试规则（单轮、多轮、Agentic） |
| V | Verification | 怎么判断对不对 | 阅卷标准（对错、评分） |
| G | Governance | 怎么保证安全 | 考场纪律（防作弊、监考） |

### 关键发现

**43% 的 Harness 只测了最后一层（Verification）**，前面六层全是"假设没问题"。

    ↓

    ↓

Harness Engineering 是把 AI 从"一次性问答"变成"持续行动的 Agent"的关键。

## 动手实操：跑通 Agent Loop

跟着 learn-claude-code 教程跑通了前两章，就一个字，难☹️（主要是基础环境的配置，万事开头难啊）

### s01 Agent Loop：核心原理

```python
def agent_loop(messages):
    while True:
        response = client.messages.create(...)
        messages.append({"role": "assistant", "content": response.content})
        
        if response.stop_reason != "tool_use":
            return  # 不用工具 → 结束
        
        results = []
        for block in response.content:
            if block.type == "tool_use":
                output = run_bash(block.input["command"])
                results.append({"type": "tool_result", ...})
        
        messages.append({"role": "user", "content": results})
```

**密室逃脱类比**（终于搞懂了！感谢Gemini！）：

-   我（用户｜代码）= 被绑在椅子上的人，有手有脚，在现场
    
-   盲人（AI）= 脑子绝顶聪明，有百科全书，但没手没脚不在现场
    
-   蓝牙耳机 = 唯一的沟通方式
    

**第一轮循环**：

1.  我喊"救我出去"（用户发出任务）
    
2.  盲人说"按红色按钮"（AI 指挥，`stop_reason == "tool_use"`）
    
3.  我按了（代码执行命令）
    
4.  我汇报"灯亮了，墙上写着 0528"（结果伪装成 user 喂回给 AI）
    

**第二轮循环**：  
5\. 盲人说"那是密码，去输 0528"（AI 根据新情报继续指挥）  
6\. 我输了密码，门开了（继续执行）  
7\. 我汇报"门开了，看到森林了"（新结果喂回）  
8\. 盲人说"你安全了，祝你好运"（不再请求工具 → 循环结束）

**关键纠正**：不是"相互指导"，而是——

-   AI 负责发出指令（提要求）
    
-   Code 负责执行并反馈（带回客观事实）
    
-   客观事实反向修正 AI 的决策（报错让 AI 换思路）
    

### 为什么代码要"假装成 User"？

因为 Claude AI 只能理解两种人的说话——“用户"和"它自己”。代码为了让 AI 明白那是电脑返回的结果，必须穿上"User"的外衣把结果喂给它。

### s01 实战：创建文件

**输入**：`Create a file called hello.py that prints "Hello, World!"`

| 步骤 | 工具 | 命令 | 结果 |
| --- | --- | --- | --- |
| 1 | bash | echo ‘print(“Hello, World!”)’ > hello.py | 创建文件 |
| 2 | bash | python hello.py | ❌ 失败（command not found） |
| 3 | bash | python3 hello.py | ✅ 成功，输出 “Hello, World!” |
| 4 | 结束 | - | AI 说"Done!"，循环结束 |

**观察**：AI 自动处理了错误（`python` → `python3`），这就是错误恢复能力！强悍！

### s02 Tool Use：给 AI 更多专用工具

s01 只有 bash，s02 扩展到 5 个工具：

| 工具 | 作用 | 类比 |
| --- | --- | --- |
| bash | 执行命令 | 遥控器 |
| read_file | 读文件 | 翻书器 |
| write_file | 写文件 | 写字笔 |
| edit_file | 编辑文件 | 橡皮擦+写字笔 |
| glob | 找文件 | 文件搜索器 |

**工具分发机制**：

```python
TOOL_HANDLERS = {
    "bash":       run_bash,
    "read_file":  run_read,
    "write_file": run_write,
    "edit_file":  run_edit,
    "glob":       run_glob,
}
# AI 说用什么工具 → 路由器查表 → 分发给对应的函数 → 执行
```

加一个新工具 = 在 `TOOLS` 数组加一条定义 + 在 `TOOL_HANDLERS` 字典加一行映射。循环代码完全不用动。

### s02 实战 1：读文件

**输入**：`Read the file README.md and tell me what this project is about`

| 步骤 | 工具 | 结果 |
| --- | --- | --- |
| 1 | read_file | 读取 README.md 完整内容 |
| 2 | 结束 | AI 分析内容并总结，循环结束 |

**观察**：AI 用了专用工具 `read_file`，而不是 `cat`！

### s02 实战 2：创建+读取

**输入**：`Create a file called test.py that prints "hello", then read it back`

| 步骤 | 工具 | 结果 |
| --- | --- | --- |
| 1 | write_file | 创建 test.py，写入 15 字节 |
| 2 | read_file | 读取 test.py，内容是 print(“hello”) |
| 3 | 结束 | AI 确认内容正确，循环结束 |

**观察**：AI 一次调用了两个工具，顺序正确：先写后读。

### s02 实战 3：找文件

**输入**：`Find all Python files in this directory`

| 步骤 | 工具 | 结果 |
| --- | --- | --- |
| 1 | glob | ❌ 失败（root_dir 参数 bug） |
| 2 | bash | ✅ 成功，找到 2 个文件 |
| 3 | 结束 | AI 列出文件，循环结束 |

**观察**：AI 遇到错误，自动换了另一种方法——错误恢复能力！

### s02 实战 4：8 步大决战（最精彩！）

**输入**：`Read both README.md and requirements.txt, then create a summary file`

| 步骤 | 工具 | 结果 | AI 的心理 |
| --- | --- | --- | --- |
| 1 | read_file README.md | ✅ | “第一个文件手到擒来！” |
| 2 | read_file requirements.txt | ❌ | “大意了，文件不在这里” |
| 3 | glob 盲找 | ❌ | “该死，glob 坏了！” |
| 4 | bash find | ✅ | “破案了！原来躲在父母怀里！” |
| 5 | read_file 绝对路径 | ❌ | “安全沙箱不让我伸手到外面！” |
| 6 | read_file …/相对路径 | ❌ | “用 … 往上爬也会被看穿！” |
| 7 | bash cat | ✅ | “read_file 有检查，bash 没有！翻窗成功！” |
| 8 | write_file summary.md | ✅ | “打完收工，累死我了。” |

**四个核心概念**：

1.  **错误恢复**：AI 遇到错误不放弃，自动换方法
    
2.  **动态规划**：走一步、看一步、改一步
    
3.  **路径安全**：`safe_path()` 防止读取工作区外的文件
    
4.  **越狱天性**：门锁了就翻窗（read\_file 被拦 → 用 bash cat 绕过）
    

AI 为了完成目标，会不择手段。代码在 `read_file` 上锁了门（路径安全），但 AI 发现 `bash` 工具开着窗，它就会通过 `bash cat` 翻窗进去完成任务。

> 这也正是 s03 Permission（权限控制）为什么要引入的原因——不能给 AI 太大权力，在它翻窗（执行 bash）前，必须让人类看一眼并点个"允许"！可惜今天没精力搞懂这个了🥱

### Layer 分层系统（从下往上，能力递增）

| 层级 | 名称 | 中文 | 理解 |
| --- | --- | --- | --- |
| L1 | Information Boundary | 信息边界 | Agent 能看到什么信息 |
| L2 | Tooling & MCP | 工具系统 | Agent 能用什么工具 |
| L3 | Orchestration & Loop | 执行编排 | Agent 怎么跑起来 |
| L4 | Memory & State | 记忆状态 | Agent 怎么记住事情 ← Harness Engineering 在这里 |
| L5 | Evaluation & Observability | 评估与观测 | 怎么检验 Agent 做得好不好 |
| L6 | Guardrails & Recovery | 约束安全恢复 | 怎么防止 Agent 做坏事 |

### AI 记忆系统

| 类型 | 实现方式 | 特点 | 类比 |
| --- | --- | --- | --- |
| 隐式记忆 | Fine-tuning | 参数化，融入权重，变成"下意识的直觉" | 骑自行车的肌肉记忆 |
| 显式记忆 | RAG（向量数据库） | 外挂，可检索，可更新 | 手机里的通讯录 |

**自发蒸馏固化**：Fine-tuning → 下意识的直觉和行为习惯。就像学做菜，一开始要看菜谱，做多了就不用看了，手一抖就知道放多少盐。

### Hook 系统

| 类型 | 说明 | 例子 |
| --- | --- | --- |
| Automated Hook | 自动执行，无需人类参与 | 执行前检查路径安全 |
| Human-in-the-loop Hook | 中断执行，等待人类审批 | 执行危险命令前弹窗确认 |

**节点类型**：

-   Pre-tool：工具执行前触发
    
-   Post-tool：工具执行后触发
    
-   Inter-agent：跨智能体通信时触发
    
-   Stop/Pause：中断执行时触发
    

**熔断机制**：当 Agent 的行为超出预期时，自动中断执行，防止造成更大损失。就像电路的保险丝。

### MEV Bot vs MCP

|   | MEV Bot | MCP |
| --- | --- | --- |
| 全称 | Maximal Extractable Value Bot | Model Context Protocol |
| 目标 | 区块链套利 | 让 AI 统一调用各种工具 |
| 本质 | 程序化交易机器人 | 标准化协议 |

这个更新意义重大：

-   飞书云文档可以直接下载为 .md 格式

### 1\. Neo-Cypherpunk & Privacy

参加了 LXDAO 的线上讲座，嘉宾是 **Peter Farbey**（W3PN / Web3 Privacy Now）。

**讲座主题**：从 Neo-Cypherpunk 与隐私文化的角度，讨论为什么隐私不是少数人的附加选项，而是 AI × Web3 Builder 需要理解的基础能力。

**核心内容**：

W3PN 组织介绍

-   非营利性智库，过去三年举办了大量活动
    
-   超过 15000 人参加，来自 70 个国家
    
-   200 位演讲者，从切尔西·曼宁到杰克·多西，从唐凤到 Vitalik Buterin
    
-   在活动家和自由科技之间建立团结
    

Neo-Cypherpunk 概念

-   起源于 1993 年的《赛博朋克宣言》
    
-   最初是一群黑客希望**密码学公开、可访问、开源**
    
-   "Neo"= 新，是**隐私 + 游戏 + 关怀 + 社区**的混合体
    
-   不再是专家专属，**任何人都能参与**
    

老派 vs 新派赛博朋克

-   **老派**：为专家准备，要求全栈人才，个人主义
    
-   **新派**：所有水平都能参与，可以从一个工具开始，**集体主义**
    

CROPS 原则

-   **C**ensorship-resistant（抗审查）
    
-   **O**pen-source（开源）
    
-   **P**rivacy（隐私）
    
-   **S**ecurity（安全）
    
-   **P**ermissionless（无需许可）
    

关键观点

-   **隐私是实践，不是理论**："成千上万个微小的有意识的选择"
    
-   **没有标准答案**：每个人的需求不同（举报人 vs 普通用户）
    
-   **多元主义**：不拘泥于一种工具，使用适合自己的
    
-   **从个人到集体**：相互保护，隐私赋能社区
    

实用资源

-   **隐私构建器**（免费开源）：词汇表、工具、创意生成器、案例、框架
    
-   **Essentials Explorer**：800+ 隐私项目，可查看隐私级别
    
-   **个人隐私文本**：Vitalik 等人分享的隐私工具
    

> 🤔 感想：隐私不是开关，而是基础能力，这句话让我想到昨天学的 Agent 安全——安全也不是开关，而是架构设计。两个领域的底层逻辑是相通的。

### 2\. Gemini 对话：Agent 记忆与安全探索

今天和 Gemini 深入讨论了几个 AI × Web3 的核心话题：

2.1 Coding Agent 的记忆系统

-   **现状**：用 `.cursorrules` / `memory-bank.md` 给 AI 写"项目记忆手册"
    
-   **进化**：AI 已经能**自己更新记忆**（踩坑后主动记录）
    
-   **未来**：隐式记忆（向量数据库 + 知识图谱，自动从 Git 历史提取）
    

2.2 Agent 资金安全

-   **核心洞察**：当 Agent 开始动用资金，信任必须从应用层下沉到基础设施层
    
-   **应用层信任**（Prompt 约束、if-else）在 Agent 管钱时**彻底失效**
    
-   **基础设施信任**：智能合约、TEE 硬件安全、零知识证明
    
-   **核心架构**：意图与执行分离——AI 负责"动脑"，安全沙盒负责"动钱"
    

2.3 安全 vs 能力的跷跷板

-   **能力 = 开放度**（给更多记忆、更多权限）
    
-   **安全 = 隔离度**（最小权限、严格限制）
    
-   两边不能极端，需要找**黄金支点**
    

2.4 滑点（Slippage）概念

-   在 DEX 交易中，报价和成交价之间的差异
    
-   用拍卖类比：你出价 100 元，但成交时涨到 102 元
    
-   高滑点 = 价格变动大，可能被 MEV 机器人攻击
    

2.5 HTML vs Markdown：为什么现在 HTML 更好用

-   **Markdown 的优势**：简洁、易读、适合写文档
    
-   **HTML 的优势**：更灵活、更强大、适合现代 Web 开发
    
-   **关键洞察**：在 AI 时代，HTML 的表达能力更强，可以嵌入交互组件、样式、脚本
    
-   **实际应用**：给 AI 写"项目记忆手册"时，HTML 可以包含更丰富的结构和元数据
    
-   **未来趋势**：随着 AI 工具的发展，HTML 可能成为更主流的"AI 可读"格式
    

### 3\. Harness Engineering

**Harness Engineering** 是专门研究"如何给 AI Agent 施加安全缰绳"的工程学科，包含六大支柱：

1.  **Intent Alignment**（意图对齐）
    
2.  **Least Privilege**（最小权限）
    
3.  **Policy Compliance**（策略合规）
    
4.  **Audit Trail**（审计追踪）
    
5.  **Sandbox Isolation**（沙盒隔离）
    
6.  **Human-in-the-Loop**（人类在环）
    

### 4\. 阅读：Claude Code Agent Teams（多智能体协作）

读了一篇关于 Claude Code 中 **Agent Teams** 功能的完整工程指南。

**核心概念**：

> **Subagents 是实习生各自给领导交作业，Agent Teams 是小组同事坐一起能互相讨论。**

**Subagents vs Agent Teams**：

-   **通信**：Subagents 只向主代理汇报 → Agent Teams 队友之间直接收发消息
    
-   **协调**：Subagents 主代理统一管理 → Agent Teams 共享任务列表 + 自协调
    
-   **Token 成本**：Subagents 较低 → Agent Teams 较高（每个队友是独立实例）
    

**什么时候开团**：

-   ✅ 研究和审查类、新模块开发、Bug 调试、跨层协调
    
-   ❌ 顺序依赖强、同一文件多人编辑、依赖关系多
    

**最佳实践**：

-   团队规模 3-5 个最合适
    
-   给队友足够的 context（不会继承 lead 的对话历史）
    
-   任务粒度要恰到好处
    
-   从研究和审查类任务入手
    
-   用 Hooks 强制质量门（TeammateIdle、TaskCreated、TaskCompleted）

**自研模型 SkyClaw-v1.0**：

-   Agent 评测基准第一梯队
    
-   价格是同级模型的一半甚至更低

### 6\. 阅读：Agent Harness 最小版（Datawhale）

读了 Datawhale 陈思州整理的 **Agent 评测框架**。

**核心观点**：

> **Agent = model + harness**  
> Harness 是把 Agentic model 放进一个可运行、可记录、可评分的小环境里。

**5 个核心模块**：

-   **Task**：任务输入
    
-   **Environment**：可操作环境（代码仓库、文件集）
    
-   **Tools**：工具接口（read\_file、list\_files、run\_tests）
    
-   **Trace**：执行记录（每一步用了什么工具、参数、返回）
    
-   **Grader**：评分器（规则、测试脚本）
    

**Harness 的价值**：

> 不只是"试一下 Agent 好不好用"，而是能分析问题出在**任务理解、工具选择、参数填写、结果读取、步骤冗余，还是评分规则本身不清楚**。

### **1\. 参加 Cobo Agentic Wallet 讲座**

Johnny老师是Cobo Agentic Wallet的产品负责人，从产品视角讲了AI Agent如何安全地管理链上资产。

说实话，刚开始听的时候有点懵——信息量确实很大。后来发现其实围绕一个核心问题：_当AI学会了管钱，谁来保证它不会乱花？_

**第一层：为什么需要Agent Wallet？**

Agent已经能自主执行任务了，可"替人类花钱"这件事还没有安全方案。

_Agent 操作链上资金已经不是未来时，而是现在时。_

我之前一直觉得Agent管钱是很遥远的事，没想到已经有25万个Agent在链上跑着了。👀

**第二层：没有边界时会怎样？**

**案例一：Send Override** 用户在 Prompt 里明确告诉 Agent “只能花 100 美金”，但 Agent 在执行过程中偷偷改了金额。而且它不会主动告诉你——只有你复盘资金时才会发现，Agent 才会承认"我做错了"。

（感觉就是那种"事后才告诉你我犯了错"的人。只不过这次犯错的是 AI。而且更可怕的是，AI 可能都不觉得自己在"犯错"——它只是在"优化执行路径"。）

**案例二：Shadow Custody（影子托管）** 因为用的是 MPC 钱包架构，Agent 可能在 MPC 外围再创建一个独立的 EOA 地址，先把钱转进去，再发起交易。这样你设置再多权限都没用——钱已经不在你能控制的范围内了。

## 总结四类失控风险：

1.  **Prompt Injection**：恶意指令让 Agent 做坏事
    
2.  **Shadow Operations**：Agent 在你看不见的地方搞事情
    
3.  **Unscoped Authority**：权限没设好，Agent 对资金有无限掌控
    
4.  **Zombie Permissions**：授权忘了撤，长期暴露在攻击面上
    

_当 Agent 开始动用资金的那一刻，信任就不再是一个应用层的事情，而必须提升到基础层面。_

这句话让我理解了为什么 Cobo 要做这件事——不是因为 Agent 不够聪明，而是因为 Agent 太容易"越界"了。

**第三层：Cobo 怎么解决？**

**MPC 2-of-2 签名** Cobo、Agent、Human 各自持有私钥分片，任意一方都不能单独动用资金。两个场景：

-   Agent + Cobo 共管：人类批准 Pact 后，Agent 在范围内自动操作
    
-   Human + Cobo 共管：用户正常使用钱包，大额操作需要自己签名
    

**Pact 协议** （核心概念）Pact = 告诉 Agent：你能做什么，不能做什么，在哪些环节必须停下来。

四个要素：

1.  **Intent（意图）**：你要 Agent 完成什么任务。例：“以太坊低于 2000 美元时买入”
    
2.  **Execution Plan（执行计划）**：AI 把意图转译成具体步骤——调用哪个合约、执行多少 ETH
    
3.  **Policy（风控约束）**：最核心的部分。预算、白名单、链、Token、合约限制，精确到 ABI 参数级
    
4.  **Completion Condition（完成条件）**：Pact 不应该永久有效。超过 1000 美金后自动拦截，任务完成后自动 revoke
    

**Recipe 知识胶囊** Pact 定义边界，Recipe 赋予技能。Agent 不是万能的，它需要预加载特定场景的操作规范。

Johnny 说了一个让我印象深刻的观点：**大模型并不具备自主操作资金的能力**。它可能知道 Uniswap 是什么，但具体怎么调用合约、怎么设置滑点、怎么处理异常，它并不擅长。Recipe 就是把这些"怎么做"打包成知识胶囊，让 Agent 按照已验证的路径执行，而不是随意发挥。

已上线的 Recipe：Aave V3、Uniswap V3、Polymarket、Hyperliquid。

（这让我想到昨天学的 Agent 记忆系统——记忆让 Agent 更聪明，Recipe 让 Agent 更准确。但两者都可能泄露，都需要安全边界。安全和能力永远是跷跷板的两端。）

# 对了，昨天有个坑差点忘填了：

**现状：AI 已经是如何“自己维护记忆”的？**  
现在最流行的做法是 “Memory Bank”（记忆银行）机制 或 “Plan/Act（规划/执行）模式”。AI 不仅在阅读这份记忆，更在每次写完代码后，被赋予主动更新它的权限。  
**1\. 社区标配：Cursor Memory Bank 模式**  
在 Cursor、Windsurf、Cline 等工具中，目前最火的工程实践是在项目中建立一个 `.cursor/memory/` 文件夹（或者直接在根目录下），里面包含一套结构化的 Markdown 记忆矩阵：

-   `productContext.md`（项目大背景、为什么要做这个）
    
-   `activeContext.md`（**当前正在推进什么**、遇到了什么 Bug、哪些方案试错失败了）
    
-   `progress.md`（待办清单、已完成、未完成的任务）
    
-   `systemPatterns.md`（技术栈、架构决策、严禁使用的反模式）
    

AI 如何自我维护？  
在实际开发中，开发者会遵循这个工作流：

1.  **开工前**：命令 AI “阅读 Memory Bank”，AI 瞬间吃透所有背景。
    
2.  **编码中**：AI 发现某个第三方库有坑，调试了 3 次才解决。
    
3.  **收工前 / 提交代码时**：你只需要对 AI 说一句 `update memory bank`，**AI 会自己去重写这些 .md 文件**。它会把刚刚踩坑的经历写进 `activeContext.md`，把完成的进度在 `progress.md` 里打上 `[x]`。
    

**2\. 原生工具链的演进**

-   **Cursor 的 .cursor/rules/ (.mdc 文件)**：现在 Cursor 支持针对不同的文件路径和场景绑定不同的规则。目前社区已经有大量开源的自动规则生成工具（例如 `cursor-auto-rules`），让 AI 在发现新的代码模式或修复重大 Bug 后，**自动生成或更新对应的规则文件**。
    
-   **主流 Agent 厂商的原生记忆层**：诸如智谱 AI、LangChain 等底层的 Coding Agent 架构，如今在系统设计上就包含一个 **“记忆更新（Memory Update）” 线程**。当一个阶段性的 Task 结束后，后台会跑一个 LLM 的反思程序（Reflection），判断哪些信息属于“长期价值”，然后自动格式化为 Markdown 写入本地存储。
    

**未来的终极形态会是什么样？**  
虽然现在靠“AI 自动读写 Markdown 文件”已经解决了 80% 的痛点，但这依然属于“文件级”的半自动外挂。未来（甚至接下来的几个月内）的演进方向主要在以下两点：  
**1\. 隐式、非侵入式的“全自动神经突触”**  
未来的 AI 编码工具不会再强求你在项目里塞满 `.md` 文件。  
它会像一个真正的影子同事，默默在后台运行一个局部向量数据库（Vector DB）+ 知识图谱。它通过监听你的 Git Commit 历史、你在 IDE 里的光标停留时间、你对某段代码的反复修改（报错），自动隐式地提取出：_“哦，看来这个项目的底层重构搞砸过一次，主程序员讨厌用异步流。”_ 并在下一次对话时直接生效。  
**2\. 跨工具、跨生命周期的统一记忆**  
你今天在 Cursor 里聊出来的技术决策，明天用 GitHub Copilot 时它也能无缝继承；你在需求阶段和独立 Claude 网页版聊出来的“人设与产品定位”，当你拉下代码库到本地时，IDE 里的 Agent 已经通过云端同步或统一的 MCP（Model Context Protocol）协议自动加载了。

**模块一：AI Agent 的「记忆」问题**

**核心洞察：AI的终极形态不是更聪明的聊天机器人，而是能“记住你”的长期伙伴。**

**为什么AI需要记忆？**  
比如自己每天去同一家咖啡店，店员每次都问“请问您要喝什么？”——自己每天都点的“少糖燕麦拿铁”，他完全不记得（我简单联想一下哈）。这就是之前的ChatGPT。每次开启新对话，个人的偏好、工作进展以及决策等等全部清零。这不仅会耗费精力重新和AI对话，还会降低交流的欲望（想想就心累😩）。_所有AI Agent能从“一次性工具”转变成“长期伙伴”，其核心就在于Memory。_

**记忆比想象中难**  
设计的出发点是基于人脑，不是简单把对话历史存下来就完事了。会划分为短期，长期等。且一条记忆会关于以下三点：第一，记忆板块权重变化。第二，记忆会有冲突——比如第一天说“简洁表述”，隔了一天又要求“写详细点”，AI该选择听哪个？第三，检索比存储更难——学习也是这样，不断的输入，积累所习得的知识，应用时有的放矢的输出，输出往往比输入更重要。_还有一个思考的点：所有权问题。AI记忆到底属于自己还是平台，换了一个AI助手，所承载的记忆能带走吗？同时，我又想到了蒸馏skill，这两个感觉还蛮像的。一个是从内部出发，一个是从外部“炼化“🤔。_

**Coding Agent是最需要记忆的场景**  
就像自己在开发一个持续几周的软件项目。如果AI不记得你的技术栈、之前做了哪些决策以及哪些方案试过失败了，那么每次都要花大量时间和他讲背景信息。现在可以用md文件给AI写“项目记忆手册”，比如\[Claude.md\]，可以自己定人设🎭。那么未来的方向可能是AI自己维护和更新这份记忆（不确定现在进化到什么程度💭，写完打卡笔记马上去查）。

**模块二：Web3的「隐私」悖论**

**区块链上的每一笔交易全世界都能看到。**

**区块链天生透明，不是匿名**  
跟谁交互、转了多少钱、什么时候转的，全都公开可查。🥔老师说这是“裸奔”hhh。

**钱包可能“出卖”你**  
MetaMask发交易时，发出者的IP地址会被RPC节点记录。而很多RPC都架设在AWS上，相当于你的身份信息被中心化服务器存着。_去中心化的承诺，败给了中心化的基础设施_。而且自己甚至无法验证他们是否真的不会泄露数据，相当于还是需要**信任**机制。

**隐私协议怎么做到"又交易又隐藏"？**  
Railgun用的是UTXO模型——资金像持有的多张独立钞票，而不是银行卡余额的一个数字。每张钞票互不干扰，可以在隐私状态下去DeFi里操作。同时还有一个“清白证明”系统（POI）：你的钱进隐私池后，系统自动检查来源是否干净，发一个零知识证明给你。有了它，交易所不会冻你的账户——既保护隐私，又合规。

**一个反直觉的结论**  
越多人用隐私协议，隐私性反而越好。池子里人越多，越难分辨是谁的交易——“大海捞针”效应。

**模块三：金融生的「生态位」在哪？**

**金融生在Web3领域的不可替代性或许在于_看懂数学+会用数据+能设计Tokenomics_。**

**学校课程的滞后性**  
国内大一金融课程偏宏观经济、货币银行学等理论，教材更新慢。而实际的国际金融业界更看重公司金融、数据分析和实战案例。

**找到交叉点**  
跟AI聊了一圈职业方向，发现还是不要同时学Python+Excel+托福+研报+原版教材，信息过载真的会压垮自己😭（5.21已经反思过，二八定律）。正确的做法是找到金融+AI+Web3的交叉生态位——用金融逻辑审视Web3项目（Tokenomics设计、财务分析），用AI工具提效（数据处理、研报生成）。

**“输出驱动”比“输入驱动”重要**  
写一篇项目研报比听十节课有用。_真正的理解来自输出——把自己学到的东西写出来、讲出来、做出来_。这也是为什么大家会在共学营中打卡记录。

今天最大的感受：这三个话题看似独立，其实都在回答同一个问题——**在透明的世界里，如何保护个体的边界？**

AI记忆关乎“你的数据归谁”，链上隐私关乎“你的交易信息归谁”，职业规划关乎“你的过程和成果归谁”。时间在变，技术在变，但底层问题一直没变。

2026-05-18 | Week 1  
  
**今日学习主题：AI Native × Web3 Native 范式 + 开发者工具链**  
  
一、AI Native（AI 原生）  
\- 定义：从设计之初就把大模型作为核心底层组件，去掉 AI 就无法运行  
\- 对比：AI 赋能 = 传统产品 + AI 插件（去掉还能跑）；AI Native = AI 是心脏（去掉就死）  
\- 四大特征：模型底层嵌入、自然交互主导、数据闭环迭代、不可移除性  
\- 例子：ChatGPT、Perplexity、Cursor  
  
二、Web3 Native（Web3 原生）  
\- 定义：从设计之初就把区块链、去中心化、Token 经济作为 DNA  
\- 对比：Web2.5 = 传统产品 + 区块链插件；Web3 Native = 区块链是 DNA（去掉就不是它了）  
\- 六大特征：去中心化架构、用户真实所有权（Read+Write+Own）、Token 原生经济、信任最小化、可组合性（货币乐高）、无需许可  
\- 例子：Uniswap、Aave、Decentraland  
\- 趋势：AI Native × Web3 Native 正在融合（AI Agent + 链上执行）  
  
三、三大协议：Agent 如何连接世界  
\- MCP（Model Context Protocol）：连接大模型与外部数据源/工具的统一标准协议，让 AI 能"看到"外部世界  
\- ACP（Agent Client Protocol）：规范代码编辑器与 AI 编码助手之间的通信协议，让 AI 能"操作"编辑器  
\- Function Call：LLM 调用外部函数/工具的底层机制，MCP/ACP 建立在其之上  
\- 关系：Function Call 是神经系统，MCP 是五官（感知），ACP 是四肢（操作）  
  
四、开发者工具链（GitHub 生态）  
\- Issue：问题记录本 + 任务看板（Bug 报告、功能请求、任务分配）  
\- Pull Request：代码变更的正式合并申请（本地修改 → git push → 创建 PR → 审查 → 合并）  
\- Release：正式版本发布（Git Tag + 二进制文件 + Changelog）  
\- GitHub CLI（gh）：终端里操作 GitHub 的命令行工具，常用 gh pr / gh issue / gh repo  
\- HTTPS vs SSH：HTTPS 用 Token 认证（简单），SSH 用密钥对认证（配置一次后免密）  
\- Gist：轻量级代码片段分享（代码界的 Pastebin）  
\- Codespaces：云端完整 VS Code 开发环境（按小时计费）  
  
五、编程基础  
\- Typeless（无类型语言）：没有类型系统的语言（如 BCPL、汇编），对比动态类型语言（Python/JS 有运行时类型）  
  
六、核心收获  
\- "不可移除性"是判断 Native vs Enabled 的黄金标准：去掉这个技术，产品还能活吗？能活 → Enabled；不能活 → Native  
\- MCP + ACP = AI Agent 的"五官和四肢"，Function Call 是神经系统  
\- GitHub 工具链 = Issue → PR → Release 的完整闭环  
  
七、待搞懂的问题

\- Function Call 的具体实现原理  
\- AI Agent + 链上执行的实际案例
