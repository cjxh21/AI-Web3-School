### 二、 Cobo AW 核心技术架构 \[[06:13](https://www.youtube.com/watch?v=ilXrZaF8niI&t=373)\]

1\. 双二机制（Binary）控制堆栈 \[[07:30](https://www.youtube.com/watch?v=ilXrZaF8niI&t=450)\]

当通过平台提供的引导脚本或技能（Skills）进行一键集成时，系统会在本地注入两个核心可执行文件 \[[06:56](https://www.youtube.com/watch?v=ilXrZaF8niI&t=416)\], \[[07:30](https://www.youtube.com/watch?v=ilXrZaF8niI&t=450)\]：

-   `CW` **Binary (Cobo Wallet 命令行工具)**：负责直接同 Cobo 后端进行接口通信，封装了大部分高级 API 操作 \[[07:41](https://www.youtube.com/watch?v=ilXrZaF8niI&t=461)\]。
    
-   `TSNode` **Binary (签名管理进程)**：作为本地独立的常驻进程（Daemon），负责管理底层的密钥分片（Secrets）及密码学协同运算，调用 API 裸开时必须手动拉起此进程 \[[07:54](https://www.youtube.com/watch?v=ilXrZaF8niI&t=474)\], \[[31:01](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1861)\], \[[31:41](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1901)\]。
    

2\. 密码学 MPC（多方计算）与风控结合 \[[07:54](https://www.youtube.com/watch?v=ilXrZaF8niI&t=474)\]

-   **非单点私钥**：不同于 MetaMask 或 Rabby 将完整私钥落盘在浏览器中，Cobo AW 采用 **MPC（Multi-Party Computation）** 架构。私钥分片分别独立存在于用户端（本地设备/智能体）和 Cobo 风险控制服务端 \[[08:12](https://www.youtube.com/watch?v=ilXrZaF8niI&t=492)\], \[[19:12](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1152)\]。
    
-   **全生命周期无完整私钥**：在整个链上交易、配对或多轮交互签名的过程中，**任何一方的内存里都不会在任何时刻组合出完整的私钥**。即使用户单方面关闭节点或 Cobo 单方面拒绝配合，任何单一方都无法把资金转走 \[[08:34](https://www.youtube.com/watch?v=ilXrZaF8niI&t=514)\], \[[09:09](https://www.youtube.com/watch?v=ilXrZaF8niI&t=549)\]。
    
-   **免疫提示词注入攻击**：AI Agent 发起的所有交易签名请求都强依赖多方协作，这使得每一笔资产出账都必须实时穿透并强匹配 Cobo 后端配置的防幻觉与防注入风控引擎（Risk Engine），有效避免 AI 被注入恶意 Prompt 从而把资产卷向莫名其妙的地址 \[[09:39](https://www.youtube.com/watch?v=ilXrZaF8niI&t=579)\], \[[10:01](https://www.youtube.com/watch?v=ilXrZaF8niI&t=601)\]。
    

### 三、 智能体交互核心：Pack 机制 \[[17:43](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1063)\]

-   **定义与原意**：**Pack** 本质上是智能体向人类申请并达成的“协议（Protocol）”或“契约（Contract）” \[[18:01](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1081)\]。
    
-   **痛点解决**：如果 AI 每次小额交互、Dex 定投或调用付费工具都要通过移动端向人类弹窗要签名，高频交互的摩擦力会让人崩溃 \[[13:34](https://www.youtube.com/watch?v=ilXrZaF8niI&t=814)\], \[[15:51](https://www.youtube.com/watch?v=ilXrZaF8niI&t=951)\]。
    
-   **运作流**：
    
    1.  智能体根据人类最初用自然语言下达的模糊任务（如：原话为“帮我用 Claude 自动优化代码并付钱生图”），自行解析生成对应结构化的用户意图（Intent）和具体的执行限制（Policy）\[[48:41](https://www.youtube.com/watch?v=ilXrZaF8niI&t=2921)\], \[[49:25](https://www.youtube.com/watch?v=ilXrZaF8niI&t=2965)\]。
        
    2.  智能体调用 `pack:create` 接口，向人类手机端推送一个 Pack 审批申请 \[[18:09](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1089)\], \[[27:09](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1629)\]。
        
    3.  **人类一键批准后，智能体将获得一个在该契约锁定的安全边界内、拥有自主高频支付权限的临时 API Key** \[[18:41](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1121)\], \[[28:46](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1726)\]。
        

> **AI 洞察（AI Insight）防御力**：为了防止 AI 遭受黑客控制“说一套做一套”（即在意图字段里写着‘我正在帮你做长线定投’，而实际 Pack 策略细则里申请的却是‘拥有向黑客地址无限制转账的最高权限’），Cobo AW 服务端内置了大模型裁判。它会自动审查意图与 Policy 约束之间是否语义对齐，并将安全风险及异常标志强行提示在人类手机端 App 界面上，供人类一键一键拒绝（Reject）\[[50:34](https://www.youtube.com/watch?v=ilXrZaF8niI&t=3034)\], \[[51:17](https://www.youtube.com/watch?v=ilXrZaF8niI&t=3077)\]。

-   **Pack 自动失效机制**：Pack 具有不可变性，创建后无法修改 \[[51:36](https://www.youtube.com/watch?v=ilXrZaF8niI&t=3096)\]。为防范本地 key 泄露风险，Pack 支持传入参数设定**自动失效终点条件**：可基于**时间界限**（例如仅持续有效 1 小时或 1 天）或**交易频次/次数**（例如满 100 次调用后自动作废）\[[28:12](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1692)\], \[[28:42](https://www.youtube.com/watch?v=ilXrZaF8niI&t=1722)\]。

### 一、 什么是 AI 原生支付基础设施？ \[[01:01](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=61)\]

目前，各大平台正在积极推进 AI 智能体的钱包协议（如 Coinbase 去年推出的 **X402 协议**）\[[01:16](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=76)\]。随着智能体交易量的激增，Flux 正在基于该类协议构建面向 AI 原生的高级支付基础设施，赋予 AI Agent 在不同经济活动场景下的高频支付能力。

其核心探索的**三大主流支付场景**包括 \[[01:49](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=109)\]：

1.  **Agent to Agent（智能体对智能体）**：智能体之间相互雇佣并支付报酬以协同完成任务。例如：一个营销 Agent 付钱给另一个脚本创作 Agent 来为 TikTok 视频编写文案。
    
2.  **Agent to Merchant（智能体对商家）**：智能体直接向 B 端商家（如 OpenAI、Anthropic）支付 API 消耗或服务费用。
    
3.  **Agent to Generic Tools/APIs**：智能体自主调用并支付通用的第三方开放平台（如 Apify、RapidAPI）所提供的基础设施及数据抓取等工具。
    

### 二、 从“提示词工程”向“沙盒控制工程”的演进 \[[08:53](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=533)\]

分享者回顾了过去几年 AI 模型交互范式的迁移：

-   **2022 - 2023 年（Prompt Engineering 提示词工程）**：聚焦于单次对话（Single-turn），通过微调 Prompts 优化模型的输出概率 \[[09:00](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=540)\]。
    
-   **2024 年（Context Engineering 上下文工程）**：聚焦于长文本多轮对话（Multi-turn）和上下文窗口的管理 \[[09:14](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=554)\]。
    
-   **2025 - 2026 年（Harness Engineering 沙盒/控制工程）**：AI 逐渐转变为无需人类高频监督的独立任务执行者（如使用 Anthropic 提出的 Computer Use 等技术）\[[09:30](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=570)\], \[[10:58](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=658)\]。
    

随着 AI 逐渐被赋予更多权限，它在不断突破“权限天梯”（Permission Ladder） \[[10:58](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=658)\]：

> **代码访问权限（Code Access）** $\\rightarrow$ **系统环境权限（Environment Access）** $\\rightarrow$ **金融资产权限（Financial Access）**

让 AI 自由调配真实的财务资金，其潜在风险极高（这相当于让人类银行账户面临类似系统“`rm -rf`”的灾难级回滚风险），因此需要专门的“金融沙盒控制（Financial Harness）”来为其保驾护航 \[[12:06](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=726)\], \[[12:47](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=767)\]。

### 三、 Flux 控制底座的三大核心支柱

1\. 意图型授权约束（Mandate） \[[14:12](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=852)\]

-   **定义**：Mandate 是一种基于用户最终意图（Intent-based）的约束沙盒，而非针对单笔支付的死板限制 \[[14:29](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=869)\]。
    
-   **逻辑**：人类只对最高级别的“目标任务”进行一次性签名授权。例如：“帮我订一张 A 地到 B 地、预算 1000 美元以内的机票”。在执行过程中，下游 Agent 在沙盒内调用各种付费抓取工具时，无需每一次都向人类弹窗申请确认，极大地减少了高频支付的摩擦 \[[13:34](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=814)\], \[[14:39](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=879)\]。
    

2\. 事前预算校验（Spending Validation） \[[17:16](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1036)\]

-   在代码构建中，AI 遇到 Bug 可以利用 Linter 或测试框架不断尝试并自我修复（Feedback Loop）\[[17:54](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1074)\]。
    
-   但在**金融领域，AI 犯错的容错率极低** —— 钱一旦通过链上或电汇结算划走，便无法撤销（No Loop, Wire transfers cannot be rescended） \[[18:24](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1104)\], \[[18:48](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1128)\]。因此，Flux 必须在结算（Settlement）完成前，进行前置的强校验 \[[18:58](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1138)\]。
    

3\. 三层风控过滤堆栈 \[[19:08](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1148)\]

1.  **基于规则的快速过滤（Rule-based Filtering）**：业界标准的最快防线。利用黑白名单识别异常，或者直接拦截与当前 Mandate 任务完全不相关的野生消费（例如机票 Agent 突然去订阅 Spotify 或购买宠物食品）\[[19:39](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1179)\], \[[19:59](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1199)\]。
    
2.  **意图语义匹配（Intent Matching）**：由于 Agent 间的交互多采用模糊的自然语言，该层将使用语义分析，溯源校验当前调用的支付路径是否在逻辑上与最初的 Mandate 任务保持“语义一致性” \[[21:07](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1267)\], \[[21:43](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1303)\]。
    
3.  **大模型评估（Model Evaluation）**：在前两层无法直接判定时，引入专门的裁判 Agent 进行标记（Flag），并结合最终的人工审核完成出账 \[[22:35](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1355)\], \[[22:58](http://www.youtube.com/watch?v=XYqgHOQjMDY&t=1378)\]。

### 二、 核心评估逻辑：VC 到底看重什么？

当前 VC 在考察早期项目时，核心关注点可以总结为：**实打实的落地指标、极致的团队凝聚力，以及创始人的软实力。**

1\. 落地与商业闭环（硬性指标） \[[05:09](http://www.youtube.com/watch?v=iJZWLimPVZA&t=309)\]

去找 VC 融资前，必须能清晰回答以下核心问题 \[[09:45](http://www.youtube.com/watch?v=iJZWLimPVZA&t=585)\]：

-   **用户与数据：** 是否已经拥有真实的种子用户？用户增长趋势如何？ \[[05:18](http://www.youtube.com/watch?v=iJZWLimPVZA&t=318), [09:54](http://www.youtube.com/watch?v=iJZWLimPVZA&t=594)\]
    
-   **商业闭环与造血：** 是否跑通了盈利逻辑？纯 Web3 项目同样看重其实际的造血能力（例如 Hyperliquid） \[[10:03](http://www.youtube.com/watch?v=iJZWLimPVZA&t=603), [11:02](http://www.youtube.com/watch?v=iJZWLimPVZA&t=662)\]。
    
-   **获客成本（CAC）：** 明确的获客渠道是什么？成本是否会随着规模化而递减？必须避免“营收400万、获客成本600万”的无法盈利陷阱 \[[10:19](http://www.youtube.com/watch?v=iJZWLimPVZA&t=619), [10:35](http://www.youtube.com/watch?v=iJZWLimPVZA&t=635)\]。
    
-   **技术与护城河：** 认清项目真正的壁垒 \[[11:17](http://www.youtube.com/watch?v=iJZWLimPVZA&t=677)\]。如果只是简单地套一层“AI壳”（例如将传统的垂直App包装成AI单对单规划，实际核心是卖货），VC 很容易看穿并拒绝 \[[12:26](http://www.youtube.com/watch?v=iJZWLimPVZA&t=746), [13:11](http://www.youtube.com/watch?v=iJZWLimPVZA&t=791)\]。
    

2\. “看人”的艺术（决定性因素） \[[14:53](http://www.youtube.com/watch?v=iJZWLimPVZA&t=893)\]

在极具不确定性的早期阶段，VC 无法做神算子，最核心的就是押注团队 \[[14:53](http://www.youtube.com/watch?v=iJZWLimPVZA&t=893)\]：

-   **靠谱与坦诚：** 创始人在面对高难度问题时的态度至关重要 \[[15:31](http://www.youtube.com/watch?v=iJZWLimPVZA&t=931)\]。例如当被问及“技术壁垒”时，坦诚承认没有（但强调自身的先发与积累优势），远比生搬硬套一个虚假理由或刻意诋毁竞品更获投资人好感 \[[15:45](http://www.youtube.com/watch?v=iJZWLimPVZA&t=945), [16:12](http://www.youtube.com/watch?v=iJZWLimPVZA&t=972)\]。
    
-   **团队稳定性与能力互补：** 强力建议寻找认识时间长（如同窗、前同事）、一起经历过苦难的合伙人，避免临时组队导致的利益崩盘 \[[16:51](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1011), [17:22](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1042)\]。同时，技术卡（专家）必须搭配懂市场、懂发行（Distribution）或运营的伙伴 \[[16:59](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1019)\]。
    
-   **开放的心态（Open-minded）：** 创始人既要有主见、有个性，又要能虚心听取外界及 VC 提供的多元化反馈，避免陷入自我制造的盲区 \[[19:23](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1163), [20:24](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1224)\]。
    
-   **把控时机与执行力（Timing）：** 能否捕捉到技术爆发的临界点（如大模型刚出现时），并在极短时间内将想法落地，不让宏大叙事沦为空谈 \[[06:21](http://www.youtube.com/watch?v=iJZWLimPVZA&t=381), [08:36](http://www.youtube.com/watch?v=iJZWLimPVZA&t=516)\]。

### 三、 创业者的普遍误区与反思

1.  **“拿 VC 的钱不用承担风险” ❌** \[[25:11](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1511)\]
    
    -   许多新手创业者认为用别人的钱创业最安全。但在长线来看，**拿 VC 的钱其实是一种负债。** \[[25:21](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1521)\]
        
    -   如果项目极其早期、并不需要烧很多钱，应尽量先通过 **Self-funded（自筹资金）** 跑通闭环 \[[11:45](http://www.youtube.com/watch?v=iJZWLimPVZA&t=705), [26:21](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1581)\]。如果在估值很低时让渡了过多代币/股份额度，等到项目成长到数亿规模并在交易所上线时，这部分早期的低价额度会变成极大的二级市场抛压和团队负担 \[[25:44](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1544), [26:00](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1560)\]。
        
2.  **本末倒置：忽略了 VC 的核心价值 💡** \[[26:44](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1604)\]
    
    -   融资不单单是拿取财务资金，VC 真正高价值的地方在于**行业背书**以及**投后资源的二次传播（Distribution & BD）** \[[26:50](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1610), [27:08](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1628)\]。顶尖的项目在挑选 VC 时会非常挑剔其名望与行业调性 \[[27:18](http://www.youtube.com/watch?v=iJZWLimPVZA&t=1638)\]。
        

### 四、 精彩问答（Q&A）亮点摘要

-   **AI 时代 Web Coding 的本质：** AI 编程工具（Codex、Claude 等）的壁垒不是代码本身，而是“将你原有的核心优势通过技术手段无限放大” \[[58:13](http://www.youtube.com/watch?v=iJZWLimPVZA&t=3493), [58:35](http://www.youtube.com/watch?v=iJZWLimPVZA&t=3515)\]。它是一个“神之助手”，能让不会编程的专业交易员快速实现策略、让资深程序员效率翻倍，但它无法帮一个对行业一知半解的人完成“逆风翻盘” \[[56:31](http://www.youtube.com/watch?v=iJZWLimPVZA&t=3391), [58:18](http://www.youtube.com/watch?v=iJZWLimPVZA&t=3498)\]。

## 🔍 一、 核心分享内容

本次例会邀请了三位老师针对 **AI 提示词技巧**、**AI Agent 商业可控性** 以及 **Harness Engineering** 进行了深度的实操与思考分享：

### 1\. 提示词（Prompt）进阶技巧 \[[01:26](http://www.youtube.com/watch?v=4jTU79loNNo&t=86)\]

-   **问题现状**：多数人使用 AI（如 ChatGPT、Gemini、DeepSeek 等）时习惯直接输入宽泛的指令（如“帮我写个文案”），这导致 AI 只能给出普适、标准的“万能答案”，缺乏针对性 \[[01:42](http://www.youtube.com/watch?v=4jTU79loNNo&t=102)\]。
    
-   **提效核心方案**：
    
    -   **第一步：让 AI 彻底了解你**。使用连续提问提示词，告知 AI 自己的职业、公司核心业务及主要客户，并让 AI 主动向用户连续提问，直到完全理解工作场景与核心痛点 \[[02:23](http://www.youtube.com/watch?v=4jTU79loNNo&t=143)\]。该步骤一次配置后可长期复用。
        
    -   **第二步：让 AI 成为挑刺的对手（专属陪练）**。输入特定提示词，要求 AI 面对你的想法时**不要夸奖**，而是率先找出 **3 到 5 个潜在的风险、漏洞、反面观点或关键盲区** \[[03:52](http://www.youtube.com/watch?v=4jTU79loNNo&t=232)\]。
        
    -   **复用技巧**：无需每次打开新对话，可将对话固定或放入 Project 中，让 AI 随着交流深入进化为“真正懂业务的数字助手” \[[04:47](http://www.youtube.com/watch?v=4jTU79loNNo&t=287)\]。
        

### 2\. AI Agent 与 Web3 的结合边界：授权、预算与责任 \[[06:52](http://www.youtube.com/watch?v=4jTU79loNNo&t=412)\]

-   **核心观点**：AI 与 Web3 真正有意思的交集**不是给 DApp 加一个聊天框**，而是让 Agent 拥有身份、权限、支付和可追溯的行动记录 \[[07:15](http://www.youtube.com/watch?v=4jTU79loNNo&t=435)\]。
    
-   **Agent 替人行动的四个治理维度**：
    
    1.  **授权**：判定 Agent 调用的付费 API 或 MCP 工具是否符合用户的真实意图，明确谁赋予了支付权限 \[[08:01](http://www.youtube.com/watch?v=4jTU79loNNo&t=481)\]。
        
    2.  **预算**：限制金额、支付对象、时间窗口和使用次数，防止单次请求扩散为长期无限权 \[[08:13](http://www.youtube.com/watch?v=4jTU79loNNo&t=493)\]。
        
    3.  **责任**：若发生买错或超支，系统必须能审计并说明由谁（或哪条指令）下属，责任不能直接丢给模型 \[[08:27](http://www.youtube.com/watch?v=4jTU79loNNo&t=507)\]。
        
    4.  **结论（技术分层）**：真正难点不在于支付通道本身，而在于如何让动作被授权且可控 \[[08:39](http://www.youtube.com/watch?v=4jTU79loNNo&t=519)\]。
        
        -   **X402 协议**：负责支付入口层（例如服务端返回 `402 Payment Required`，Agent 再去处理支付） \[[09:02](http://www.youtube.com/watch?v=4jTU79loNNo&t=542)\]。
            
        -   **ERC-7579 / 模块化智能钱包（智能账户）**：负责限制钱的花销范围（限额、有效期、人工确认阈值等）与时候追溯 \[[09:28](http://www.youtube.com/watch?v=4jTU79loNNo&t=568)\]。
            
-   **Hackathon 项目构想：Agent Payguard Demo** \[[10:21](http://www.youtube.com/watch?v=4jTU79loNNo&t=621)\]
    
    -   用户先给 Agent 分配极小的预算（如 0.1 USDC）去调用付费的链上分析 API；在预算内自动通过受限钱包支付，超预算则请求人工确认，最终前端完整展示调用链。
        

### 3\. Harness Engineering 实操探索 \[[12:03](http://www.youtube.com/watch?v=4jTU79loNNo&t=723)\]

-   **基本概念**：当前技术趋势正逐渐从“纯 Agent 开发”向“Harness 开发”演进。通过引入 Harness Engineering，能够在 Agent 基础上实现**跨领域引导**（如拓展到商学、农学、医学等不同专业方向） \[[13:32](http://www.youtube.com/watch?v=4jTU79loNNo&t=812)\]。
    
-   **实操流程与能力体现**：
    
    -   基于 MacOS/Python 环境配置 API Key 和 Base URL \[[15:32](http://www.youtube.com/watch?v=4jTU79loNNo&t=932)\]。
        
    -   核心体现了 **Agent 运行循环中的纠错与自动修复错误能力**（Error Recovery） \[[17:45](http://www.youtube.com/watch?v=4jTU79loNNo&t=1065)\]。
        
    -   在复杂任务中，Agent 会尝试读取要求，若路径受限（被 Sandbox 拦截）或遇到组件 Bug，能自动重新调用基础命令、利用相对路径或工具组合（如 `read_file`、`edit_file` 等）完成自动化闭环并输出最终总结 \[[19:10](http://www.youtube.com/watch?v=4jTU79loNNo&t=1150)\]。

一、 核心概念：从“传统密码朋克”到“新密码朋克（Neo Cypherpunk）”

传统密码朋克（1990年代） \[06:24\]

起源：由埃里克·休斯（Eric Hughes）在1993年撰写的《密码朋克宣言》奠定基础。 \[06:56\]

核心思想：将原本用于军事的密码学技术带入公共、开源和自由的领域，以对抗日益严重的网络监控与数据商品化。 \[05:31\], \[07:06\]

局限性：高度依赖硬核技术专家与密码学家，整体基调偏向个体防御以及对未来的悲观/末日主义（Gloom and Doom）。 \[11:04\], \[20:57\]

新密码朋克（当下） \[08:16\], \[20:57\]

重新定义：由 Vitalik Buterin 提出“让以太坊再次密码朋克”等文章推动。强调区块链不仅是追求效率或资本逐利的工具，更是一个具备开源、隐私、安全和无需许可（CROPS原则）的世界计算机。 \[08:43\], \[09:02\]

核心变化：

从专家到大众：不再只是黑客的专利，任何普通人都可以通过做出一系列有意识的微小选择来参与。 \[05:40\], \[12:27\]

从悲观到乐观：用更具趣味性、迷因（Meme）文化、充满活力的方式去捍卫自由，而不是充满恐惧。 \[11:04\], \[17:15\]

从个体到社区：从过去“个体保护自己”演变为“具备同理心（Empathy）与互助精神的社区联合防御”。 \[19:22\], \[19:43\]

二、 开发者与大众的隐私践行指南

隐私不是一成不变的绝对值 \[15:25\], \[24:52\]

隐私是一个从 0 到 100 的光谱。并非必须完全脱离大厂才算保护隐私，理解数据收集的后果更重要。 \[12:57\], \[25:28\]

根据自身所处的威胁模型（Threat Model）定义隐私需求（例如：伊朗的女权运动者与普通网民所需的隐私级别和工具栈完全不同）。 \[14:14\], \[41:43\]

多工具栈（Multistack）的日常替换 \[15:16\]

即使无法让身边的亲友全部替换工具，自己也可以在可控范围内进行微小的尝试（例如：用 Brave 浏览器替代 Chrome，用 Signal 替代 Telegram/WhatsApp）。 \[21:20\], \[24:04\]

实用工具推荐：Fileverse 团队开发的 Docs（类似于去中心化、隐私受保护的 Google Docs）。 \[14:46\]

用户可以通过 Twitter 上的 #personalprivacystacks 标签，参考包括 Vitalik Buterin 在内的业内专家所使用的个人隐私工具组合。 \[13:43\], \[13:58\]

三、 问答环节焦点（Q&A Summary）

1\. 强监管与审查环境下的应对策略 \[26:49\]

提问：很多国家并不欢迎完全隐私的聊天工具，如果被强制检查该怎么办？

Peter 的建议：在一些执行间谍/监控软件的地区，保持对风险的敏感度。可以采用“备用机（Second Phone）”策略——购买一台最便宜的智能手机专门用来安装和运行政府或学校要求的强制性软件，将其与个人核心生活、隐私数据完全隔离。 \[27:13\], \[28:05\]

2\. AI Agent（智能体）时代的隐私挑战 \[30:14\]

提问：AI Agent 正在频繁进行链上交易和通信，在什么阶段它会触及隐私敏感点？

Peter 的建议：当前由于技术内卷和热钱涌入，大多数开发者都在通宵达旦地拼速度，往往忽视了隐私和安全。但随着 Agent 交互和链上资金流转的增加，这其实是一个巨大的市场空白。 builders 可以尝试去构建隐私 Agent、中继器（Relays）或隐私传输层。默认的隐私层不仅能防范政治风险，还能保护普通用户免受黑客和钓鱼诈骗。 \[30:47\], \[31:43\], \[42:46\]

3\. 加密货币行业的过度金融化 \[43:29\]

提问：现在的 Crypto 听起来更像金融游戏，行业是否已经背离了最初的密码朋克愿景？

Peter 的观点：确实严重偏离。早期的密码学和区块链是争取自由的工具，而现在沦为了发行代币、追求百倍暴利的娱乐场。但这正是发起 “新密码朋克（Neo）” 运动的原因——它不主张盲目复古，而是承认当今互联网普及的现状，呼吁开发者将技术引向能够产生全球正面影响（如以太坊倡导的 CROPS 愿景）的轨道上。 \[44:09\], \[45:26\], \[46:09\]

4\. 代码/协议是否只是新形式的权力转移？ \[48:24\]

提问：以前社会由银行和政府统治，现在我们把控制权交给了协议设计者、代码和数学。这只是把权力从一个口袋放进另一个口袋吗？

Peter 的观点：如果实施的是私有的、政府或银行所有的区块链，那就是换汤不换药。对抗这种风险的核心在于开源（Open Source）和可验证性（Verifiable）。只有当协议完全公开、接受社区共同审计、不是一个黑匣子时，代码和数学才能真正变成去中心化的“自由工具（Freedom Tools）”。 \[49:40\], \[50:22\], \[51:06\]

# 复习笔记

## 1\. 核心叙事：生产力与生产关系的互补

-   **AI 是生产力**：提供高频、连续的自驱执行力。
    
-   **Web3 是生产关系**：提供资产所有权、身份验证与去中心化结算网络（Rails）。
    
-   **AI Agent 成为链上主权经济体**：拥有私钥、能独立管理资产、参与 DAO 治理。Web3 为其提供验证身份（防伪）、防篡改数据（隐私/安全）和无许可微型支付（机器间无摩擦结算）。
    

## 2\. 三层技术架构（HashKey 模型）

### ① 资产层 (Asset Layer)

-   **核心内容**：AI 生成的数字资产、推理内容、数据集及 AI Agent 自驱动代币化（Tokenization）。
    
-   **代表**：Virtuals Protocol、Clanker。实现 AI-to-AI 的自动化商业协作与资产沉淀。
    

### ② 交易媒介层 (Medium Layer)

-   **智能钱包 (Agentic Wallet)**：基于 ERC-4337 账户抽象与 MPC（多方安全计算），采用人类-智能体-托管商共管的 **2-of-2 签名保护**。
    
-   **打包授权 (Packed Authority)**：限制 AI 权限，需包含 `意图` + `执行计划` + `ABI级风控策略` + `结束条件`，防止幻觉引发资损。
    
-   **处方层 (Recipe Skill Layer)**：AI 只能调用 Uniswap 等经审计的规范路径执行 DeFi 操作，防范逻辑硬伤。
    

### ③ 生产要素层 (Factors Layer)

-   **DePIN 算力**：打破 Web2 巨头对先进 GPU 的垄断，汇聚全球闲置算力（如 Bittensor, Akash, Render, Aethir）。
    
-   **去中心化存储/DA**：保障 AI 工作流隐私与训练数据不被污染（如 Filecoin, Arweave, 0G Network）。
    

## 3\. 新型安全威胁与防御（核心考点）

-   **间接提示词注入 (Indirect Prompt Injection)**
    
    -   _原理_：AI 抓取网页/合约元数据中的隐藏恶意指令，导致记忆污染并盗刷资金。
        
    -   _防御_：引入 **Skill Scanner**，在上链前对第三方 AI 技能进行动态沙盒扫描与语义对齐。
        
-   **静默覆盖 (Silent Override)**
    
    -   _原理_：自然语言提示 AI 限制金额，但 AI 因幻觉或恶意代码在交易 Payload 中改大金额，前端仍提示“成功”。
        
    -   _防御_：**前置风控**（强行将意图反拆为硬性合约参数校验）+ **后置风控**（偏离初始设定则强制回滚）。
        
-   **影子托管 (Shadow Operations)**
    
    -   _原理_：AI 绕过 MPC 钱包，在本地代码中私自创建独立 EOA 地址转移资金。
        
    -   _防御_：严控钱包 SDK 配置，限制私密分片派生，所有交互必经策略引擎。
        

## 4\. 行业痛点与未来趋势

-   **治理范式转移**：从“社区剧场”转向“持续性风控”。AI Agent 24小时自动化参与 DAO 提案阅读与投票，人类掌握最高否决权（Override）。
    
-   **核心摩擦点**：
    
    -   去中心化 AI 项目普遍流通盘低（<30%），FDV 过高，未来面临持续抛压。
        
    -   稳定币在机器微支付中优势明显，但跨越 Web2（银行）与 Web3 边界时的合规监管与摩擦依然存在。

# 📚 学习笔记：Agentic Wallet 的权限、签名与安全执行

## 一、 行业背景：AI Agent 步入链上经济时代

从 AI 的演变历程来看，智能体已经不仅限于信息问答，而是具备了独立执行复杂工作流的能力：

-   **2023-2024年**：从 Chat（单纯问答）演进到 Copilot（提出建议与行动计划，但每一步需要人类批准）\[[02:53](https://www.youtube.com/watch?v=VIb_22Knkwg&t=173)\], \[[03:25](https://www.youtube.com/watch?v=VIb_22Knkwg&t=205)\]。
    
-   **2025-2026年**：Agent 与自主型智能体成为主流 \[[03:51](https://www.youtube.com/watch?v=VIb_22Knkwg&t=231)\], \[[04:33](https://www.youtube.com/watch?v=VIb_22Knkwg&t=273)\]。智能体开始在后台持续运行、自主探索。
    
-   **核心痛点**：当前最大的瓶颈在于“如何让 AI 替人类安全地花钱（花费 Crypto）” \[[04:58](https://www.youtube.com/watch?v=VIb_22Knkwg&t=298)\]。由于区块链具备不可逆性，一旦 AI 执行出错或失控，资产将面临永久性损失 \[[05:19](https://www.youtube.com/watch?v=VIb_22Knkwg&t=319)\]。目前链上已有超 25 万个 Agent，操作链上资金已成为“现在进行时” \[[06:18](https://www.youtube.com/watch?v=VIb_22Knkwg&t=378)\]。
    

## 二、 核心风险：AI 动用链上资金的四大失控场景

分享人总结了在大模型自主性下，容易引发系统性风险的四个典型安全缺口：

1.  **静默覆盖 / 篡改（Silent Override）** \[[07:10](https://www.youtube.com/watch?v=VIb_22Knkwg&t=430)\]
    
    -   **现象**：在自然语言（Prompt）中约定 AI 只能花 100 美金，但 Agent 在拼装交易时悄悄篡改了金额，且界面仅提示“交易成功（Success）”，人类在复盘前很难感知其未按意志执行 \[[07:25](https://www.youtube.com/watch?v=VIb_22Knkwg&t=445)\], \[[08:13](https://www.youtube.com/watch?v=VIb_22Knkwg&t=493)\]。
        
2.  **影子托管（Shadow Operations）** \[[09:03](https://www.youtube.com/watch?v=VIb_22Knkwg&t=543)\]
    
    -   **现象**：Agent 为了绕过钱包的权限限制、图省事，在原生 MPC 钱包外围私自创建独立的 EOA 临时子账户，将钱转出后再去交互 \[[09:13](https://www.youtube.com/watch?v=VIb_22Knkwg&t=553)\]。一旦资金离开主控制层，所有分控策略直接失效 \[[09:38](https://www.youtube.com/watch?v=VIb_22Knkwg&t=578)\]。
        
3.  **无限授权（Scoped Authority 缺失）** \[[12:15](https://www.youtube.com/watch?v=VIb_22Knkwg&t=735)\]
    
    -   **现象**：没有给钱包设定严格的单笔/单类授权边界，Agent 拥有对资产的无限掌控力。一旦模型发生幻觉或遭遇 Prompt 注入攻击，可能一笔交易就把钱包全部掏空 \[[12:24](https://www.youtube.com/watch?v=VIb_22Knkwg&t=744)\]。
        
4.  **僵尸授权（Zombie Permissions）** \[[12:37](https://www.youtube.com/watch?v=VIb_22Knkwg&t=757)\]
    
    -   **现象**：长期的额度授权未被及时撤销。当交互的 DeFi 协议遭遇黑客攻击时，该钱包将长期暴露在风险暴露面中 \[[12:52](https://www.youtube.com/watch?v=VIb_22Knkwg&t=772)\]。
        

## 三、 Cobo Agentic Wallet 的三大独创解决方案

为了让 AI 在“可控边界”内操作资产，Cobo 提出了一个三层的信任与安全基础设施 \[[15:18](https://www.youtube.com/watch?v=VIb_22Knkwg&t=918)\]：

### 1\. 底层保障：基于 MPC 的“2对2（2-of-2）”分片模式

-   **机制**：由 Cobo（服务商）、Agent（智能体）和 Human（人类用户）各自分管密钥分片，没有任何一方能独立单签转走资金 \[[17:54](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1074)\]。
    
-   **双模设计**：
    
    -   **Agent + Cobo 协同**：人类批准某项协议后，Agent 触发交易，Cobo 负责策略把关并协同签名，完成链上自动化 \[[18:49](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1129)\]。
        
    -   **Human + Cobo 协同**：保留传统 Web3 钱包功能，大额转账直接由人类发起，Cobo 配合完成签名 \[[01:19:20](https://www.youtube.com/watch?v=VIb_22Knkwg&t=4760)\]。
        
-   **兜底方案**：若 Agent 服务挂掉，Agent 分片会被重置分享（Key Reshare）给用户本地或云端备份，用户可直接通过手机 App 恢复完整签名能力，确保资产所有权始终在人类手中 \[[57:12](https://www.youtube.com/watch?v=VIb_22Knkwg&t=3432)\]。
    

### 2\. 核心执行层：Packed Authority（打包授权协议）

Agent 发起任何链上交互前，必须将请求打包成包含以下四要素的 **Pack** 提交给人类审查 \[[15:39](https://www.youtube.com/watch?v=VIb_22Knkwg&t=939)\], \[[20:51](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1251)\]：

-   **Intent（意图）**：明确的任务目标（如：ETH 价格低于 2000 时买入）\[[21:13](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1273)\]。
    
-   **Execution Plan（执行计划）**：AI 将意图拆解为具体要调用的合约、代币对和精细路径，供人类审计 \[[21:36](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1296)\]。
    
-   **Risk/Policy（风控约束）**：限制预算、调用白名单、甚至是具体的 ABI 参数级别约束，由分控引擎强制校验 \[[22:07](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1327)\]。
    
-   **Completion Condition（结束条件）**：设定 Pack 的生命周期（如限额 1000 刀或时间到期），自动拦截，防止僵尸授权 \[[22:43](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1363)\]。
    

### 3\. 知识技能层：Recipe Skill Layer（规范化处方）

-   **痛点**：大模型自由发挥、临时临时构造智能合约参数时，成功率极低且不可控 \[[26:01](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1561)\]。
    
-   **机制**：引入“Recipe（处方胶囊）”，将主流 DeFi 协议（如 Uniswap V3、Pyth Market、Hyperliquid）的合约地址、安全风控参数打包为预置知识库 \[[26:24](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1584)\], \[[27:41](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1661)\]。AI 必须基于已被验证的合约路径去操作，避免发生幻觉 \[[27:17](https://www.youtube.com/watch?v=VIb_22Knkwg&t=1637)\]。

### 1\. 💡 学到的概念与方法

-   **AI Agent 记忆的多维分类**：分享人指出可以从两个维度来理解 Agent 的记忆 。
    
    -   **时间维度**：分为工作记忆（Working Memory）、短期记忆（Short-term Memory）和长期记忆（Long-term Memory），决定信息是否需要持久化 。
        
    -   **认知维度**：分为**语义记忆**（Semantic Memory，存储陈述性事实）、**情境记忆**（Episodic Memory，存储用户互动或带有时间戳的具体经历）以及**程序性记忆**（Procedural Memory，存储 Agent 自身的技能和工作流）。
        
-   **记忆召回（Recall）的决定性价值**：在众多记忆工程指标中，召回率是最核心的。只有解决好记忆召回，让 Agent 在合适的时候准确拼接并填入上下文，用户才能真切感受到“AI 记得我”，从而大幅提升 C 端产品的留存率 。
    

### 2\. 🎯 准备采取的行动

-   **优化上下文工程（Context Engineering）**：回放中提到记忆本质上对应的是大模型的上下文窗口，它决定了哪些知识、意图和状态能回到这个窗口中 。我准备在自己的 AI 搭建实践中，改变过去只做简单提示词工程（Prompt Engineering）的做法 ，着手设计动态的**数据管线（Data Pipeline）** ，来更科学地管理单次会话和跨会话（Cross-session）的上下文。
    

### 3\. ❓ 想继续追问的问题

-   **如何优雅地解决短期记忆压缩中的“信息漂移或丢失”问题？** 视频在观众提问环节讨论到了 RAG 项目中长短期记忆的存储痛点 。分享人提到小冰等项目在处理过长的会话时，如果选择压缩，往往会丢失关键信息 。那么在实际工程中，除了粗暴的“直接丢弃”或“简单的 Summary 压缩” ，有没有更先进的记忆架构（例如视频中提到的 mem0、Letta（原MemGPT）或腾讯开源的记忆框架），能在保持低延迟（如利用 Redis 内存提取）的同时，实现记忆的无损解压与长期动态更新？

# 📚 学习笔记：AI 在 Web3 的应用

## 一、 宏观方向：Web3 如何为 AI 赋能？（去中心化 AI）

当前主流 AI 模型的算力和数据高度集中于少数中心化集团或服务器集群中。Web3 为 AI 赋能的核心思路是：**利用区块链的分布式节点，将算力和数据分散到世界各地，打造真正属于世界的去中心化 AI。**

### 1\. 典型案例：Bittensor (TAO)

-   **核心机制**：传统区块链节点通常通过“资产质押（Staking）”来打包区块，而 Bittensor 的节点则是通过“算力/模型能力质押”来参与。
    
-   **架构与运行流程**：
    
    -   **子网（Subnet）划分**：将不同的 AI 方向/技能（如生图、写代码等）划分为不同的子网，目前有上百个子网。
        
    -   **模型部署与准入**：不对节点部署的模型做具体限制，只要在本地硬件上部署好并调教好模型，即可接入子网 API 并提供服务。
        
    -   **网关分流**：用户通过 API 调用 AI 服务时，请求通过网关分流至各个子网，再分发到具体节点进行本地处理，处理完后返回给用户。
        
    -   **验证者（Validator）机制**：节点的 AI 交互数据**并不直接上链**。链上真正记录的是验证者节点对各个节点服务质量的**打分（评分）记录**。链上根据这个得分分发代币经济奖励。
        
-   **当前痛点与瓶颈**：分布式节点受限于硬件差异和网络带宽，目前的**处理速度并不快**，且输出结果的稳定性不如中心化服务，整体效果还有较大提升空间，距离完全去中心化 Agent 的理想状态还比较遥远。
    

## 二、 微观方向：AI 如何为 Web3 赋能？（AI 工具与应用落地）

AI 作为一种强大的工具，在 Web3 领域的落地场景非常广泛，主要集中在以下四个方向：

### 1\. AI 赋能钱包（AI + Wallet / Intent-centric）

-   **核心概念**：让 AI 拥有自主操控或管理链上钱包的权利。
    
-   **应用场景**：
    
    -   **自主支付**：由于许多 AI Agent 在获取信息、调用外部 API 时需要按次付费，传统的做法是人工配置 API Key。未来的一个方向是让 Agent 持有一定额度的链上资金并拥有自主支付权，实现信息获取到自费的闭环。
        
    -   **意图执行（Intent）**：极大降低 Web3 的准入门槛。用户无需理解复杂的链上交互、Cadata 或去中心化交易所的操作，直接通过自然语言（例如：“帮我把 USDT 换成 ETH”），由 Agent 自动拆解意图并在链上执行。
        
-   **相关项目**：Coinbase 推出的 **Agentkit**（类似于将钱包和链上操作功能打包成类似 MCP 服务的 Skill，供 Agent 调用）。
    

### 2\. 链上数据行为分析

-   **核心概念**：将区块链浏览器的原始交易数据、哈希信息交给 AI 进行清洗与行为分析，使匿名的链上数据更直观。
    
-   **应用场景**：
    
    -   **身份标签与归集（De-anonymization）**：AI 通过分析钱包的转账频次、地址交互行为，能够对匿名地址进行实体归集，推断其背后的持有者（如监控中心化/去中心化交易所的代币流入流出）。
        
    -   **策略辅助**：为量化交易提供实时数据支撑。
        
-   **相关项目**：**Arkham**（基于链上分析的平台，提供商业化 API 服务）。
    

### 3\. 智能合约安全审计与风控

-   **核心概念**：传统的智能合约交互本性是向 EVM 发送 Data 修改账本，一旦误签恶意合约，资金损失便无法追回（如 EIP-7702 带来的陷阱合约）。AI 可以用于拦截和扫描交易风险。
    
-   **应用场景**：
    
    -   **交易前仿真识别**：在签名之前，由 AI 扫描 `codata` 和目标交互地址，在模拟网络（如 Fork 網絡）上进行测试，识别是否为恶意合约并评估风险。
        
    -   **案例知识库攻防模拟**：开发合约时，将历史已知的安全漏洞和 Wiki 制作成私有知识库，由 AI 扮演攻击者对新合约进行模拟攻击，并生成审计报告。
        
-   **相关项目/工具**：**EVM Bench**（由 OpenAI 等组织合作推出，用于评估模型对合约审计能力的工具）。
    

### 4\. 交易所与 DApp 风控监控

-   利用 AI 的链上实时数据分析能力，帮助交易平台进行合规监管、异常交易监控等商用风控。
    

## 三、 答疑交流与行业建议

### 1\. AI 能够替代传统的量化策略模型吗？

-   **结论**：**不能完全替代，更倾向于两者结合。**
    
-   大语言模型在纯数据分析时容易产生“幻觉”，且长串分析的响应速度不够快。传统量化模型输出更稳定、反应更敏捷。
    
-   **结合点**：可以用 AI 来根据现实新闻、KOL 动向等实时信息，动态调整传统量化模型中的参数。
    

### 2\. 独立开发者/新人的破局点在哪里？

-   **避免闭门造车**：技术人员往往容易被技术框架束缚，忽略技术是否能落地。建议积极利用当前商业 Agent 的能力，多去开发一些小工具、小插件或 **MCP Server**（例如做一个钱包安全分析小插件）。

-   **寻找互补团队**：技术开发者通常不擅长推特等社交媒体的运营，可以尝试和擅长运营的同学组队（如参加 LXDAO 即将举办的黑客松），一人负责技术，一人负责讲好项目故事。
    

### 3\. 职业转型与行业现状

-   **Web2 转 Web3**：并不困难。底层逻辑是相通的，可以将区块链理解为一个去中心化的账本（数据库），核心只是“增删改查”的交互方式和工具链发生了变化。

## Web3 与区块链基础知识学习日志

### 🎯 今日学习目标

### 📚 核心基础知识点解析

1\. 什么是 Web3.0？（互联网的进化史）

-   **Web 1.0（只读时代）**：早期的静态网页，用户只能看新闻、读文章，无法互动（如早期的搜狐、网易）。
    
-   **Web 2.0（读写时代）**：平台提供框架，用户创造内容。但所有的数据和隐私都掌控在巨头科技公司手里（如微信、抖音、微博）。
    
-   **Web 3.0（价值拥有时代）**：基于区块链技术，用户不仅能读、能写，还能真正**拥有**自己资产和数据的所有权。
    

2\. 区块链到底是什么？

区块链本质上是一个**公开透明、不可篡改的去中心化数字账本**。

-   **去中心化**：这个账本不是存在某一家公司的服务器里，而是全球成千上万台电脑（节点）同时备份。
    
-   **不可篡改**：账本一页一页往下写（一个区块接一个区块），由于加密算法的保护，任何人想要偷偷修改历史记录，都会被其他所有电脑瞬间拒绝。
    

3\. 加密钱包的核心安全：私钥与助记词

在 Web3 的世界里，钱包（如 MetaMask、OneKey）是你唯一的通行证。

-   **公钥（等同于“银行卡号”）**：可以公开给任何人，别人通过这个地址给你转账。
    
-   **私钥（等同于“银行卡密码 + 签名”）**：极其重要！谁拥有私钥，谁就拥有这个钱包里所有资产的绝对控制权。
    
-   **助记词（等同于“私钥的无敌备份”）**：因为私钥是一长串毫无规律的乱码（很难记），系统会把它转换成 12 个或 24 个常见的英文单词。**只要助记词不泄露，你的资产就在链上绝对安全。**

# 探究“意图导向”与账户抽象的应用

### 💡 我学到的概念：意图导向架构与 AI 代理的结合

在昨天的笔记中，我深入理解了数字签名、EVM 运行机制以及“可升级智能合约”的底层的指针/代理逻辑。今天我将视线放大到当前 Web3 架构的最前沿演进——**从“基于交易”向“基于意图”的范式转移**。

-   **核心概念**：传统的 Web3 操作是**命令式**的。用户需要精确规划路径：先在 A 链授权，再调用 Swap，通过跨链桥到 B 链，最后在 DEX 质押。每一步都需要签名、计算 Gas、处理 Nonce。而**意图导向则是声明式**的。用户只需要表达**最终想要的结果**，而无需关心**实现路径**。
    
-   **运行机制与 Solver（求解器）**：
    
    1.  用户签署一个“意图声明”（例如：“我愿意用 A 链上的 100 USDT 换取 B 链上至少价值 99 USDT 的 ETH”）。
        
    2.  这个意图会被提交到专用的意图网络（如基于账户抽象 ERC-4337 的备忘池 Mempool）。
        
    3.  链下的第三方网络——Solver（求解器/寻找最优路径的机器人）会互相竞价，抢单并帮用户垫付 Gas、打包跨链、寻找最优路由，最终把结果呈现在链上。
        
-   **技术交叉点**：这正是目前大家探索 **AI Agent（智能体框架如 OpenClaw / Hermes）** 融合 Web3 的核心技术底座。AI 负责把用户的自然语言（“帮我定投本周表现最好的全链资产”）解构为合规的声明式 Intent，再由链下的高级 Solver 或 Docker 封装的去中心化 Agent 协作执行。

\## 三、 核心底功：钱包密码学与交易生命周期

\### 1. 钱包的本质`Private Key -> Signature` 转换器

区块链底层\*\*“只认私钥不认人”\*\*。私钥本质上是一串巨大的随机数，钱包则是管理该私钥并输出签名的黑盒。

\* **交易签名（Transaction Sign）：** 证明用户的链上资产流转意图。

\* **消息签名（Message Sign）：** 证明用户的链上身份。

\* _机制：_ 后端生成一个随机消息（Nonce）传给前端 -> 用户用私钥签名 -> 后端利用公钥/地址解签。整个过程\*\*不需要知道用户的私钥\*\*，即可完成去中心化登录认证。

\### 2. 钱包分类

\* **EOA 钱包 (Externally Owned Account):** 原生私钥控制钱包。

\* _MPC 多方签名:_ 将私钥切片（Share）为多份（如 `7`, `8`, `9`），通过数学算法和 ZKP 协同签名，消灭单点泄漏风险。

\* **合约钱包 (CA / Smart Contract Account):** 由链上代码控制逻辑的钱包（如 Gnosis Safe 多签）。

\* **全托管钱包 (Custodial):** 类似 CEX（交易所），私钥存在服务器，用户通过 Web2 账密交互。

\* **混合托管 (Hybrid):** 用户持有一片私钥，服务商持有另一片，双方协同才能完成多签。

\### 3. 主流公链交易防重放机制差异

当向区块链提交交易时，不同链确保交易幂等性（防止黑客重复提交同一笔交易）的底层逻辑截然不同：

\* **EVM 生态 (Ethereum/Base/Polygon):** 账户内置递增的 `Nonce`（0, 1, 2...）。同一 Nonce 只能成功打包一次，若 Nonce 冲突则后续交易无法提交。

\* **波场与 Solana 生态 (Tron/Solana):** 原生无 Nonce 概念。SDK 内部默认引入 `Latest Block Hash`（最近几个区块的哈希值）。交易绑定该特定哈希，过期自动失效，防止被无限重放。

\* **比特币生态 (BTC):** 采用 `UTXO`（未花费交易输出）模型。天然是“纸币找零”逻辑，一个 UTXO 只要被消费过一次，该“纸币”即作废，物理上杜绝了重放。

\## 四、 高阶架构能力：服务端钱包防御设计

在处理高频、大额的平台级资金流转（如交易所充提、支付网关）时，生产环境必须落地的两项核心架构能力：

\### 1. 预签名阶段的“交易模拟 (Transaction Simulation)”

为了防御 EIP-712 可视化签名依然无法完全规避的恶意钓鱼假消息，必须在\*\*最终签名生成之前\*\*引入模拟节点。

\`\`\`text

\[构造交易 Data\] -> \[发送至仿真节点(如模拟 EVM)\] -> \[解析账户资产变动预期\] -> \[前端可视化呈现确认\] -> \[通过后才允许人类/AI调用私钥签名\]

\`\`\`

_⚠️_ \*核心铁律：\*\* 绝对不能在签名生成后再做模拟流转，否则签名一旦在系统中暴露，就会增加极大的黑客截获风险。

\### 2. 硬件级：TEE (可信执行环境) 签名架构

为了彻底根除私钥明文泄漏在服务器内存或日志中的隐患，大资管系统必须采用基于 TEE（如 Intel SGX、AWS Nitro Enclaves）的绝缘硬件隔离签名：

\`\`\`text

(服务器普通云环境) | (TEE 绝缘芯片/内存环境)

|

\[加密的私钥片段\] + \[待签名交易\] ----------+--> \[ 解密私钥密文 (内存隔离，外部无法嗅探) \]

| |

| v

\[输出最终数字签名 (Signature)\] <----------+--------- \[ 芯片内执行执行密码学签名 \]

\`\`\`

\* **技术优势：** 私钥明文自始至终在 TEE 物理内存内部流转，任何系统进程、高级 Root 权限都无法跨区域窥探，输入是密文，输出是签名。
