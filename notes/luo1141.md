# 2026-05-21

五大方向构成 Agent 经济栈

1\. 去中心化 AI 基础设施（资源层）

核心问题：AI 生产要素（算力、存储、数据、模型推理、服务）怎么被组织。

关键理解：主流路径为链下计算，链上激励/结算/验证/服务发现；中心化云是平台化资源市场，而Bittensor、Render、Akash是更开放的资源市场实验；关键难点在于质量验证、反作弊、稳定性、价格效率及企业级SLA。

项目案例：

\- Bittensor（激励网络）：Subnet 定义任务 → Miner 提供服务 → Validator 评分 → 链上共识 → 优质者获奖励

\- Render Network（GPU）：闲置 GPU 进入开放市场

\- Akash（云计算）：去中心化云计算市场

\- Filecoin / Arweave（存储）：AI 数据资产长期存储

\- [Fetch.ai](http://Fetch.ai) / ASI（服务网络）：AI agent 之间交易服务

2\. AI Agent + 钱包（行动层）

AI 拥有链上身份、资产账户、支付能力和执行权限。案例：Coinbase AgentKit

3\. AI 链上分析工具（认知层）

把链上数据（地址、资金流、项目风险）翻译成人能理解的判断。案例：Arkham

4\. 智能钱包和安全助手（用户保护层）

在签名/授权/转账前，帮助识别真实后果。案例：Blockaid — 交易模拟 + 欺诈识别

5\. 交易所和 DeFi 智能风控（机构风控层）

用模型、规则、链上监控发现异常交易、攻击和系统性风险。案例：Chainalysis

技术栈分层

底层资源（算力/存储、数据/模型） → 经济账户（钱包/身份、支付/权限） → 认知工具（链上分析、资金流解释） → 安全防线（欺诈识别、签名前保护） → 系统风控（KYT/预警、攻击监控）

核心观点

\- 资源：AI负责模型推理、数据处理；Web3负责算力/存储市场、激励分发。

\- 账户：AI负责理解意图、决策；Web3负责身份认证、支付、权限。

\- 认知：AI负责分析链上数据；Web3负责可验证数据来源。

\- 安全：AI负责风险识别；Web3负责透明审计。

\- 风控：AI负责异常检测；Web3负责链上监控。

1.  Web3 的核心本质并不是单纯的软件技术更新，而是互联网规则和控制权的改变。通过今天的课程，我发现它其实是三门学科的结合：利用密码学来确保数据的真实性与所有权，利用 Gas 费和惩罚机制等经济手段来规范用户的行为，最终在社会学层面实现大家共同制定规则的去中心化管理。它相当于把原本由大公司掌控的信任，变成了由底层代码和数学公式来保证。
    
2.  准备亲自动手验证一下公链的数据开放性。结合课堂上讲到的主流区块链“公开账本和伪匿名”的特点，以及不同应用之间可以互相调用的特性，我打算利用课后时间，自己找几个公开的区块链数据接口，写一个简单的程序来读取和分析一段链上的交易记录。通过亲眼看到资金是如何在各个不同的平台之间流转的，来真正搞懂它的公开可验证性，同时也感受一下数据完全公开所带来的隐私问题。

# 2026-05-18
<!-- DAILY_CHECKIN_2026-05-18_START -->





今天这场分享的信息密度很大，涵盖的议题比较多，从开发心态到具体技术架构，再到链上交互的细节，很多观点需要记下来反复琢磨。

AI与开发者的关系，这个话题上的一些观点，纠正了我之前的一些想法。之前有人会觉得有了AI写代码，人可以少学原理，但会上tc老师提出的观点是，AI时代对人的要求反而更高了。AI改变了开发的方式，确实写代码更快，学习新东西也更容易，它极大减少了精力消耗，相当于代替了人的手和脚去执行任务。但AI没有改变的是系统的复杂度。人依然需要负责设计方案，AI负责编码执行，而AI编码质量的关键在于它能够进行自我验证。如果开发者自己缺乏基础知识和架构能力，就根本不知道AI生成的代码对不对，方向有没有偏。一个很现实的问题是责任归属，如果AI生成的代码最终出现安全风险，担责的还是开发者自己。所以开发者必须确保AI在可控范围内，对所有输出进行审核和验收。

关于Web3与Web2的关系，Web3并不是与Web2完全独立的体系，大多数Web3项目中，Web2的能力占比相当高。像币安、Bybit这类交易所，虽然核心业务涉及加密资产，但实际上它们的Web2含量非常高。没有前端页面和后端状态管理，项目就无法提供良好的用户体验。区块链只是其中负责价值流转和共识的那一层，不能割裂地去看。要在Web3方向走得长远，Web2的功底必须稳固。

安全设计的理念是反复被强调的重点。有一个原则贯穿始终，安全应该源于设计，贯穿项目整个生命周期，而不是事后补救。审计只是锦上添花，是某个时间点的第三方检查，不能等同于安全本身。Web3项目因为直接涉及资金交互，安全性更加敏感。会上提到一个很根本的衡量标准，如果你无法解释系统的行为，就无法保证资产的安全。这其实也对开发者提出了更高的要求，对每一个状态变化和资金流向都必须有清晰的理解。

Web3支付系统设计的例子，用一个具体案例串联了很多概念。先梳理了Web2支付流程里各个参与方之间的交互，包括用户、电商平台、支付服务、银行机构这条链路。接着讲解怎么把这个系统升级为支持稳定币的Web3支付，核心区别在于把银行机构替换为区块链，同时增加链上监听服务来处理链上交易的确认。这个案例让抽象的概念落到实际架构层面，理解了请求流和资金流之后，再去想象链上部分的对接，会清晰很多。

钱包的分类在这次分享里讲得比较系统。钱包是Web3中身份和资产管理的核心工具，形式上分了很多种。按类别记下来，EOA钱包是私钥直接控制的外部账户，合约钱包把逻辑写在智能合约里，AA钱包做账户抽象以改善体验，多签钱包适合多人共管，硬件钱包把私钥隔离在物理设备里，隐私钱包隐藏交易细节，分层钱包从一个种子派生大量子密钥。托管方式也有自托管、全托管和混合托管之分，差别在于私钥归谁管。用什么类型的钱包，取决于安全需求和使用场景。

私钥是所有权凭证，不能暴露，签名用私钥完成授权而不泄露私钥本身，任何签名前都必须仔细核对内容。交易从构造、签名到广播上链的整个生命周期，每一步都依赖于私钥安全、合约透明和网络共识。
