# 核心概念

## LLM

语言学习模型（LLM）的基本工作原理： LLM 基于上下文以概率方式生成文本：给定一个输入，模型预测最可能的下一个词元序列。它们擅长语言理解、代码生成和推理，但在准确的事实回忆、确定性计算和跨会话状态维护方面表现不佳。

四个控制层： 上下文窗口是模型的“工作记忆”，控制模型在任何给定时间可以看到的信息量。系统指令设定身份、语气和行为边界。提示传达当前任务的意图。工具调用将模型从说话者转变为执行者。

实际调用 LLM API： MaaS 允许您通过 API 密钥按令牌调用顶级模型，无需 GPU。核心参数： model 、 messages 、 temperature 、 max\_tokens 。

提示、工作流和代理之间的界限： 提示是指让模型给出答案，由人做出决策。工作流是一个预定义的任务流程，其中模型是其中一个节点，路径是固定的。代理是指模型能够自主规划，动态调用工具，并在各个回合之间管理状态。这三者在故障模式、风险敞口和可调试性方面存在根本差异。

## AI 编码工具的价值与局限性：

Claude Code、Codex CLI 和 Cursor 可以快速生成样板代码、解释不熟悉的库并加速原型开发。然而，代码审查、测试设计和架构决策却无法委托给它们。

为什么人工智能的输出必须始终经过验证： 事实错误是最常见的陷阱：模型会自信地捏造信息，因此关键事实必须通过外部验证。模型在引用方面同样不可靠，它们经常会编造论文标题、URL 和数据来源，因此永远不要相信模型提供的任何链接。在更长的上下文中，推理漂移会成为一种风险，逻辑链会悄然断裂，结论偏离最初的前提，因此逐段验证至关重要。当涉及智能体时，还会出现两种额外的故障模式：执行越权，即智能体超出其授权范围执行操作（需要设置防护措施和人工干预检查点）；以及工具误用，即模型调用了错误的工具或传递了错误的参数（需要跟踪以监控执行过程）。

## Agent 的核心技术组件：

-   状态管理：多个节点共享对同一状态对象的读写访问权限。
    
-   长期记忆：跨会话存储和检索信息。
    
-   MCP：LLM 与外部工具之间的统一连接协议。
    
-   skills：支持自动发现和动态生成的可重用高级指令集。
    
-   Tool calling：模型发出结构化请求；框架执行该请求并返回结果。
    
-   跟踪：可视化代理的执行链。
    
-   防护机制：输入/输出验证规则；如果违反规则，则执行停止。
    
-   交接：子任务完成后转移控制权。
    
-   错误恢复：失败时进行重试、回滚或升级到人工干预。
    

## 何时真正需要智能体？

当目标不明确、需要多个工具协作、中间结果决定下一步操作或状态需要在会话间保持时，可以使用智能体。如果是一次性问答（使用提示）、流程固定（使用脚本）、合规性要求严格（使用人工审核节点）或数据确定性至关重要（使用数据库查询），则应坚持使用更简单的解决方案。复杂性和风险越高，就越应该谨慎避免过度使用智能体。
