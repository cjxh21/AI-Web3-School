## OpenClaw 是什么

**一句话定义**：**开源、本地优先、自托管的 AI 智能体（Agent）运行框架**，外号 “龙虾”，基于 TypeScript 开发，核心是让大模型**不只是聊天，而是能自动帮你干活、跨软件执行任务**。

简单理解：

-   ChatGPT/Claude：云端对话机器人，只能回答问题
    
-   OpenClaw：**本地 AI 员工**，跑在你电脑 / 服务器，通过微信、Telegram、飞书等聊天软件指挥，自动执行文件操作、代码、浏览器、定时任务、调用 API、子代理分工。
    
    核心特点：**本地运行、隐私可控、模型无关、插件可扩展、多 IM 渠道接入**。

## 二、底层概念

### 1）AMM 自动做市商

-   没有订单簿，靠**资金池+公式**定价
    
-   V2：`x*y=k`
    
-   V3/V4：在选定区间内用 `x*y=k`，区间外价格不移动
    

### 2）流动性池（Pool）

-   两种代币按比例存入，供用户兑换
    
-   V2：每个 Pair 一个池
    
-   V3：每个（代币A+代币B+费率）一个池
    
-   V4：全部池子在一个合约里
    

### 3）滑点 Slippage

区块链交易不是即时确认的。从用户在钱包点下“确认”到区块打包完成，中间可能有利差交易者或其他人抢先交易，导致价格变动。

-   交易时价格波动导致实际兑换变少
    
-   **让用户设置滑点（0.1%–1%）→ 计算最小接收金额**
    

滑点设置（Slippage）： 前端必须提供一个 UI 允许用户设置可接受的滑点百分比（如 0.5%）。前端计算出用户最少必须收到的代币数量（amountOutMinimum）并传给合约。如果不设防，用户极易遭受夹子攻击（MEV 恶意三明治攻击），资产被白白榨干。

交易期限（Deadline）： 一个 Unix 时间戳（例如当前时间 + 10 分钟）。如果交易在打包时超过了这个时间，合约会自动拒绝（Revert），防止用户的交易在网络拥堵时卡住，在几小时后以极差的价格被动执行。

### 4）路由 Router

用户在兑换时，直接查询某一个池子往往拿不到最优价格。通常“代币 A ➔ 代币 B ➔ 代币 C”的多跳路径比直接兑换更划算。

前端不能盲目调用合约，而是需要先请求 Uniswap 官方的 Auto Router API 或 1inch 等聚合器 API，获取当前最优的路由路径、预期获得的代币数量（Quote）以及预估 Gas 费，并动态渲染到界面上。

-   前端不直接调用 Pool，而是调用 **Router（路由合约）**
    
-   自动找最优路径（如 ETH→USDC→DAI 比直接 ETH→DAI 更好）
    

### 5）V3/V4：Tick（刻度） / 价格区间

-   Tick：价格的离散刻度（`price = 1.0001^tick`）
    
-   LP 提供流动性必须选 **\[tickLower, tickUpper\]**
    
-   前端要画价格区间、选区间、计算仓位
    

注意事项： 当用户在 UI 输入“1000 ~ 2000”时，前端必须调用 Uniswap SDK（如 v3-sdk），将输入的价格转化为合约能看懂的最近的 TickLower 和 TickUpper。

### 6）V4 专属：Hooks

-   8 个生命周期钩子：`beforeSwap` / `afterSwap` / `beforeAddLiquidity`…
    
-   前端要：**识别池子是否带 Hook → 展示特殊逻辑（限价、动态费）**
    

### 7）授权 Approval

-   ERC20 代币必须先授权给 Router/NFTManger，才能交易/加流动性
    
-   前端流程：**检查授权 → 未授权则发起授权交易 → 授权后再执行 swap/mint**
    

### 8）V4 核心：解锁回调与差额管理

V4 为了省 Gas 费，采用了单例模型和闪电记账。这改变了前端或周边合约的调用逻辑：任何交易（Swap、Add Liquidity）必须先调用 PoolManager.unlock()。

合约被解锁后，会触发一个回调函数，前端（或路由合约）在回调中把“兑换”、“添加流动性”等操作打包成一个批处理指令。

# web3 密码学

区块链密码学主要由两条支柱撑起：**哈希函数（确保数据不可篡改）和非对称加密**（确保所有权与授权）。

## 哈希函数

它用于生成区块哈希、交易哈希及地址。

**特点**

-   抗碰撞性
    
-   不可逆性
    
-   雪崩效应
    

**常用算法**

-   SHA-256: 用于工作量证明（POW）
    
-   Keccak-256: 以太坊 POS 里用来生成地址、交易哈希、区块哈希、状态树节点哈希、智能合约
    

```markdown
虽然区块链主要使用非对称加密（如ECC），但AES在以下场景中非常重要：

| 应用场景     | 用途                      | 示例                   |
| ------------ | ------------------------- | ---------------------- |
| **钱包加密** | 用AES加密私钥，存储到本地 | MetaMask、硬件钱包     |
| **数据传输** | 加密节点间的通信数据      | P2P网络加密            |
| **智能合约** | 链下数据加密              | 可信执行环境（TEE）    |
| **隐私保护** | 加密敏感数据              | 零知识证明中的辅助加密 |
```

## 非对称加密

非对称加密使用一对密钥：加密密钥和解密密钥是不同的，分别称为\*\*私钥 (Private Key)\*\* 和 \*\*公钥 (Public Key)\*\*。私钥签名，公钥验证。

私钥一般需要通过随机数算法生成，公钥可以根据私钥生成。公钥一般是公开的，他人可获取的；私钥一般是个人持有，他人不能获取。  
  
代表算法包括：

-   RSA
    
-   ElGamal
    
-   椭圆曲线（Elliptic Curve Crytosystems, ECC）：核心用途是做离线签
    
    -   ECDSA：以太坊、BSC、所有 EVM 链**主力签名算法**
        
        secp256k1
        
    -   EDDSA：Solana、波卡、TON 等公链主流签名Ed25519
        

```markdown
| 项目        | 曲线      | 用途               |
| ----------- | --------- | ------------------ |
| **比特币**  | secp256k1 | 交易签名           |
| **以太坊**  | secp256k1 | 交易签名、账户地址 |
| **Solana**  | Ed25519   | 交易签名           |
```

# 基本概念

### 助记词（Mnemonic）

-   由**12/24个英文单词**组成
    
-   作用：**批量生成无数私钥**，是最高级备份
    
-   特点：离线保存，丢了全丢，泄露资产全没
    
-   来源：钱包首次创建自动生成
    

### 私钥（Private Key）

-   64位16进制字符串，一串乱码字符
    
-   作用：**唯一签名凭证**，能签名发起交易、转走币
    
-   关系：**助记词 → 推导算出私钥**
    
-   权限：拥有私钥=拥有账户所有资产
    

### 公钥（Public Key）

-   由**私钥通过椭圆曲线算法单向算出**
    
-   单向：私钥能推公钥，公钥推不回私钥
    
-   作用：用来生成钱包地址
    

### 钱包地址（Address）

-   0x开头42位字符串，日常收款转账用
    
-   生成：**公钥哈希压缩 = 钱包地址**
    
-   作用：对外收款、查余额、公开可见
    
-   无任何转账权限，只是收款账号
    

### 账户（Account）

以太坊链上两种账户

1.  **EOA外部账户**：普通人钱包账户，由地址标识，能发交易
    
2.  **合约账户**：部署智能合约生成，无私钥，不能主动发交易
    

### 钱包（Wallet）

-   软件/工具：MetaMask、小狐狸、TP钱包等
    
-   本质：**帮你管理助记词、私钥、地址的容器**
    
-   功能：签名交易、收发币、调用合约，不存币，只管钥匙
    

### 关系

**助记词 → 私钥 → 公钥 → 钱包地址**

1.  记好助记词 = 掌控所有私钥
    
2.  私钥唯一对应一个地址
    
3.  地址公开随便给，私钥绝对不能外泄
    

# 钱包的种类

### **1\. 热钱包（Hot Wallet）**

### 定义

私钥**存储在联网设备**，随时联网签名交易

### 常见类型

-   手机钱包：TP、imToken、麦子钱包
    
-   浏览器插件：小狐狸MetaMask、Coinbase钱包
    
-   交易所钱包：币安、OKX 资金账户
    

### 优点

-   打开即用，转账、合约交互、swap、挖矿**最快最方便**
    
-   无需插设备，一键调用DApp
    

### 缺点

-   联网易被盗、钓鱼、木马、劫持
    
-   安全性**最低**
    

### 适用

日常小额交易、玩DeFi、刷链、交互合约

* * *

### 2\. 冷钱包（Cold Wallet）离线钱包

### 定义

**私钥永远不联网**，断网生成、断网签名

### 常见类型

-   硬件冷钱包：Ledger、Trezor、库神
    
-   离线纸钱包：手抄私钥/助记词
    
-   离线断网手机/电脑生成密钥
    

### 原理

联网设备只传**交易明文** → 离线设备签名 → 再传回联网设备广播上链  
全程**私钥不上网**

### 优点

-   几乎无法被盗，安全性**最高**
    

### 缺点

-   操作麻烦，插拔设备、确认签名繁琐
    
-   不能直接一键交互DApp
    

### 适用

**大额囤币、长期持仓、资产兜底**

* * *

### 3\. 温钱包（Warm Wallet）折中钱包

### 定义

介于冷热之间，**半离线半在线**

### 主流形态

1.  本地离线导出私钥，平时不导入热钱包，用时临时导入用完删除
    
2.  自建本地节点钱包、桌面轻钱包
    
3.  多签钱包（Safe多签最典型温钱包）
    

### 逻辑

-   常用小额放热钱包
    
-   大额资产放温钱包隔离
    
-   不长期联网暴露私钥
    

### 优点

-   比热钱包安全，比冷钱包方便
    
-   兼顾便捷与风控
    

### 缺点

安全性中等，不如纯冷钱包

### 适用

中量级资产、项目持仓、工作室常用资产

* * *

# 三类钱包安全等级排序

**冷钱包 ＞ 温钱包 ＞ 热钱包**

1.  **90%大额资产** → 冷钱包（囤着不动）
    
2.  **7%中端资产** → 温钱包（项目、定投）
    
3.  **3%流动资金** → 热钱包（日常swap、交互、Gas费）

# 核心概念

## LLM

语言学习模型（LLM）的基本工作原理： LLM 基于上下文以概率方式生成文本：给定一个输入，模型预测最可能的下一个词元序列。它们擅长语言理解、代码生成和推理，但在准确的事实回忆、确定性计算和跨会话状态维护方面表现不佳。

四个控制层： 上下文窗口是模型的“工作记忆”，控制模型在任何给定时间可以看到的信息量。系统指令设定身份、语气和行为边界。提示传达当前任务的意图。工具调用将模型从说话者转变为执行者。

实际调用 LLM API： MaaS 允许您通过 API 密钥按令牌调用顶级模型，无需 GPU。核心参数： model 、 messages 、 temperature 、 max\_tokens 。

提示、工作流和代理之间的界限： 提示是指让模型给出答案，由人做出决策。工作流是一个预定义的任务流程，其中模型是其中一个节点，路径是固定的。代理是指模型能够自主规划，动态调用工具，并在各个回合之间管理状态。这三者在故障模式、风险敞口和可调试性方面存在根本差异。

## AI 编码工具的价值与局限性：

Claude Code、Codex CLI 和 Cursor 可以快速生成样板代码、解释不熟悉的库并加速原型开发。然而，代码审查、测试设计和架构决策却无法委托给它们。

为什么人工智能的输出必须始终经过验证： 事实错误是最常见的陷阱：模型会自信地捏造信息，因此关键事实必须通过外部验证。模型在引用方面同样不可靠，它们经常会编造论文标题、URL 和数据来源，因此永远不要相信模型提供的任何链接。在更长的上下文中，推理漂移会成为一种风险，逻辑链会悄然断裂，结论偏离最初的前提，因此逐段验证至关重要。当涉及智能体时，还会出现两种额外的故障模式：执行越权，即智能体超出其授权范围执行操作（需要设置防护措施和人工干预检查点）；以及工具误用，即模型调用了错误的工具或传递了错误的参数（需要跟踪以监控执行过程）。

## Agent 的核心技术组件：

-   状态管理：多个节点共享对同一状态对象的读写访问权限。
    
-   长期记忆：跨会话存储和检索信息。
    
-   MCP：LLM 与外部工具之间的统一连接协议。
    
-   skills：支持自动发现和动态生成的可重用高级指令集。
    
-   Tool calling：模型发出结构化请求；框架执行该请求并返回结果。
    
-   跟踪：可视化代理的执行链。
    
-   防护机制：输入/输出验证规则；如果违反规则，则执行停止。
    
-   交接：子任务完成后转移控制权。
    
-   错误恢复：失败时进行重试、回滚或升级到人工干预。
    

## 何时真正需要智能体？

当目标不明确、需要多个工具协作、中间结果决定下一步操作或状态需要在会话间保持时，可以使用智能体。如果是一次性问答（使用提示）、流程固定（使用脚本）、合规性要求严格（使用人工审核节点）或数据确定性至关重要（使用数据库查询），则应坚持使用更简单的解决方案。复杂性和风险越高，就越应该谨慎避免过度使用智能体。
