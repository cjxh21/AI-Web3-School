\## 核心收获

\### API 是什么

API 是两个程序之间说话的规则和通道。就像餐厅的服务员——你不需要冲进厨房，只要按菜单格式下单，服务员（API）把结果端给你。WCB 开放了 API，我可以直接用代码拿到课程数据，不用手动复制粘贴。

\### 浏览器 vs 服务器

\- **服务器**：一台24小时不关机的电脑，存着数据，任何时候都能响应请求。就像24小时便利店。

\- **浏览器**：做两件事——从服务器拿数据，再把数据渲染成人能看的页面（颜色、排版、按钮）。

\- **API**：跳过渲染步骤，把原始数据直接给程序用。

\### API Key 是通行证

服务器不是谁都能进。API Key 是你的身份证明，每次请求都带着它，服务器验证通过才返回数据。这就是为什么 Key 暴露了要立刻作废——相当于钥匙丢了。

\### 环境变量是本地保险箱

Key 不能写进代码（上传 GitHub 就泄露），不能发进对话（对话有记录）。存到操作系统的环境变量里，程序需要时去取，但不会出现在任何文件或对话记录里。这是行业标准做法。

\### 为什么要去开发者面板拿 programId

API 文档告诉你怎么调用，但有些参数是占位符，比如 `<programId>`。具体的值只在你的账号数据里有。开发者工具（F12）的 Network 面板能看到浏览器和服务器之间真正传了什么——真实 ID 就藏在里面。这个技能可以直接迁移到爬虫。

\### 关于服务器部署

自己电脑上跑的 Agent，关机就停了。把 Agent 部署到服务器上，它能24小时运行、随时响应。这是做商业 Agent 必须走到的一步。

LLM（大型语言模型）的基本工作方式是**基于上下文进行概率生成**——给定一段输入文本，模型通过对海量语料训练所形成的参数，预测最合理的下一个 token 序列，从而逐步生成连贯的输出。这一机制使其在**语言理解、代码生成与逻辑推理**等任务上表现出色，能够灵活应对多样化的自然语言需求。

然而，这种概率生成的本质也决定了它的局限性：LLM 并不具备精确的事实记忆能力，容易产生"幻觉"；面对需要严格精度的数学计算时也可能出错；此外，由于每次对话相互独立，它**无法跨会话保持状态**，不具备持续学习与记忆的能力。理解这些边界，是合理使用 LLM 的前提。
