\## Handbook 模块 35：Governance AI（治理 AI）  
  
\### 核心问题  
  
DAO、协议、公共物品和开源社区往往积累大量提案、论坛帖、会议纪要、预算申请和贡献记录。信息过载时，\*\*少数人掌握上下文，多数人只能被动投票\*\*——治理效率低，但决策正当性不能靠「让 AI 替人拍板」来换。  
  
\> 一句话：\*\*Governance AI 是提高公共决策的信息质量，不是替代人的政治判断。\*\*  
  
\*\*第一性原理\*\*：  
\- \*\*来源可追溯\*\*：每个关键摘要都能回到提案、论坛、会议或链上记录  
\- \*\*立场要分开\*\*：事实、推断、观点、建议不能混写  
\- \*\*人保留决策权\*\*：AI 辅助阅读与整理；投票、授权、拨款仍由人或明确规则完成  
  
与 Day 6 Module G 一致：适合 AI 做提案总结、会议转行动项、贡献记录；\*\*不适合\*\*直接做价值判断、预算批准、代表社区发起不可逆动作。  
  
\---  
  
\### 与 Payment 主线的关系  
  
Week 2 主方向是 Payment，但治理与支付在链上常\*\*同一条链路\*\*：  
  
| Payment 场景 | Governance AI 能辅助什么 |  
|--------------|-------------------------|  
| 协议金库拨款给 x402 服务方 | Budget Check：预算项、里程碑、收款地址、历史拨款是否完成 |  
| Agent 消费公共 API / 开源 infra | Contribution Graph：谁维护底层库、是否重复拨款 |  
| Escrow 争议升级 | Proposal Summary：把争议材料整理成可比较的结构，附来源 |  
| CAW / Pact 策略变更 | Meeting Action：谁负责改 policy、截止日、是否需要投票 |  
| 社区对「Agent 自主支付上限」表决 | Plurality：展开 LP / 开发者 / 金库管理者等不同视角，不压成单一结论 |  
  
Handbook 强调：\*\*治理比交易更重视正当性\*\*。AI 辅助工具必须透明、可质疑、可复核——尤其不能悄悄替用户投票或隐藏反对意见。  
→ 与 Day 9 AI Security 中「治理提案 / 交易 memo 可作为 Prompt Injection 载体」形成对照：治理 AI 读提案，也要防\*\*信息操纵\*\*。  
  
\---  
  
\### 七大知识节点  
  
\#### 1. Proposal Summary（初级）  
  
把长提案整理成固定结构，而非缩短成宣传稿。  
  
| 固定字段（建议） | 说明 |  
|------------------|------|  
| 提案目标 / 背景 | 要解决什么问题 |  
| 预算、付款地址、里程碑 | 可核对的事实项 |  
| 执行主体、时间线、影响范围 | 谁做、何时、影响谁 |  
| 支持理由 / 反对理由 | 保留争议，不单列一方 |  
| 未决问题 | AI 不能假装已回答 |  
| 相关历史提案 / 链上交易 | 避免重复讨论 |  
  
\*\*关键区分\*\*：「提案请求 50,000 USDC」是\*\*事实\*\*；「预算是否合理」是\*\*判断\*\*——混写会让读者误以为 AI 的判断就是治理结论。  
  
\---  
  
\#### 2. Meeting Action（初级）  
  
把会议讨论转成\*\*可执行、可检查\*\*的事项。  
  
| 不是 Action | 才是 Action |  
|-------------|-------------|  
| 「我们应该研究一下支付方案」 | 「Alice 下周五前整理三种 x402 集成方案并链到 issue」 |  
  
还需标注\*\*决策状态\*\*：已决定 / 待确认 / 需要投票 / 需要预算 / 需要安全复核——避免把不同成熟度的事项混在一份纪要里。  
  
与 Agent Workflow 衔接：Action 应有 \*\*owner + deadline + 上下文链接 + 后续检查方式\*\*。  
  
\---  
  
\#### 3. Contribution Graph（中级）  
  
帮助社区看见：谁在做什么、贡献如何连接、哪些工作被低估。  
  
| 数据来源 | 注意 |  
|----------|------|  
| GitHub、论坛、链上支付、会议记录、项目管理工具 | 勿把贡献简化成 commits 数或单一分数 |  
  
\*\*价值\*\*：  
\- 发现「隐形贡献」：长期 review、协调会议、维护文档、带新人  
\- 公共物品资助：避免重复拨款、漏拨款；显示项目依赖与协作网络  
  
AI 整理证据；\*\*评分与拨款规则仍由人 + 透明流程决定\*\*。  
  
\---  
  
\#### 4. Budget Check（中级）  
  
审查资金申请是否清楚、合理、可追踪——\*\*不是自动砍预算\*\*，而是让社区看清问题。  
  
| 检查项 | 示例问题 |  
|--------|----------|  
| 预算项完整性 | 是否覆盖开发、审计、运营？ |  
| 里程碑具体性 | deliverables 能否验证？ |  
| 历史拨款 | 上次拨款是否交付？ |  
| 金额与范围 | 是否重复申请？ |  
| 链上要素 | 收款地址是否属提案方？多签？vesting？异常提款？ |  
  
与 \`payment-flow.md\` Escrow、x402 结算对齐：\*\*付款节奏应与里程碑绑定\*\*；Budget Check 是治理层的「验收 checklist」，Escrow 是链上的「条件释放」。  
  
\---  
  
\#### 5. Source Traceability（初级）  
  
治理 AI 的\*\*底线\*\*：每条关键总结附来源（论坛链接、提案编号、会议时间戳、链上 tx、投票记录、预算表）。  
  
| 要求 | 反例 |  
|------|------|  
| 读者能点回原文 | 无来源摘要用于投票依据 |  
| 预算结论能点回表格与交易 | AI 编造「社区普遍支持」 |  
| 无法给出来源 → 标「推断」或「不确定」 | 无来源推动集体决策 |  
  
与 Day 9 Audit Log、CAW 可审计日志一致：\*\*治理与支付系统都要能复盘「依据从哪来」\*\*。  
  
\---  
  
\#### 6. Deep Funding（高级）  
  
公共物品与复杂贡献的分配：影响非线性（底层库、协调者、文档项目往往被 KPI 低估）。  
  
| AI 可辅助 | 人必须保留 |  
|-----------|------------|  
| 证据包：依赖图、使用案例、贡献者网络、历史资助、用户反馈 | 多元评审、公开理由、申诉机制 |  
| 整理评审材料 | 价值判断与最终分配 |  
  
模型\*\*不应\*\*成为黑箱拨款者。  
  
\---  
  
\#### 7. Plurality（高级）  
  
在多元群体中\*\*保留差异、支持协商\*\*，而非输出最响亮或最常见的观点。  
  
| 产品含义 | 示例 |  
|----------|------|  
| 摘要含少数派担忧、角色差异、可协商空间 | 协议费用提案：LP / 交易者 / 开发者 / 金库 / 长期 holder 视角分开呈现 |  
| 不压成「平均意见」 | 避免「AI 建议投赞成票」式单一结论 |  
  
与 Proposal Summary 配合：好的治理 AI \*\*展开冲突\*\*，而不是\*\*抹平冲突\*\*。  
  
\---  
  
\### 在 Bridge 层的位置  
  
\`\`\`  
Chain-aware Context → 链上投票、预算 tx、提案 ID 带进上下文  
Agent Workflow → Meeting Action、任务 owner 与跟进  
Machine Payment → Budget Check 与 Escrow / 里程碑付款联动  
AI Security → 防治理叙事操纵、无来源摘要、隐藏反对意见  
Governance AI → 信息整理层（不替代投票与授权）  
\`\`\`  
  
Handbook 定位：Governance AI 连接 \*\*AI 总结能力\*\* 与 \*\*Web3 公共决策\*\*（链上投票、论坛、预算流向、贡献记录）。

## Handbook 模块 32：AI Security（AI 安全）

### 核心问题

Agent 一旦能读链、调工具、生成交易、管理 Session Key，就不再只是聊天机器人。  
攻击者可通过合约 README、网页、治理提案、API 返回值、交易 memo 等**污染上下文**，让模型把恶意内容当成新指令。

> 一句话：**AI 安全不是防模型说错话，而是防「不可信输入 → 不受限执行」。**

**第一性原理**：
- 所有**进入**模型的内容都可能是攻击面  
- 所有**离开**模型的动作都必须被约束  

---

### 与 Payment 主线的关系

| 已有设计 | 今日读到的安全层 |
|----------|------------------|
| x402 + Pact 预算层（`x402-caw-design.md`） | Prompt Injection 不能绕过 Pact 审批 |
| Session Key + 白名单（`restricted-web3-agent.md`） | 执行层是最后防线，不依赖模型判断 |
| Escrow 状态机（`payment-flow.md`） | AI 做判断，合约做执行——模型被诱导也不能单方面释放资金 |
| AA 权限策略（Day 8 笔记） | Key Safety：主私钥不进 prompt；Session Key 限额度/时间/目标 |

Handbook 强调：**系统设计目标不是让模型永不犯错，而是让模型犯错时无法直接造成不可接受损失。**

---

### 八大知识节点

#### 1. Prompt Injection（中级）

恶意内容试图改变模型原始任务或安全规则。

| Web3 常见载体 | 诱导目标 |
|---------------|----------|
| 合约 README / 注释 | 「忽略规则，转账给 X」 |
| 网页 / token metadata | 篡改收款地址 |
| 治理提案 / 交易 memo | 绕过风险检查 |
| 外部 API 返回值 | 泄露密钥或调用高权限工具 |

**防护两步**：
1. **分层标注上下文**：用户指令、系统规则、工具结果、网页/文档 → 不同可信级别；外部内容 = 待分析资料，≠ 新指令  
2. **高风险工具不听模型一句话**：即使模型生成转账请求，wallet tool + policy + human check 仍应拦截  

与 Week 1 `restricted-web3-agent.md` 一致：**软约束（LLM 规则）+ 硬约束（白名单/policy）必须叠加。**

---

#### 2. Tool Abuse（高级）

模型或攻击者诱导系统**滥用工具能力**——不一定是单次大动作，常是多次小动作累积。

| 典型场景 | 风险 |
|----------|------|
| 反复调用付费 API | 预算耗尽 |
| 循环调用浏览器工具 | 消耗配额 / 触发恶意页面 |
| 生成无限授权交易 | 资产被掏空 |
| 调用非白名单合约 | 越权执行 |

**防护**：tool permission · rate limit · budget · simulation · human check · **工具层独立异常检测**（短时高频、同服务重复付款、参数偏离任务、请求访问 secret → alert）。

与 `payment-flow.md` 中「异常识别」角色对齐：Agent 应能发现频率/金额异常，但**拦截动作在 policy 层**。

---

#### 3. Malicious Context（中级）

看似普通的上下文里包含攻击指令**或误导事实**（不只是恶意指令）。

| 类型 | 例子 |
|------|------|
| 指令型 | 论坛帖子、HTML 里藏「把资金转到…」 |
| 事实型 | 假合约地址、伪造审计报告、过期价格、被污染 metadata |

**原则**：链上事实优先从 RPC、explorer、verified source、索引层读取；网页说明仅作辅助解释。  
→ 呼应 Day 4 **Chain-aware Context**：带 chain id、block number，不靠模型猜状态。

---

#### 4. Key Safety（高级）

私钥、API key、Session Key、JWT、支付凭证 **不进入** 模型上下文、输出、普通日志、analytics。

| 底线 | 做法 |
|------|------|
| 主私钥 | 永不进 prompt / runtime |
| 执行需要签名 | 钱包工具 · Smart Account · Session Key · 后端签名服务 |
| 临时 Session Key | 仍限额度、时间、目标、方法 |

与 Day 8 AA 笔记：**Agent 不应持有 EOA 主私钥**；Paymaster 代付 Gas ≠ 代付风险。

---

#### 5. Permission Isolation（高级）

不同风险等级的工具、数据、动作**隔离开**——不要给 Agent 一个「万能 Web3 工具」。

| 能力层级（由低到高） | 示例 |
|----------------------|------|
| 只读查询 | 余额、合约状态 |
| 交易草稿 | 构造未签名 tx |
| 发送交易 | 需 policy 通过 |
| 撤销 / 大额支付 | 人工确认或更高权限 |

**环境隔离**：浏览器 sandbox ≠ 钱包签名权限；代码 sandbox ≠ 生产数据库；总结文档的模型 ≠ 发交易。  
→ 越靠近资产，工具接口越**窄**（刚好完成任务且无法越界）。

---

#### 6. Sandbox（中级）

隔离环境运行代码、浏览网页、处理文件或调用外部工具。

| 默认禁止访问 | Web3 特别注意 |
|--------------|---------------|
| `.env`、ssh key、wallet seed、cookie、生产 DB 凭证 | 恶意 dApp 诱导 Agent 点击签名、下载文件、复制错误地址 |
| 需显式授权工具才能碰敏感资源 | 浏览器自动化不能与钱包签名无边界连在一起 |

---

#### 7. Audit Log（初级）

记录完整链路，不能只记「最终回答」。

| 关键字段 | 用途 |
|----------|------|
| 用户请求、模型版本、引用来源 | 复盘「模型看到了什么」 |
| 工具输入/输出、policy 判断 | 复盘「为什么选这个工具、是否通过策略」 |
| 交易哈希、用户确认、错误 | 复盘「最终执行了什么」 |

高价值系统：日志 hash 锚定链上，或签名记录关键事件 → 防篡改。

与 CAW/Pact 设计（`x402-caw-design.md`）的「可审计执行日志」一致。

---

#### 8. Alert（中级）

发现异常并**及时打断**自动化——只有告警没有响应，安全收益有限。

| 可监控信号 | 响应动作（需预先定义） |
|------------|------------------------|
| 异常工具频率、预算快速消耗 | 暂停 Agent |
| 非白名单地址、session key 越界尝试 | 撤销 Session Key |
| 连续失败交易、大额授权 | 冻结 Escrow / 通知用户 / 人工审核 |
| prompt injection 命中 | 进入人工审核 |

告警分级：低风险 → 后台队列；高风险资产动作 → 立即打断。

---

### 在 Bridge 层的位置

AI Security **贯穿**所有桥接模块：

```
Chain-aware Context  → 防恶意上下文（Malicious Context）
Web3 Tool Use        → 防工具滥用（Tool Abuse）

Machine Payment      → 防预算穿透、支付路径篡改
Governance AI        → 防信息操纵
````

\## Handbook 模块 17：Account Abstraction（账户抽象）  
  
\### 核心问题  
  
传统 \*\*EOA\*\* 把「谁能发交易、怎么验证签名」写死在协议层：一把私钥 = 全部权力。  
AI Agent、自动化支付、限额/撤销/代付 Gas 等需求，需要的是 \*\*在合约里自定义验证与权限规则\*\*——这就是账户抽象要解决的问题。  
  
\> 一句话：\*\*把账户从「私钥驱动」升级为「策略驱动」。\*\*  
  
\---  
  
\### EOA vs 智能账户（AA）  
  
| 维度 | EOA | 智能账户（AA / ERC-4337） |  
|------|-----|---------------------------|  
| 账户形态 | 地址由私钥推导，无合约代码 | 链上合约账户，逻辑可编程 |  
| 验证位置 | 协议层：ecrecover 检查签名 | 合约层：\`validateUserOp\` 等自定义规则 |  
| 权限模型 | 全有或全无 | 可设 Owner、Guardian、Session Key、日限额 |  
| 恢复与限额 | 基本不支持 | 社交恢复、多因素、消费上限等可配置 |  
| 适合 Agent | ❌ 不宜把私钥交给程序 | ✅ Session Key / 策略引擎限权 |  
| 主要风险 | 私钥泄露即失控 | 合约漏洞、策略配置错误、升级权限被滥用 |  
  
与 Week 1 \[\`eoa-account-compare.md\`\](../submissions/week1/eoa-account-compare.md) 对照：多签解决「多人共同管钱」；AA 解决「同一账户内灵活、可编程的授权」。  
  
\---  
  
\### ERC-4337 核心思路（不改造共识层）  
  
协议不直接改「交易」格式，而是引入 \*\*UserOperation（用户操作）\*\* 抽象层：  
  
\`\`\`  
用户 / Agent 意图  
→ 构造 UserOperation（目标、calldata、gas 等）  
→ Bundler 打包进区块  
→ EntryPoint 合约统一校验 & 执行  
→ 智能账户合约 validateUserOp → 通过则执行  
\`\`\`  
  
| 角色 | 职责 |  
|------|------|  
| \*\*Smart Account\*\* | 链上账户合约；定义「什么签名/什么条件算合法」 |  
| \*\*UserOperation\*\* | 类似「待执行意图」的结构化包，不是传统 tx |  
| \*\*Bundler\*\* | 收集 UserOp、代发上链（可收服务费） |  
| \*\*EntryPoint\*\* | 标准入口；校验、扣费、回调账户逻辑 |  
| \*\*Paymaster\*\* | 可选；代付 Gas 或允许用 ERC-20 付 Gas |  
  
\*\*关键洞察\*\*：验证逻辑从「节点验 ECDSA」变成「EntryPoint 调账户合约验规则」——因此才能做 Session Key、限额、批量操作等。  
  
\---  
  
\### 与 Agent / Payment 方向的衔接  
  
| 场景 | AA 提供的机制 | 与本周任务的联系 |  
|------|---------------|------------------|  
| Agent 代用户付 x402 账单 | Session Key：白名单合约 + 金额上限 + 有效期 | → \`wallet-permission.md\` 权限策略表 |  
| 支付后撤销/紧急止损 | Owner 调 \`revokeSessionKey\` 或暂停账户 | → 对齐 \`restricted-web3-agent.md\` 30 秒撤销路径 |  
| 用户无 ETH 付 Gas | Paymaster 代付或 ERC-20 付 Gas | → 降低 Agent 自动化摩擦 |  
| 批量结算 | 同一 UserOp 批处理多笔调用 | → 对齐 x402 Batch Settlement 思路 |  
  
\*\*边界（与 Escrow / Payment 一致）\*\*：  
  
\- \*\*AI 能做\*\*：理解意图、起草 UserOp、解释 calldata、在策略内建议是否执行  
\- \*\*链上/账户合约做\*\*：最终是否接受签名、是否超限额、是否命中白名单  
\- \*\*人不能省掉的\*\*：首次授权、提高限额、改 Owner、approve 大额、升级实现  
  
\---  
  
\### Session Key（Agent 接入 AA 的常用模式）  
  
临时密钥 + 链上/链下策略，而非把主私钥交给 Agent：  
  
| 约束维度 | 典型配置 | 目的 |  
|----------|----------|------|  
| 时间 | 24h 或单次任务结束即失效 | 缩小泄露窗口 |  
| 金额 | 单笔上限 + 日累计上限 | 控制损失上限 |  
| 目标 | 合约地址白名单 / 方法选择器 | 防止调用任意合约 |  
| 撤销 | Owner 主动 revoke；Bundler 拒绝已撤销 Key | 攻击后快速止血 |  
  
与 Week 1 跟读 \*\*Elytro\*\* 的笔记呼应：Session Key 是「有限签名权」，不是「完整钱包权」。  
  
\*\*仍待深挖（可记入 handbook-feedback）\*\*：接近日上限的连续小额攻击，链上策略 vs 链下异常检测如何分工。  
  
\---  
  
\### 授权设计四维度（预习 \`wallet-permission.md\`）  
  
结合 Day 6 Module D 框架，读 AA 时可按四维记笔记：  
  
1\. \*\*对象\*\*：Session Key 代表谁（用户智能账户 / 团队 Safe）  
2\. \*\*范围\*\*：可调用的合约、代币、金额、时间窗  
3\. \*\*执行策略\*\*：全自动 / 超阈值人工确认 / 仅模拟不广播  
4\. \*\*恢复机制\*\*：撤销、暂停、审计日志、多签升级路径  
  
\*\*反例\*\*：界面只有「用自然语言发交易」，没有展示权限范围与失败原因——对用户和审计都不友好。  
  
\---  
  
\### 与 Pact / CAW 的关系（概念层）  
  
| 机制 | 层级 | 作用 |  
|------|------|------|  
| \*\*ERC-4337 / AA\*\* | 链上账户与交易验证 | 「这笔链上操作是否被允许执行」 |  
| \*\*Pact / CAW（任务级授权）\*\* | 应用/预算层 | 「围绕哪次任务、多少预算、何时失效」 |  
  
二者可叠加：Pact 定义任务预算与审计要求 → AA Session Key 在链上落实白名单与限额 → x402 / Escrow 处理支付与释放。  
  
\---  
  
\### 核心 Takeaway（≤ 5 条）  
  
1\. \*\*AA 的本质\*\*是把「账户规则」从协议硬编码搬到合约可编程，是 Agent 安全上链的前提设施之一。  
2\. \*\*ERC-4337\*\* 用 UserOperation + EntryPoint 在不改共识的前提下实现智能账户生态。  
3\. \*\*Agent 不应持有 EOA 主私钥\*\*；推荐路径是 \*\*智能账户 + Session Key + 可撤销策略\*\*。  
4\. \*\*Paymaster\*\* 解决 Gas 体验，但不替代权限设计——代付 Gas ≠ 代付风险。

## Handbook 模块 27：Settlement & Escrow

### 核心问题

付款方与交付方之间存在**时间差**：谁先动都有风险。Escrow 合约作为中间层，解决这个信任问题。

### 关键机制

| 机制 | 说明 |
|------|------|
| 资金锁定 | 买方资金先转入合约，任何一方都无法单独取出 |
| 完成条件 | 必须在合约部署时**明确定义**，不能事后修改 |
| 状态定义 | 合约需要预先枚举所有可能状态，状态迁移规则写入代码 |
| 自动释放 | 条件满足 → 状态变更 → 合约自动执行转账，无需人工干预 |

### 状态机模型

```
PENDING → LOCKED → DELIVERED → RELEASED
                       ↓
                   DISPUTED → (仲裁) → RELEASED / REFUNDED
```

### AI 的角色

- **能做**：解析用户意图、识别交付凭证、判断完成条件是否满足
- **不能做**：单方面触发释放资金（必须通过合约执行）
- **关键边界**：AI 是"建议层"，合约是"执行层"，两者不可混淆

### 安全增强方向

- 多签（Multi-sig）：多个验证者共同确认
- 时间锁（Timelock）：超时自动退款
- Oracle：引入链外数据验证交付
- 仲裁机制（Arbitration）：争议时的人工介入通道

---

## Handbook 模块 28：Agent Identity

### 核心问题

AI Agent 在无信任环境中行动，必须能被其他系统**可验证地识别**——"我说我是谁"不够，需要密码学证明。

### Agent 身份 vs 普通用户身份

| 维度 | EOA（普通用户） | Agent Identity |
|------|----------------|----------------|
| 控制者 | 人类持有私钥 | 程序/人类共同控制 |
| 身份内容 | 地址 + 签名 | 地址 + 能力声明 + 所有者 + 权限范围 |
| 信任来源 | 私钥签名 | 链上 DID / ERC-8004 声明 |
| 可审计性 | 交易历史 | 交易 + 行为记录 + 声誉 |

### Agent 身份应包含的内容

1. **所有者（Owner）**：谁授权了这个 Agent
2. **能力声明（Capability）**：这个 Agent 能做什么、不能做什么
3. **权限范围（Scope）**：可操作的资产上限、目标合约白名单
4. **身份绑定（Binding）**：与链上地址 / DID 的对应关系
5. **声誉记录（Reputation）**：历史行为的可验证记录

### 相关标准

- **DID（去中心化身份标识）**：W3C 标准，链上可解析
- **ERC-8004**：Agent Profile 草案标准，专为 AI Agent 设计
- **Session Key**：临时授权密钥，限制单次操作权限，用完即废

### 核心洞察

> 身份不是"我说我是谁"，而是"我能证明我能做什么"。  
> Agent 的可信度 = 身份可验证 + 权限可审计 + 行为可追溯。

# Day 6 阅读笔记

> 来源：AI × Web3 School Handbook + Week 2 课程内容  
> 课程页：[https://ethpanda.notion.site/Week-2-AI-Web3-354bbd63be87818a83abdca6da1e50cf](https://ethpanda.notion.site/Week-2-AI-Web3-354bbd63be87818a83abdca6da1e50cf)  
> 今日阅读：AI 基础 → Vibe Coding（07）、MCP（08）、Evaluation（09）；Week 2 框架 Module A–G

* * *

## AI 基础补读：Vibe Coding（模块 07）

**核心理解**：Vibe Coding 是用 AI 作为协作者驱动编码过程——不是写完整代码再让 AI 检查，而是用自然语言描述意图，让 AI 生成草稿，人负责方向判断和验收。核心是"人定目标，AI 生成，人审查迭代"的节奏。

**关键点**：

-   AI 生成的代码要审查，不能盲目运行——特别是涉及权限、网络请求、文件写入的部分
    
-   Vibe Coding 适合快速出原型、重复性代码、测试脚本，不适合直接用于安全敏感的生产代码
    
-   **和 Web3 的关系**：合约代码不能 vibe coding 后直接部署主网——生成→审查→测试网验证是最低要求
    
-   Cursor 就是 Vibe Coding 工具的典型实现：这整个学习 repo 本身就是 Vibe Coding 的产物
    

* * *

## AI 基础补读：模型上下文协议 MCP（模块 08）

**核心理解**：MCP（Model Context Protocol）是一个开放标准，让 AI 模型能以统一接口连接外部工具和数据源。解决的问题是：每次接入新工具都要定制集成，MCP 把这个接口标准化了——工具提供者只需实现 MCP server，模型就能直接调用。

**关键点**：

-   **三角关系**：Host（Cursor/Claude）→ Client（MCP 客户端）→ Server（工具实现）
    
-   MCP 解决的是 **tool-to-model 接口** 问题，而不是 agent-to-agent 的协作问题（那是 A2A 解决的）
    
-   典型 MCP server：文件系统、数据库、GitHub、Etherscan、Uniswap 数据源
    
-   **和 Web3 的关系**：链上数据（余额、合约 ABI、交易历史）可以通过 MCP server 注入 context，这正是 Chain-aware Context 的实现层
    

**MCP vs A2A 的区别**（Week 2 Module C 里会用到）：

|   | MCP | A2A |
| --- | --- | --- |
| 解决什么 | 模型 ↔ 工具的接口标准 | Agent ↔ Agent 的协作协议 |
| 通信对象 | 工具/数据源 | 另一个 Agent |
| 典型场景 | 查链上数据、读文件 | 委托另一个 Agent 执行子任务 |

* * *

## AI 基础补读：评估 Evaluation（模块 09）

**核心理解**：Evaluation 是判断 AI 系统表现是否达到预期的方法论。不是"感觉还不错"，而是能被测量、被重复、被对比的指标体系。没有评估就没有改进方向。

**关键点**：

-   **评估维度**：正确性（事实是否准确）、相关性（是否回答了问题）、安全性（有无有害输出）、一致性（相同输入是否稳定）
    
-   **评估方法**：人工标注 / LLM-as-judge（用另一个模型评分）/ 自动化测试（有标准答案的任务）
    
-   **Benchmark 的局限**：榜单上的好成绩不等于实际任务好用——评估要贴近真实使用场景
    
-   **和 Web3 的关系**：Agent 做链上决策时，必须能评估模型输出的可靠性——错误的链上操作不可逆，评估是安全前提

## Module B：Payment / Commerce 核心知识点

**关键理解**：Payment 只是链路中的一段；commerce 还包括发现服务、协商价格、执行任务、验收、争议和结算。

**深入路径**：

1.  **场景层**：Agent 买的是什么？（API 调用 / 数据 / 算力 / 人工服务）
    
2.  **流程层**：发现 → 报价 → 授权预算 → 执行 → 交付 → 验收 → 付款/退款/争议
    
3.  **验证层**：交付能否自动验证？不能则谁来验收？争议如何处理？
    
4.  **协议层**：需要 checkout / invoice / receipt / escrow / reputation，还是完整 marketplace/settlement layer？
    

**Cobo CAW 在此链路的作用**：

-   预算与权限控制：Pact 机制——围绕具体任务批准预算和执行范围，超出边界自动拒绝
    
-   审计与可验证记录：完整执行日志（授权内容、链上交易、状态变化、异常）
    

**反例**：只让 Agent 帮用户点传统支付按钮，没有预算控制、交付证明、可验证记录——这是普通自动化，不是 AI × Web3 强方向。

* * *

## Module C：Identity / Reputation 核心知识点

-   identity 解决"你是谁"，capability 解决"你能做什么"，reputation 解决"别人为什么信你"
    
-   MCP、A2A、ERC-8004、MPP 处于**不同层级**，解决不同问题，不互相替代
    
-   **反例**：只给 Agent 发 NFT 名片，但没有能力声明、调用接口、交付记录——不构成完整 identity/reputation 方向
    

* * *

## Module D：Wallet / Permission 核心知识点

-   账户抽象（ERC-4337）原理：把"谁能发交易"的逻辑从协议层移到合约层，实现灵活权限控制
    
-   Pact（任务级授权）思路：不给 Agent 长期权限，而是围绕一次具体任务生成临时授权，任务结束后权限失效
    
-   **授权四维度**：对象（代表谁）+ 范围（合约/金额/时间）+ 执行策略（自动/人工）+ 恢复机制（暂停/撤销/日志）
    
-   **反例**：只能展示"自然语言发交易"，但不能解释权限限制、失败处理——是危险 demo，不是可靠产品
    

* * *

## Module F：Security / Privacy 核心知识点

**常见攻击面**：

-   Prompt Injection（恶意内容注入指令）
    
-   Tool Abuse（工具返回值被污染）
    
-   敏感数据泄露（API Key、私钥、上下文）
    
-   模型供应商依赖（单点故障、数据托管）
    
-   不可审计操作（无日志、无回滚）
    

**安全设计要回答**：Agent 能看到什么、能调用什么、能花多少钱、能代表谁做决定、出了错谁负责

* * *

## Module G：Governance / Coordination 核心知识点

| 适合 AI 辅助 | 不适合直接交给 AI |
| --- | --- |
| 提案总结 | 价值判断 |
| 会议转行动项 | 预算批准 |
| 贡献记录 | 惩罚/激励决策 |
| 预算执行 checklist | 代表社区发起不可逆动作 |

**反例**：治理 assistant 自动生成并通过预算提案，没有人工确认、公开讨论和追责——是治理风险，不是效率提升。

# Day 5 阅读笔记

> 来源：AI × Web3 Handbook  
> 今日阅读：AI 基础 → 框架（Frameworks）；Web3 基础 → 网络（Network）；Bridge → 机器支付（Machine Payment）

* * *

## AI 向：框架（Frameworks）

**核心理解**：Framework 不是为了少写几行 API，而是把**模型、工具、状态、检索、评估、部署**组织成可维护的系统。框架解决的是**工程组织问题**，不替你定义产品价值，也不保证输出正确。

**关键点**：

-   **先画工作流，再决定用不用框架**：许多项目失败不是因为没用框架，而是先引入框架再让产品逻辑迁就它
    
-   **简单链路保持简单**：一次模型调用 + 一次检索 + 一次格式化，不需要复杂 Agent 框架
    
-   **长流程需要显式状态**：多步骤 + 工具调用 + 人工确认 + 失败恢复 → 应该有可查询的 state，而不只靠聊天历史
    
-   **框架要能退出**：换不掉模型、换不掉向量库的框架，长期成本很高
    

**主要框架速览**：

| 框架 | 定位 | 适合场景 |
| --- | --- | --- |
| LangChain | 组件库：模型接入、prompt、工具调用、retriever | 快速把模型能力和外部系统接起来 |
| LangGraph | 状态机 + 工作流 graph | 多轮工具调用、可恢复、有分支的长流程 |
| OpenAI Agents SDK | Agent + handoff + guardrails + tracing | 组织多 Agent 协作、工具权限、运行追踪 |
| DSPy | 把 prompt pipeline 写成可优化程序 | 有明确数据集和评估指标的任务，用 optimizer 自动改进 |

**和 AI × Web3 的联系**：

-   AI Framework 管理 prompt / tools / state / eval / trace
    
-   Web3 基础设施管理账户 / 签名 / 合约 / 链上状态
    
-   **框架不能替代 Web3 权限控制**——它可以组织 Agent，不能替用户承担资产风险
    
-   Day 4 的 AgentKit 和 Eliza 都是"框架 + Web3 工具"的组合：Eliza 用 LangChain-like 插件架构，AgentKit 的 Action Layer 就是一套 tool schema
    

* * *

## Web3 向：网络（Network）

**核心理解**：区块链网络不是"存数据的服务器"，而是让**互不信任的参与者对状态变化达成一致**的系统。一笔交易从钱包签名到用户看到结果，要经过：钱包 → RPC → mempool → 区块 → 共识 → 执行 → 确认 → 索引 → 浏览器。

**关键点**：

-   **区块是同步节奏**：状态按区块批量推进，不是每毫秒连续更新
    
-   **共识是信任来源**：用户相信状态，不是因为某家公司说，而是因为网络按规则达成一致
    
-   **测试网 ≠ 主网安全保证**：测试网验证流程，不验证经济机制安全（流动性、攻击动机、用户规模都不同）
    

**网络层级**：

| 层级 | 代表 | 特点 |
| --- | --- | --- |
| L1 主网 | Ethereum | 最高安全性，费用最高，速度最慢 |
| L2（Rollup） | Base、Arbitrum、Optimism | 继承 L1 安全，费用低、速度快，多了桥/提现等待 |
| 测试网 | Sepolia | 资产无价值，用于开发测试 |

**Rollup 两类**：

-   **Optimistic Rollup**（Arbitrum / Base）：默认有效，7 天挑战期，提现慢
    
-   **ZK Rollup**（zkSync / StarkNet）：用零知识证明即时验证，提现快，但开发难度更高
    

**和 AI × Web3 的联系**：

-   AI Agent 执行链上操作必须知道当前在哪条网络——chain id、合约地址、RPC，不能靠模型猜
    
-   Day 4 的"链感知上下文"里要求带 chain id 和 block number，原因就在这：不同链、不同时间的状态完全不同
    
-   AgentKit 默认用 Base（L2）而不是主网：Gas 低 → Agent 高频操作成本可控
    

**今天我用过的**：Sepolia 测试网 → 部署 Hello.sol + Counter.sol；理解了"测试网资产无价值、换了 chain id 余额就不同"

* * *

## Bridge 向：机器支付（Machine Payment）

**核心理解**：Machine Payment 不是"AI 会花钱"，而是让 Agent 之间的支付**可限制、可验证、可追踪**。核心是把"付款意图"和"实际结算"拆开，每一步都有凭证。

**关键点**：

-   **Agent 不能有无限支付能力**：只应拿到具体任务、预算和收款方范围内的支付权限
    
-   **预算先于执行**：没有预算边界就没有安全自动支付——预算最好分多层（全局 / 任务 / 单次上限 / 服务方范围）
    
-   **Quote 必须有过期时间**：价格随时变，过期 quote 不能被 Agent 使用
    
-   **Payment Intent**：表达"用户授权为某个任务付款"，但不等于已经结算——先有意图，再有执行
    

**关键概念**：

| 概念 | 一句话 |
| --- | --- |
| Stablecoin Payment | 稳定币适合机器支付：价格稳定、可编程、合约可验证；但仍要处理 chain id / token address / allowance |
| Budget | 用户授权给 Agent 的最大支出范围；应进入 wallet policy 或 session key，不能只存在聊天里 |
| Quote | 服务方给 Agent 的可执行报价；必须包含价格、有效期、收款地址、交付条件 |
| x402 | 用 HTTP 402 返回付款要求，Agent 像处理 401 登录一样处理付款——读取要求 → 检查预算 → 完成付款 → 重放请求 |
| Micropayment | 高频小额支付；单次上链成本可能超过服务价值，需要 L2 / payment channel / 批量结算 |

# Day 4 阅读笔记｜

> 来源：AI × Web3 Handbook  
> 今日阅读：AI 基础 → 微调（Fine-tuning）；AI 基础 → 推理服务（Inference）；Bridge → 开发栈（Dev Stack）

* * *

## AI 向：微调（Fine-tuning）

**核心理解**：Fine-tuning 是在一个预训练好的大模型基础上，用**特定领域的数据继续训练**，让模型"专精"某一任务或风格。不是从头造模型，而是在巨人肩膀上加装专用模块。

**关键点**：

-   预训练（Pre-training）：用海量通用数据训练出底座模型，成本极高（GPT-4 级别动辄千万美元）
    
-   微调（Fine-tuning）：在底座上用少量任务数据继续优化，成本低得多
    
-   RLHF（从人类反馈强化学习）：ChatGPT 变"有礼貌、会拒绝"的核心技术——人工标注「好回答」去强化模型行为
    
-   LoRA / PEFT：参数高效微调方法，只训练模型的一小部分参数，显著降低算力门槛
    
-   **Fine-tuning vs. RAG 的选择原则**：
    
    -   想改变模型「说话方式/风格/价值观」→ 用 Fine-tuning
        
    -   想让模型「知道最新信息/私有数据」→ 用 RAG
        
    -   两者可以叠加使用
        

**和 Web3 的联系**：有项目在用 Fine-tuning 训练「智能合约审计模型」——在大量历史漏洞代码上微调，让模型更擅长识别 Reentrancy / Integer Overflow 等模式。这是 RAG 做不好的场景（行为特化，不只是知识检索）。

* * *

## AI 向：推理服务（Inference）

**核心理解**：Inference（推理）是指把训练好的模型**实际运行起来服务用户**的过程。训练是"造模型"，推理是"用模型"。推理服务解决的是：如何让模型又快、又便宜、又稳定地跑起来。

**关键点**：

-   推理的核心指标：**延迟**（用户等多久）、**吞吐量**（每秒能处理多少请求）、**成本**（每 1000 token 多少钱）
    
-   调用方式：
    
    -   API 调用（OpenAI / Anthropic / Gemini）：最简单，按 token 计费，不用管底层
        
    -   自托管（vLLM / Ollama / llama.cpp）：自己跑模型，成本可控，但需要硬件
        
-   量化（Quantization）：把模型从 FP32 压缩到 INT4/INT8，牺牲少量精度换取大幅降低内存和速度提升
    
-   **去中心化推理**：Bittensor、Gensyn 等项目尝试用区块链激励分布式节点提供推理算力——这是 AI × Web3 的真实交叉点之一
    

**新问题**：去中心化推理网络如何保证推理结果的正确性？（链上无法直接验证 LLM 输出）→ 这可能用到 ZK Proof 或可信执行环境（TEE）。

* * *

## Bridge 向：开发栈（Dev Stack）

**核心理解**：Dev Stack 是构建一个完整应用时用到的工具链组合。AI × Web3 项目需要同时懂**两套工具**，但不需要精通所有——关键是知道每层用什么工具，遇到时会查。

**AI 开发工具栈**：

| 层级 | 工具 | 用途 |
| --- | --- | --- |
| 模型调用 | OpenAI SDK、Anthropic SDK | API 调用大模型 |
| 编排框架 | LangChain、LlamaIndex | 管理 Prompt、工具调用、记忆 |
| 向量数据库 | Pinecone、Chroma、Weaviate | RAG 的检索层 |
| Agent 框架 | AutoGPT、CrewAI、Eliza | 多步骤 Agent 工作流 |

**Web3 开发工具栈**：

| 层级 | 工具 | 用途 |
| --- | --- | --- |
| 合约语言 | Solidity | 写智能合约 |
| 本地开发 | Hardhat、Foundry | 编译/测试/部署合约 |
| 在线 IDE | Remix | 快速试验合约（我们用过） |
| 前端交互 | ethers.js、wagmi、viem | JS 调用合约、签名、读链上状态 |
| 测试网 | Sepolia（Ethereum）、Mumbai（Polygon） | 免费测试（我们用过） |

**AI × Web3 Bridge 层**：

-   `ethers.js` + `OpenAI SDK` 可以直接在一个 Node.js 脚本里共存——这就是最小的 AI × Web3 集成
    
-   Coinbase AgentKit = 帮你把这两套工具粘合起来的框架，让 Agent 能直接调用链上操作
    
-   关键思路：**AI 负责推理决策，Web3 工具负责链上执行**，中间需要人工确认节点
    

**和前几天的联系**：Day 3 学的 Agent Wallet 是理念，Dev Stack 是实现那个理念的具体工具清单。知道有 `ethers.js` + LangChain 这个组合，就能理解为什么 Agent 能"读链上数据再推理再发交易"。

# Day 3 阅读笔记

> 来源：AI × Web3 Handbook  
> 今日阅读：AI 基础 → Agent、RAG；Web3 基础 → Smart Contract；Bridge → Agent Workflow；Bridge → Agent Wallet

* * *

## AI 向：Agent（智能体）

**核心理解**：Agent 不是"更聪明的 LLM"，而是**能感知环境、制定计划、调用工具、执行动作**的系统。LLM 是 Agent 的"大脑"，但 Agent 比 LLM 多了手脚。

**关键点**：

-   Agent = 感知（Perception）+ 推理（Reasoning）+ 行动（Action）+ 记忆（Memory）
    
-   和纯 LLM 的核心区别：Agent 可以**调用外部工具**（搜索、写文件、发交易），并在结果反馈后继续推理
    
-   Agent 的危险性也来自"能执行"——权限越大，人工确认节点越关键
    
-   常见模式：ReAct（Reason → Act → Observe → Reason…）循环
    

**和我学习的联系**：这个对话里的 Learning Agent 就是一个受限 Agent——它能读写仓库文件（工具调用），但涉及推送 / 隐私操作需要我确认（人在回路）。

* * *

## AI 向：RAG（检索增强生成）

**核心理解**：RAG 是给 LLM 加"实时查阅能力"的方案——不是把所有知识塞进模型训练，而是**先检索相关文档片段，再把它们塞进 Prompt，让模型基于这些内容回答**。

**关键点**：

-   RAG 流程：用户提问 → Embedding 向量化 → 向量数据库检索相似片段 → 拼入 Prompt → LLM 生成答案
    
-   解决的核心问题：LLM 训练数据有截止日期 + 私有知识库无法微调训练
    
-   RAG vs. 微调（Fine-tuning）：RAG 适合"知识经常更新"的场景；微调适合"改变模型行为风格"的场景
    
-   检索质量决定回答质量：垃圾进 → 垃圾出，向量检索的召回率是关键
    

**和 Agent 的联系**：Agent 的"外部记忆"模块经常就是 RAG——Agent 需要查某个文档时，不是靠训练里的记忆，而是实时检索后再推理。这解释了为什么 Agent 可以"知道"它从未见过的内容。

* * *

## Web3 向：Smart Contract（智能合约）

**核心理解**：Smart Contract 是**部署在区块链上的代码**，一旦满足条件自动执行，无需第三方介入。“代码即法律”——逻辑写在链上，任何人都能验证，任何人都无法单方面篡改。

**关键点**：

-   合约部署后地址固定，逻辑不可更改（除非合约本身内置升级机制）
    
-   执行消耗 Gas（链上算力费用），由调用方支付
    
-   合约漏洞 = 永久风险：代码有 Bug 不能"打补丁"，只能新部署或事前审计
    
-   典型场景：ERC-20 代币、NFT（ERC-721）、DeFi 协议（借贷 / 兑换）
    

**和昨天的联系**：昨天学的钱包是"签名工具"，Smart Contract 是"被签名调用的对象"——钱包签了一笔交易 → 合约收到调用 → 自动执行链上逻辑，这条链现在完整了。

* * *

## Bridge 向：Agent Workflow（Agent 工作流）

**核心理解**：Agent Workflow 是把一个复杂任务拆解为**多步骤、可回溯的执行流程**。Agent 不是一次性给答案，而是循环"想→做→看结果→再想"。

**关键点**：

-   单步 LLM call vs. 多步 Workflow：后者能处理依赖关系、失败重试、条件分支
    
-   工具调用（Tool Call）是 Workflow 的核心动作单元：每次调用都有入参、出参、可审计
    
-   人工检查点（Human-in-the-loop）可以插入任意步骤，不必只在最后
    
-   好的 Workflow 设计原则：最小权限、每步可观测、失败有降级路径
    

**新问题**：Agent Workflow 出错时，如何判断是"大脑（LLM）推理错了"还是"工具调用失败了"？（值得后续深入）

* * *

## Bridge 向：Agent Wallet（Agent 控制钱包）

**核心理解**：Agent Wallet 是让 AI Agent **代替或协助人管理链上签名**的机制。这是 AI × Web3 最核心、也最危险的交叉点。

**关键点**：

-   Agent 控钱包 = Agent 可以发起链上交易 → 后果不可逆
    
-   实现方式：MPC（多方计算）/ 智能合约钱包（Account Abstraction）/ 策略规则引擎
    
-   **受限原则**：签名前必须有人工确认 + 交易规则白名单 + 金额上限 + 紧急暂停机制
    
-   AA（Account Abstraction）钱包天然支持"自定义验证逻辑"，适合给 Agent 设策略
    

**关键判断**：Agent Wallet 的安全性 ≠ 技术安全性，更多是**授权边界设计**——Agent 应该只能做"你明确允许"的事，而不是"它觉得合理"的事。

* * *

## 今日阅读小结

| 概念 | 一句话 |
| --- | --- |
| Agent | 有工具、有计划、能行动的 LLM 系统；能力越大，人工节点越重要 |
| RAG | 先检索再生成；给 LLM 接上实时/私有知识库的标准方案 |
| Smart Contract | 链上自执行代码，不可改、不可停，逻辑即合约 |
| Agent Workflow | 多步骤循环推理-执行流程，每步可审计、可插入人工检查点 |
| Agent Wallet | Agent 控制签名权限的机制；安全核心是授权边界，而非技术加密 |

# Day 2 阅读笔记｜2026-05-19

> 来源：AI × Web3 Handbook  
> 今日阅读：AI 基础 → Prompt / 上下文；Web3 基础 → 钱包

* * *

## AI 向：Prompt（提示词）

**核心理解**：Prompt 是你对 LLM 的"指令+上下文"组合，决定了模型的输出方向。不是单纯的「问题」，而是一次对话的完整设定。

**关键点**：

-   一个好的 Prompt 通常包含：角色设定（你是谁）+ 任务描述（做什么）+ 约束（怎么做 / 不做什么）+ 示例（可选）
    
-   Prompt 的清晰度直接影响输出质量；模糊的 Prompt → 模糊的答案
    
-   “Prompt Engineering” 本质上是在管理模型的注意力和边界
    

**一个例子**：

-   差的 Prompt：「帮我写代码」
    
-   好的 Prompt：「你是一个 Solidity 初学者教练，帮我用不超过 20 行写一个能存储和读取一个字符串的合约，加上注释，适合在 Remix 上直接部署」
    

**和我学习的联系**：[CLAUDE.md](http://CLAUDE.md) 里对 Agent 的角色设定本身就是一个 Prompt——定义了边界和行为规则。

* * *

## AI 向：上下文（Context Window）

**核心理解**：模型每次只能「看到」有限的文字量，这个范围叫 Context Window（上下文窗口）。超出窗口的内容模型不记得。

**关键点**：

-   Context Window 包含：System Prompt + 对话历史 + 当前输入 + 模型输出
    
-   越长的对话 → 越早的内容越可能被截断或降权
    
-   这就是为什么 Agent 需要「外部记忆」（写入文件/数据库）来跨会话保留状态
    

**陷阱**：  
以为 AI 「记得」上次聊的内容——实际上一旦超出窗口它就不知道了，必须重新喂入。

**和我学习的联系**：这个对话用到 summary + [CLAUDE.md](http://CLAUDE.md) 来给模型补充上下文，就是在绕过 Context Window 限制。

* * *

## Web3 向：钱包（Wallet）

**核心理解**：Web3 钱包不是「存钱的地方」，而是**管理私钥的工具**。链上资产实际在区块链上，钱包只是帮你签名。

**关键点**：

-   钱包 = 私钥管理器 + 签名工具 + 链上交互入口
    
-   助记词（Seed Phrase）→ 私钥 → 公钥 → 地址，这条推导链单向不可逆
    
-   助记词泄露 = 钱包完全失控；助记词丢失 = 无法恢复
    
-   Hot Wallet（热钱包，如 MetaMask）：联网，方便，风险高
    
-   Cold Wallet（冷钱包，如硬件钱包）：离线，安全，适合存放大额资产
    

**和昨天的联系**：EOA 账户的控制权就在于这把私钥 / 助记词；所有签名操作都要通过钱包完成，这是人工确认环节最关键的出口。

**安全原则**：  
助记词永远不要输入任何网站、不要截图、不要存在云端、不要告诉任何人（包括 AI）。
