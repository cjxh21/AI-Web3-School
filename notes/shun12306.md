# 你应该先建立的判断框架

我建议你把 AI 能力改进分成 4 层，从轻到重：

## Level 1: 改 Prompt

做法：

-   改系统提示词
    
-   改 few-shot 示例
    
-   改输出格式要求
    
-   增加约束条件
    

适合：

-   想让模型回答更结构化
    
-   想让模型更像某种角色/语气
    
-   任务本身并不复杂，只是输出不稳定
    

不适合：

-   模型根本不知道领域知识
    
-   模型长期风格不稳定
    
-   任务需要稳定遵守细粒度规则
    
-   任务要在很多样本上保持高一致性
    

* * *

## Level 2: RAG / 外挂知识

做法：

-   给模型检索文档
    
-   喂 handbook / repo / FAQ / docs
    
-   让模型回答时引用外部上下文
    

适合：

-   问题主要是“知识不在模型里”
    
-   内容常更新
    
-   你不想重新训练模型
    
-   你希望可追溯、可替换知识源
    

不适合：

-   模型不是不知道，而是“不会按你想要的方式做”
    
-   风格、格式、决策模式需要长期稳定
    
-   某种输出行为需要被“内化”
    

* * *

## Level 3: Workflow / Agent 约束

做法：

-   工具调用
    
-   多步骤链路
    
-   先规划再执行
    
-   检查器 / 审核器 / guardrails
    

适合：

-   问题是执行流程不稳定
    
-   需要调用工具
    
-   需要验证、回退、人类确认
    
-   需要长期任务组织
    

不适合：

-   单次生成本身质量太差
    
-   任务模式本身没有被模型学会
    

* * *

## Level 4: 微调 / 模型适配

做法：

-   SFT（监督微调）
    
-   LoRA / QLoRA
    
-   PEFT
    

适合：

-   你有稳定、重复、明确的输入→输出模式
    
-   你想长期固化某种行为或风格
    
-   prompt 太长、太脆弱、太贵
    
-   同一任务会高频重复执行
    
-   你有可用的数据集和评估方法
    

不适合：

-   只是偶尔用一次
    
-   数据很少、很乱
    
-   你都还没搞清楚任务定义
    
-   其实用 prompt / RAG 就能解决
    

* * *

# 你现在最该掌握的 3 个概念

## 1) 微调不是“教模型所有知识”

很多初学者误解成：

> 我给它一些资料，它就“学会了”。

更准确地说：

**微调更像是让模型更稳定地执行某类任务模式。**

比如：

-   更稳定地输出课程学习计划
    
-   更稳定地把 Web3 项目描述转成统一 proposal 格式
    
-   更稳定地生成你想要的日报/笔记模板
    
-   更稳定地做某种领域问答风格
    

但如果你想让它“知道最新课程内容”，更适合的通常是：

-   RAG
    
-   检索 handbook / repo / FAQ / docs
    

不是微调。

* * *

## 2) LoRA / PEFT 是什么

你可以这样理解：

### 全量微调

-   改整个模型的大量参数
    
-   成本高
    
-   资源要求高
    
-   小团队/个人通常不优先考虑
    

### PEFT

PEFT = **Parameter-Efficient Fine-Tuning** 意思是：**只调整少量参数，低成本地适配模型**

### LoRA

LoRA 是 PEFT 里最常见的一种方法。 可以简单理解成：

> 不大动原模型，只外挂一小层“适配补丁”，让模型学会你的任务风格。

为什么它流行：

-   显存需求低
    
-   成本低
    
-   训练快
    
-   适合个人实验
    

* * *

## 3) 可控性到底在说什么

“可控性”不是让模型更聪明，而是让它：

-   更稳定
    
-   更一致
    
-   更少跑偏
    
-   更容易预测输出
    
-   更容易满足格式/风格/边界要求
    

比如你做 AI × Web3 学习助手，可控性可能是：

-   输出必须包含“目标 / 输入 / 风险 / 下一步”
    
-   不能把高风险链上动作说成自动可执行
    
-   必须区分“事实、推测、建议”
    
-   必须提醒人工确认钱包签名/转账动作
    

这类需求，本质上都在追求“可控”。

* * *

## 你要能回答这 4 个问题

### Q1. 什么情况只改 prompt 就够了？

答：

-   任务简单
    
-   数据不多
    
-   输出只是偶尔不稳
    
-   更关心开发速度而不是长期固化
    

### Q2. 什么情况应该上 RAG？

答：

-   问题核心是缺知识
    
-   文档频繁更新
    
-   需要引用官方资料
    
-   希望可追溯
    

### Q3. 什么情况该做 workflow / agent？

答：

-   任务有步骤
    
-   要调工具
    
-   有验证和回退
    
-   有人机协作
    

### Q4. 什么情况才值得微调？

答：

-   高频重复任务
    
-   规则稳定
    
-   数据集足够
    
-   有评估指标
    
-   prompt/RAG 已经试过但不够稳

围绕 **Web3 工程化与链上执行** 做了一个最小实践，不再只把链上交互理解成“连接钱包并发交易”，而是把它拆成了一条完整反馈链：**合约开发 → 测试 → 部署 → 钱包签名 → 交易广播 → tx hash → 浏览器验证 → repo 留痕**。

我的主要收获有三点：

**执行边界会随权限和资金规模快速升级**

读取状态和写链完全不是一个风险级别，涉及授权、owner 权限或更复杂账户结构时，必须重新设计权限边界和确认流程。

**真正困难的不只是交互，而是风险控制**

基础链上交互只是入口，实际工程中更难的是链切换、错误恢复、revert 解释、pending 状态管理、地址/ABI 校验，以及如何让用户理解自己正在确认什么。

**链上执行必须形成可验证证据链**

我把测试记录、部署网络、合约地址、交易哈希等内容写入 repo，目的是让链上行为不仅“发生了”，而且“可以被复现和验证”。

今日打卡：AI 在智能钱包和 Web3 合约链上监控中的作用

今天在想一个很有意思的问题：

**AI 在 Web3 里到底最先会落地在哪？**

我现在越来越觉得，除了大家常说的 AI Agent、链上交易助手，**智能钱包**和**链上监控**其实是两个非常现实、也很有价值的方向。

1\. AI 可以让智能钱包从“工具”变成“助手”

传统钱包更像一个“保管资产和签名交易的工具”，用户要自己判断：

-   这个合约能不能签
    

-   这个代币是不是风险项目
    

-   这笔授权是不是过大
    

-   这次跨链/转账路径合不合理
    

但如果加入 AI，钱包就不只是弹出一个“Confirm”按钮，而是能先帮用户做一层理解和判断。

比如：

-   自动解释一笔交易到底会发生什么
    

-   识别可疑授权、钓鱼合约和高风险交互
    

-   根据用户习惯推荐更合理的操作路径
    

-   把复杂的链上行为翻译成人能看懂的话
    

这其实很重要。

因为 Web3 最大的门槛之一，不只是技术复杂，而是**用户经常不知道自己到底签了什么**。

2\. AI 在链上监控里的价值更直接

链上数据是公开的，但公开不代表容易理解。

真正难的是：**海量信息里，哪些是值得立刻关注的异常信号？**

AI 在这里可以做几件很实用的事：

-   监控大额转账和异常资金流动
    

-   追踪“聪明钱”地址的行为模式
    

-   识别项目方、巨鲸、做市商的异常动作
    

-   发现合约被攻击前后的异常调用特征
    

-   对链上事件做自动总结和预警
    

换句话说，AI 不只是“看数据”，而是能帮用户做第一层筛选和解释。

这对投资、风控、安全、研究都很关键。

3\. 我觉得这背后的本质是“降低认知成本”

不管是智能钱包，还是合约链上监控，核心都不是炫技，

而是把原本只有专业用户才能处理的信息，变成普通用户也能理解和使用的能力。

以前链上世界的问题是：

**信息很多，信号很少；工具很多，理解很难。**

AI 最有机会补上的，就是这块。

4\. 我的一个判断

我现在会觉得，AI + Web3 的早期爆发点，不一定先来自“完全自主的 Agent”，

反而更可能来自这些**更贴近真实需求的基础设施层产品**：

-   更聪明的钱包
    

-   更实时的链上预警系统
    

-   更自然的链上数据解释工具
    

-   更个性化的风险提示和资产管理助手
    

这些东西不一定最酷，但很可能最先产生真实用户价值。

今日收获

今天这点思考让我更明确了一件事：

**AI 在 Web3 里的价值，不只是“替用户操作”，更重要的是“帮用户理解、判断和降低风险”。**

如果把这件事做好，Web3 的可用性可能会提升一个层级。

5.31 今日打卡

AI x Web3 方向选择

三个更聚焦的方向，按推荐程度排序：

**1\. RWA / 股票永续风险监控助手**

这是我最推荐的。

现在 tokenized stocks、股票/商品/FX perps 确实在变热，比如 Hyperliquid HIP-3、Ondo Perps、xStocks、Relentless、B3X 等都在做“用稳定币交易传统资产敞口”。这个方向的痛点很明确：

普通用户以为自己在“买股票敞口”，但实际承担的是一组复杂风险：

\- 标的价格风险：TSLA/NVDA/MSTR 等本身波动。

\- 资金费率风险：perp funding 长期侵蚀收益。

\- 清算风险：杠杆、保证金、波动率。

\- 预言机/锚定风险：链上价格和真实市场价格偏离。

\- 交易时段风险：美股休市时链上 24/7 交易，价格发现可能失真。

\- 平台/流动性风险：OI、深度、滑点、异常爆仓。

\- 法规/可用性风险：很多 tokenized equities 对地区有限制。

这个可以做成：

\> **AI RWA Perps Risk Copilot：给用稳定币交易股票/商品永续的人做风险解释和事件监控。**

Demo 可以是：用户输入 `NVDA perpTSLA perp` 或自己的仓位，系统输出：

\- 当前仓位风险等级

\- funding 是否异常

\- perp price 与 TradFi reference price 是否偏离

\- OI/volume 是否异常

\- 是否接近清算

\- 今日相关事件：财报、FOMC、CPI、盘后大波动

\- AI 生成“你现在真正承担了什么风险”的报告

这个方向商业化比泛链上分析更强，因为用户对象清楚：\*\*RWA/perp 交易者、稳定币资产配置用户、DeFi vault 管理者、做 RWA 交易入口的钱包/交易终端\*\*。

**2\. Stablecoin Yield / Vault 风险监控**

如果你更想贴近“普通人用稳定币配置资产”，这个也很好。

不是监控 BTC/ETH 交易，而是监控用户把 USDC/USDT 放进哪些协议、vault、LP、perp vault 后的风险。

可以叫：

\> **Stablecoin Risk Radar：稳定币收益仓位的 AI 风险雷达。**

它监控：

\- vault APY 是否异常升高

\- TVL 是否快速下降

\- 大户是否撤出

\- 策略是否依赖 perp funding / lending / LP 做市

\- 协议合约是否刚升级

\- 管理员权限是否有变化

\- 稳定币是否脱锚

\- 风险解释：这个 APY 到底从哪来

这个商业价值也很清楚：很多人愿意拿稳定币赚 5%-20% APY，但看不懂风险来源。你的工具把“收益”翻译成“风险构成”。

**3\. BTC/ETH 巨鲸与市场结构监控**

这个最容易做，但竞争最大。

\- CEX 净流入/净流出

\- 巨鲸钱包异动

\- ETF/机构相关地址变化

\- 稳定币流入交易所

\- 清算密集区

我的建议是选 **方向 1：RWA / 股票永续风险监控助手**。

原因很简单：它同时满足三点：

1\. **叙事新**：RWA、tokenized stocks、onchain perps 都是当前热点。

2\. **用户痛点明确**：普通用户以为只是“用 USDC 买美股敞口”，但实际上风险复杂。

3\. **两周能做出来**：不需要做完整风控系统，只要做“风险解释 + 事件监控 + 仓位报告”。

黑客松项目可以这样定义：

\> **PerpLens AI：面向稳定币用户的 RWA 永续合约风险分析助手。输入一个股票/商品 perp 或用户仓位，AI 自动解释锚定偏离、资金费率、清算、流动性和宏观事件风险。**

MVP 不要做太大。只支持 3-5 个标的就够，比如：

\- `NVDA`

\- `TSLA`

\- `MSTR`

\- `BTC`

\- `ETH`

\- 可选 `GOLD`

核心 demo 流程：

1\. 用户连接钱包或手动输入仓位`Long NVDA perp, 3x leverage, margin 1000 USDC`

2\. 系统拉取 perp price、reference price、funding、OI、volume、volatility

3\. 系统判断：

\- 是否溢价/折价异常

\- funding 是否偏贵

\- 清算距离

\- 美股是否休市

\- 是否有财报/宏观事件

4\. AI 输出一页风险报告：

\- “你不是在买 NVDA 股票，而是在持有一个 24/7、带杠杆、带 funding、依赖 oracle 的合成敞口。”

\- 当前最大风险是什么

\- 建议降低杠杆/等待开盘/设置止损/避免跨财报夜

这个比“链上调查员”更垂直，也比“交易所风控系统”更适合黑客松。它的商业化路径也更自然：先做 trader 工具，再接钱包、交易终端、RWA perp DEX、DeFi vault。

今日打卡｜Agent Commerce 领域的现状

今天我想梳理一下 **Agent Commerce** 这个方向目前的发展状态。

我理解的 Agent Commerce，不只是“AI 推荐商品”，而是让 Agent 直接参与甚至执行交易流程，比如：搜索商品、比较价格、理解用户偏好、自动下单、售后跟进，甚至在数字资产和链上场景里完成更复杂的决策与执行。

我看到的几个现状：

**1\. Agent Commerce 还在早期，但方向已经很明确。**

现在很多产品还停留在“AI 助手 + 电商导购”阶段，真正能闭环完成交易的 Agent 还不多。原因不只是模型能力问题，更关键的是支付、授权、风控、责任归属这些基础设施还不成熟。

**2\. 真正的壁垒不只是模型，而是交易执行能力。**

大家都能做一个会聊天、会推荐商品的 Agent，但难的是让它真正完成“从需求理解到下单成交”的整条链路。这里面涉及：

-   用户身份与权限确认
    

-   支付与钱包调用
    

-   多平台商品/服务接口接入
    

-   风险控制与异常处理
    

-   售后和可追责机制
    

所以 Agent Commerce 拼的不只是智能，更是 **系统能力 + 商业基础设施能力**。

**3\. Web2 和 Web3 的机会不太一样。**

在 Web2 里，Agent Commerce 更像是平台内购物助手或消费决策助手，机会大，但平台门槛也高。

在 Web3 里，Agent 更容易直接接入钱包、合约和链上协议，因此“可执行性”反而更强，比如自动换币、自动找收益、自动执行策略等。

但问题也很明显：链上交互虽然开放，但用户体验、资产安全和信任成本还是很高。

**4\. 目前最现实的落地方向，可能不是“全自动消费”，而是“半自动决策 + 人类确认”。**

也就是说，短期内更容易成立的模式不是让 Agent 完全替人花钱，而是：

-   Agent 帮用户筛选信息
    

-   Agent 给出购买/交易建议
    

-   Agent 生成执行方案
    

-   最后由用户确认支付或签名
    

这比完全自动化更符合当前市场接受度，也更容易通过风控。

我的一个判断：

Agent Commerce 的核心，不是“AI 会不会卖货”，而是：

**Agent 能不能获得可信的执行权，并在交易链路里承担一部分可验证、可控制的职责。**

谁能先解决“授权、支付、风控、信任”这几个问题，谁才更可能真正把 Agent Commerce 做起来。

今天的收获

我觉得 Agent Commerce 已经不是一个纯概念，它正在从“AI 提建议”走向“AI 参与执行”。

但现在行业还处在从 **assistant（助手）** 到 **agent（执行者）** 的过渡阶段，距离大规模成熟还有一段路。

## 今日学习打卡

今天学习的主题是：**隐私在商业世界中的作用，以及 Web3 隐私功能的应用**。

### 一、隐私在商业世界中的作用

隐私不仅仅是个人信息保护的问题，在商业世界中，它也是一种重要的信任机制和竞争优势。

在传统互联网商业模式中，很多平台依赖用户数据进行广告推荐、用户画像、精准营销和风控。但随着数据泄露、过度收集、滥用个人信息等问题越来越严重，用户、企业和监管机构都开始重新审视数据隐私的重要性。

隐私在商业中的核心作用主要体现在几个方面：

1.  **建立用户信任** 如果一个产品能够明确告诉用户：哪些数据被收集、如何使用、是否会共享，用户会更愿意长期使用这个产品。
    
2.  **降低合规风险** 随着 GDPR、数据安全法、个人信息保护法等法规出现，企业如果不重视隐私保护，可能面临高额罚款和品牌信任危机。
    
3.  **形成差异化竞争力** 在同质化严重的产品市场中，隐私保护可以成为品牌定位的一部分。例如一些通讯、浏览器、钱包、身份类产品，会把“隐私优先”作为核心卖点。
    
4.  **保护商业机密与交易安全** 对企业来说，隐私不仅是保护用户，也包括保护自身的业务数据、客户关系、交易策略和供应链信息。
    

### 二、Web3 中隐私功能的应用

Web3 强调开放、透明、可验证，但这也带来了一个问题：链上数据默认公开。钱包地址、交易记录、资产变化、交互行为都可能被追踪和分析。

因此，Web3 中的隐私功能非常重要，它不是为了“隐藏一切”，而是在透明和隐私之间取得平衡。

常见的 Web3 隐私应用包括：

1.  **隐私交易** 通过零知识证明、混币协议、隐私链等方式，让用户可以在不暴露完整交易细节的情况下完成资产转移。
    
2.  **隐私身份 DID** 用户可以证明自己满足某些条件，比如年龄、资格、信用、会员身份，而不需要暴露完整个人信息。
    
3.  **零知识证明 ZK** ZK 可以实现“证明我知道某个信息，或我满足某个条件，但不透露具体信息”。这在身份验证、合规、投票、信用评分等场景中很有价值。
    
4.  **隐私 DeFi** 在 DeFi 场景中，交易策略、持仓规模、清算风险等信息如果完全公开，可能被 MEV、跟单或攻击利用。隐私功能可以保护用户的金融行为。
    
5.  **企业级 Web3 应用** 企业如果要把供应链、金融结算、数据协作放到链上，通常不会希望所有商业信息完全公开。因此隐私计算、权限控制和可验证计算会成为重要基础设施。
    

### 三、我的理解

今天最大的收获是： **隐私不是 Web3 的反面，而是 Web3 大规模商业化的重要前提。**

如果一个系统只有透明，没有隐私，那么普通用户和企业都很难放心使用。真正有价值的 Web3 隐私能力，应该是在保证可验证、可信任的同时，让用户保留对自己数据和资产行为的控制权。

未来 Web3 的隐私功能可能会在以下方向更重要：

-   链上身份认证
    
-   合规隐私交易
    
-   企业级区块链应用
    
-   隐私 DeFi
    
-   用户数据资产化
    
-   AI 与 Web3 结合中的数据授权与隐私计算
    

### 四、今日总结

今天通过学习隐私在商业世界中的作用，以及 Web3 隐私功能的应用，我认识到：

> 隐私不是单纯的技术问题，而是产品信任、商业模式、用户权益和监管合规共同作用的结果。

在 AI × Web3 的学习方向中，隐私会是一个非常值得持续关注的主题，尤其是在零知识证明、去中心化身份、数据所有权和隐私计算等领域。

1.  Harness Engineering 不只是 Prompt  
    传统 Prompt 更像“一次性提问”，而 Harness 更强调给 Agent 搭建完整的工作环境，包括代码仓库、文档、测试、工具链、执行规则等，让 Agent 能在约束中持续完成复杂任务。
    
2.  Agent 的能力高度依赖上下文工程  
    一个优秀的 Harness，需要提前设计好目录结构、任务拆分、Memory、工具权限和反馈机制。很多时候，Agent 做不好并不是模型不行，而是上下文和工作流设计不合理。
    
3.  TDD 与 Agent 协作的重要性  
    课程中提到，很多 AI Coding Workflow 会结合 Test Driven Development。通过先写测试，再让 Agent 修改代码，可以显著降低代码失控和“幻觉修改”的问题。
    
4.  Harness 更像“AI 的操作系统”  
    它本质上是在解决：如何让 Agent 稳定、可重复、可扩展地完成任务。未来 Agent 工程可能会从“写 Prompt”演化为“设计 AI 工作系统”。

1.  Agent 的记忆不只是“聊天记录”  
    传统大模型更多依赖上下文窗口，而 Agent 会将长期信息、任务状态、用户偏好等内容结构化保存。这样即使会话中断，Agent 依然能够恢复任务进度，形成持续协作能力。
    
2.  记忆通常分为短期记忆与长期记忆  
    短期记忆主要是当前任务上下文，例如最近的对话、工具调用结果、当前目标等；长期记忆则更像知识库或用户档案，可以通过向量数据库、RAG 等方式进行检索。Agent 会根据任务动态召回相关记忆，而不是一次性加载全部信息。
    
3.  Memory 是 Agent “智能工作流”的核心  
    很多复杂任务并不是一次完成，而是“规划 → 执行 → 反思 → 更新记忆 → 再执行”的循环。记忆系统能够帮助 Agent 记录历史决策、失败原因和执行结果，从而不断优化后续行为。
    
4.  Web3 场景下的 Agent Memory 很有想象空间  
    例如链上身份、信誉系统、交易历史、协作记录等，都可以成为 Agent 的长期记忆来源。未来 AI Agent 可能会拥有自己的链上声誉与行为历史，从而形成更加可信的协作网络。

# **AI Agent 入门：Hermes 从 0 到 1 学习笔记**

## **一、角色与目标**

-   **个人 Learning Agent 的定位**：AI × Web3 School 为学员提供了个人学习助理。Learning Agent 并不是替学员完成所有任务，而是帮助学员理解课程、规划每日任务并维护个人学习仓库，同时生成打卡草稿，并提醒学员同步到 WCB/打卡平台。
    
-   **固定入口**：日常使用时建议常用以下链接：Handbook（课程手册）、WCB 课程和 Learning 页面、WCB Agent API 文档、GitHub 官网及 CLI 等 。
    

## **二、初始化流程**

1.  **确认学员画像**：
    
    -   问题涉及学员的 AI 基础、Web3 基础、编程能力、目标方向、每日投入时间和输出语言等 。
        
    -   每轮提问控制在 2 – 3 个问题，收集信息后需复述给学员确认。
        
2.  **引导安装和登录 GitHub CLI**：
    
    -   学员需先注册或登录 GitHub，随后安装 GitHub CLI 并通过 `gh auth login` 完成授权 。
        
    -   重要提示：不要要求学员发送 Personal Access Token、密码或验证码 。
        
3.  **创建个人学习仓库**：
    
    -   推荐仓库名为 `ai-web3-school-cohort-0`，设置为 public，并将本地目录设在 `~/ai-web3-school-cohort-0` 。
        
    -   仓库描述建议为「Personal learning journal and proof‑of‑work for AI × Web3 School」。
        
    -   仓库默认公开，禁止上传任何隐私信息、API key、助记词或未公开的联系信息 。
        
    -   创建仓库后，若 Agent 对本地 repo 做了任何变更（创建、修改笔记或整理反馈），应在学员确认后执行 `git add .` → `git commit` → `git push`，并保持简短提交信息 。
        
4.  **初始化仓库结构**：
    
    -   建议在仓库中创建以下文件和目录：
        
        -   `README.md`：存放 AI × Web3 School 简介、Handbook 链接、WCB 课程链接和隐私提醒 。
            
        -   `profile.md`：记录个人信息与学习画像。
            
        -   `learning-plan.md`：制定学习计划。
            
        -   子目录：`daily/`（每日笔记）、`tasks/`（任务记录）、`experiments/`、`handbook-feedback/`、`hackathon/`、`submissions/`。
