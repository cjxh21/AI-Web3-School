# 📚 Hermes 微信机器人搭建 & 维护全流程笔记

## 一、首次搭建成功流程回顾

1.  **安装 Hermes**
    
    bash
    
    运行
    
    ```
    curl -fsSL https://raw.githubusercontent.com/miurahr/hermes/main/install.sh | bash
    ```
    
2.  **配置微信网关**
    
    bash
    
    运行
    
    ```
    hermes setup
    ```
    
    -   按提示绑定微信，设置你的微信 ID 为 Home Channel
        
    -   配对模式开启，只允许你自己对话
        
3.  **配置 DeepSeek 模型**
    
    bash
    
    运行
    
    ```
    nano ~/.hermes/config.yaml
    ```
    
    写入配置：
    
    yaml
    
    ```
    model:
      provider: deepseek
      api_key: "你的DeepSeek API Key"
      base_url: "https://api.deepseek.com"
    gateways:
      weixin:
        enabled: true
        send_delay: 2  # 防微信限流
    ```
    
    保存退出：`Ctrl+O` → `Enter` → `Ctrl+X`
    
4.  **设置 API Key 永久生效**
    
    bash
    
    运行
    
    ```
    echo 'export DEEPSEEK_API_KEY="你的DeepSeek API Key"' >> ~/.bashrc
    source ~/.bashrc
    ```
    
5.  **启动机器人**
    
    bash
    
    运行
    
    ```
    hermes gateway run
    ```

## 三、常见问题 & 解决方案

### 1\. 安全拦截 / 工具执行报错（Dangerous command requires approval）

-   原因：Hermes 安全机制拦截了 AI 生成的命令（如查天气 / 物流）
    
-   解决：
    
    -   单次放行：微信里发 `/approve`
        
    -   会话内临时放行：发 `/approve session`（重启机器人后失效）
        
    -   避免触发：给机器人发提示词：`优先用内置知识回答，除非我说“帮我查一下”，否则不要调用外部工具`
        

### 2\. 不回复消息（连 “你好” 都不回）

按顺序排查：

1.  终端按 `Ctrl+C` 停掉机器人，再重启：`hermes gateway run`
    
2.  微信里发 `/reset` 重置会话
    
3.  检查 API Key：终端执行 `echo $DEEPSEEK_API_KEY`，看是否能正常显示 Key
    
4.  测试 DeepSeek 接口：
    
    bash
    
    运行
    
    ```
    curl https://api.deepseek.com/v1/chat/completions \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer 你的DeepSeek API Key" \
      -d '{"model": "deepseek-chat", "messages": [{"role": "user", "content": "你好"}]}'
    ```
    
    能返回结果则 Key 有效，否则 Key 失效 / 无额度
    

### 3\. 微信限流 / 连接异常

-   报错：`rate limited`
    
-   解决：
    
    1.  停掉机器人，等待 10-15 分钟再启动
        
    2.  配置文件里的 `send_delay: 2` 已生效，无需额外修改
        
    3.  不要短时间内频繁发消息，一条一条回复
        

### 4\. 配置文件格式错误

-   验证命令：`cat ~/.hermes/config.yaml`
    
-   重点检查：
    
    -   `api_key` 前后无多余空格 / 引号
        
    -   YAML 缩进为 2 个空格，不要用 Tab 键
        
    -   `provider: deepseek` 拼写正确

## 一、LLM 核心原理 拓展解读

### 1\. Token 补充说明

Token 是模型最小处理单元，**不等同于汉字 / 单词**：中文约 1 个 Token 对应 1.5~2 个汉字，英文约 1 个单词对应 1~2 个 Token。

-   计费、上下文窗口、`max_tokens` 均以 Token 为统计单位；
    
-   上下文窗口满额会触发**截断**（优先舍弃早期对话内容），是长对话失效的主要原因。
    

### 2\. LLM 能力短板 延伸

-   精确计算：大数运算、复杂公式、统计汇总易出错，必须依赖计算器 / 代码工具；
    
-   短时记忆局限：仅依赖当前上下文，会话结束后默认丢失所有状态，无法主动留存信息；
    
-   时效性缺陷：训练数据有时间截止点，无法获取实时新闻、动态数据、最新接口文档。
    

### 3\. 典型适用 / 不适用场景

✅ 适合：文案创作、语义理解、对话问答、代码片段生成、逻辑推理、文本总结翻译

❌ 不适合：高精度数值计算、实时数据查询、强合规凭证生成、离线长期状态维护

* * *

## 二、四大控制层面 细化讲解

四者层层配合，共同约束模型行为，是调优提示词、设计流程的基础：

1.  **上下文窗口（工作内存）**
    
    容量决定单次可加载的历史对话、参考文档、任务资料上限。窗口越小，越容易丢失前置信息；大窗口模型（如长文本版 GLM、Claude）适合文档分析、长会话。
    
2.  **系统指令（System Prompt）**
    
    全局规则，**优先级最高**。用于定义角色、输出格式、行为红线、禁用操作，全程生效。
    
    示例：`你是代码工程师，只输出可运行代码，不做多余解释，禁止编造接口信息`。
    
3.  **提示词（User Prompt）**
    
    单次任务指令，传递具体需求、输入数据、约束条件。优秀提示词可大幅降低模型出错概率。
    
4.  **工具调用（Function Calling）**
    
    打破纯文本能力边界，让模型联动外部系统（数据库、接口、代码解释器、搜索引擎），从 “语言生成” 转向 “动作执行”。
    

* * *

## 三、LLM API 调用 实操拓展

### 1\. MaaS 模式优势

模型即服务，无需搭建算力、部署模型、维护环境，开箱即用，是个人 / 小型项目首选。计费模式：**输入 Token + 输出 Token 分开计价**。

### 2\. 核心入参 深度释义

表格

| 参数 | 作用 | 调参建议 |
| --- | --- | --- |
| model | 指定调用模型版本 | 简单任务用轻量模型；复杂推理 / 长文本用旗舰模型 |
| messages | 对话上下文数组 | 固定格式：role(系统/用户/助手) + content(内容)，按对话顺序拼接 |
| temperature | 随机性 / 创造力（0~2） | 0 = 逻辑严谨、结果固定（计算 / 问答）；1 = 创意丰富（文案） |
| max_tokens | 限制模型最大输出长度 | 避免无限生成，根据任务预设上限 |

### 3\. 上手路径建议

1.  优先跑通官方 **Quick Start 示例**（Python/HTTP 请求二选一）；
    
2.  本地编写简单脚本，测试单轮问答、多轮对话；
    
3.  进阶测试：工具调用、超长文本输入、不同温度值的效果差异；
    
4.  备选国内模型：智谱 GLM、通义千问、文心一言（网络更稳定，适配国内场景）。

## 四、Prompt / Workflow / Agent 边界与选型（重点区分）

三者本质是**自动化程度、决策主体、流程自由度**的区别，也是项目选型核心依据。

### 1\. 核心对比

1.  **Prompt（纯提示词）**
    
    -   流程：人提需求 → 模型单次输出 → 人判断结果、做决策
        
    -   特点：无自动化，路径完全由人把控；调试简单、风险极低
        
    -   适用：临时问答、短句创作、单次翻译
        
2.  **Workflow（固定工作流）**
    
    -   流程：预先编排线性 / 分支流程，模型只是流程中的一个节点，**步骤固定、跳转规则写死**
        
    -   特点：可批量自动化，流程可控；无法应对突发情况
        
    -   适用：固定格式文档处理、标准化数据提取、流水线代码生成
        
3.  **Agent（智能代理）**
    
    -   流程：模型自主分析目标 → 规划子任务 → 动态选工具 / 调步骤 → 自主迭代直到完成
        
    -   特点：高度自主、流程动态可变；复杂度高、失控风险大
        
    -   适用：开放式复杂任务、多工具联动、需要动态决策的场景
        

### 2\. 选型口诀（落地参考）

-   一问一答 → 只用 Prompt
    
-   步骤固定、批量执行 → 搭建 Workflow
    
-   目标模糊、步骤未知、需要边做边调整 → 考虑 Agent
    
-   高合规、高确定性场景：**优先拒绝 Agent，能用固定流程就不用自主决策**

## 五、AI 编码工具：价值、局限与使用规范

### 1\. 主流工具定位

-   **Cursor**：IDE 集成，日常编码、改 bug、读项目代码首选
    
-   **Claude Code / Codex CLI**：命令行工具，适合批量脚本、服务端代码、工程化任务
    

### 2\. 核心价值

-   快速生成模板代码、脚手架、测试用例；
    
-   解读陌生框架、老旧代码、第三方库用法；
    
-   辅助排查报错、改写语法、跨语言转换。
    

### 3\. 不可替代的工作（红线）

软件架构设计、核心业务逻辑编写、代码安全审计、性能优化、单元测试设计、线上故障复盘，必须由人工主导。

### 4\. 使用原则

AI 生成代码 = **草稿**，务必逐行审查参数、逻辑、权限、依赖，严禁直接上线。

## 六、AI 输出必验证：风险分类 + 落地校验方案

原有 5 类风险，补充**校验方法**，可直接落地执行：

1.  **事实编造（幻觉）**
    
    校验：关键数据、人物、事件、知识点，交叉检索外部权威来源。
    
2.  **引用造假**
    
    校验：论文、链接、版本号、文档地址，手动点开 / 检索核验，不直接采信。
    
3.  **推理漂移（长文本失效）**
    
    校验：长逻辑链**分段拆解**，逐段核对前提与结论是否一致。
    
4.  **执行越权（Agent 重点风险）**
    
    校验：配置**防护规则 (Guardrails)** + 人工介入节点 (Human-in-the-loop)，限制操作范围。
    
5.  **工具误用**
    
    校验：开启链路追踪 (Tracing)，查看工具调用名称、入参、返回值，排查参数错误。
    

> 通用准则：**模型输出只做参考，不做最终结论**。

* * *

## 七、Agent 核心组件 全解（架构层面）

Agent 是一套完整运行框架，各组件各司其职，构成闭环，按执行链路排序：

1.  **状态管理 State**
    
    全局共享数据池，存储任务进度、中间结果、用户信息，所有节点可读写，保证流程连贯。
    
2.  **长期记忆 Long-term Memory**
    
    跨会话持久化存储（区别于临时上下文），常用向量数据库实现语义召回，实现 “记住历史对话”。
    
3.  **MCP（模型 - 工具连接协议）**
    
    标准化接口，统一 LLM 调用各类外部工具的格式，降低多工具接入成本。
    
4.  **Skills 技能集**
    
    可复用指令 / 动作模板，比如 “数据查询”“文件解析”，支持框架自动调用、动态扩展。
    
5.  **Tool Calling 工具调用**
    
    模型输出结构化 JSON 指令，框架解析后执行工具，再把结果回传给模型，形成循环。
    
6.  **Tracing 链路追踪**
    
    可视化全流程：每一步思考、工具调用、参数、耗时、报错，是 Agent 调试的核心手段。
    
7.  **Guardrails 安全围栏**
    
    输入过滤、输出校验、权限拦截，阻止违规提问、越权操作、敏感内容输出。
    
8.  **Handoff 控制权移交**
    
    子任务完成后，自动把权限 / 状态移交到下一任务节点，实现多子任务串联。
    
9.  **错误恢复机制**
    
    异常兜底：自动重试、回滚操作、降级到人工处理，提升 Agent 稳定性。
    

* * *

## 八、Agent 落地判断：该不该用？（决策清单）

### ✅ 适合搭建 Agent 的场景

1.  任务目标开放，无法提前定义全部执行步骤；
    
2.  必须联动**多个不同工具 / 系统**协作；
    
3.  中间结果会直接改变后续执行策略（动态决策）；
    
4.  需要跨多次会话，持续保留用户状态与历史信息。
    

### ❌ 不建议用 Agent（轻量化方案更优）

1.  单次简单问答、指令明确 → 纯 Prompt
    
2.  流程固定、步骤不变的批量任务 → 脚本 + Workflow
    
3.  强合规、强审计、高风险业务 → 增加人工审核节点
    
4.  要求 100% 数据准确、确定性计算 → 直接用数据库 / 程序计算，绕开 LLM
    

### 重要提醒：警惕「过度 Agent 化」

很多简单任务强行做成 Agent，会增加开发成本、故障风险、调试难度。**能简单就不复杂，能固定流程就不用自主代理**。

| 治理 AI | Governance AI | 用 AI 辅助 / 参与去中心化治理，如提案分析、投票预测、合规校验 | 提升 DAO 治理的效率和公平性，降低治理成本 | DAO 提案的 AI 审核、链上治理的自动化投票分析 |

# AI × Web3 Bridge 笔记（超易懂版）

## 一句话总览

这部分讲的是：**AI 怎么跟区块链打交道**，就像给 AI 装了一套 “链上生存装备”，让它能看懂、能操作、能在 Web3 里干活。

* * *

### 1\. 链感知上下文（Chain-aware Context）

大白话：**给 AI 装一双 “链上眼睛”**

-   以前的 AI：你说 “帮我转 1000 块”，它不知道你钱包里有没有钱，也不知道现在转账手续费贵不贵，纯纯 “盲操作”。
    
-   有了链感知的 AI：它能直接看到你的钱包余额、链上实时手续费、你之前的交易记录，甚至能判断 “你钱不够，转不了”“现在手续费太贵，晚点再转吧”。
    
-   为啥重要？：不然 AI 很容易瞎操作，要么转不出去，要么白白亏手续费。
    

* * *

### 2\. Web3 工具调用（Web3 Tool Use）

大白话：**给 AI 一套 “链上快捷工具包”**

-   以前的 AI：想在链上操作，得写一堆复杂代码，普通人根本搞不定。
    
-   有了工具包的 AI：就像手机里的快捷指令，比如 “一键转账”“一键换币”“一键存币理财”，AI 点一下就能执行，不用你教它写代码。
    
-   举个例子：你说 “帮我把 ETH 换成 USDC”，AI 直接调用兑换工具，自动完成操作，不用你一步步点 APP。
    
-   要注意啥？：这些工具得设好权限，比如只能用固定额度，不然 AI 乱转钱就麻烦了。
    

* * *

### 3\. 智能体工作流（Agent Workflow）

大白话：**给 AI 一套 “链上任务流程图”**

-   以前的 AI：只能做一步操作，比如 “帮我换币”，换完就停了，不会接着帮你存起来。
    
-   有了工作流的 AI：能自己按步骤干完整件事，比如你说 “帮我把 ETH 换成 USDC，再存起来赚利息”，它会：
    
    1.  先看你钱包里的 ETH 够不够
        
    2.  再看手续费贵不贵，太贵就告诉你
        
    3.  然后帮你把 ETH 换成 USDC
        
    4.  最后把 USDC 存到理财里
        
    5.  中间失败了会自动重试，还会告诉你哪里出问题了
        
-   为啥重要？：不用你一步一步指挥，它自己就能把复杂事干完，省超多事。
    

* * *

### 4\. 智能体钱包（Agent Wallet）

大白话：**给 AI 一个 “专属链上钱包”**

-   以前的 AI：没有自己的身份，没法在链上签名、交易，只能帮你查信息，干不了实事。
    
-   有了钱包的 AI：就像给它办了一张 “链上身份证 + 银行卡”，有自己的地址，能帮你签名交易、管理资产，而且你可以设上限，比如每天最多转多少钱，不怕它乱花。
    
-   举个例子：你授权这个钱包只能用 100 块，AI 就算想乱转也转不出去，很安全。
    

* * *

## 最后总结（一句话版）

链感知让 AI “看得懂” 链上情况，工具调用让 AI “能操作” 链上事情，工作流让 AI “会安排” 复杂任务，智能体钱包让 AI“有身份、能签名、能管钱”。

这四个加起来，AI 才能真正在 Web3 里帮你干活，而不是只会说空话。

📘 **今日学习：Web3 基础四件套**

今天一口气串了 Web3 基础的四个章节，最大收获是它们能连成一条线：

**密码学** → 链上身份来自数学证明，不是平台发放的。核心：私钥=控制权，哈希=防篡改，签名=授权证据。

**钱包** → 钱包不是"登录按钮"，是管理链上控制权的入口。每次弹窗都要分清楚：连接（读信息）、签名（证明身份）、发交易（改状态）。

**网络** → 链上系统运行在按区块推进的公开状态机上。L2 降低成本但引入桥和提现等新问题。AI Agent 读链上必须知道自己在哪条网络。

**智能合约** → 合约 = 部署在链上的公开程序。写合约不是写普通函数，是在管理真实资产。ABI 告诉你"能调什么"，不告诉你"安不安全"。

最大的认知转变：Web3 把信任从"相信平台"变成了"验证数学关系"。这既是机会（开放、可组合），也是责任（私钥丢了没人帮你找回）。

## 一、安装 WSL（Linux 子系统）


老师们好，由于本职工作周一周二最忙所以这两天有点掉队今天主要学习的是AI的基础知识

**LLM（大语言模型）**  
通过海量文本训练的“下一个词预测器”，能理解和生成文本，但困在训练数据的时空中，会“幻觉”，没有手脚去触碰真实世界。

**Prompt（提示词）**  
你写给模型的“任务小纸条”，通过设定角色、边界和格式把模糊需求翻译成模型能稳定执行的指令，纸条本身不包含逻辑。

**Workflow（工作流）**  
把 LLM 调用、工具操作和代码判断按固定顺序钉死的“标准作业程序”，稳定高效但路径写死，不懂变通。

**Agent（智能体）**  
以 LLM 为大脑、配上了眼睛和双手的自主决策者，在“观察→思考→行动→再观察”的循环里灵活追求目标，而不仅是机械执行流程。

**Tool Use（工具使用）**  
模型通过生成结构化指令去调用外部 API 的“工具箱”，让它从只会说的理论家变成能查、能算、能操作的实干派。

**AI Coding（AI 编程）**  
以上所有概念在编程领域的复合应用：把代码库当上下文，把读写文件和终端当工具，组装成一个帮你理解项目、写代码、找 bug 的编程搭子。

## **串联记忆：**

## 从**一个人**开始，一层一层给他加装备，最后变成**你的编程搭子**。

* * *

**第一层：裸人（LLM）**  
你雇了一个脑子极好、读过天下书的高材生，但他天生没有手脚，关在密室里，只能靠说话输出。他能写方案、能推理、能解释，但问他“外面天气怎样”，他只能瞎猜——因为他碰不到真实世界，也不知道今天是几号。

**第二层：递纸条（+ Prompt）**  
每次找他干活，你得递一张纸条，写清楚“你是谁、要干啥、什么格式、别瞎编”。纸条写得越细，他回答越靠谱。纸条本身没魔法，只是沟通界面。

**第三层：固定流水线（+ Workflow）**  
简单重复的事，你直接给他订一套标准作业流程：先做A，再做B，最后做C。他不用动脑子，照着跑就行。稳定高效，但不会拐弯。

**第四层：装上手脚（+ Tool Use）**  
你觉得光说不练不行，给他接上了外部工具——能搜网页的线、能执行命令的终端、能读写文件的手。他终于能碰到真实世界了：查资料、跑代码、读文件，不再只是动嘴。

**第五层：自主决策（= Agent）**  
有了手和眼睛，你不必再一步步指挥他了。你说“把这事办成”，他自己观察情况、选工具、看结果、发现错了就拐弯重来。他不是在执行流程，而是在追求目标。这就是 Agent——在循环里自己决策的高材生。

**第六层：编程搭子（= AI Coding）**  
把上面所有装备塞进一个编程环境里：他能看见你的代码库（Context），能读文件、写文件、跑终端（Tool Use），你递纸条说“修一下登录的 bug”（Prompt），他就自己查、自己改、自己验证，全程像 Agent 一样自主规划。这一整套组合拳，就叫 AI Coding。
