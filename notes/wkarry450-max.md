# AI × Web3 最小实践笔记

## 目标

这次实践要完成三个最小动作：

完成一次 AI 工具实践；  
完成一次测试网交互；  
完成一个 AI × Web3 的交叉实验。

重点不是做复杂项目，而是先跑通一个完整闭环。

## 一、AI 工具实践

AI 工具不要只用来聊天，而是要用来解决一个具体问题。

本次可以让 AI 帮助完成：

解释 Web3 基础概念；  
拆解测试网交互步骤；  
生成操作清单；  
分析交易哈希；  
整理实践复盘。

可用 Prompt：

```
我是 Web3 新手，请你帮我设计一个测试网交互任务。
要求：
1. 不需要真实资金；
2. 步骤简单；
3. 每一步说明目的；
4. 最后给我一个复盘模板。
```

记录重点：

使用了什么 AI 工具；  
问了什么问题；  
AI 给出了什么可执行结果。

## 二、测试网交互

测试网的作用是让用户在不花真实资产的情况下，模拟链上操作。

最小操作流程：

先准备钱包，比如 MetaMask。

然后切换到测试网，比如 Sepolia、Base Sepolia 或其他测试网络。

接着通过水龙头领取测试币。

拿到测试币后，完成一次简单交互。新手可以先做测试网转账，也可以尝试 Mint、Swap 或调用测试合约。

交互完成后，复制交易哈希，到区块浏览器查看交易状态。

需要记录的信息：

钱包地址；  
测试网络；  
领取测试币的数量；  
交互类型；  
交易哈希；  
交易是否成功；  
Gas 消耗。

## 三、AI × Web3 最小交叉实验

这个实验的核心是：让 AI 帮助理解一次真实的链上操作。

做法很简单：

先完成一次测试网交互。

然后把测试网络、交易哈希、操作步骤和交易结果发给 AI。

让 AI 帮你分析这笔交易做了什么，交易哈希代表什么，Gas 是什么，以及这次操作说明你掌握了哪些 Web3 基础概念。

可用 Prompt：

```
这是我完成的一次测试网交互，请你帮我分析：

测试网络：
交易哈希：
交互类型：
交易状态：
Gas 消耗：
我的操作步骤：

请你说明：
1. 这笔交易做了什么；
2. 交易哈希有什么作用；
3. Gas 代表什么；
4. 这次实践说明我掌握了哪些 Web3 基础概念；
5. 下一步可以做什么更进阶的实验。
```

## 四、核心理解

AI 的作用是解释、拆解、分析和总结。

Web3 的作用是提供钱包、测试网、交易哈希、区块浏览器和链上数据。

AI × Web3 的最小结合，就是用 AI 帮助理解和复盘一次链上交互。

这个实验不需要很复杂。只要完成一次测试网操作，并用 AI 把过程解释清楚，就已经形成了一个最小闭环。

## 五、注意事项

不要泄露助记词和私钥。

不要用主钱包随便连接陌生网站。

测试币没有真实价值，只用于练习。

交互前确认网络是否正确。

每次操作都要记录交易哈希。

AI 给出的建议要自己验证，尤其是涉及钱包、合约和资产操作时。

# AI 基础：从 LLM 到 Agent Workflow

## 1\. LLM 到底如何“基于上下文生成”

LLM 本质上不是数据库，也不是逻辑推理机，而是一个**基于上下文预测下一个 token 的模型**。输入给它的所有内容，包括系统指令、用户问题、历史对话、工具返回结果、文件片段，都会被组织成一个上下文，然后模型根据这些内容生成后续 token。

OpenAI 对 tokenizer 的解释是：大语言模型处理的是 token，并学习 token 之间的统计关系，从而生成后续内容。也就是说，LLM 不是直接“理解整篇文章”，而是把输入切成 token 后，在上下文限制内做条件生成。

可以把一次 LLM 调用理解成：

```
输入上下文：
系统指令 + 开发者指令 + 用户消息 + 历史消息 + 工具结果 + 约束

模型内部：
根据上下文计算下一个 token 的概率分布

输出：
一个 token 一个 token 地生成答案、工具调用参数或结构化结果
```

核心点：**LLM 的回答质量强依赖上下文质量**。上下文里没有的信息，它可能猜；上下文里有冲突的信息，它会按指令优先级和概率倾向处理；上下文太长或太乱，会降低稳定性。

* * *

## 2\. 上下文窗口、系统指令、提示词、工具调用分别控制什么

### 上下文窗口：控制“模型能看到多少”

上下文窗口是模型一次调用中能处理的最大 token 数。它包括输入和输出，不只是你发给模型的问题。

它决定：

```
模型当前能看到多少历史信息
能放多少文档、代码、工具结果
输出最多能写多长
长任务中是否需要压缩、摘要、检索
```

常见误区：上下文窗口大，不代表模型一定会更聪明。窗口越大，越要管理信息密度。低价值上下文会稀释注意力。

实践建议：

```
短任务：直接给完整上下文
中等任务：只给相关片段
长任务：用 RAG / 文件搜索 / 摘要记忆
复杂任务：分阶段执行，保留关键状态
```

* * *

### 系统指令：控制“角色、边界、优先级”

系统指令或开发者指令是高优先级约束，用来定义模型应该如何行为。OpenAI 的 Chat Completions 文档中提到，developer messages 是开发者提供的指令，模型应遵循这些指令；在较新的模型中，developer messages 替代了之前的一些 system messages 用法。

系统指令适合放：

```
角色定位：你是客服 / 教练 / 代码审查员
安全边界：不能泄露隐私，不能执行危险操作
输出风格：简洁、结构化、中文回答
任务规则：必须引用来源，必须先验证再总结
工具规则：什么时候搜索、什么时候调用数据库
```

不适合放：

```
临时问题
大量业务资料
一次性格式要求
用户上传的长文档
```

系统指令像“宪法”，不是“任务内容”。

* * *

### 提示词 Prompt：控制“这次要做什么”

Prompt 是用户或应用在当前任务中给模型的具体指令。它控制本轮任务目标、输入材料、输出格式和评价标准。

好 prompt 通常包含：

```
任务目标：你要完成什么
背景信息：为什么做
输入材料：依据什么做
输出格式：表格 / JSON / 分点 / 代码
质量标准：准确、可执行、不要编造
限制条件：字数、语气、语言、受众
```

弱 prompt：

```
帮我优化一下
```

强 prompt：

```
请把下面这段产品介绍改成适合官网首页的中文文案。
目标用户是没有技术背景的小企业主。
语气要可信、简洁，不要夸张。
输出 3 个版本，每个版本 80 字以内。
```

Prompt 控制的是**当前任务行为**，系统指令控制的是**长期行为边界**。

* * *

### 工具调用：控制“模型能做什么动作”

工具调用让 LLM 不只生成文字，还能调用外部能力，例如搜索网页、查数据库、读文件、发邮件、运行代码、调用 API。

OpenAI API 文档中说明，模型可以生成 tool calls，例如 function calls；同时也提醒函数参数是模型生成的，可能不总是合法 JSON，也可能包含未在 schema 中定义的参数，因此调用前要在代码中验证。

工具调用的本质流程：

```
用户提出任务
LLM 判断是否需要工具
LLM 生成工具名和参数
系统执行工具
工具返回结果
LLM 基于结果继续生成答案
```

注意：**LLM 不是真正执行工具的一方**。它只是提出“我想调用什么工具、参数是什么”。真正执行的是外部运行时。

工程上必须做：

```
参数校验
权限控制
危险操作确认
工具结果截断与摘要
失败重试
日志记录
```

## 3\. LLM API 的 model / messages / temperature / max\_tokens

以常见聊天式 API 为例，一次调用通常有几个关键参数。

### model：选择“大脑”

`model` 决定使用哪个模型。不同模型在能力、速度、上下文窗口、价格、工具调用能力、多模态能力上不同。OpenAI API 文档也说明，不同模型有不同能力、性能特征和价格。

选择模型时看：

```
任务难度：简单分类还是复杂推理
输入类型：文本 / 图片 / 音频 / 代码
上下文长度：是否要读长文档
延迟要求：实时对话还是后台任务
成本限制：单次调用预算
可靠性：是否需要稳定结构化输出
```

* * *

### messages：放入上下文的对话结构

`messages` 是模型看到的对话内容，通常包含不同 role：

```
[
  {"role": "system", "content": "你是一个严谨的学习教练。"},
  {"role": "user", "content": "解释什么是上下文窗口。"}
]
```

常见 role：

```
system / developer：高优先级规则
user：用户请求
assistant：模型历史回答
tool：工具返回结果
```

messages 不是普通字符串拼接，而是带角色结构的上下文。

* * *

### temperature：控制随机性

`temperature` 越低，输出越稳定、保守、可重复；越高，输出越发散、有创造性。

建议：

```
0 - 0.3：事实问答、代码修复、结构化抽取
0.4 - 0.7：教学解释、产品文案、一般写作
0.8+：创意发散、头脑风暴、故事生成
```

但要注意：temperature 不是“智商旋钮”。它控制采样随机性，不直接提升准确率。

* * *

### max\_tokens / max\_output\_tokens：控制输出长度

这个参数限制模型最多生成多少 token。它影响：

```
答案能写多长
成本上限
是否可能中途截断
是否适合生成长文档
```

实践建议：

```
短问答：几百 token
结构化总结：1000-3000 token
长报告：分段生成，不要一次全压
代码任务：按文件或模块拆分
```

## 4\. Prompt、Workflow、Agent 的边界

这是非常关键的一组概念。

### Prompt：一次性指令

Prompt 是“你告诉模型怎么回答”。

例子：

```
请总结这篇文章，输出 5 个要点。
```

特点：

```
单轮或少量轮次
主要依赖语言指令
没有复杂状态
没有自动决策流程
```

适合：

```
改写
总结
翻译
分类
解释概念
生成简单代码
```

* * *

### Workflow：固定流程编排

Workflow 是“你设计一套步骤，让模型按流程参与”。

例子：

```
1. 读取用户需求
2. 提取关键信息
3. 检索知识库
4. 生成初稿
5. 用规则检查
6. 输出最终答案
```

特点：

```
流程由开发者写死或半固定
模型只负责某些节点
可测试、可监控、可复现
适合生产系统
```

典型 workflow：

```
客服问答：分类 -> 检索 -> 回答 -> 置信度判断
代码审查：读取 diff -> 找风险 -> 建议修改 -> 生成评论
写作助手：收集要求 -> 生成大纲 -> 写草稿 -> 自检
```

* * *

### Agent：自主决策的任务执行者

Agent 是“模型可以根据目标，自主决定下一步做什么、调用什么工具、是否继续”。

OpenAI Agents SDK 文档把 agent 定义为配置了 instructions、tools，以及可选 handoffs、guardrails、structured outputs 等运行时行为的 LLM。

Agent 的典型循环：

```
接收目标
理解当前状态
制定下一步
调用工具
观察结果
修正计划
继续执行
直到完成或失败
```

适合：

```
代码库修改
复杂资料调研
多步骤数据分析
自动化运维
长期任务协作
```

风险：

```
成本不可控
路径不可预测
错误会累积
工具误用
权限风险
调试困难
```

一句话区分：

```
Prompt：让模型回答
Workflow：让模型按流程工作
Agent：让模型自己决定怎么工作
```

## 5\. AI Coding / Learning Agent 工具

### Claude Code

Claude Code 是 Anthropic 的 agentic coding system。官方介绍称它可以读取代码库、跨文件修改、运行测试并交付提交后的代码。

它的核心价值不是“补全一行代码”，而是：

```
理解整个代码库
执行多文件修改
运行命令和测试
解释复杂代码
处理 git 工作流
```

适合场景：

```
重构模块
修 bug
写测试
理解陌生项目
生成 PR
把自然语言需求转成代码变更
```

使用心法：

```
先让它读代码，不要马上改
让它提出计划
要求它列出会改哪些文件
小步提交
每一步跑测试
最终人工 review diff
```

## 6\. AI 输出验证：不要直接相信模型

LLM 输出有几个天然问题：

```
可能编造
可能遗漏
可能格式错误
可能引用不存在来源
可能工具参数错误
可能在复杂任务中前后矛盾
```

所以工程上要做验证。

### 常见验证层

```
格式验证：是不是合法 JSON / Markdown / SQL
Schema 验证：字段是否完整、类型是否正确
事实验证：是否有来源支持
业务规则验证：是否符合公司规则
安全验证：是否含敏感信息、危险操作
执行验证：代码能否运行，测试是否通过
人工验证：关键节点 human review
```

### 输出验证策略

```
低风险任务：模型自检 + 格式校验
中风险任务：规则校验 + 检索引用
高风险任务：人工确认 + 审计日志
生产任务：自动测试 + 回滚机制
```

重要原则：**模型负责提出答案，系统负责验证答案。**

## 7\. Tracing：让 Agent 行为可观察

Tracing 是 agent workflow 的“黑匣子记录仪”。OpenAI Agents SDK 文档说明，tracing 会收集一次 agent run 中的 LLM 生成、工具调用、handoffs、guardrails 和自定义事件，并可用于调试、可视化和监控。

Tracing 应记录：

```
用户输入
模型选择
prompt / messages 摘要
工具调用名称
工具调用参数
工具返回结果
中间推理状态摘要
错误信息
重试次数
最终输出
人工接管节点
```

Tracing 的价值：

```
定位错误来自哪里
分析成本和延迟
发现失败模式
评估 agent 是否越权
优化 prompt 和工具设计
为后续评测积累数据
```

没有 tracing 的 agent 系统，很难进入生产环境。

* * *

## 8\. Guardrails：给 Agent 装护栏

Guardrails 是输入输出和行为层面的约束。OpenAI Agents SDK 文档提到，guardrails 可以并行运行输入验证和安全检查，并在不通过时快速失败。

常见 guardrails：

```
输入护栏：
- 是否包含敏感信息
- 是否越权请求
- 是否恶意 prompt injection
- 是否任务范围过大

输出护栏：
- 是否泄露隐私
- 是否包含危险建议
- 是否格式不合规
- 是否缺少引用
- 是否违反业务规则

工具护栏：
- 是否允许调用该工具
- 参数是否安全
- 是否需要人工确认
- 是否超出预算
```

典型例子：

```
用户：删除所有数据库备份
Agent：必须触发人工确认，不能直接执行

用户：总结客户邮件
Agent：可以读邮件，但不能把隐私发给外部 API

用户：修改支付逻辑
Agent：可以提出 diff，但必须要求人类 review 后合并
```

* * *

## 9\. Handoff：把任务交给更合适的 Agent 或人

Handoff 是任务转交机制。OpenAI Agents SDK 文档提到，handoffs 可以让 agent 把特定任务委派给其他 agent。

常见 handoff 类型：

```
Agent -> Agent：
客服 agent 转给退款 agent
代码 agent 转给测试 agent
研究 agent 转给写作 agent

Agent -> Human：
低置信度时转人工
高风险操作前转人工
用户投诉时转人工
合规问题转专家
```

好的 handoff 需要带上上下文：

```
任务目标
已完成步骤
当前状态
失败原因
相关证据
建议下一步
```

差的 handoff：

```
“我处理不了，你来吧。”
```

好的 handoff：

```
“用户想退款。订单号是 X。已查到订单超过 30 天，但用户声称商品未拆封。政策 A 允许主管例外审批。建议转人工客服确认是否批准。”
```

* * *

## 10\. 错误恢复：Agent 不可能一次全对

Agent workflow 必须默认会失败。

常见失败：

```
工具调用失败
网络超时
参数格式错
检索结果为空
模型输出跑偏
代码测试失败
权限不足
用户需求不清
上下文过长
```

错误恢复策略：

```
重试：适合临时网络错误
降级：复杂模型失败时用简单流程
澄清：需求缺失时问用户
回滚：代码或数据修改失败时恢复
中止：高风险或不确定时停止
转人工：超过自动处理能力
```

成熟的 agent 不应该假装永远成功，而应该能说：

```
我完成了 A 和 B。
C 失败了，原因是权限不足。
我没有继续执行危险操作。
下一步需要你确认或提供权限。
```

* * *

## 11\. Human-in-the-loop：人在关键节点控制方向

Human-in-the-loop 不是“AI 不够强”的补丁，而是生产系统的必要设计。

适合人工介入的节点：

```
高金额交易
删除、发送、发布、合并代码
法律、医疗、金融建议
低置信度判断
用户情绪强烈
模型多次失败
权限升级
```

常见模式：

```
Approve：人类批准后执行
Review：AI 生成，人类审核
Edit：人类修改 AI 草稿
Escalate：AI 转交给专家
Override：人类否决 AI 决策
```

你的目标不是让 AI 完全替代人，而是：

```
AI 做高频、低风险、可验证的部分
人做低频、高风险、高判断力的部分
```

* * *

## 12\. 一个完整 Agent Workflow 示例

以“AI 帮你修改代码 bug”为例：

```
1. 用户描述 bug
2. Agent 读取相关文件
3. Agent 总结当前代码逻辑
4. Agent 提出修复计划
5. Guardrail 检查：是否会改敏感文件

```

关键不是“让 AI 写代码”，而是建立：

```
可观察
可验证
可回滚
可接管
可评估
```

的工作流。

## 1\. 账户：链上的“身份”

在以太坊等公链里，账户通常分两类：

**外部账户 EOA**  
也就是普通用户账户，由私钥控制。你常见的钱包地址就是这种账户，例如：

```
0xA1b2...9F3c
```

它可以发起交易、转账、调用合约。

**合约账户 Contract Account**  
也就是部署在链上的智能合约地址。它没有私钥，不能主动发起操作，只能被外部账户或其他合约调用。

简单说：

```
人控制的钱包地址 = 外部账户
代码控制的地址 = 合约账户
```

* * *

## 2\. 钱包：管理账户和私钥的工具

钱包不是“装币的软件”，更准确地说，它是：

```
管理私钥、生成地址、发起签名、发送交易的工具
```

常见的钱包如 MetaMask、Rabby、OKX Wallet、Phantom 等。

你钱包里看到的资产，其实并不“存放在钱包里”。资产状态记录在链上，钱包只是帮你读取和操作这些链上数据。

钱包最核心的是：

```
助记词 / 私钥 → 控制账户 → 签名交易
```

谁掌握私钥，谁就能控制这个账户。

* * *

## 3\. 签名：证明“这是我授权的操作”

当你想进行链上操作时，比如转账、Mint NFT、Swap 代币、授权合约使用你的 USDT，钱包会让你签名。

签名的作用是证明：

```
这笔操作确实由这个账户的私钥授权
```

注意，签名不一定都是交易。

有两种常见签名：

**消息签名**  
比如登录某个 Web3 网站时，它让你签一段消息，证明你拥有这个钱包地址。通常不花 Gas，也不直接改变链上状态。

**交易签名**  
比如转账、Swap、Mint、Approve，这类签名会生成一笔链上交易，通常需要支付 Gas，并可能改变链上状态。

你要特别小心签名内容。很多盗币行为就是诱导用户签恶意授权或恶意交易。

* * *

## 4\. 交易：把操作提交到链上执行

交易是链上操作的基本单位。

一笔交易可能是：

```
转 ETH
转 ERC-20 代币
调用智能合约
部署智能合约
授权某个合约使用你的代币
参与 DeFi、NFT、DAO 操作
```

一笔交易通常包含：

```
from：谁发起
to：发给谁 / 调用哪个合约
value：转多少原生币
data：调用合约时携带的指令数据
gas：愿意支付的执行费用
nonce：账户发出的第几笔交易
signature：签名证明
```

例如，你用 Uniswap swap ETH 换 USDC，本质上不是“点了一下按钮”，而是：

```
你的账户 → 签名一笔交易 → 调用 Uniswap 合约 → 合约执行兑换逻辑 → 链上状态更新
```

* * *

## 5\. Gas：链上操作的手续费

Gas 是你让区块链节点执行计算和存储操作时支付的费用。

可以理解成：

```
Gas = 链上计算资源费
```

不同操作消耗不同 Gas：

```
普通转账：较少 Gas
转 ERC-20：更多一点
Swap：更多
Mint NFT：视合约复杂度而定
部署合约：通常更贵
```

Gas 通常用链的原生币支付：

```
以太坊：ETH
BNB Chain：BNB
Polygon：POL / MATIC
Arbitrum：ETH
Base：ETH
Solana：SOL
```

所以即使你有 USDT，如果没有链的原生币，也可能无法发交易。

* * *

## 6\. 合约：链上的自动执行程序

智能合约是部署在链上的代码。

它可以定义规则，例如：

```
谁可以 Mint NFT
代币总量是多少
Swap 如何计算价格
质押收益怎么发放
DAO 投票如何统计
借贷清算如何触发
```

用户通常不是直接“操作链”，而是通过钱包调用合约。

例如 Mint NFT：

```
你的钱包地址
→ 签名交易
→ 调用 NFT 合约的 mint() 函数
→ 支付 Gas
→ 合约检查条件
→ 铸造 NFT
→ NFT 归属写入链上
```

## 7\. 测试网：低成本练习环境

测试网是模拟主网的链上环境，用来开发、测试和学习。

例如：

```

```

领取测试币

理解 Gas 和区块浏览器
```

它的作用是让你在不损失真实资产的情况下熟悉链上操作。

Arbiscan
BscScan

Solscan

```

交易哈希 Tx Hash

Gas 消耗

## 一条完整链上操作链

以“用钱包在测试网上 Mint 一个 NFT”为例：

```
1. 创建账户
   ↓
2. 用钱包管理账户
   ↓
3. 连接 DApp
   ↓
4. DApp 构造一笔调用 NFT 合约的交易
   ↓
5. 钱包弹窗显示交易内容
   ↓
6. 用户确认并用私钥签名
   ↓
7. 交易被发送到区块链网络
   ↓
8. 节点验证签名、nonce、Gas 等信息
   ↓
9. 交易进入区块，被矿工/验证者打包
   ↓
10. 合约执行 mint 逻辑
   ↓
11. 链上状态改变：NFT 归属到你的账户
   ↓
12. 扣除 Gas
   ↓
13. 你用区块浏览器查看交易结果
```

简化成一句话：

```
账户由钱包控制，钱包用私钥签名交易，交易支付 Gas 后调用合约，合约执行后改变链上状态，最后可以在区块浏览器中验证结果。
```

这就是大多数 Web3 操作的底层逻辑。

\# AI 应用基础概念笔记

\## 1. LLM（大语言模型）

LLM 是 Large Language Model 的缩写，即大语言模型。它通过学习大量文本数据，具备理解语言、生成文本、总结信息、回答问题、写代码等能力。

常见用途：

\- 问答

\- 写作

\- 翻译

\- 总结

\- 代码生成

\- 信息整理

可以理解为：LLM 是 AI 应用的大脑。

\---

\## 2. Prompt（提示词）

Prompt 是用户给 AI 的指令或问题，用来告诉模型要做什么、怎么做、输出什么格式。

好的 Prompt 通常包含：

\- 任务目标：要 AI 完成什么

\- 背景信息：相关材料或要求

\- 输出格式：例如列表、表格、思维导图

\- 限制条件：如字数、语气、不要编造

可以理解为：Prompt 是和 AI 沟通的指令。

\---

\## 3. Workflow（工作流）

Workflow 指完成任务的一整套步骤流程。它把复杂任务拆成多个环节，让 AI 按顺序执行。

例子：

用户上传音频 → 转文字 → 提取重点 → 生成摘要 → 输出思维导图

可以理解为：Workflow 是 AI 做事的流程图。

\---

\## 4. Agent（智能体）

Agent 是能够根据目标自主完成任务的 AI 系统。它不仅能回答问题，还可以规划步骤、调用工具、处理信息并给出结果。

Agent 的特点：

\- 能理解任务目标

\- 能分解步骤

\- 能调用工具

\- 能根据结果继续调整

可以理解为：Agent 是能帮你办事的 AI 助手。

\---

\## 5. Tool Use（工具调用）

Tool Use 是指 AI 在完成任务时调用外部工具，例如搜索、计算器、数据库、语音识别、图片生成、代码运行等。

例子：

\- 调用搜索工具查资料

\- 调用语音工具转文字

\- 调用思维导图工具生成结构图

\- 调用代码工具运行程序

可以理解为：Tool Use 是 AI 使用外部能力完成任务。

\---

\## 6. AI Coding（AI 编程）

AI Coding 是指利用 AI 辅助编写、修改、解释和调试代码。它可以帮助程序员提高开发效率，也可以帮助初学者理解代码逻辑。

常见用途：

\- 生成代码

\- 修改 Bug

\- 解释代码

\- 写注释

\- 生成测试用例

\- 优化程序结构

可以理解为：AI Coding 是用 AI 辅助写代码。

\---

\# 概念之间的关系

LLM 是基础模型，Prompt 是用户给模型的指令。Workflow 把任务拆成流程，Agent 根据目标执行流程，并在需要时进行 Tool Use。AI Coding 是这些能力在编程场景中的具体应用。

简单关系：

用户需求 → Prompt → LLM 理解任务 → Agent 规划执行 → Tool Use 调用工具 → Workflow 完成任务 → 输出结果

\---

\# 一句话总结

LLM 是 AI 的核心能力，Prompt 是指令，Workflow 是流程，Agent 是执行任务的智能助手，Tool Use 是调用工具的能力，AI Coding 是 AI 在编程中的应用。

# AI与Web3的深度融合：架构、生态与前沿范式

## 一、 核心融合逻辑：生产力与生产关系的重构

-   **AI是顶级生产力**：专注于数据的处理、模型的训练以及高效的内容与决策生成（解决效率与自动化问题）。
    
-   **Web3是核心生产关系**：通过去中心化账本、智能合约和Token经济学，解决资产确权、价值分配、信任机制与数据隐私（解决公平、安全与协作问题）。
    
-   **双向赋能**：AI赋予Web3更聪明的“大脑”，让DApp和链上生态智能化；Web3为AI构建去中心化的“土壤”，打破中心化巨头的算力与数据垄断。
    

* * *

## 二、 关键技术交汇层

### 1\. 去中心化算力网络（AI + DePIN）

-   **长尾算力聚合**：通过区块链网络将全球闲置的GPU（如RTX 4090等消费级显卡到企业级服务器）聚集起来，形成去中心化的超级计算集群。
    
-   **降低供给成本**：相比AWS、Google Cloud等中心化云服务，去中心化算力网络通常能降低50%-70%的AI训练与推理成本。
    
-   **自动结算机制**：利用智能合约，根据算力节点贡献的Slot和工作量证明（PoW）实现毫秒级的Token自动化结算。
    

### 2\. AI Agent（智能体）的链上经济学

-   **原生加密账户**：赋予AI Agent独立的链上身份（DID）和加密钱包。AI不再只是一个工具，而是成为链上经济的独立参与者。
    
-   **Agent to Agent（A2A）交易**：AI Agent之间可以为了完成复杂任务而相互雇佣。例如：A智能体付费给B智能体进行数据清洗，B再付费给C智能体获取算力。
    
-   **预测市场与套利**：AI能以毫秒级的速度分析链上数据和市场情绪，在去中心化预测市场（如Polymarket）或DeFi协议中自主执行长线预测与套利策略。
    

### 3\. 数据资产化与可信确权（DataFi）

-   **AIGC版权追溯**：AI生成的内容（文字、图片、代码）极易被无偿复制。通过将其铸造为链上资产（NFT），可以天然记录创作者、生成时间和流转路径。
    
-   **数据贡献者激励**：高质量的训练数据是AI的核心资产。Web3可以通过构建数据DAO，让用户提供隐私数据并获得Token回报，实现“数据即挖矿”。
    
-   **反深度伪造（Anti-Deepfake）**：利用区块链的不可篡改性，为真实世界的内容打上链上数字水印，帮助用户识别AI伪造的音视频信息。
    

### 4\. 链上隐私计算与零知识证明（Crypto + ZKML）

-   **可信模型验证（ZKML）**：零知识机器学习（Zero-Knowledge Machine Learning）允许模型所有者在不泄露模型具体参数（商业机密）的情况下，向用户证明该输出确实是由特定模型生成的。
    
-   **数据可用不可见**：结合完全同态加密（FHE）和多方安全计算（MPC），AI可以直接在加密数据上进行训练和推理，用户无需担心自身的敏感信息泄露给大模型公司。
    

* * *

## 三、 典型生态应用与赛道

### 1\. 智能化DeFi（Intelligent DeFi）

-   **智能资产管理**：AI驱动的资产管理协议，能够实时监控链上流动性、借贷利率和滑点，自动在Uniswap、Aave等不同协议间调整仓位以获取最大化收益。
    
-   **动态风控与审计**：AI全天候扫描智能合约代码和链上异常交易，预测潜在的闪电贷攻击或黑客行为，实现秒级的主动防范。
    

### 2\. 去中心化AI模型平台

-   **模型开源与协同**：创作者将微调（Fine-tuning）后的模型上传至去中心化网络，用户按次付费调用，收益直接通过区块链路由给模型开发者，形成去中心化的“Hugging Face + App Store”。
    

### 3\. 下一代Web3游戏与元宇宙

-   **高智能NPC**：游戏中的NPC不再依赖死板的对话树，而是接入大模型，拥有独立的性格、记忆与链上资产，能够根据玩家的行为产生长期的动态动态反馈。
    
-   **AI驱动的UGC世界**：玩家通过自然语言指令，直接在链上生成游戏道具、地图和剧情，并自动转化为可交易的NFT。

## 四、 当前面临的系统性挑战

-   **性能与带宽瓶颈**：AI的吞吐量与区块链的TPS（每秒交易量）存在量级差距。如何将高频的AI计算与低频的链上结算完美对齐，是Layer 2和应用链需要解决的问题。
    
-   **验证的复杂性（Verification Dilemma）**：在去中心化网络中，如何作弊检查（即如何证明一个算力节点真的运行了复杂的AI模型，而不是随机返回了一个虚假结果），需要极高成本的密码学证明。
    
-   **安全与黑天鹅风险**：当AI Agent拥有链上资产的完全控制权时，如果其底层大模型受到“提示词注入攻击（Prompt Injection）”或逻辑漏洞利用，可能会导致成千上万美金的加密资产瞬间被黑客诱骗盗走。
