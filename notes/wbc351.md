# 2026-06-05
<!-- DAILY_CHECKIN_2026-06-05_START -->
AI Agent × 链上自主经济

\- Autonomous Agent 爆发：2025-2026 年大量框架（ElizaOS、Virtuals、AI16Z 等）让 AI Agent 能自主管理钱包、发推、交易。核心逻辑：LLM 做决策 → 链上执行 → 社交媒体反馈循环。

\- "Agent 即资产"：链上 Agent 发自己的 Token，用收入回购或分配，形成微型经济体。风险是庞氏化严重，多数 Agent 币归零。

\- 关键瓶颈：Agent 的链上行为无法被用户有效验证，存在"黑箱操作"风险。

DePIN + AI 算力

\- 去中心化算力网络竞争：Render、Akash、[io.net](http://io.net) 等争抢 AI 训练/推理的闲置 GPU 市场。[io.net](http://io.net) 因"虚假算力"争议信誉受损。

\- 推理证明（Proof of Inference）：新方向。如何让链上验证"某个 AI 推理确实发生且结果未被篡改"。Several projects（如 Ritual、Gensyn）在做 ZKML / OPML 方案，但离商用远。

\- 边缘 AI：DePIN 节点跑小模型（SLM）做本地推理，减少延迟和成本，适合物联网场景。

ZKML / 可验证 AI

\- 核心命题：把 AI 模型推理过程用零知识证明封装，让链上合约能验证 AI 输出。

\- 现状：证明生成成本仍高，复杂模型（LLM）几乎无法实时证明。当前落地集中在简单模型（线性回归、小 CNN）。

\- 关键项目：EZKL、Risc Zero、Modulus Labs。

SocialFi + AI

\- AI 生成内容链上确权：用 NFT 给 AI 生成内容（图像、音乐、文本）做版权锚定，但法律认可度低。

\- 去中心化数据标注：通过 Token 激励全球用户做数据标注/RLHF，试图打破 OpenAI/Google 的数据垄断。实际质量参差不齐。

一个观察

当前 AI×Web3 的交集，真正能跑通的只有两件事：

1\. 投机/叙事：Agent 发币、炒概念

2\. 算力租赁：有真实需求（中小团队租不起 AWS A100），但利润率薄

可验证 AI、链上推理、AI DAO 等方向长期有价值，但 2026 年仍处在极早期，基础设施没ready。

一、为什么需要链上 AI Agent

传统 AI 的局限：

\- 你问 ChatGPT"帮我买 100 美元的 ETH"，它只能告诉你"我无法执行交易"。

\- 你让它"持续监控某个 DAO 提案并投票"，它做不到 7×24 小时在线。

链上 Agent 补上的能力：

\- 自主执行：拥有加密钱包，能签名交易、调用合约、转移资产。

\- 持续运行：部署在服务器或去中心化网络里，不需要人类一直盯着。

\- 可验证行为：链上留下完整操作记录， Agent 做了什么、为什么做，都能审计。

生活类比：

以前你雇了个聪明但只能动嘴的顾问（传统 AI），现在给他配了银行卡、公章和 24 小时不打烊的办公室（链上钱包 + 智能合约 + 云服务器），他能自己进货、卖货、记账。

二、链上 Agent 的四层架构

| 层级 | 功能 | 对应组件 |

|--------|--------------------|----------------------------------------------------|

| 感知层 | 读取外部信息 | 链上数据（The Graph、预言机）、社交媒体、API |

| 决策层 | 理解任务并制定计划 | LLM（GPT-4、Claude、开源模型） |

| 执行层 | 与区块链交互 | 钱包（EOA/智能合约钱包）、链上合约调用 |

| 记忆层 | 保存上下文和经验 | 链下数据库（向量数据库）、链上存储（IPFS/Arweave） |

关键难点：执行层的安全

Agent 掌握私钥，意味着它被黑了就是直接丢钱。解法：

\- 智能合约钱包（Safe、ERC-4337）：多签、社交恢复、交易限额。

\- TEE（可信执行环境）：在硬件隔离区里运行 Agent，连服务器管理员都看不到私钥。

\- 人机回环（Human-in-the-loop）：大额操作必须人工确认，小额自动执行。

三、搭建一个最小可用 Agent（概念流程）

不需要写完整代码，理解流程即可：

Step 1：定义目标

"监控 Uniswap 上 ETH/USDC 池子的价格，当 ETH 跌破 1800 美元时，用我钱包里的 USDC 买入 0.1 ETH。"

Step 2：感知层配置

\- 用 The Graph 或 Alchemy API 订阅链上价格变动事件。

\- 每 60 秒查询一次 Chainlink 预言机价格。

Step 3：决策层 Prompt 设计

你是交易 Agent。当前 ETH 价格：{price}。

规则：

\- 如果 price < 1800，执行 buy\_eth(amount=0.1)

\- 如果 wallet\_usdc\_balance < 所需金额，发送警报并停止

\- 每天最多执行 1 次交易

LLM 解析后输出结构化指令：{"action": "buy\_eth", "params": {"amount": 0.1}}

Step 4：执行层签名广播

\- Agent 用私钥构建交易：swapExactTokensForTokens

\- 通过 Bundler（如 Pimlico）提交 ERC-4337 UserOperation

\- 等待链上确认，记录交易哈希

Step 5：记忆层记录

\- 把这次交易的价格、时间、结果存入向量数据库。

\- 下次决策时检索："上次 1800 买入后，3 天内涨了多少？"形成反馈循环。

适合场景: 复杂 DAO 自动化、多策略资管

特点: 成熟的 Agent 注册发现网络，自带钱包和消息协议

────────────────────────────────────────

特点: 低门槛 launch Agent + 代币，强调娱乐和社交属性

适合场景: Meme Agent、社交机器人

适合场景: 特定策略、高频交易、科研

五、当前最实的落地场景

1\. AI 资管

\- 链上执行定投、再平衡、止损。

\- 优势：7×24 小时，不受情绪影响，策略透明可查。

2\. DAO 治理自动化

\- 监控提案、分析内容、按预设规则投票。

\- 优势：解决 DAO 投票率低、专业度不够的问题。

3\. MEV 保护/套利

\- Agent 监测 mempool，用私有 RPC（Flashbots）提交交易，防止被三明治攻击。

\- 注意：纯套利竞争激烈，散户 Agent 基本赚不到。

4\. 内容+收益闭环

\- AI 生成内容 → 自动发布到链上社交平台（如 Lens）→ 收取小费或订阅费。

\- 优势：创作者不用维护多个平台。

六、风险清单

| 风险 | 说明 | 缓解 |

|--------------|---------------------------------|-----------------------------------|

| 私钥泄露 | Agent 代码或服务器被入侵 | TEE、多签钱包、限额 |

| 提示词注入 | 恶意数据让 Agent 执行非预期操作 | 输入过滤、动作白名单 |

| 智能合约漏洞 | Agent 调用的合约有后门 | 只用审计过的协议 |

| 链上成本 | 每笔交易都有 Gas 费 | 用 L2（Base、Arbitrum）、批量执行 |

| 监管 | Agent 自主募资可能触发证券法 | 法律咨询、限制自主募资功能 |

AI×Web3 知识笔记

一、为什么要结合

AI 当下的核心问题：

\- 黑箱化：OpenAI、Google 等巨头控制模型，用户只能调用 API，无法验证模型是否被篡改、数据是否被污染。

\- 数据白嫖：你的社交数据、聊天记录、创作内容被用来训练模型，但你没拿到收益，甚至不知情。

\- 算力垄断：训练大模型需要几千张 H100，全球 90% 以上的高端 AI 算力集中在少数几家公司手里。

Web3 能补上的板：

\- 透明可验证：模型参数、训练数据、推理过程可以用密码学手段验证并留下不可篡改的记录。

\- 所有权回归：你的数据你做主，想卖就卖，不想卖就留着；卖的时候还能要求对方按约定用途使用。

\- 激励对齐：通过代币让数据提供者、算力提供者、模型验证者都能分到钱，而不是平台通吃。

生活类比

AI 像是一个超级学霸，但住在黑箱里考试，你不知道他是不是作弊、是不是偷看了答案。Web3 像是一个透明考场 + 公平交易市场：学霸必须公开答题过程（可验证），而且让出卷人（数据提供者）、租电脑的（算力提供者）、改卷的（验证者）都能分到钱。

二、五大核心赛道

1\. 去中心化算力（Decentralized Compute）

\- 是什么：把全球闲置的 GPU/CPU 连成一个网络，按需分配给 AI 训练或推理任务。

\- 为什么：一台 H100 一小时租金约 2-3 美元，但全球有数百万张游戏显卡和高性能机器在闲置。

\- 生活类比：像滴滴打车，但拉的不是人，是算力。你有空车（闲置显卡）就接单，没空就挂着，系统自动撮合。

2\. 去中心化数据市场（Data Marketplace）

\- 是什么：用户把自己的数据（医疗记录、消费习惯、创作内容）加密后上架，AI 公司付费购买，收益归数据所有者。

\- 为什么：现在 AI 公司免费爬取全网数据训练模型，创作者和数据主体被白嫖。

\- 生活类比：以前你的日记被出版社偷偷印成书卖钱，你一分拿不到。现在你自己开个网店，谁想看你的日记，先付钱，而且对方只能看脱敏版。

3\. ZKML（零知识机器学习）

\- 是什么：证明"AI 模型确实按声称的方式完成了推理"，但不用暴露模型参数和输入数据。

\- 为什么：如果你用一个 AI 医生诊断，你怎么知道它真的用了正确的模型，而不是随便猜的？

\- 生活类比：老师出了一道难题，你请家教辅导。家教不想让你看到他脑中的完整解题过程（怕被偷学），但需要向老师证明他确实辅导了你，而且答案是对的。ZKML 就是这个"证明过程"。

4\. AI Agent + 加密经济（Agentic Economy）

\- 是什么：AI 代理拥有加密钱包，可以自主签订合同、支付费用、赚取收入，在人类不干预的情况下完成任务闭环。

\- 为什么：未来的 AI 不只是聊天机器人，而是能独立执行任务的数字员工。它们需要钱来完成任务（调用 API、租用算力、购买数据）。

\- 生活类比：你雇了一个 24 小时不打烊的虚拟助理，你给它一张带密码的银行卡（加密钱包）。它自己判断要买什么资料、付什么费用，每个月给你报账。

5\. DePIN + AI（去中心化物理基础设施）

\- 是什么：用代币激励搭建 AI 所需的物理基础设施（GPU 集群、数据采集设备、传感器网络）。

\- 为什么：中心化云厂商（AWS、Azure）垄断了 AI 基础设施，价格和可用性由他们说了算。

\- 生活类比：像共享单车，但共享的是 AI 机房。你出一台机器，项目方给你发代币；别人用你的机器训练模型，用代币付费。

三、关键技术栈

区块链层

\- 不可篡改日志：记录谁训练了什么模型、用了什么数据、推理结果是什么。

\- 智能合约：自动执行"数据购买者付款 → 数据提供者收款 → 数据解密授权"的全流程。

密码学层

\- ZK-SNARKs/STARKs：让模型所有者证明"输入 X 时输出确实是 Y"，而不泄露权重。

\- 全同态加密（FHE）：在加密数据上直接做计算，解密后的结果等于明文计算结果。

\- 联邦学习 + 区块链：多个参与方各自在本地训练模型，只把梯度更新上传到链上聚合，原始数据不出本地。

代币经济层

\- 算力代币：激励节点提供 GPU（如 Render、Akash）。

\- 数据代币：激励用户上传高质量数据（如 Ocean Protocol）。

\- 模型代币：让社区共同拥有和治理 AI 模型（如 Bittensor 的子网机制）。

四、代表项目速览

\- Bittensor：去中心化机器学习网络，不同子网专攻不同任务（文本、图像、预测），用 TAO 代币按贡献分配收益。

\- Render Network：去中心化 GPU 渲染网络，后扩展至 AI 推理。

\- Akash Network：去中心化云计算市场，支持以容器形式部署 AI 模型。

\- Ocean Protocol：数据交易市场，支持隐私计算和数据代币化。

\- Worldcoin：Sam Altman 发起，用虹膜扫描做"人格证明"，区分真人与 AI，并探索 UBI（全民基本收入）代币分发。

\- Fetch.ai / Autonolas：AI Agent 框架，让多个 Agent 在链上自主协作完成任务。

五、当前挑战

1\. 链上性能瓶颈

区块链 TPS 有限，大规模模型训练和推理不可能全部上链。

\- 解法：链下计算 + 链上验证（ZKML、Optimistic Rollup）。

2\. 数据隐私悖论

用户想卖数据赚钱，但又不想暴露隐私。

\- 解法：差分隐私、联邦学习、同态加密。

3\. 激励设计困难

怎么判断数据/算力/模型输出的质量？怎么防止刷量？

\- 解法：Bittensor 的 Yuma 共识、预测市场验证、链上 RLHF（人类反馈强化学习）。

4\. 监管不确定

SEC 对代币的定性、数据跨境流动限制、AI 生成内容的版权归属仍不清晰。

1\. AI Agent 自主经济体

\- 模式：个人/矿场把闲置 GPU 挂链上，按需出租给 AI 推理任务

\- 关键：延迟、验证（怎么证明算力真的跑了模型）、支付粒度（按 token 计费）

\- 代表：Akash、Render、io.net、以及新一批专注推理的链

3\. ZKML（零知识机器学习）

\- 解决 AI 黑盒问题：模型说"这是猫"，怎么信它？

\- 应用场景：链上信用评分、AI 预言机、可验证的游戏 NPC

\- 逻辑：Web3 激励降低采集/部署成本，AI 消耗这些数据/算力

5\. On-chain AI 与模型市场

\- 把 AI 模型本身资产化：训练好的模型作为 NFT/Token 交易

\- 模型创作者靠调用费持续盈利，而不是一次性卖 API

\- 挑战：大模型上链不现实，目前集中在小模型、LoRA、Embedding 层

6\. 监管与风险

\- SEC 对 AI 代理发币的态度模糊：如果 AI 自主发起募资，责任算谁的？

\- 深度伪造 + 链上诈骗：AI 生成的 KOL 视频+钓鱼合约组合攻击

\- 各国开始讨论"AI 代理的法律人格"问题

今日思考

AI×Web3 目前最实的落地是算力层（DeAI）和Agent 自动化交易，概念最炫的是 ZKML 和 AI 自治经济体。但大部分项目还在用 Web3 叙事套 AI，真正的 PMF（产品市场契合）很少。观察指标：链上 AI Agent 的真实收益和用户留存，而不是 TVL 和币价。

AI × Web3 专题笔记（新视角：AI智能体+链上自治经济方向）

​

2. 互补逻辑：Web3提供可信底层、数据主权、去中心化资产与价值流转；AI补齐决策、感知、自动化、交互体验短板。

​

3. 核心目标：打造无需人工持续干预的链上自治智能经济。

二、核心载体：Web3 AI Agent（链上智能代理）

\### 三条工程原则

1\. **可信来源要显式标注** — 系统状态、用户输入、检索文档、工具结果分区放置

2\. **上下文要可刷新** — 权限、价格、版本不能长期缓存后继续使用

3\. **记忆要可撤销** — 用户偏好不能变成隐藏权限或永久身份假设

\### 知识节点

\- **Context Window**：窗口大 ≠ 模型完美使用每个细节，"看见了但没抓住重点"是常见问题

\- **Context Engineering**：设计上下文进入模型的方式，目标不是塞满窗口，而是让模型在正确的信息层级里工作

\- **Memory**：跨请求保留的信息（偏好/历史/配置），\*\*不能替代实时授权\*\* — 过去允许的不代表现在仍允许

\- **Knowledge Base**：解决知识过期但不会自动保证正确，至少记录：来源/URL、更新时间、适用版本、是否需要人工复核

\### Context 五层分层

1\. 指令层（系统规则、禁止事项）

2\. 任务层（用户目标、会话参数）

3\. 事实层（链上状态、工具结果、simulation）

4\. 知识层（文档、标准、历史案例）

5\. 记忆层（用户偏好、项目配置）

\> 层级越清楚，越容易做权限控制、prompt injection 防护和审计。

\---

\## 三章关系

\`\`\`

Prompt（接口）→ Context（信息治理）→ LLM（推理引擎）→ Code+Guard+Human（校验确认）→ 执行

\`\`\`

\## 问题与思考

1\. Prompt Injection 在 Web3 风险更大 — Agent 能调钱包签名，被注入可能直接丢资产

2\. Context Engineering 是 Agent 开发核心技能 — 不是写好 prompt 就行，要设计"什么信息以什么身份进入模型"

3\. 结构化输出是 LLM 接入真实系统的桥梁 — 没有 schema 约束，输出就是不可控散文

4\. "交易解释器"和"交易风险摘要"值得做个原型试试

114 ## 四、兼容性注意事项

115

116 Hermes 对 JavaScript 标准的支持与 V8 不完全一致，以下是一些需要注意的点：

117

118 ### 1. 不支持 `eval()` 和 `new Function()`

119

120 Hermes 出于安全和性能考虑，禁用了动态代码执行：

121

122 \`\`\`javascript

123 // ❌ 不支持

124 eval('console.log("hello")');

125 new Function('return 1')();

126

127 // ✅ 替代方案：使用静态逻辑替代动态执行

128 \`\`\`

129

130 ### 2. 部分 ES2020+ 特性需确认支持

131

132 - `Optional chaining?.`）：支持

133 - `Nullish coalescing??`）：支持

134 - `Promise.allSettled`：支持

135 - `Top-level await`：\*\*不支持\*\*（需包在 async 函数中）

136

137 ### 3. `Proxy` 和 `Reflect` 的限制

138

139 Hermes 对 `Proxy` 的支持有限，某些 trap 可能不完全兼容：

140

141 \`\`\`javascript

142 // ⚠️ 可能有问题的用法

143 const proxy = new Proxy(target, {

144 apply: function(target, thisArg, args) {

145 // 某些 trap 行为可能与 V8 不一致

146 }

147 });

148 \`\`\`

149

150 ### 4. `Symbol` 的部分特性

151

152 `Symbol.toPrimitiveSymbol.species` 等内置 Symbol 的支持程度需要逐个测试。

179 ## 六、实践心得

180

181 ### 心得 1：渐进式启用，不要一步到位

182

183 不要在主干分支直接开启 Hermes。建议：

184

185 1. 新建一个 `hermes-test` 分支

186 2. 开启 Hermes 后全面回归测试

187 3. 重点测试边界场景（复杂表单、长列表、动画页面）

188 4. 确认无误后再合并到主干

189

190 ### 心得 2：关注第三方库的兼容性

191

192 这是最容易踩坑的地方。很多 npm 包在设计时假设运行环境是 V8，可能会用到 Hermes

不支持的特性：

193

194 - `eval` / `new Function`（常见于模板引擎、序列化库）

195 - 完整的 `Proxy` 支持（常见于状态管理库）

196 - 某些 `Symbol` 特性（常见于响应式框架）

197

198 \*\*建议\*\*：逐步替换有兼容问题的库，优先选择微信小程序生态内的库。

199

200 ### 心得 3：Hermes 对大体积小程序效果更明显

201

202 如果你的小程序页面多、逻辑复杂、JS 包体积较大（> 500KB），Hermes 的 AOT 编译优

势会非常明显。反之，如果小程序本身很轻量，提升可能不太显著。

203

204 ### 心得 4：字节码不可跨平台

205

206 Hermes 的预编译字节码是平台相关的。这意味着：

207

208 - iOS 和 Android 需要分别编译

209 - 不同 Hermes 版本的字节码不通用

210 - 升级 Hermes 版本后需要重新编译

211

212 ### 心得 5：持续关注官方更新

213

214 Hermes 在微信小程序中的支持还在不断演进，建议：

215

216 - 关注 \[微信开放社区\]([https://developers.weixin.qq.com/community/](https://developers.weixin.qq.com/community/)) 的公告

217 - 定期升级开发者工具和基础库

218 - 参与社区讨论，分享遇到的问题和解决方案

219

220 ### 心得 6：性能优化不止于引擎

221

222 Hermes 是一个引擎层面的优化，但小程序性能是一个系统工程：

223

224 \`\`\`

225 性能优化 = 引擎优化 + 代码质量 + 渲染优化 + 网络优化

226 \`\`\`

227

228 不要把所有希望都寄托在 Hermes 上，同时要做好：

229

230 - 减少不必要的 `setData` 调用

231 - 使用分包加载降低首包体积

232 - 优化图片和静态资源

233 - 合理使用缓存策略

237 ## 七、常见问题 FAQ

238

239 \*\*Q1：开启 Hermes 后小程序白屏了怎么办？\*\*

240

241 A：检查是否有第三方库使用了 `eval` 或 `Proxy` 等不兼容的 API。可以逐个注释页面

和组件，定位问题模块。

242

243 \*\*Q2：Hermes 模式下如何调试？\*\*

244

245 A：微信开发者工具的调试器仍然可用，但部分高级功能（如 CPU Profile）可能受限。

建议结合 `console.log` 和性能面板进行调试。

246

247 \*\*Q3：开启 Hermes 后包体积变大了？\*\*

248

249 A：正常情况下字节码应该比源码小。如果变大，可能是编译配置问题，检查是否同时开

启了不必要的 polyfill。

250

251 \*\*Q4：Hermes 和 Skyline 渲染引擎可以同时使用吗？\*\*

252

253 A：可以。Skyline 是渲染引擎，Hermes 是 JS 引擎，两者互不冲突，可以组合使用以获

得最佳性能。

环境

WSL2 + Ubuntu，Hermes Agent 通过 pip 安装。初次启动后会在 ~/.hermes/ 下自动生成配置目录。

核心配置文件

┌───────────────────────┬────────────────────────────────────────────────────┐

│ 文件 │ 作用 │

├───────────────────────┼────────────────────────────────────────────────────┤

│ ~/.hermes/config.yaml │ 主配置，模型、provider、超时、压缩、显示等所有参数 │

├───────────────────────┼────────────────────────────────────────────────────┤

│ ~/.hermes/.env │ API Key 等敏感信息，和 config.yaml 配合使用 │

└───────────────────────┴────────────────────────────────────────────────────┘

config.yaml 内容很多，但大部分默认值不用动，主要关注顶部的 model 段落就行。

模型切换

默认用的是 DeepSeek，想换成别的模型（比如小米 MiMo 代理），改三个地方就行：

config.yaml 顶部：

model:

default: mimo-v2.5-pro

provider: openai

base\_url: [https://token-plan-sgp.xiaomimimo.com/anthropic](https://token-plan-sgp.xiaomimimo.com/anthropic)

.env 里换成对应的 key：

OPENAI\_API\_KEY=你的key

OPENAI\_BASE\_URL=[https://token-plan-sgp.xiaomimimo.com/anthropic](https://token-plan-sgp.xiaomimimo.com/anthropic)

provider 选 openai 就行，因为 MiMo 代理走的是 OpenAI

兼容接口。大多数第三方代理服务都支持这个格式，所以换模型基本就是换个 base\_url 和 API Key 的事。

其他实用配置

\- max\_turns（agent 段）：单次对话最大轮数，默认 90，够用了。

\- compression：上下文压缩，开了之后长对话不容易爆 context，建议保持默认开启。

\- display.language：改成 zh 可以让界面显示中文。

\- terminal.timeout：命令执行超时，默认 180 秒，跑长脚本可以调大。

\- fallback\_model：配置备用模型，主模型挂了自动切换，config.yaml 底部有注释示例。

踩的坑

\- 改完配置要重启 Hermes 才生效，光改文件没用。

\- .env 里的变量名要和 provider 对应，用 openai provider 就得是 OPENAI\_API\_KEY，写成 DEEPSEEK\_API\_KEY 会读不到。

\- Hermes 的 provider 是 openai 而不是 anthropic，即使底层模型是 Claude，走 OpenAI 兼容接口就选 openai。

\- config.yaml 里有个 _config_version 字段，别手动改，Hermes 升级时会自动处理。

428 ## 十、常见问题

429

430 ### Q1: 国内无法访问 Claude API？

431

432 使用阿里云百炼/百度千帆等国内服务商中转。参考第三节配置。

433

434 ### Q2: 权限弹窗太多？

435

436 设置 `"permissions": { "defaultMode": "auto" }` 自动放行安全操作，或用 `/fewer-permission-prompts` 命令优化。

437

438 ### Q3: Hooks 不生效？

439

440 - 检查 Hook 事件名是否\*\*区分大小写\*\*（如 `PostToolUse` 不是 `posttooluse`）

441 - Hook 默认超时 60 秒，长任务需设置 `timeout`

442 - 退出码 `2` 会阻止操作（PreToolUse 场景）

443

444 ### Q4: MCP 服务器连接失败？

445

446 - 确认 `command` 和 `args` 中的路径正确

447 - 检查环境变量是否通过 `env` 字段正确传递

448 - 运行 `claude doctor` 诊断

449

450 ### Q5: 如何在项目中共享配置？

451

452 将 `.claude/settings.jsonCLAUDE.md.mcp.json` 提交到 Git 仓库，队友 `git clone` 后开箱即用。
