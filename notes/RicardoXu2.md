## AI Agent 时代下的 Web3 应用——成本重构与开发范式演进

摘要

传统区块链项目在落地时，往往面临成本与收益不匹配的困境。高昂的性能损耗、隐私限制、用户教育门槛及开发复杂性，阻碍了大量“中间地带”应用的链上化。随着 Layer 2 等基础设施的成熟，以及 AI Agent 的深度介入，区块链的采用成本正在大幅降低，收益转化变得更加直接。应用构建的核心逻辑正在从“为什么要用区块链”向“为什么不用区块链”发生根本性转变。

### 一、 区块链采用的传统四大成本分析

在评估一个项目是否需要去中心化、不可篡改、透明等特性时，本质上是一场收益与成本的博弈 (ROI Analysis)。过去，以下四类成本往往超出了项目所能获得的收益：

1\. 性能与费用成本 (Performance & Gas Fees)

-   区块链作为去中心化网络，处理速度天生弱于中心化服务器。
    
-   用户在与链交互时需要忍受较高的网络延迟，并支付 Gas 手续费。
    

2\. 隐私限制 (Privacy Constraints)

-   区块链强调“数据公开、状态可验证”，这虽然是信任的基础，但对于部分需要商业机密或数据隔离的应用来说，反而构成了底层限制。
    

3\. 用户教育成本 (User Education Barrier)

-   与 Web2 传统的账密体系不同，Web3 涉及诸多反直觉的复杂概念（如助记词管理、私钥签名机制、Gas 费计算等）。
    
-   极高的学习门槛直接阻断了大量圈外普通用户的流入。
    

4\. 开发成本与工程复杂性 (Development Complexity)

-   学习曲线陡峭：例如 Rust 等智能合约语言虽然强大，但入门极难。
    
-   人力资源短缺：市场上熟练的 Web3 工程师薪资溢价高，招聘困难。
    
-   项目周期拉长：智能合约对安全性的极高要求（不可篡改性）导致开发和测试周期显著变长。
    

### 二、 临界点的偏移：AI Agent 与基础设施的双重驱动

过去，只有强依赖去中心化信任的场景（如纯线上金融/DeFi）才愿意承担上述成本。但现在，天平正在倾斜。

1\. 基础设施带来的“硬成本”下降

-   扩容方案普及：以太坊生态通过 Layer 2 方案持续提升吞吐量并大幅降低了交易费用。
    
-   高性能公链崛起：Solana 等高性能 Layer 1，以及不断升级的底层架构（目前处理速度已可达 15,000 TPS），基本解决了性能瓶颈。
    

2\. AI Agent 带来的“软成本”减负

-   降低用户门槛：AI Agent 可以作为用户的智能交互中介，将复杂的“私钥、签名、Gas”等底层逻辑封装在后台，用自然语言完成交互，极大抹平了用户教育成本。
    
-   降低开发门槛：复杂的代码编写和基础架构搭建可以交由 AI 完成，将开发复杂度大幅降低。
    

核心结论：随着基础设施和 AI Agent 的双重发力，采用区块链的成本正在迅速变轻，而其带来的资产确权和信任红利却更容易兑现。未来的常态将是大量非金融类的普通应用也能无缝集成区块链技术。

### 三、 AI Agent 参与下的 Web3 开发实战指南

在 AI 时代，即使是非技术背景的项目发起人，也能够通过管理 AI Agent 来主导应用开发。这需要引入软件工程中的经典理论来进行管理。

1\. 理论基础：本质复杂性与附属复杂性 (No Silver Bullet)

根据 1987 年《人月神话》作者的理论，软件开发的困难分为两类：

本质复杂性 (Essential Complexity)：决定产品要“解决什么问题”。这涉及到业务逻辑的设计、概念结构的搭建和规格说明。这是人的核心工作。

附属复杂性 (Accidental Complexity)：决定“怎么解决这个问题”。这涉及具体的代码编写、环境配置和语法错误。这是AI Agent 擅长解决的工作。

在实际操作中，人需要专注把控项目的“本质”，而将写代码等“附属”工作外包给 AI。

2\. AI Agent 的执行模式与任务流

在面对长周期或复杂的开发任务时，推荐采用合理的任务拆解机制：

单线长程模式：适合简单的脚本或小型应用，让 AI 一次性贯穿执行。

独立子任务循环 (Rough Loop/Agentic Loop)：面对复杂项目，将整体需求拆解为多个子任务。每次派发一个子任务给一个拥有“全新上下文 (Clean Context)”的 AI Agent 执行。完成后清空记忆，再开启下一个。这能有效避免 AI 在超长对话中出现“幻觉”或逻辑混乱，保证单点代码质量。

3\. 项目管理与验收机制 (Quality Control)

让 AI 写代码，核心在于“验收管理”：

1\. 明确对齐目标：执行前，必须让 AI 充分理解产品需求文档（PRD）和业务决策逻辑。

2\. 自我 Review 机制：交付代码前，强制要求 AI 根据 PRD 进行一轮 Self-Review，比对输出结果与初始目标是否一致，提升代码健壮性。

3\. 实施笔记 (Implementation Notes)：强烈建议让 AI 维护一份开发日志。由于人类的初始设计不可能面面俱到，AI 在执行中（如调用第三方库、选择本地部署方式等）会产生自己的补充决策。通过这份日志，项目方可以获得极佳的“可观察性”，清晰掌控底层发生的技术变动。

### 总结：

AI Agent 并非单纯的代码生成器，而是重塑 Web3 商业模式的催化剂。它不仅在微观层面改变了项目的开发协作流程，更在宏观层面打破了区块链高昂的试错成本，让去中心化技术真正走向大规模应用成为可能。

AI Agent 的执行边界与金融 Harness 机制

摘要

本次直播分享围绕 “Harness Engineering for Financial Intelligence” 展开，核心讨论的是：当人工智能从聊天机器人进一步演化为可执行任务的 Agent 之后，如何让它安全地调用外部工具、使用资金，并在金融场景中完成闭环操作。课件首先指出，LLM 正在从“被更多工具增强”走向“从 chatbot 走向 agent”的阶段；随后进一步提出，Agent 不仅需要工具，更需要钱包和支付能力，才能真正参与人类经济活动。分享中还重点解释了 financial harness、mandate、intent-based constraints、spending validation 等概念，试图建立一套面向金融支付的安全执行框架。结合直播中的演示内容，本文认为，这类机制的意义不仅在于提升自动化效率，更在于为 Agent 进入现实世界提供可控、可审计、可约束的基础设施。

一、问题提出：为什么 Agent 不只是“更聪明的聊天机器人”

课件开头提出了一个很直接的问题：LLMs 现在正在被越来越多的工具增强，从 chatbot 逐步走向 agent，那么它们还能做得更多吗。紧接着，分享者列举了 coding agent 的内部工具，例如 grep、read/write files、web fetch、bash 等，并进一步追问：它能不能生成图片、写 PPT、做深度研究，甚至订酒店、买票、线上购物。这个问题实际上是在强调，Agent 的能力边界不在“会不会说话”，而在“能不能执行”。

结合这次直播的 Demo 来看，老师演示的并不只是“让 AI 生成一张图”，而是让 Agent 通过上下文理解任务、自动生成 Prompt、调用图像模型、完成营销素材产出。也就是说，真正的重点不是某个单点功能，而是 Agent 如何在多个工具之间形成连续动作。这个思路很像金融工作里的流程化作业：真正重要的不是某一个环节，而是整个链条是否能稳定、可控地跑通。

二、从工具到支付：Agent 为什么需要 Wallet

课件里有一句话很关键：Agents Need to be able to use Money。它的逻辑非常直接——AI 已经开始变成执行者，而支付能力决定了它能否主动参与人类的经济活动。换句话说，Agent 如果只能“建议”，不能“购买”或“支付”，它离现实业务其实还差一截。

这一点对我来说特别有触动。因为我原本是从金融视角去理解这些内容的，所以会本能地去想：一个系统真正进入业务流程，最核心的不是它多会分析，而是它有没有权限去完成闭环。对人来说，钱包是参与经济活动的入口；对 Agent 来说，Wallet 可能就是它进入现实世界的入口。没有钱包，Agent 只能停留在“建议层”；有了钱包，它才开始有“执行层”的意义。

三、Prompt Engineering 之后，为什么需要 Harness Engineering

课件把三件事放在了一起：

Prompt engineering 解决的是单轮表达问题，即如何把需求说清楚。

Context engineering 解决的是多轮规划问题，即如何让 AI 在一个会话里保持准确的情境理解。

而 harness engineering 则更进一步，它试图构建一个基础环境，让 Agent 默认就拥有所需上下文，并且不需要人类每次执行都进行监督。也就是说，harness engineering 的目标是把人类反复劳动，转移到系统结构本身。

这个概念其实很像从“手工操作”走向“制度化流程”的过程。以前我们依赖人工盯每一步；现在则希望系统自己知道边界、知道上下文、知道何时可以继续、何时必须停下。对我这种偏金融背景的人来说，这个转变很好理解：就像风控不是每一笔都靠人眼盯，而是依赖规则、系统和授权边界来完成一样，Agent 也需要类似的结构。

四、Permission Ladder：为什么金融权限是最难的一层

课件中提到一个很有意思的概念：Intent and the Permission Ladder。它指出，不同层级的权限风险不一样，但也代表着信任不断向前推进。代码访问已经有比较成熟的 harness，环境访问仍在探索，而金融访问则“largely unaddressed”。

这个判断很现实。因为代码错了可以回滚，环境出问题可以重试，但金融行为一旦落地，往往就很难撤销。也正因为这样，金融权限不是简单加一个“支付按钮”就能解决的。它需要一整套围绕授权、预算、时间窗、范围约束和事后验证的机制。这个层次感，正是这次分享最有价值的地方之一。

五、Financial Harness 的核心：Mandate 不是“付款授权”，而是“任务授权”

课件中对 financial harness 的定义非常清楚：当 Agent 被指派执行一个需要外部付费 API 调用的任务时，它先向人类请求 payment approval，形成 mandate；随后在后续大量 API 调用中，Agent 就不必每一步都反复询问，而是根据这个授权继续执行。风险控制则围绕 mandate 对所有支付进行管理。

这里最重要的不是“允许花钱”，而是“允许完成一个任务”。课件里还明确指出：The authorization object is the task itself, not any individual payment. 也就是说，授权对象是任务本身，而不是单笔支付。

这一点我觉得特别像金融风控里的“授权边界”思维。并不是每一步都单独判断，而是先给出一个明确的任务目标，再在这个目标之下设置预算上限、用途范围、时间窗口。Mandate as a sandbox，本质上就是给 Agent 划一个可执行、可追踪、可回收的空间。

六、Spending Validation：金融场景为什么需要“前置验证”

课件把金融和编程做了一个很精彩的对比。在编程里，典型反馈回路是：写代码、构建、测试、报错、回馈上下文、修复，直到工作通过审查。可是金融不一样，因为错误可能已经结算完成，一旦 wire transfer 出去，就不能轻易撤销。所以 financial harness 需要的是一种发生在 settlement 之前的验证机制。

这部分我觉得是全场最有“金融味”的地方。因为它把“风险控制”从事后处理，前移到了执行之前。课件给出了三层验证思路：

第一层是规则过滤，特点是快、确定、成本低，比如识别黑名单类别、明显超出授权范围的行为，或者重复且相同的交易。

第二层是 intent matching，也就是每笔交易都携带一个 intent ID，判断这笔交易在语义上是否真的符合原始任务目标，支付路径是否合理。

第三层是 model evaluation 或 agent-to-agent supervision。当天然规则和 intent matching 都无法判断时，再由模型来判断行为是合理、幻觉式的，还是带有对抗性的。

从我的理解看，这其实是在模仿传统金融里的“规则—人工—模型”分层风控，只不过这里的对象从人类交易变成了 Agent 交易。

七、结合直播 Demo：Agent 走向“能执行、能付费、能交付”的闭环

直播现场演示的内容和课件是高度一致的。老师通过 Claude Code、Codex、OpenClaw 等工具，展示了如何让 Agent 根据一篇文章自动生成营销图片，并且通过 wallet 和 mandate 机制完成多次调用而不需要每次人工确认。你前面发来的逐字稿里也能看到，他反复强调“无需每次都手动 approve”“一次授权后可以连续调用”“每张图生成大约需要 25 到 30 秒”等细节，这实际上就是 financial harness 在实际工作流里的样子。虽然具体工具名和转写存在不少误识别，但大方向和课件是一致的：Agent 的执行能力，必须和支付能力绑定在一起，才算真正进入可用阶段。

八、结合我的背景：我为什么会对这场分享有感觉

从你的背景来看，你并不是纯技术出身，而是带着金融基础、正在重新理解 AI 和 Web3 的人。这样的人看这场分享，其实会比纯技术背景更容易抓住它的本质。

因为你会天然去问几个问题：

这个东西怎么产生价值？

它怎么完成交易？

它怎么控制风险？

它的授权边界在哪？

它出错之后怎么处理？

这些问题，本质上就是金融思维的问题。

所以当老师讲到 wallet、mandate、budget ceiling、intent scope、time window 的时候，你应该会比别人更容易意识到：这不是单纯的 AI 工程技巧，而是一种“让智能体进入经济系统”的制度设计。

对我来说，最大的感受是：

以前我们讨论 AI，更多是在讨论“能不能回答”；

现在这场分享讨论的是“能不能做成事”。

这中间差的不只是工具，而是执行权、支付权和责任边界。

九、结语

总体来看，这次分享让我对 Agent 的理解更完整了一层。

如果说 Prompt Engineering 解决的是“怎么让 AI 听懂”，那么 Harness Engineering 解决的就是“怎么让 AI 真正做完”。而金融 harness 的关键，不只是让 Agent 会花钱，而是让它在明确的授权边界里、安全地、可审计地花钱。对于未来的 AI Agent 来说，Wallet 可能会像浏览器之于互联网、支付工具之于电商一样，成为它进入现实世界的基础入口。

对我个人而言，这次分享最重要的启发不是某个模型名字，也不是某个 Demo，而是我开始更清楚地看到：AI 和 Web3 真正有交集的地方，不只是“生成内容”，而是“执行交易”。这也让我更确信，接下来继续学习的重点，应该放在 Agent 如何与支付、授权、风控和资产流转结合起来。

## 从VC视角理解AI+Web3创业逻辑：关于融资、团队与个人成长的思考

### 一、课程内容概述

这次直播课主要从VC的视角，系统讲了AI+Web3创业项目在当前环境下应该如何被判断。课程里最核心的几个点包括：项目是否真正落地、团队是否靠谱、商业闭环是否成立、是否具备执行力，以及在AI快速迭代的背景下，项目是否仍然拥有持续存在的价值。

如果说过去Web3行业更强调叙事能力，那么现在行业明显变得更现实了。VC不再只听故事，更关注真实数据、真实用户和真实增长。换句话说，市场已经从“讲得好不好”，转向“做得成不成”。

这一变化其实不只是投资逻辑的变化，也是整个行业成熟度提升后的必然结果。

### 二、我对课程中几个关键观点的理解

**1\. AI正在降低创业门槛，但也在抬高竞争强度**

课程里提到，随着Codex、Gemini等工具的发展，很多原本需要多人协作完成的基础工作，现在一个人就能借助AI完成。这个判断让我印象很深。

以前我总觉得创业首先比的是团队规模、资源多少、启动资金多少，但现在看，真正的变化是：创业的技术门槛在下降，验证速度在变快，竞争反而更激烈了。

这意味着，未来能活下来的项目，不一定是最“豪华”的团队，而可能是最会利用工具、最能快速迭代、最懂行业问题的人。

这对我来说是一个很现实的提醒。因为我自己的背景并不是技术出身，所以我以前会下意识觉得，进入Web3或AI领域，必须先把技术学到很强。但这节课让我意识到，技术当然重要，可真正稀缺的，未必只是代码能力，而是把行业问题看清楚、把需求抓准确、把产品推进落地的能力。

**2\. 叙事很重要，但真正决定成败的是执行**

课程反复强调一个词：execution。

这让我想到很多时候我们容易被“故事感”吸引。一个项目如果叙事够强，很容易让人产生想象空间。但课程里讲得很直白：VC现在越来越务实，光有PPT和概念已经远远不够了。

这个点对我触动很大。因为在很多学习场景里，我们都很容易停留在“我理解了一个概念”的层面，但真正往前走一步，往往是最难的。

\*\*二、 新密码朋克（Neo-Cypherpunk）范式的演进\*\*

传统的密码朋克运动起源于1993年左右，其核心理念是利用密码学工具保护个人隐私免受互联网和权力机构的监控。然而，随着技术普及和全球网络环境的变化，一种被称为“新密码朋克”的理念正在形成。这种新范式具有以下显著特征：

\* \*\*从去中心化极客到普惠大众：\*\* 传统的隐私工具往往面向密码学家和技术专家，而新密码朋克理念强调隐私属于所有人，适合所有技术背景和水平的参与者。

\* \*\*从零和博弈到社区互助：\*\* 区别于早期强调绝对个人主义的隐匿，新理念主张将隐私视为公共品，用社区互助替代单一的竞争关系。

\* \*\*非二元对立的隐私工具栈：\*\* 隐私保护并非要求用户实现100%的绝对隐身，而是一个从0到100的渐进尺度。建设者和用户可以通过构建“个人隐私工具栈（Personal Privacy Stacks）”，根据自身的具体处境，灵活选择如Brave浏览器或Signal通讯软件等替代方案，逐步夺回数据自主权。

\*\*三、 金融化挑战与Web3价值观的重塑\*\*

从经济与金融的视角审视，当前的加密货币行业已经严重偏离了早期的密码朋克精神，表现出强烈的金融激励导向。市场参与者往往过度追逐代币发行和高额的资本回报。尽管资金的流入推动了行业发展，但也导致技术在一定程度上沦为纯粹的金融投机工具。

面对这一现状，行业需要重新校准价值锚点。以太坊提出的“CROPS”原则为这种重塑提供了方向：

\* \*\*C\*\*ensorship resistance（抗审查）

\* \*\*R\*\*eusion/Open source（开源）

\* \*\*P\*\*rivacy（隐私）

\* \*\*S\*\*ecurity（安全）

将系统的控制权交由代码和数学算法，如果缺乏开源和独立性作为保障，仅仅是将权力从传统机构（如银行）的口袋转移到了另一个黑盒中。开源且独立的基础设施才是确保技术不被滥用的核心约束力。此外，链上巨鲸的交易透明度与隐私保护之间的矛盾，本质上也是一个复杂的伦理与金融监管议题，需要根据社会效益与个人权利进行权衡。

\*\*四、 AI Agent的隐私盲区与开发契机\*\*

在AI与Web3的融合过程中，AI Agent的爆发式增长带来了前所未有的挑战。目前，AI市场因大量资金涌入而处于高速且狂热的扩张期，导致多数开发者在追求迭代速度和市场份额时，严重忽视了Agent在链上通信和执行交易过程中的隐私与安全问题。

这种行业现状为非传统技术背景的建设者提供了一个极具价值的切入点。在设计AI结合Web3的应用模型或原型程序时，可以着重考量以下几点：

\* \*\*默认隐私机制：\*\* 借鉴传统金融体系中银行交易的默认隐私设定，为AI Agent的链上交互构建基础的隐私保护协议，以规避潜在的资产风险。

\*\*五、 结论\*\*

在AI与Web3技术交汇的当下，建设者不仅需要理解技术的底层逻辑，更需要建立一种宏观的、包含伦理与经济考量的世界观。摒弃单纯追逐资本回报的短视行为，将开源、隐私、安全等核心理念注入产品设计，尤其是填补当前AI Agent领域的隐私空白，是构建一个更加自由、互助且具备社区韧性的Web3生态的必由之路。

AI与Web3融合语境下的隐私文化重构：基于Neo-Cypherpunk视角的应用实践与产品哲学

摘要：

随着人工智能大模型对海量数据的依赖与Web3网络默认公开特性的碰撞，用户数据隐私正面临前所未有的系统性风险。本文以Neo-Cypherpunk的文化演进为切入点，探讨隐私在AI与Web3融合时代的战略意义。研究指出，隐私已从早期极客的附加选项转变为去中心化应用构建者必须具备的基础素养。文章系统梳理了个人主权、数据最小化等五个核心框架，并针对应用层面的设计逻辑提出了具体的实践路径，以期为AI与Web3复合应用的开发提供兼具理论深度与落地价值的参考。

一、 引言：数字全景监狱与隐私文化的转向

当前技术的演进呈现出显著的双刃剑效应。一方面，AI模型的训练和个性化交互需要汲取庞大的用户数据，极易在无形中构建出一种数字“全景监狱”（Panopticon），使个体的行为被无死角地监控与分析；另一方面，Web3技术虽然在资产所有权上实现了去中心化，但其底层账本数据的默认透明性，使得用户的链上行为与链下身份极易被精准画像并产生强关联。

在此背景下，Peter Farbey等研究者提出的Neo-Cypherpunk理念提供了破局思路。与早期Cypherpunk侧重于纯粹的密码学技术对抗（如加密邮件、匿名网络）不同，Neo-Cypherpunk更强调“技术与文化”的双轮驱动。它不再仅仅将隐私视为一种防御工具，而是将其提升为21世纪数字空间中一项基本的自由表达权利，即为个体保留“做真实自己”的数字空间。

二、 Neo-Cypherpunk的多维文化层次解析

在构建下一代互联网架构时，理解隐私的价值需从多个维度的文化层次进行系统解构：

1\. 个人层面（主权与自治）：隐私的核心落脚点在于“个人主权”（Sovereignty）。这意味着用户必须对其产生的数据、身份标识及交互行为拥有最终且绝对的控制权，这种控制权是原生的，而非受制于中心化平台的暂时授权。

2\. 产品层面（设计即隐私）：隐私不应是产品上线后面临监管压力时的合规补丁，而应贯彻“Privacy by Design”（隐私源于设计）的原则。在项目的初始构思阶段，隐私保护机制即需作为核心架构被嵌入产品逻辑之中。

3\. 社会层面（抗监控与多元化）：在一个趋向于过度数据化的社会体系中，隐私是保护创新自由与多元表达的最后防线，是对抗系统性监控社会的重要基础设施。

三、 核心实践框架：五维修建隐私安全边界

对于涉足AI与Web3领域的开发者而言，将文化理念转化为可执行的产品逻辑至关重要。基于上述理论，可提炼出以下五个核心关联维度，作为构建应用时的参考标尺：

1\. 确立个人主权：重塑数据所有权模型，确保产品在架构上支持用户真正拥有数据，并具备随时撤回、迁移或销毁数据的技术能力。

2\. 践行数据最小化：在数据采集端保持极度克制。仅收集维持核心业务逻辑运行的绝对必要信息，对于非必要数据，需在系统层级采取默认不记录或定期自动清除的机制。

3\. 提升抗监控能力：在底层技术选型上，应积极引入零知识证明（ZKP）、端到端加密及数据混淆等前沿密码学技术，从技术底层阻断第三方对用户行为轨迹的系统性追踪与关联。

4\. 阻断身份暴露风险：警惕链上地址、社交图谱与现实身份的过度耦合带来的衍生风险。在系统设计中，优先采用去中心化身份（DID）、可验证凭证（VC）及伪匿名化机制，建立多层次的身份隔离墙。

5\. 重塑应用设计范式：在产品需求分析阶段，开发者必须进行深度的逆向思考：明确划定哪些数据流、交互记录（特别是与AI的对话及生成内容）必须对外部默认屏蔽，并确保系统支持用户的低调使用模式。

四、 商业实践与产品落地建议

在实际的商业与产品开发闭环中，隐私保护不仅是道德约束，更是提升产品核心竞争力的战略资产。具有高度隐私意识的用户群体，往往具备更高的付费意愿与品牌忠诚度。

首先，在项目管理流程中，应在产品需求文档（PRD）中强制引入“Privacy Checklist”评估环节，以此作为项目进入开发阶段的前置约束条件。其次，针对AI对话、数据投喂等高敏感交互，优先探索数据本地化处理及用户端授权的物理级开关机制。最后，开发者在构思早期的Demo或验证性原型时，就应主动审视潜在的数据泄露节点。通过预演最坏情况下的数据滥用后果，反推系统设计的安全性，从而在产品冷启动阶段即建立起坚实的信任基础。

五、 结语

Neo-Cypherpunk不仅是一场技术层面的演进，更是一种具有深远影响的产品哲学。在构建AI与Web3的生态中，尊重并捍卫用户的数据隐私，是每一位行业建设者不可推卸的责任。唯有将隐私文化深植于代码、协议与商业模式之中，才能真正创造出具备长期主义价值、且被用户广泛信任的去中心化应用。

### **AI Agent 链上自主支付的权限控制与安全边界探索——基于 Cobo Agentic Wallet 的案例分析**

**摘要**

随着人工智能从问答式对话（Chatbot）向具备自主执行能力的智能体（Agent）演进，AI 介入加密经济的核心痛点已由“意图理解”转变为“链上资产的安全执行”。本文基于 Cobo Agentic Wallet 的产品架构，探讨 AI Agent 在操作链上资金时所面临的风控、审计与权限管理挑战，并系统分析 MPC 钱包架构、Pact 授权协议以及 Recipe 技能层在构建可信执行链路中的作用。此套架构为构建客观安全的 Web3 世界观提供了基础设施参考，也为非技术背景下的 AI 与 Web3 结合实践（如自动化交易策略部署）提供了可行路径。

**1\. 引言：Agent 经济的操作层缺失**

在 2026 年的当下，AI 具备了在后台持续处理复杂工作流的能力。然而，由于链上资金具有不可逆性，当 Agent 需要代替人类执行代表意志的消费（如代币转移、合约交互）时，纯粹的自然语言指令缺乏强制约束力。

缺乏可执行、可信任的授权操作层，是目前 Agent 经济发展的最大阻碍。若赋予智能体独立的资金转移权限，一旦大模型产生幻觉或遭遇外部注入攻击，将引发系统性的资金风险。因此，构建一个具备硬性边界约束的基础设施，是实现大规模 Agent 链上应用的前提。

**2\. Agent 链上资金交互的核心风险模型**

在没有策略引擎和审计跟踪的无约束边界下，Agent 动用资金主要面临以下四类典型的失控风险。从金融风控的角度来看，这些风险覆盖了操作风险、授权敞口及长尾系统性风险。

| 风险类别 | 场景描述 | 潜在危害与风控盲区 |
| --- | --- | --- |
| Prompt Injection(提示词注入) | 模型产生幻觉或受到恶意输入影响，导致 Agent 执行未经授权的交易路径。 | 交易意图与最终执行结果脱节，自然语言无法提供强制校验。 |
| Shadow Operations(影子托管/静默操作) | Agent 在主控钱包（如 MPC）外围私自创建 EOA（外部账户），将资金转移至子账户后再进行合约操作。 | 资金脱离主控审计范围（Send Override），人类无法通过常规交易回溯识别资金流向。 |
| Unscoped Authority(无边界授权) | 未对 Agent 设定明确的动用资金上限与权限约束，使其具备对链上资产的无限掌控力。 | 极大的单点风险敞口。一旦私钥泄露或模型失控，可能导致全部资产流失。 |
| Zombie Permissions(僵尸授权) | 授权任务完成后，权限未被及时撤销（Revoke），长期暴露在系统中。 | 类似于 DeFi 的合约授权漏洞，长期面临遭受系统性攻击的隐患。 |

**3\. 可信执行链路的系统架构设计**

为解决上述风险，Cobo Agentic Wallet 提出了一种“Human-in-the-loop”（人类在回路中）的强管控架构，确保 Agent 的每一步操作都在可信节点内流转。

3.1 底层安全：基于 MPC 的 2-of-2 签名模式

相较于依赖 TEE 或临时分发 API Key 的中心化单点风险方案，MPC（多方安全计算）保障了最底层的私钥安全。Cobo 采用 2-of-2 阈值签名机制，涵盖两种共管场景：

-   Human + Cobo：用户作为钱包 Owner 保留最高控制权，发起大额交易时通过生物识别完成主观意愿签名。
    
-   Agent + Cobo：当 Agent 获得明确授权后，由 Agent 发起交易请求，Cobo 依据安全策略进行校验与协同签名，确保单方无法私自挪用资金。
    

3.2 权限边界：Pact 授权协议

Pact 是约束 Agent 行为的核心风控层，本质是一份人类与 AI 之间的可执行数字契约。一份完整的 Pact 包含四个严谨的控制要素：

1\. Intent (意图)：明确 Agent 需要完成的具体目标（例：监控以太坊二级市场价格，逢低买入）。

2\. Execution Plan (执行计划)：将意图转译为透明、可审计的链上动作（调用何种合约、路由及代币对）。

3\. Policy (风控策略)：设定的硬性约束条件，涵盖整体预算限制、地址白名单、滑点容忍度以及 ABI 参数级的限制。这是阻断过度授权的关键。

4\. Completion Condition (终止条件)：设定任务的生命周期（如达到 1000 美元预算上限后即刻自动拦截交易），从机制上消除僵尸授权（Zombie Permissions）风险。

3.3 知识封装：Recipe 技能层

鉴于当前主流大模型在处理复杂链上交互时的高失败率，Recipe（知识胶囊）扮演了预加载知识库的角色。它将主流 DeFi 协议（如 Uniswap、Aave）的合约地址、交互规范及安全参数打包。此机制限制了模型的自由发挥空间，将其行为框定在已验证的安全路径内执行，大幅降低了由于模型幻觉导致的操作风险。

4\. 实践启示：非技术视角的 AI x Web3 应用构建

这种通过机制设计而非纯代码逻辑来规范 AI 行为的架构，为构建客观、严谨的 Web3 交互模型提供了新思路。

在尝试设计 AI 与 Web3 结合的 Demo（如二级市场自动化交易助手或 DeFi 资金管理工具）时，核心关注点可以从底层的智能合约开发，向上转移至风控规则（Policy）的设计与业务逻辑（Intent）的定义。

通过多 Agent 架构，可以针对不同场景（如 BTC/ETH 网格交易、小额高频的测试币交互等）设定相互隔离的资金池与独立的 Pact 协议。这种将金融风控逻辑直接映射到链上执行层的模式，极大降低了构建安全链上业务的门槛，使得依靠业务洞察与管理思维驱动 Web3 创新成为可能。

**5\. 结语**

AI 介入加密经济的深水区，必然要求信任机制从应用层下沉至基础执行层。通过 MPC 保障资产托管底座，借助 Pact 限定权限边界，并利用 Recipe 规范执行路径，可以在控制系统性风险的前提下，最大化释放 Agent 在链上交互中的效能。在完全去中心化的自主执行网络成熟之前，以人类授权为核心的可信执行链路，是当前兼顾效率与安全的最优实践。

Agent 记忆系统的演进路径与架构解构

​摘要

​伴随大语言模型（LLM）从静止的文本生成器向具备自主规划与执行能力的智能体（Agent）演进，传统的单次提示词交互与有界的上下文窗口已无法满足复杂、长链路任务的工程需求。本文将 Agent 记忆（Memory）系统定位为一种“跨越时间维度的意图与状态管理系统”，并深入探讨了其在应对人类交互涌现性、模糊性时的核心价值。通过对消费级通用大模型与专业级工程 Agent 记忆管理范式的对比，本文系统性地解构了由 RAG、上下文工程及强化学习交织而成的技术矩阵，分析了记忆全生命周期管理中的核心工程挑战，并对行业构建通用记忆层的趋势进行了思辨性展望。

​一、 引言：从有界上下文到无界状态管理

​在LLM技术发展的初期，交互模式主要依赖于单次的提示词工程（Prompt Engineering）。这种模式本质上是无状态的（Stateless），模型的输出完全取决于单次输入的质量。随后，上下文工程（Context Engineering）的出现允许维持短期的对话历史，但其依然受限于底层模型物理层面的最大上下文窗口（Context Window）。

​然而，当大模型走向 Agent 化，其面对的人类交互环境呈现出高动态与高复杂度的特征：

​意图的渐进涌现（Emergent Interaction）： 人类在解决复杂问题时，初始意图往往是模糊且高度概括的。长链路的真正约束条件、工具调用的中间状态以及阶段性产物，是在交互过程中“动态涌现”出来的。

​状态的跨时域延续（Cross-time State Management）： 智能体需要在一个超越单次会话（Session）、甚至跨越自身运行生命周期的维度上，保持对用户偏好、过往策略及环境反馈的感知。

​因此，Agent 记忆（Memory）系统的本质并不是对物理窗口的盲目扩容，而是一套面向人类用户、大模型自身以及运行时（Runtime）环境的、跨越时间的意图与状态管理系统。 它将碎片化的单次交互，拼接成了具备时间连续性的智力运行流。

​二、 概念重塑与记忆的多维分类模型

​在探讨技术架构之前，必须厘清行业内常被混淆的工程概念。Memory 是处于最顶层的系统级概念，它决定了智能体“应该记住什么”以及“如何使用记忆”；而 RAG（检索增强生成）仅仅是支撑这套系统落地的数据管线（Data Pipeline）之一；检索与生成则是管线上的具体动作；向量模型（Embedding Model）则是实现自然语言到高维数学表征转换的底层工具。

​为了在工程上实现精准管理，我们将 Agent 记忆体系解构为两个核心交织的维度：

​1. 时间时序维度

​工作记忆（Working Memory）： 类似于计算机的内存。它聚焦于当前任务流的上下文，承载着即时的动态意图和工具执行的中间状态，随着当前任务的结束或会话的关闭而释放。

​长期记忆（Long-term Memory）： 类似于计算机的硬盘。它跨越了会话的物理边界，将有价值的信息长久地固化在外部存储介质中，用于跨越数天、数月后的冷启动唤醒。

​2. 认知内容维度

​语义记忆（Semantic Memory）： 沉淀关于世界或特定对象的通用事实知识。在 Agent 语境下，主要表现为静态的用户画像、长期的行为偏好与环境约束。

​事件记忆（Episodic Memory）： 记录特定时间、特定序列下发生过的“历史事件”。例如 Agent 过去在某个时间点调用某个外部接口的成功或失败记录，为当下的决策提供“前车之鉴”。

​程序记忆（Procedural Memory）： 固化下来的技能与工作流（Workflow）。即 Agent 掌握的“如何执行某一复杂任务”的结构化步骤与自动化策略。

​三、 典型产品记忆范式的分化与对比

​不同的应用场景决定了智能体对“记忆”的侧重点截然不同。通过对比三类典型产品，可以清晰地看到记忆系统的分化趋势：

​1. ChatGPT：基于人际关系维持的偏好分层

​ChatGPT 的核心价值在于建立长期的用户粘性与情感/工作陪伴。其记忆设计采用了 Saved Memory（显式留存偏好） 与 Chat History（隐式历史滚动） 的分层模式。为了确保用户对数字资产的绝对掌控权与隐私安全，其记忆层引入了强烈的自主管控机制，允许用户对固化的偏好进行精准的显式擦除与修正。

​2. Cloud Code：基于高连续性工程上下文的状态机

​作为面向工业级编码场景的智能体，Cloud Code 的交互几乎不存在“闲聊”属性，其记忆系统完全是为了解决工程项目的连续性与长期异步运行。它需要精准承载整个代码库（Repository）的拓扑结构、依赖关系以及多任务长时间并行时的断点状态。这种记忆是极其硬核、精准且高密度的，对容错率的要求达到了工业级。

​3. “小龙虾”类端侧 Agent：多模态长执行链的工程痛点

​这类以移动端/桌面端自动化操控为核心的开源 Agent 架构，其长链路执行特性将记忆模块推向了风口浪尖。在动辄数十步的跨应用操作中，Agent 不仅要接收文本输入，还要处理大量的屏幕截图（多模态数据）以及环境不确定性带来的突发错误。开源架构的碎片化使得记忆流极易断裂。如何在漫长的执行链条中对抗“记忆漂移”与“上下文噪声”，是当前端侧自动化面临的最主要技术壁垒。

​四、 技术栈透视：记忆全生命周期的管理矩阵

​一个高可用的记忆系统，在底层需要应对极其复杂的全生命周期管理流。信息并不是简单地“存入”即可，系统必须实时解决：增（何时写入新知识）、删（如何剔除无用噪声）、改（如何修正过时偏好）、查（如何精准唤醒）、权重调整（判断信息重要程度）以及冲突处理（当新旧信息矛盾时如何仲裁）等一系列动态工程问题。

​五、 开源生态演进与通用记忆层的思辨

​当前，开源社区正试图打破单一应用的藩篱，如 Memory0 等项目正致力于打造一套独立于具体大模型的、通用的标准化记忆层，力求在不同的智能体生态间实现记忆的无缝迁移与共享。

​然而，我们必须对这种“通用性”保持清醒的观望态度。不同领域的 Agent，其知识密度和记忆维度存在天然的鸿沟（例如：Web3 金融交易 Agent 追求绝对的数字化状态和实时清算，而情感陪伴 Agent 追求对人类情绪波动的细腻感知）。由于标准化协议尚未成型，且隐私数据跨平台的安全合规性存在天然壁垒，试图毕其功于一役打造全场景“通用记忆层”的尝试，在短期内仍面临极大的工程不确定性。

​六、 结语与共学感悟

​作为一名具备金融与经管背景、正在系统学习 AI 与 Web3 交叉融合的研一学生，这次关于“Agent Memory”的直播学习与课件研读，引发了我跳出技术本身的跨学科思考。

​在金融和 Web3 的底层逻辑中，我们最核心关注的范畴之一就是“状态（State）”。区块链本质上就是一个跨越全网、分布式、不可篡改的“全球状态机”与信任账本；而今天我赫然发现，AI 领域为了让智能体走向真正具备商业价值的实体，同样在围绕“状态管理”进行艰苦的工程探索。Web3 通过去中心化的底层链条来锚定和管理价值的状态，而 AI 则通过 Memory 模块来承载和管理智力与意图的状态。 两者在逻辑终点上表现出了惊人的同频共振。

​从非技术背景的视角审视，构建 Agent 记忆系统的过程，与我们每个人在数字时代打造“个人品牌（IP）”的逻辑如出一辙。人类在 Twitter 上持续输出深度见解、垂直领域的思考，本质上是在向数字世界写入自己的“语义记忆”；而我们在共学营、在行业社群中建立链接、做自我介绍、沉淀合作项目，则是在数字空间中编织不可磨灭的“事件记忆”。

​一个缺乏高效记忆系统的 AI，纵使算力再强，也终究只是一个高能的瞬时计算器；只有当它能够跨越时间的尘埃，精准地接住人类模糊的意图，并将其固化为持续运行的状态时，它才真正具备了金融级与商业级的生产力价值。这次学习让我确信，未来在探索 AI x Web3 的结合（如智能自动化交易、去中心化 AI 协作等）时，深入理解并优化 Agent 的记忆与状态管理，将是通往构建高价值资产级别 Demo 的关键钥匙。

开放智能体经济 (Open Agentic Economy)：以太坊作为 AI Agent 的底层基础设施

摘要

在过去 18 个月中，大型语言模型 (LLM) 经历了能力跃升，AI 正在从单一的问答工具转变为能够执行代码、转移资金并自主决策的“新型经济参与者”。然而，现有的传统互联网基础设施专为人类设计，无法为 AI 智能体 (Agent) 提供安全的运行轨道。本笔记探讨了以太坊(Ethereum) 作为去中心化协调与结算层的核心作用，阐述了其如何通过特定协议（如 ERC-8004 和 x402）赋能机器经济，同时确保人类保留最终控制权。

一、 核心痛点：传统基础设施对 AI Agent 的不适配

随着 AI 智能体数量呈指数级增长，其进行交易、协商和支付的基础设施将直接决定未来经济的运作模式。目前，Agent 面临以下严峻的基建限制：

1\. 隐私与状态存储缺陷：Agent 无法像人类一样“保密”（如密码），因为信息必须存储在上下文窗口中。

2\. 金融交互的高风险：Agent 在调用传统银行账户时，极易受到提示词注入 (Prompt) 的诱导而产生资金安全问题。

3\. 缺乏社会信任层：Agent 无法签署传统法律合同，也无法通过人类社会的渠道建立有效的商业信誉。

二、 解决方案：以太坊作为中立的协调与结算层

以太坊作为一个公共、可编程的区块链，允许在没有中间人的情况下实现数字所有权和点对点协调。其能够成为 AI Agent 可信运行轨道的原因，在于其底层遵循的 CROPS 设计原则：

-   Censorship Resistance (抗审查)：任何单一实体都无法阻止网络上有效交易和智能合约的执行。
    
-   Open Source and Free (开源且自由)：代码可审计、可分叉，开发者可以像搭乐高积木一样组合不同的模块进行构建。
    
-   Privacy (隐私)：通过零知识证明 (ZK Proofs) 等技术，在不暴露底层数据的前提下验证真伪。
    
-   Security (安全)：代码能够按照既定逻辑持续稳定运行。
    

基于 CROPS 原则，身份、声誉、支付和可执行协议才得以在区块链上建立。

三、 核心构建模块：身份、声誉与支付协议

为了让 Agent 能够真正参与链上商业活动 (Agentic Commerce)，目前正在建立关键的标准与协议：

1\. ERC-8004：Agent 的身份与信任层

ERC-8004 是一个为数字软件实体（Agent、API、预言机等）提供可验证身份和声誉的标准。

-   它由三个核心部分组成：验证机制、注册表 (Registry) 和声誉系统。
    
-   任何用户或其他 Agent 在与之交互前，都可以明确知晓该软件的信任状态和声誉分数。未来路线图中还计划整合基于 TEE（可信执行环境）的验证。
    

2\. x402：机器对机器 (M2M) 支付标准

-   这是一种针对机器环境的“需支付”状态码协议，无需信用卡或 API 密钥即可通过稳定币完成结算。其核心优势在于结合了智能合约的“可编程货币”特性：
    
-   人类可以通过智能合约为 Agent 设定硬性护栏（Guardrails），严格限制其支出上限和支付对象（例如：限定只能通过特定供应商支付最多 10 美元）。
    
-   这种机制从代码层面保证了执行结果必然与人类的指令一致。
    

四、 实践案例：AI 参与公共物品资金分配

AI × Web3 共学营day5

AI与Web3融合下的Agent经济栈构建及五大应用方向探究

摘要

随着人工智能逐渐从“回答问题”向“执行任务”演进，单一的模型能力已无法满足其在复杂经济网络中的交互需求。本文探讨了AI与Web3结合的核心范式：AI负责理解与决策，Web3提供去中心化的身份、支付、结算与审计等基础设施。研究表明，两者结合的本质并非简单的代币化，而是构建一套完整的“Agent经济栈”，从而赋予AI代理在授权范围内的真实经济执行能力。

1\. 引言

当前AI代理（Agent）的演进路径面临缺乏原生经济系统支持的瓶颈。当AI开始租用算力、购买数据或与其他代理协作时，需要一个能够被授权、付款和记录追责的经济层。Web3的去中心化账本、密码学身份和智能合约体系恰好填补了这一空白，形成了涵盖资源、行动、认知、安全和风控五大核心方向的互补架构。

2\. Agent经济栈的五大核心层级

为了系统性理解这一融合，我们可以将各类Web3项目映射到Agent运作的具体经济环节中：

| 经济栈层级 | 核心解决问题 | 核心运作逻辑 | 代表性项目案例 |
| --- | --- | --- | --- |
| 资源层 (去中心化基础设施) | AI生产要素如何被组织与激励 | 链下计算配合链上验证、激励与服务发现 | Bittensor, Render, Akash |
| 行动层(Agent与钱包) | AI如何获取身份、支付与执行权限 | 为大语言模型(LLM)连接真实的链上钱包与操作动作 | Coinbase AgentKit, MoonPay |
| 认知层 (链上分析工具) | 链上公开数据如何转化为可理解的判断 | 通过地址聚类、标签化将底层哈希转为实体画像 | Arkham, Nansen, Dune |
| 用户保护层 (智能钱包安全) | 普通用户如何安全进行签名与交互 | 将单私钥账户升级为权限系统，提供交易模拟与拦截 | Blockaid, Safe, Rabby |
| 机构风控层 (智能风控系统) | 系统如何提前发现异常与攻击风险 | 将海量行为转化为异常模式，驱动告警与合约冻结 | Chainalysis, Forta |

2.1 资源层：生产要素的去中心化重构

主流发展路径在于链下计算配合链上的激励、结算与验证，而非将庞大的AI模型直接上链运行。

该层级的核心技术难点集中在服务质量验证、反作弊机制构建、系统稳定性保障以及企业级SLA（服务等级协议）的实现。

以Bittensor为例，其通过Subnet定义任务，利用验证者评分和代币激励机制来分配收益，探索了AI服务如何通过开放市场机制组织生产的路径。

2.2 行动层：从建议者到具备经济实体的执行者

没有独立钱包的Agent仅能提供文字建议，而接入钱包后的Agent才能在授权范围内完成转账、合约调用及收款等真实经济动作。

这种执行能力的安全形态建立在“人类授权范围内的自动执行”之上，必须辅以预算限制、白名单、单笔上限及撤销权限等保护措施。

Coinbase AgentKit作为典型代表，展示了如何将大语言模型的工具调用直接映射并连接到真实的链上执行动作。

2.3 认知层：隐匿数据的语义化解析

链上数据虽具备完全的公开性，但缺乏直观的可理解性，AI在此的核心价值是对高维数据进行清洗、聚类分析及生成行为解释假设。

该方向的受众需求呈现分化：面向个人用户时强调直观的“看懂”，而面向机构时则严格要求数据的可验证性与可审计追踪能力。

Arkham通过引入地址聚类和资金归因算法，成功将底层的多链哈希数据转化为更接近现实世界的实体级情报画像。

2.4 用户保护层：UX重构与签名前置防线

Web3领域最大规模采用的UX（用户体验）瓶颈并非界面设计，而是用户无法准确预判底层密码学签名的真实经济后果。

智能钱包机制通过引入多签、社会化恢复和权限时间锁重构了账户体系，而安全助手则扮演了“交易解释与后果拦截器”的角色。

Blockaid项目通过在资产转移前实时扫描域名与合约，并在模拟执行后匹配恶意模式，实现了将风险在签名阶段进行前置拦截的安全检测闭环。

2.5 机构风控层：全局系统的智能化防御

在应用侧重上，中心化交易所（CEX）更聚焦于资金来源、反洗钱与合规审查，而DeFi协议则高度关注智能合约攻击预警与流动性参数崩溃风险。

AI风控并非替代既有规则，而是将海量的链上交互行为转化为高优先级的异常模式信号，并直接驱动告警、冻结或暂停合约等自动化响应动作。

Chainalysis作为机构级合规标准的标杆，有效帮助合规团队识别各类可疑行为并严守监管合规底线。

3\. 风险边界探讨

在评估AI与Web3结合的长期价值时，必须警惕将其包装为“万能叙事”的投机倾向，并清晰划定以下风险边界：

执行权不能直接让渡于模型：AI模型不可直接等同于最高执行权，真实的资金动作必须受到人类的强制授权、限额及撤销机制的约束，以防范因提示词注入（Prompt Injection）导致的资产非预期转移。

链上标签不具备绝对法律效力：AI生成的链上地址聚类与风险评分仅属于概率判断，不能作为绝对的法律事实认定，金融机构在应用时仍需严密的证据链与人工复核。

体验升级与安全托管的博弈：免助记词和免Gas等旨在降低用户门槛的体验升级措施，往往会引发关于资产托管边界模糊及权限滥用的新型安全隐患挑战。

4\. 结论

综上所述，看懂并判断AI与Web3融合项目的真实内在价值需牢牢把握三个核心基准判断：

1\. Agent需要原生经济账户：一旦AI代理跨越聊天边界开始实际行动，必然面临复杂的账户、支付、授权与追责问题。

2\. 价值依托于具体环节的解决能力：有效的创新项目不应是孤立的宏大叙事，而应当精准解答Agent经济栈中诸如算力获取、上链执行或安全阻断等某一个具体环节的问题。

3\. 基建价值大于代币机制：代币本身不构成价值壁垒，AI+Web3的真正红利在于促使软件主体融入一个高度透明、可授权、可结算且可审计的开放经济基础设施网络中。

今天尝试复盘钻研一下布老师昨天最后留的四个问题。

1\. 资产自托管：如何提高安全性、降低私钥管理复杂度？

我的理解是，自托管最大的优点，是让我真正拥有资产的控制权；但它的代价也很明显，就是责任几乎全落在自己身上。以前在平台里，密码忘了、账号异常了，通常还能找回；可在 Web3 里，系统只认私钥和签名，不认识“现实中的我”，所以私钥一旦丢了或者泄露了，后果会很直接。对新手来说，最难的地方不是“有没有资产”，而是“能不能把钥匙保住”。所以我觉得，自托管真正要解决的，不只是“把钥匙交给用户”，而是让用户在不懂技术的情况下，也能安全地用它。

自托管不是把风险消灭，而是把风险从平台转移到用户，并且要求工具把这个风险尽量做小。 这也是我现在最能理解的一点。

2\. 没有中心化机构 / 税收时，公共基础设施谁来维护？

我的理解是，这其实是在问：如果没有一个“统一出钱、统一负责”的中心，那么公共品靠什么活下去。站在新手角度看，我觉得 Web3 的答案不太像传统社会那种“政府税收—财政支出”的逻辑，而更像是一种把成本和激励写进系统里的方式。也就是说，维护网络的人不会是“白白做贡献”，而是通过 Gas、质押奖励、罚没机制、协议激励等方式，让参与者有动力去维护系统。课件里也明确提到，Gas 的作用之一就是给出块者激励、让资源有价格。

但我也会觉得，这种机制并不是完全没有问题。因为它能解决“谁来参与”，却不一定天然解决“谁来承担长期公共责任”。所以我理解的Web3 更像是在尝试用经济机制替代一部分传统公共治理，但它是不是足够稳定，还要看制度设计能不能长期成立。这个问题我现在不能说自己懂了，只能说我知道它是在问“公共物品怎么被持续供养”。

3\. 没有审查且隐私很强时，如何界定并治理有害信息 / 黑产？

我的理解是，这个问题其实在问 Web3 的另一面：如果一个系统强调开放、抗审查、隐私和无许可，那它也可能会被坏人利用。也就是说，越开放的系统，越容易同时承载正向协作和负向滥用。 课件里提到，主流公链默认是“公开账本 + 伪匿名”，不是默认隐私系统，同时它也强调了抗审查、开放可验证、可组合性这些特征。对我这种新手来说，这就意味着：Web3 不是“天然更道德”，它只是把规则变得更公开，把权力分散了，但风险并没有自动消失。

我感觉治理黑产不能只靠“删掉”或者“封掉”，因为去中心化系统里没有一个中心按钮可以按；更现实的方式，可能是通过协议规则、地址风控、审计、社区共识和前端限制等多个层面一起做。 我现在还不懂具体技术怎么实现，但差不多能理解为什么这会成为 Web3 必须面对的问题。

4\. 去中心化协作下，如何实现公平、可信的分配？

我的理解是，这个问题不是在问“怎么平均分”，而是在问：当大家不是在一个公司里上班，也没有一个老板拍板时，怎样让分配规则还算公平、还算可信。 在 Web3 里，规则通常写在合约里，执行记录公开可查，大家可以去验证它是不是按规则运行。课件里也说得很直接：智能合约的价值之一，就是让规则能看见、少一点临时拍板、信任对象从“某个机构”转向“代码 + 共识 + 审计/治理”。所以我会理解为，Web3 想要的公平，不是“人情上的公平”，而是“规则上的公平”。

从我的角度看，这种公平也不是自动实现的。因为规则谁来写、参数谁来调、漏洞谁来修，这些背后还是有人在决定。所以我现在对这个问题的理解是：去中心化不是没有分配，而是把分配变成了可验证、可讨论、可追责的过程。 这已经比我一开始以为的“区块链就是自动公平”要成熟很多了。

个人感悟：

感谢布老师的分享，这些问题我会继续观察真实项目，再去思考。通过这四个思考题，我对 Web3 的理解比之前清楚了一些。以前我总觉得 Web3 更像是一些很抽象的技术名词，离普通人很远，但现在慢慢意识到，它真正讨论的其实是“谁来掌握权力、谁来承担责任、规则应该由谁来制定”。从资产自托管到公共基础设施，从隐私和审查到公平分配，这些问题表面上是技术问题，实际上背后都和信任机制、激励机制以及治理方式有关。对我来说，Web3 最吸引人的地方，不只是它“去中心化”的概念，而是它试图把很多原本依赖中心机构的事情，改造成一种更透明、更可验证的协作方式。同时，我也感受到 Web3 并不是一个“天然更好的答案”。它把很多权力交给了用户，也就意味着把更多责任交给了用户；它强调开放和自由，也会带来安全、治理和滥用的问题。所以我现在对 Web3 的态度不是盲目乐观，而是保持兴趣和谨慎并存。它可能代表一种新的组织方式，但这种方式能不能真正落地，关键还是看它能不能在安全、效率、公平和可持续之间找到平衡。对我这样的新手来说，现阶段最重要的不是急着下结论，而是先把它的底层逻辑慢慢看懂。

一、 安全与AI的底层逻辑（商科风控视角）

1\. 安全的“非对称性”（0和1的概念）

金融视角解读：安全在Web3中是无法被“证伪”的。项目做了100项安全防护，可能依然被黑客找到第101个漏洞攻破；但有些项目即便是“裸奔”（毫无防护），短期内也可能侥幸没事。

核心结论：安全是贯穿项目生命周期的底层设计，而不是事后打补丁。你做的防护越多，项目的相对安全概率就越高。如果团队对底层行为没有绝对的掌控力，任何安全架构都是“空中楼阁”。

2\. AI时代的“复杂度悖论”

误区：有了AI自动编码，技术变简单了。

真相：AI没有降低系统复杂度，反而拉高了对人的要求。AI加速了代码的堆砌，但如果架构师自身的基础不扎实，就根本无法判断AI端上桌的究竟是“屎山”还是“珍馐”。安全边界的最终审核权和责任，永远在人手里。

二、 核心主线：Web2 vs Web3差异对比

老师用了一个非常性感的表达：Web3的区块链在抽象层面上就是Web2的银行机构。

信息流的方向变了：Web2 是“顺差请求”，支付服务要先找银行要数据，银行再主动通知支付服务；Web3 是“逆向监听”，区块链只管记账，外面的系统必须自己派“哨兵”来偷看账本变化。

​信任的载体变了：Web2 信任的是“开户银行这个权威实体”；Web3 信任的是“底层的开源代码与数学加密算法”。

​错误成本变了：Web2 的银行账本做错了可以冲正、冲账（有反悔机制）；Web3 的转账一旦被节点打包，就是宇宙级的不可逆，漏洞即归零。

三、 钱包的本质：非对称加密与确权

1\. 记账逻辑的转变

Web2：钱在银行的账户里，银行的中央账本上记着“张三10块，李四20块”，你输入的支付密码只是划扣账目的“请求信号”。

Web3：钱包的输入是一串随机的大数字——私钥（Private Key）。钱包的本质是资产的控制权和身份的证明器。

2\. 非对称加密的商科大白话

意图交易确权：你用私钥去签名一个消息，数学算法可以保证：\*\*外界在完全不知道你私钥的情况下，仅凭签名就能确认“这笔交易确实是持币人本人的真实意图”\*\*。

\*不管区块链怎么升级（Layer 1 还是 Layer 2），只要数学签名能对上，链就认账。这就是去中心化世界最底层的“清结算协议”。

个人感悟：听完TC老师的这堂课，我最大的感悟是：Web3 并不是要把传统金融的一切推倒重来，它其实是一场对“信任成本”的工业革命。作为商科生，以前我觉得技术离我很远。但今天看到支付模型的对比图时，我突然通透了Web2 支付之所以复杂，是因为中间有太多的“信任摩擦”。为了防范欺诈，银行和支付服务之间要套上无数层 API Key、风控模型和人工对账流程。而 Web3 用区块链和非对称加密，直接在底层用“数学”把信任成本降到了零。但硬币的另一面是，Web3 把风控的责任从“中心化机构”无情地推给了“每一个个体”。在 Web2 里，密码输错了可以找回，银行挂单了可以冲正；而在 Web3，容错空间是绝对的零。AI 时代的到来，虽然让写代码变快了，但如果对金融逻辑和系统安全的工程严谨性没有敬畏之心，AI 只会用更快的速度制造灾难。这也让我坚定了在这个训练营里，必须先把钱包、交易这些“技术底座”的业务逻辑彻底搞懂的决心。
