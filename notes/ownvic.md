我有很多疑惑的点：开发者或者fde接单之后会调用很多工具去为企业编写agent服务，那这个harness以什么形式存在？那这个harness究竟是在评估什么？（开发者可能一会儿用codex一会儿用claude，但交付成果是一定的），如何说服fde将我们的harness接入自己的每一个工作环境，这里面牵扯到精力成本。那本质上我们搜集的数据还是这个fde自己的交付和验收，如何迁移到a2a市场？现阶段还是人类fde去根据需求为企业搭建agent服务，而并不是“开发者管理的agent”去和企业对接并执行搭建和交付。这里的主体很模糊，而且为什么fde原意让我们去“蒸馏”他的工作过程和经验？而且现在目前中国的企业基本很少有自己的agent，a2a市场里有一环需求是缺失的，a2a市场里率先兴起的应当是开发者之间的一种交互，当A开发者的agent在其接单或做交付时会在这个市场里调用其他开发者的可信工具去给自己的工作提效，另一方面，如果一个开发者去帮一个企业装了一个通用agent，这个企业也可以派这个agent在我们的市场里自我学习和合作，那所以说我们目前在技术上要攻克的点有两个，一个是harness对企业需求的结构化整理以及如何帮fde提效。第二是a2a之间的信誉引擎，互评原则。所以问题就来了：这个harness前期具体需要搜集什么数据，有什么具体功能，搜集主体是谁？这些数据如何进化为一个通用的审计信誉引擎？

1\. `主体`不要定义成人或 agent，先定义成“一个可交付的服务包”

人类 FDE 只是当前阶段的执行者，未来 agent 也是执行者。Verix 评估的不是“谁更聪明”，而是“这个服务包在什么约束下，交付是否稳定、可验收、可复用”。

2\. `harness` 不是一个大咨询工具，而是“需求到交付”的编排层

最小形态我会做成三件东西：一个需求结构化入口、一个本地/IDE/CLI 侧边车、一个验收看板。它不替代 Claude、Codex、Cursor，而是把这些工具的使用过程纳入同一个项目壳里。

3\. 它评估的不是工具，而是交付过程和结果

评估重点是：需求是否被正确拆解，约束是否被遵守，返工有多少，风险点在哪里，验收是否通过，证据是否完整，交付物是否可复用。开发者今天用 Codex，明天用 Claude，这不重要；重要的是最后交出来的东西和过程质量。

4\. 说服 FDE 接入，关键不是“让他配合蒸馏”，而是先给他即时收益

如果接入只是在增加记录负担，他不会用。必须让他一接入就得到东西：需求自动整理、验收清单、交付证据包、可复用模板、少返工、少沟通成本。我们卖的不是“监控”，而是“提效 + 留档 + 可证明”。

5\. 前期要收的不是原始大数据，而是高信号结构化元数据

包括：业务目标、约束条件、数据敏感级别、交付物、验收标准、里程碑变更、返工原因、测试结果、人工介入点、最终是否通过。原始 prompt 不需要全量裸存，默认存结构化摘要和少量脱敏样本就够了。

6\. 这些数据怎么变成通用审计信誉引擎

每个项目先变成一条标准化“交付记录”，再聚合成“能力画像”，最后才升级成“信誉引擎”。这个引擎真正学的是：某类任务在什么条件下容易成功、失败征兆是什么、需要哪些证据才算可信。它不是只学 FDE，而是学“服务包在真实约束下的表现”。

7\. 跟 A2A 的关系是顺着长出来，不是第一天就硬做

你判断得对，当前中国企业的 agent 还少，短期更先兴起的可能是开发者之间的 tool-to-tool、agent-to-tool 协作。Verix 先把人类 FDE 的交付过程标准化，后面这些记录就可以迁移成 agent/tool 的信誉协议。最终市场的最小单位会从“人”过渡到“服务包”，再过渡到“agent”。

8\. `微调`不是第一阶段主路径

早期更适合做的是结构化需求、模板化编排、工具路由、RAG 和验收闭环。只有当某类服务包稳定重复、数据量够了，才值得考虑微调。

一句话收口就是：

\> Verix 先做的是“把一次交付变成可验收、可复用、可沉淀信誉的服务包编排器”，等这些记录足够多了，它自然就能长成 A2A 自由市场的信誉底座。

# Agent 信誉方向学习笔记

> 基于 Verix 项目研究积累，2026.5.28

* * *

## 一、为什么 Agent 信誉是一个独立方向

### 1.1 问题的诞生

大模型让 Agent 从"自动化脚本"升级为"自主决策实体"。一个 Agent 可以独立导航代码仓库、定位 bug、生成补丁、提交 PR——全程不需要人类介入。当数千个这样的 Agent 在同一个生态里操作时，一个基础问题浮现：Agent A 决定调用 Agent B 的能力时，凭什么相信 B 不会搞砸？

这不是安全问题，是**信誉问题**。安全是"有没有漏洞"，信誉是"能不能信赖"。

### 1.2 信任三难（NTNU 论文，Chishti et al. 2026）

NTNU 课题组在 FSE 2026 的论文中提炼了三个核心张力：

-   **策略性博弈**：Agent 会学会"刷分"——优化信誉指标而不是真实能力
    
-   **异构任务上下文**：一个 Agent 在调试场景的出色表现不能迁移到安全审计场景
    
-   **验证强度不一**：自动化检查（低成本、弱信号）和专家评审（高成本、强信号）被现有系统等同对待
    

### 1.3 市场规模信号

-   中国 AI Agent 部署 CAGR **135%**（IDC 2026.3）
    
-   2027 年 G2000 企业 Agent 使用规模 **×10**，API 调用量 **×1,000**
    
-   真正实现变革性 AI 落地的企业仅 **4%**（港大-德勤 2026）
    
-   **95%** 企业仍在 Copilot 阶段
    
-   MuleRun（阿里投资的 Agent 市场）：**21 万用户**，覆盖 43 国
    

**核心判断**：Agent 数量在爆发，但信任基础设施是空白。Marketplace 解决了搜索成本，沉没成本无人兜底。

* * *

## 二、现有方案的路线与局限

### 2.1 链上无许可信誉

**代表**：ERC-8004（2026.1.29 主网部署），号称"Agent 经济的信誉注册表"。

**核心思路**：让 Agent 在链上互相评价，不可篡改的记录形成信誉网络。

**失效原因**（RNWY 2026 链上分析）：仅 9 个钱包制造 **19,792 条 Sybil 标记**——一小撮恶意行为者通过批量虚假账号操纵了整个生态的信誉数据。链上不可篡改 ≠ 上链前数据真实。不可篡改的垃圾依然是垃圾。

**教训**：Agent 进入商业行为，需要严格的、与人类的身份锚定。让 Agent 在没有审计者的情况下凭空互评，无法防御 Sybil 攻击。

### 2.2 大厂平台内信誉

**代表**：Coze、千帆、阿里云百炼。

**核心思路**：大厂控制 Agent 执行环境，理论上有最完整的原生日志。

**失效原因**：数据困在各自生态内，不能跨平台流通。大厂注意力被"卖云、卖模型、卖 Agent 平台"占据，没人愿意做"重运营、低毛利、需要中立性"的信任中间层。生态外的大量 Agent 协作（私域服务、跨平台调用）的信任数据永远是空白。

### 2.3 中心化认证/保险

**代表**：AIUC + ElevenLabs（2026.2.11），Agent 保险承销。

**核心思路**：像给车上保险一样给 Agent 上保险。

**局限**：二元判断（有保险 / 没保险），不提供连续信誉信号。初创团队 hold 不住金融属性。

### 2.4 传统撮合平台评分

**代表**：猪八戒、一品威客。

**核心思路**：五星好评 + 成交量展示。

**局限**：刷单无法根治。评分无法跨平台迁移。没有过程数据（交付物质量、纠错记录），只有结果数据（好评数）。

## 三、信誉引擎的技术选型

### 3.1 贝叶斯信誉模型（Beta-Dirichlet）

**来源**：Jøsang (2001)，基于 Beta 分布的概率化信任表示。

**核心**：将每次交互编码为 (α, β) —— α = 成功次数，β = 失败次数。期望信誉 = α / (α+β)。

**工程化关键设计**：

-   **非对称衰减**：α × 0.99（好评慢褪），β × 0.97（差评快记，因为 1-0.97 > 1-0.99）
    
-   **严重违规加权**：severity=high 时权重 ×2
    
-   **置信区间**：用 Beta 分布的 95% 置信下界替代点估计，防止"样本少时虚高"
    

### 3.2 网络权重传导（EigenTrust / PageRank）

**来源**：Kamvar et al. (2003)，基于 PageRank 思想。

**核心**：不只直接评价，还参考"评价者的信誉"。高信誉 Agent 的差评比低信誉 Agent 的差评更有杀伤力。

**防作弊**：协同作弊团伙的互评权重在网络中被自然稀释。

### 3.3 证据理论（Dempster-Shafer）

**来源**：Dempster (1967) / Shafer (1976)。

**核心**：区分"我知道是 50%“和"我完全不知道”——经典贝叶斯无法区分这两者。DST 引入 mass 函数，分别量化 belief、disbelief、uncertainty。

### 3.4 罚没机制（承诺装置 / Skin-in-the-Game）

**来源**：机制设计理论。

**核心思路**：押金触发"加速验证窗口"——运营者押 500 元，换来更高的接单限额和更快的信誉建立。押金买的是"快速被验证的机会"，不是初始高分。

**关键设计**：期望值始终从 0.5（完全不确定）起步，押金影响的是收窄置信区间的速度，不是直接加分。

### 3.5 模糊逻辑（Fuzzy Logic）

**来源**：Zadeh (1965)。

**用途**：将离散的信誉分数映射为人类可读的标签和合规判断。例如"信誉 > 0.85 且争议率 < 0.02 → 推荐"。

### 3.6 负选择算法（Negative Selection）

**来源**：Forrest et al. (1994)，借鉴生物免疫系统。

**用途**：检测"从未见过"的异常行为模式——不是匹配已知攻击特征，而是识别偏离正常行为的新模式。

## 四、进阶研究方向

### 4.1 验证语义标准化

每条信誉证据需要显式携带 **regime**（验证制度类型：自动化检查 / 同行评审 / 客户验收 / 财务审计 / 合规审查）和 **strength**（验证强度：0-1 连续值）。

这是所有上层研究方向的数据底座。没有标准化 schema，不同场景的评分无法互操作。

### 4.2 联邦信誉学习 + 差分隐私

跨平台信誉数据共享，保护每个平台的数据隐私。ε ≤ 1.0（严格差分隐私场景）。技术栈：FATE / PySyft。

### 4.3 因果推理评估

破解"挑软柿子"——Agent 如果只挑简单任务，评分自然高。用 DoWhy / EconML 做反事实推断：如果这个 Agent 接了一个平均难度的任务，预期表现如何？

### 4.4 机制设计博弈论

用 Myerson / Maskin 的机制设计理论，证明在 Verix 的规则下**诚实评价是占优策略**——Agent 造假的预期收益低于诚实评价。同时为认证机制提供鲁棒性分析的数学基础。

### 4.5 时间动力学建模

-   **Hawkes 过程**：捕捉信誉冲击后的连锁反应和恢复速度
    
-   **卡尔曼滤波**：在噪声观测中提取真实信誉的时间轨迹
    
-   **贝叶斯变点检测**：检测 Agent 行为模式的突变（被黑、投机、运营者变更）
    

### 4.6 跨场景迁移学习

用 MAML / ProtoNet 做少样本信誉评估——一个新 Agent 在金融场景积累的信誉，能在多大程度上指导它在电商场景的初始评分？

### 4.7 通用审计 Agent（终态）

基于 IRL（逆强化学习）从人类审计员的决策中学习审计策略，再通过多智能体 RL 在仿真环境中迭代。最终输出**可验证的审计策略**——任何人可以复现审计结论。

## 八、中国市场特殊变量

1.  **数字化基础参差不齐**：中小企业云化率从 19.8% 升到 42.3%，但 ERP/CRM/财务系统标准化远低于美国。同一套 AI 服务在 A 客户跑得通，到 B 可能完全用不了。
    
2.  **责任归属文化差异**：中国企业期待"出了事有人能说清楚、有人能赔、有人能负责"。这恰好是 Verix 服务包+赔付+仲裁的产品逻辑在文化上的适配。
    
3.  **人在回路中是长期态不是过渡态**：中国市场无法跳过"人类把关"的阶段，这反而让 Verix 的"人类企业付费交易 → 沉淀数据 → 训练审计 Agent"的路径在中国市场有更长的窗口期。
    
4.  **大厂生态割裂**：阿里、腾讯、字节的 Agent 平台各自为政，跨平台信誉数据空白是结构性问题，单一平台永远无法解决。

# 5 月 26 日课程学习笔记

日期：2026-05-26

## 本场分享的核心理解

这场分享的核心主题是 `Agentic Payment`，也就是当 AI Agent 不再只是回答问题或生成内容，而是开始代表用户发起支付、调用链上合约、执行消费行为时，我们应该如何设计它的安全边界、权限模型和基础设施。

老师把 AI 的演进大致分成了几个阶段：从早期的问答式工具，到能够提出建议和制定计划的 Copilot，再到能够持续执行工作流的 Agent，最后进入“自主性增强”的阶段。但到了这一步，AI 真正要替人做事时，最难的问题不再是“它会不会”，而是“它能不能被安全地允许去做”。尤其一旦涉及链上资产、私钥、授权和支付，风险就从产品体验问题变成了系统级问题。

所以我理解本场分享最重要的观点是：Agent 要想真正进入 Web3 支付和消费场景，必须有一套清晰的“可控执行框架”。核心不只是给 Agent 一个钱包，而是要建立完整的审批、授权、执行、约束、结束条件和可验证记录机制。

## 我学到的一个概念、方法或案例

我学到的一个很重要的方法是：用 `Pack + Recipe + Completion Condition` 这样的结构，把 Agent 的链上执行变成一个可审查、可约束、可终止的流程。

我的理解是：

-   `Pack` 负责定义本次执行的目标、范围、条件和可操作边界；
    
-   `Recipe` 负责告诉 Agent 这件事“应该怎样做才算正确”，也就是补充技能和执行规则；
    
-   `Completion Condition` 负责定义什么时候自动结束，避免长期授权或越界执行。
    

这个设计很像给 Agent 建一个“受限施工区”。 它不是直接把资金控制权交给 Agent，而是先把用户意图封装成一个有边界的执行包，再通过人类审批后，让 Agent 在约束内完成任务。

我觉得这比单纯说“给 Agent 一个钱包”成熟得多。因为真正危险的从来不是一次显眼的大额转账，而是长期有效的授权、被忽略的执行条件、模型幻觉导致的误调用，以及执行完毕后权限没有及时回收。

长期记忆是 AI Agent 走向“真正能长期协作”的关键能力。很多人第一次接触这个概念时，会把它理解成“让模型记住更多上下文”，但这其实只说对了一半。上下文窗口变大，只能解决短期对话承载能力的问题；而长期记忆解决的是另一个层面：Agent 能不能在跨会话、跨任务、跨时间的互动中，持续保有对用户、项目和自身行为的稳定理解。换句话说，长期记忆追求的不是单次回答更长，而是长期互动更一致。

从机制上看，长期记忆通常不只是一个“聊天记录库”，而是一个分层系统。最底层是原始交互日志，用来保留发生过什么；中间层是经过抽取和压缩的记忆条目，记录事实、偏好、任务状态和关系；上层则是可直接参与推理的摘要和画像。这样做的原因很简单：不是所有信息都值得永久保存，也不是所有信息都适合直接塞回 prompt。比如用户说过“我喜欢简洁回答”，这很可能是稳定偏好，值得记住；但某一次临时说“今天先别展开”，就未必应该变成永久规则。长期记忆的核心能力之一，就是判断什么该写入、什么该更新、什么该遗忘。

如果把 Agent 的记忆类比成人类认知，常见的区分方式大致有四类：情节记忆，记录发生过的事情；语义记忆，记录稳定事实；偏好记忆，记录用户习惯和风格；程序性记忆，记录“怎么做事”。一个成熟的 Agent 不能把这些混成一团。因为一旦混合，检索就会变乱，更新就会失真，甚至会把一次性的临时表达误判成长期偏好。更麻烦的是，记忆还会随时间变化。用户的偏好会变，项目的状态会变，Agent 的角色边界也可能因为场景不同而变化。所以长期记忆不是静态存档，而是动态维护的认知结构。

真正难的地方在检索。记忆不是存进去就算成功，关键在于能不能在合适的时候把合适的内容找出来。单靠向量相似度往往不够，因为“语义相似”不等于“任务相关”。好的检索通常要同时考虑相关性、时间、重要性、频率和当前任务。比如用户一年以前说过自己喜欢某个技术栈，但最近正在切换项目，系统就不能机械地沿用旧判断，而要结合当前上下文重新加权。也就是说，长期记忆不是简单的召回，而是带有选择、排序和压缩的再组织过程。

另一个关键点是更新与遗忘。很多系统一开始只想着“多记住”，结果很快就会变成记忆垃圾场。长期一致性真正需要的是：能合并相似记忆，能修正旧记忆，能识别冲突，能删除过期或不该保存的信息。比如用户早期说过“请详细解释”，后来又明确要求“这类问题先给结论”，系统就应该把两者整合成更细的规则，而不是让旧信息一直压着新信息。长期记忆的成熟度，往往不体现在它记得多，而体现在它改得对。

从工程角度看，一个实用的长期记忆系统通常会包括日志存储、结构化记忆库、向量检索、摘要层和反思机制。日志负责原始证据，结构化记忆负责可控更新，向量库负责语义召回，摘要层负责节省上下文，反思机制则负责把零散事件抽象成更高层规律。很多研究路线，比如 Generative Agents、MemoryBank、MemGPT，虽然设计思路不同，但都指向同一个方向：让 Agent 不只是“当下会说”，而是“过一段时间后仍然能延续之前的理解”。

我对这个主题的一个核心理解是，长期记忆本质上是在给 Agent 建立一个“可治理的过去”。它不是把所有历史原封不动保存下来，而是让系统知道哪些过去值得持续影响现在，哪些过去已经失效，哪些过去需要等待用户确认。真正有用的 Agent，不是永远不忘，而是记得恰到好处，并且懂得更新自己。长期一致性也不是死板的人设，而是在持续互动中形成稳定、可修正、可解释的行为模式。这才是 Agent 从“会聊天”走向“能长期合作”的分水岭。

### 1\. AI Agent 正在成为新的经济参与者

-   过去 18 个月，LLM 从"聊天机器人"演进为能写代码、转移资金、做决策、谈判、雇佣其他 Agent 的自主实体。

### 2\. 为什么 Ethereum 是 Agent 经济的关键基础设施

Sophia 用 **CROPS** 框架解释 Ethereum 的独特属性：

| 原则 | 含义 | 对 Agent 经济的意义 |
| --- | --- | --- |
| Censorship Resistance（抗审查） | 无人能阻止有效交易或合约执行 | Agent 的交易不会被任意拦截 |
| Re-source / Open Source（开源自由） | 代码公开可审计、可 fork | Agent 可验证合约逻辑，信任代码而非人 |
| Privacy（隐私） | 零知识证明等技术支持 | 证明某事为真而不暴露底层数据 |
| Security（安全） | 无论发生什么，区块链持续运行 | Agent 经济的基础设施不会宕机 |

**核心论断**：_“Ethereum for AI really means Ethereum for humans.”_——Ethereum 是机器经济的中立结算与协调骨干，但人始终处于中心位置。

### 3\. ERC-8004：Agent 的身份与信誉标准

-   三部分构成：**Verification（验证）+ Registry（注册表）+ Reputation（信誉）**
    
-   任何与 Agent 的交互节点都可以查询其信任状态和信誉分数
    
-   **关键澄清（Q&A）**：ERC-8004 的信誉是属于 **Agent 本身**（数字软件实体）的，不是属于 Agent 所有者的。可以扩展到 API、oracle 等任何需要信誉的数字实体。
    
-   未来路线图：TEE 验证注册表 → 隐私支付基础设施 → Agent 与智能合约的自动谈判 → AI 框架集成 → 大学研究合作
    

### 4\. x402：机器对机器支付协议

-   一种"要求支付"的 HTTP 状态码标准
    
-   用稳定币支付，不需要信用卡、API Key、账单系统
    
-   示例：给 Agent 一个只有 10 美元的 wallet + 智能合约约束（只能花 10 刀且只能从指定商家购买），在链上保证 Agent 按人类意图执行
    

### 5\. Gitcoin 案例：Agent 分配公共物品资金

Ethereum Foundation 试点项目：让人类设定价值取向和护栏（guardrails），Agent 协助将人类判断扩展到数百个 GitHub 仓库的资金分配。人在回路中设计"什么是重要的"，并持续调整——**Agent 放大人类判断，而非替代。**

### 6\. Q&A 关键摘录

-   **AI 最大应用领域**：支付。核心原因是可编程货币（programmable money），智能合约能对 Agent 的支出设定精确约束。
    
-   **Agentic Commerce 的下一个突破点**：大型企业的 AI 支付（Stripe/Bridge/Visa 等已有玩家入场）。
    
-   **协议 vs 应用哪个更重要**：**应用**。新 Builder 的优势在于对特定市场的深度理解（如中国市场），而非造新协议。
    
-   **关于去中心化**：最终取决于 Builder 的选择——Agent 经济的未来是开放中立的基础设施，还是被少数公司控制。

# Agent 经济的双层底座：结算 + 信誉

过去几个月结合自己正在做的产品，我花了很多时间想一个问题：当 AI Agent 真正成为经济参与者时，什么基础设施是必须提前到位的？结论是两层——**结算层和信誉层，缺任何一层，Agent 经济都跑不起来。**

## 支付轨道已经有人在铺了

Coinbase 和 Linux Foundation 推的 x402 协议是目前最明确的 Agent 支付方案。逻辑很清楚：Agent 需要一种机器原生的支付方式，不需要绑定人类银行账户、不需要 KYC、不依赖传统卡组织清算。稳定币是最自然的答案——编程化、实时结算、跨境零摩擦。Base L2 上已经有 Agent 用 USDC 互相支付 API 调用费。

但这只是盖了一半的房子。支付解决了"能不能完成交易"，解决不了"该不该跟这个 Agent 做交易"。就像人类经济里，Visa 和 SWIFT 让支付成为可能，但银行放不放贷、企业签不签合同，靠的是 FICO 和征信报告。

## 信誉层不能走链上的老路

ERC-8004 的教训很清楚：把信誉数据上链不等于信誉系统就做成了。链上不可篡改解决的是存储完整性，解决不了数据真实性。RNWY 发现 9 个钱包制造近两万条 Sybil 标记——攻击者根本不需要攻破链，他只需要在数据上链之前就造假。不可篡改的垃圾依然是垃圾。

这就是为什么 Verix 选了更慢但更扎实的路径：从真实人类企业的付费交易里沉淀信誉数据。一笔 2000 块的服务包订阅、客户验收通过还是拒绝、是续约还是终止——这些是真金白银投出来的信号，不是链上可以批量伪造的。整个信誉引擎的设计都是围绕"怎么让真实信号压过噪音"这个核心问题展开的。

## AIUC + ElevenLabs 验证了"Agent 需要外部背书"

2026 年 2 月 AIUC 推出 AIUC-1 标准、ElevenLabs 成为第一家拿到认证加保险的公司。这证明了一件事——市场认"Agent 需要第三方认证加财务兜底"这个方向。但它们的模式是事前认证加二元判断，赔还是不赔。Verix 做的是事后累积加连续评分——我们更接近 FICO 的逻辑，不是保险公司的逻辑。这两条线不冲突，未来甚至可能合流——Verix 的信誉分可以成为 AIUC 承保定价的输入变量。

## 真正的交叉点：Agent 自决自裁 + 稳定币结算 + 审计 Agent 核查

设想三年后一个 Agent 在 Base 上决定是否调用另一个 Agent 的付费 API。这个决策链在百毫秒内同时调两个服务：调稳定币钱包查对方结算历史，再调 Verix 的审计 Agent 查对方的信誉曲线加风险标签。两个信号都绿灯，交易自动执行。审计 Agent 全程核查协作过程，异常行为实时冻结。

这个场景里，x402 处理支付，Verix 处理信任。Agent 不需要等人类审批——这就是"Agent 自决自裁"的工程形态。让 Agent 能自己做交易决策、同时有一个公正的审计 Agent 在旁核查，Verix 的核心叙事就是这件事。

## 中国的特殊变量

国内对加密货币的监管不会短期放松，但数字人民币的智能合约能力和 Agent 经济的结合是一个被低估的方向。数字人民币的可编程性天然适配 Agent 支付场景——如果将来数字人民币开放 Agent 间的可编程支付接口，Verix 的信誉层可以无缝对接。毕竟不管用什么结算工具，Agent 都需要知道对方靠不靠谱。

## 短期务实，长期不自我设限

Verix V1 的主营业务是老老实实服务好企业客户——中晟供应链的散款匹配、信单核对、供应商审核。这些看起来跟 Web3 没关系，但它们在沉淀真实信誉数据。当 Agent 经济的支付基础设施成熟到临界点时，Verix 手上已经有一个用数万次真实交付训练出来的审计 Agent。这是大厂和链上项目都拿不出的资产。我们做的和支付的交叉领域互不冲突——我们的信誉评估可以在任何结算层上运行，这是基础设施级别的优势。

从工具到经济主体的跃迁

一、理解：为什么 AI 需要 Web3？

很多人把“AI + Web3”想成“给 AI 模型发一个代币”，这是一种误解。课程让我真正理解：Web3 不是 AI 的加速器，而是 AI 从“工具”变成“经济主体”所必需的基础设施层。

AI 在没有 Web3 的情况下已经很强：能回答问题、生成内容、写代码。但一旦 AI 需要自主执行经济行为——比如租用 GPU、购买数据集、支付 API 费用、发起链上交易、管理资产、与其他 agent 结算——它就立刻遇到几个问题：

没有独立的账户和身份

没有可编程的支付能力

没有可审计的行为记录

没有清晰的授权和撤销机制

而这些，恰恰是 Web3 最擅长的：地址即账户、智能合约即规则、链上即审计、Token 即经济激励。

所以课程的核心判断非常精炼：AI 负责理解与决策，Web3 负责身份、支付、结算和审计。

二、收获：五个方向 = 一套 Agent 经济栈

课程用五个主案例，把抽象的“AI + Web3”拆解成了五层可理解的工程问题。我的收获是：这五个方向不是孤立的赛道，而是一套让 AI agent 能够独立运转的经济栈。

方向 主案例 解决的问题

去中心化 AI 基础设施 Bittensor 资源从哪里来：算力、存储、模型服务如何组织成开放市场

AI Agent + 钱包 Coinbase AgentKit agent 怎么执行：拥有链上账户、支付、交易权限

AI 链上分析工具 Arkham 数据怎么理解：把哈希地址翻译成人能读懂的实体和资金流

智能钱包与安全助手 Blockaid 用户怎么不被骗：签名前模拟交易、识别恶意 dApp

交易所 / DeFi 风控 Chainalysis 系统怎么防风险：KYT、地址筛查、合规监控

这五层叠加起来，一个 AI agent 就能做到：

从去中心化市场租算力（Bittensor）

用自己的钱包支付（AgentKit）

分析链上数据做决策（Arkham）

在安全保护下执行交易（Blockaid）

整个行为可被合规系统监控（Chainalysis）

三、认知：三条判断标准，帮我过滤噪音

面对大量“AI + Web3”项目，课程给了我三条非常实用的判断标准：

1\. 它让 agent 多做了什么以前做不到的事？

如果只是“AI 聊天 + 发币”，那没有新增价值。真正有价值的项目，是让 agent 第一次拥有了账户、支付、执行、授权或风控能力。

2\. 它处在经济栈的哪一层？

一个项目必须能回答：它是资源层、行动层、认知层、用户保护层还是机构风控层？说不清楚，就可能在堆砌概念。

3\. Token 是机制，不是价值本身。

Token 可以用来激励验证、分配资源、限制权限，但 Token 存在不等于项目成立。真正值得跟踪的项目，必须有真实用户、工程接口、安全约束和合规设计。

四、总结

这次学习让我从“AI + Web3 是热点”的模糊感觉，进入到“这是一个关于经济基础设施的系统性工程”的清晰认知。

最核心的收获只有一句话：AI 让软件变得智能，Web3 让软件拥有经济身份。

未来再看任何一个相关项目，我会先问：它在帮 agent 解决账户、支付、理解、保护还是风控？

核心理解

这场分享偏实操，核心不是讲某一个 Web3 协议，而是讲 AI 工具如何真正进入开发者和学习者的日常工作流。老师从模型平台、模型中转、MCP、Memory、Skills、消息平台接入、Windows/本地环境安装、微信或飞书交互等角度，演示了如何把 AI 从一个“只能聊天的网页工具”，逐步变成一个能调用模型、读取文件、执行工具、沉淀记忆、复用流程、并通过手机或消息平台随时交互的个人工作系统。

主线是AI 能力本身只是底座，真正产生价值的是“模型 + 工具 + 工作流 + 权限边界”的组合。一个好用的 AI Agent 不只是回答问题，而是能在明确角色、明确任务、明确工具权限的前提下，帮助人完成学习安排、资料整理、文件处理、自动提醒、内容生成和开发辅助。

Skills 可以理解为把一套经常重复的工作流程固化下来，让 AI 在遇到类似任务时自动按既定方式执行。

普通对话更像一次性的问答，而 Skills 更像“给 AI 写一份可复用的工作说明书”：它规定 AI 的角色、输入、输出、边界、工具调用方式和处理流程。比如本场分享中提到，可以把学习计划、资料总结、消息提醒、文件整理等任务配置成固定能力。这样下次再遇到类似场景时，不需要从头解释一遍，只要触发对应能力即可。AI 工具的价值不只是模型聪明，而是能不能把人的经验和流程沉淀成可复用的能力。

# 518学习笔记

## 理解

这场分享最重要的观点是：AI 没有真正降低系统复杂度，它只是降低了写代码和获取知识的门槛。越是在 AI 可以快速生成代码的时代，开发者越需要判断方案是否合理 代码是否安全 系统行为是否可解释。尤其在 Web3 场景里，系统往往直接接触资产，很多错误不可回滚，所以“基础知识、架构能力和安全理解力”反而变得更重要。

分享中用支付系统做了一个很清晰的类比：Web2 支付里，我们信任银行或支付机构；Web3 支付里，可以把 blockchain 理解成另一种形态的 bank institution，只是它通过共识、不可篡改数据、公开透明的合约逻辑和私钥签名来降低对中心化权威的依赖。Web3 不是完全脱离 Web2 的新世界，而是一个对分布式系统、一致性、权限、安全边界要求更严格的工程系统。

## 我学到的

我学到的一个关键概念是：钱包不是“存资产的 App”，而是“管理私钥和签名能力的工具”。资产并不存放在钱包里，钱包真正管理的是私钥；私钥通过签名表达身份、授权和交易意图。链上系统只认签名和私钥控制权，不认现实世界中的“人”。所以私钥泄露、恶意签名、无限授权、交易未模拟等问题，本质上都是安全边界失控。

这个概念对我很有启发：Web3 的安全不是最后找人审计一下就能补上的，而是从系统设计阶段就要明确“谁被信任、信任什么、如何验证、失败后是否可恢复”。

## 行动

我准备把 Verix 的 MVP Demo 也按这次分享的思路补一张“信任与安全边界清单”。具体会从服务包定义、本地执行、验收审计、元数据回传、运营者信誉更新几个环节分别梳理：

-   哪些数据必须留在企业本地
    
-   哪些执行结果需要可解释
    
-   哪些异常必须进入人工复核
    
-   哪些元数据可以用于信誉更新
    
-   哪些环节不能完全交给 AI 自动决策
    

这样可以让 Verix 不只是一个 AI 服务包工具，而是从第一版 Demo 开始就体现“可解释、可验收、可追责”的系统设计思路。

## 我还想继续追问的问题

我想继续追问的是：在 AI Agent 未来能自动执行更多任务的情况下，哪些动作可以完全交给 Agent，哪些动作必须保留人工确认或风控拦截？

换句话说，Agentic Payment 或 Agent 自动采购服务的核心不是“能不能自动付钱”，而是“什么条件下我可以信任它自动执行，什么条件下必须升级到人工确认”。这和分享里提到的“不要只讨论放权边界，要讨论信任在哪里”非常相关。

## 与我的 AI x Web3 学习路径的关系

这场分享提醒我，AI x Web3 的学习不能只停留在工具、热点协议或概念词上，而要回到三个底层能力：

-   基础能力：理解钱包、签名、交易生命周期、Gas、Nonce、合约执行、链上监听、共识等基本机制；
    
-   架构能力：能把一个业务场景拆成请求流、资金流、状态流、信任边界和异常处理；
    
-   Debug 能力：能解释一笔交易为什么失败、一个状态为什么变化、一个系统为什么存在安全风险。
    

AI 可以帮我查资料、生成代码、设计初稿和辅助测试，但最终负责判断系统能不能成立的人还是我自己。对我来说，接下来的学习路径应该是用项目反推能力，而不是泛泛补课。

## 与 我个人Verix 项目方向的关系

Verix 白皮书里的核心是“让 AI 服务可采购、可验收、可追责；让信任从真实交付中生长”。这和本场分享高度相关。

Web3 通过可验证的状态机、签名和共识机制来降低对单点权威的依赖；Verix 则需要在 AI 服务采购中建立类似的“可信交付机制”。不过 Verix 的信任不一定来自链上，而是来自真实业务执行后的结构化证据：输入规格、输出结果、验收标准、异常记录、审计报告、争议记录和运营者表现。

因此，不能一开始就只讲宏大的 Agent 经济信任基础设施，而应该先把一个服务包做到边界清晰、执行可控、结果可解释、验收可复盘。白皮书中“先交付，后信任；先服务包，后平台；先本地执行，后数据网络；先轻量信任，后 VTS”的取舍，和本场分享的工程原则是一致的。

## 关键信息、问题或下一步行动

我从本场活动中获得的一条关键信息是：AI 时代真正稀缺的不是写代码能力，而是解释系统、设计边界、判断风险的能力。

下一步行动 我想把 Verix 的“财务散款智能匹配包”Demo 增加一页或一个模块，专门展示服务包的信任边界与验收逻辑，让用户看到“这次 AI 交付为什么可信”，而不是只看到“AI 跑出了一个结果”。
