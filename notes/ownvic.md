# AI & Web3 School Bonus Session: Open Agentic Economy

* * *

## 一、核心内容复盘

### 1\. AI Agent 正在成为新的经济参与者

-   过去 18 个月，LLM 从"聊天机器人"演进为能写代码、转移资金、做决策、谈判、雇佣其他 Agent 的自主实体。
    
-   核心观点：**AI 不再只是一个被查询的工具，它正在成为一种新型经济参与者。**
    
-   如果 Agent 是经济参与者，它需要什么样的基础设施？——支付轨道、身份、信誉、可执行协议。
    

### 2\. 为什么 Ethereum 是 Agent 经济的关键基础设施

Sophia 用 **CROPS** 框架解释 Ethereum 的独特属性：

| 原则 | 含义 | 对 Agent 经济的意义 |
| --- | --- | --- |
| Censorship Resistance（抗审查） | 无人能阻止有效交易或合约执行 | Agent 的交易不会被任意拦截 |
| Re-source / Open Source（开源自由） | 代码公开可审计、可 fork | Agent 可验证合约逻辑，信任代码而非人 |
| Privacy（隐私） | 零知识证明等技术支持 | 证明某事为真而不暴露底层数据 |
| Security（安全） | 无论发生什么，区块链持续运行 | Agent 经济的基础设施不会宕机 |

**核心论断**：_“Ethereum for AI really means Ethereum for humans.”_——Ethereum 是机器经济的中立结算与协调骨干，但人始终处于中心位置。

### 3\. ERC-8004：Agent 的身份与信誉标准

-   三部分构成：**Verification（验证）+ Registry（注册表）+ Reputation（信誉）**
    
-   任何与 Agent 的交互节点都可以查询其信任状态和信誉分数
    
-   **关键澄清（Q&A）**：ERC-8004 的信誉是属于 **Agent 本身**（数字软件实体）的，不是属于 Agent 所有者的。可以扩展到 API、oracle 等任何需要信誉的数字实体。
    
-   未来路线图：TEE 验证注册表 → 隐私支付基础设施 → Agent 与智能合约的自动谈判 → AI 框架集成 → 大学研究合作
    

### 4\. x402：机器对机器支付协议

-   一种"要求支付"的 HTTP 状态码标准
    
-   用稳定币支付，不需要信用卡、API Key、账单系统
    
-   示例：给 Agent 一个只有 10 美元的 wallet + 智能合约约束（只能花 10 刀且只能从指定商家购买），在链上保证 Agent 按人类意图执行
    

### 5\. Gitcoin 案例：Agent 分配公共物品资金

Ethereum Foundation 试点项目：让人类设定价值取向和护栏（guardrails），Agent 协助将人类判断扩展到数百个 GitHub 仓库的资金分配。人在回路中设计"什么是重要的"，并持续调整——**Agent 放大人类判断，而非替代。**

### 6\. Q&A 关键摘录

-   **AI 最大应用领域**：支付。核心原因是可编程货币（programmable money），智能合约能对 Agent 的支出设定精确约束。
    
-   **Agentic Commerce 的下一个突破点**：大型企业的 AI 支付（Stripe/Bridge/Visa 等已有玩家入场）。
    
-   **协议 vs 应用哪个更重要**：**应用**。新 Builder 的优势在于对特定市场的深度理解（如中国市场），而非造新协议。
    
-   **关于去中心化**：最终取决于 Builder 的选择——Agent 经济的未来是开放中立的基础设施，还是被少数公司控制。

## 二、一个学到的概念/方法/案例

**概念：CROPS 设计原则框架**

CROPS（抗审查、开源自由、隐私、安全）不只是一个缩写，它提供了一套判断"某条链/基础设施是否适合 Agent 经济"的评估维度。过去我更多从性能（TPS、Gas 费）或生态（DeFi 规模）角度评价公链，Sophia 的框架让我意识到：对 Agent 经济而言，**中立性、可验证性、抗审查能力**可能比性能更重要。Agent 需要的不是最快的网络，而是一个不会被任意关停或篡改的结算层。

# Agent 经济的双层底座：结算 + 信誉

过去几个月结合自己正在做的产品，我花了很多时间想一个问题：当 AI Agent 真正成为经济参与者时，什么基础设施是必须提前到位的？结论是两层——**结算层和信誉层，缺任何一层，Agent 经济都跑不起来。**

## 支付轨道已经有人在铺了

Coinbase 和 Linux Foundation 推的 x402 协议是目前最明确的 Agent 支付方案。逻辑很清楚：Agent 需要一种机器原生的支付方式，不需要绑定人类银行账户、不需要 KYC、不依赖传统卡组织清算。稳定币是最自然的答案——编程化、实时结算、跨境零摩擦。Base L2 上已经有 Agent 用 USDC 互相支付 API 调用费。

但这只是盖了一半的房子。支付解决了"能不能完成交易"，解决不了"该不该跟这个 Agent 做交易"。就像人类经济里，Visa 和 SWIFT 让支付成为可能，但银行放不放贷、企业签不签合同，靠的是 FICO 和征信报告。

## 信誉层不能走链上的老路

ERC-8004 的教训很清楚：把信誉数据上链不等于信誉系统就做成了。链上不可篡改解决的是存储完整性，解决不了数据真实性。RNWY 发现 9 个钱包制造近两万条 Sybil 标记——攻击者根本不需要攻破链，他只需要在数据上链之前就造假。不可篡改的垃圾依然是垃圾。

这就是为什么 Verix 选了更慢但更扎实的路径：从真实人类企业的付费交易里沉淀信誉数据。一笔 2000 块的服务包订阅、客户验收通过还是拒绝、是续约还是终止——这些是真金白银投出来的信号，不是链上可以批量伪造的。整个信誉引擎的设计都是围绕"怎么让真实信号压过噪音"这个核心问题展开的。

## AIUC + ElevenLabs 验证了"Agent 需要外部背书"

2026 年 2 月 AIUC 推出 AIUC-1 标准、ElevenLabs 成为第一家拿到认证加保险的公司。这证明了一件事——市场认"Agent 需要第三方认证加财务兜底"这个方向。但它们的模式是事前认证加二元判断，赔还是不赔。Verix 做的是事后累积加连续评分——我们更接近 FICO 的逻辑，不是保险公司的逻辑。这两条线不冲突，未来甚至可能合流——Verix 的信誉分可以成为 AIUC 承保定价的输入变量。

## 真正的交叉点：Agent 自决自裁 + 稳定币结算 + 审计 Agent 核查

设想三年后一个 Agent 在 Base 上决定是否调用另一个 Agent 的付费 API。这个决策链在百毫秒内同时调两个服务：调稳定币钱包查对方结算历史，再调 Verix 的审计 Agent 查对方的信誉曲线加风险标签。两个信号都绿灯，交易自动执行。审计 Agent 全程核查协作过程，异常行为实时冻结。

这个场景里，x402 处理支付，Verix 处理信任。Agent 不需要等人类审批——这就是"Agent 自决自裁"的工程形态。让 Agent 能自己做交易决策、同时有一个公正的审计 Agent 在旁核查，Verix 的核心叙事就是这件事。

## 中国的特殊变量

国内对加密货币的监管不会短期放松，但数字人民币的智能合约能力和 Agent 经济的结合是一个被低估的方向。数字人民币的可编程性天然适配 Agent 支付场景——如果将来数字人民币开放 Agent 间的可编程支付接口，Verix 的信誉层可以无缝对接。毕竟不管用什么结算工具，Agent 都需要知道对方靠不靠谱。

## 短期务实，长期不自我设限

Verix V1 的主营业务是老老实实服务好企业客户——中晟供应链的散款匹配、信单核对、供应商审核。这些看起来跟 Web3 没关系，但它们在沉淀真实信誉数据。当 Agent 经济的支付基础设施成熟到临界点时，Verix 手上已经有一个用数万次真实交付训练出来的审计 Agent。这是大厂和链上项目都拿不出的资产。我们做的和支付的交叉领域互不冲突——我们的信誉评估可以在任何结算层上运行，这是基础设施级别的优势。

从工具到经济主体的跃迁

一、理解：为什么 AI 需要 Web3？

很多人把“AI + Web3”想成“给 AI 模型发一个代币”，这是一种误解。课程让我真正理解：Web3 不是 AI 的加速器，而是 AI 从“工具”变成“经济主体”所必需的基础设施层。

AI 在没有 Web3 的情况下已经很强：能回答问题、生成内容、写代码。但一旦 AI 需要自主执行经济行为——比如租用 GPU、购买数据集、支付 API 费用、发起链上交易、管理资产、与其他 agent 结算——它就立刻遇到几个问题：

没有独立的账户和身份

没有可编程的支付能力

没有可审计的行为记录

没有清晰的授权和撤销机制

而这些，恰恰是 Web3 最擅长的：地址即账户、智能合约即规则、链上即审计、Token 即经济激励。

所以课程的核心判断非常精炼：AI 负责理解与决策，Web3 负责身份、支付、结算和审计。

二、收获：五个方向 = 一套 Agent 经济栈

课程用五个主案例，把抽象的“AI + Web3”拆解成了五层可理解的工程问题。我的收获是：这五个方向不是孤立的赛道，而是一套让 AI agent 能够独立运转的经济栈。

方向 主案例 解决的问题

去中心化 AI 基础设施 Bittensor 资源从哪里来：算力、存储、模型服务如何组织成开放市场

AI Agent + 钱包 Coinbase AgentKit agent 怎么执行：拥有链上账户、支付、交易权限

AI 链上分析工具 Arkham 数据怎么理解：把哈希地址翻译成人能读懂的实体和资金流

智能钱包与安全助手 Blockaid 用户怎么不被骗：签名前模拟交易、识别恶意 dApp

交易所 / DeFi 风控 Chainalysis 系统怎么防风险：KYT、地址筛查、合规监控

这五层叠加起来，一个 AI agent 就能做到：

从去中心化市场租算力（Bittensor）

用自己的钱包支付（AgentKit）

分析链上数据做决策（Arkham）

在安全保护下执行交易（Blockaid）

整个行为可被合规系统监控（Chainalysis）

三、认知：三条判断标准，帮我过滤噪音

面对大量“AI + Web3”项目，课程给了我三条非常实用的判断标准：

1\. 它让 agent 多做了什么以前做不到的事？

如果只是“AI 聊天 + 发币”，那没有新增价值。真正有价值的项目，是让 agent 第一次拥有了账户、支付、执行、授权或风控能力。

2\. 它处在经济栈的哪一层？

一个项目必须能回答：它是资源层、行动层、认知层、用户保护层还是机构风控层？说不清楚，就可能在堆砌概念。

3\. Token 是机制，不是价值本身。

Token 可以用来激励验证、分配资源、限制权限，但 Token 存在不等于项目成立。真正值得跟踪的项目，必须有真实用户、工程接口、安全约束和合规设计。

四、总结

这次学习让我从“AI + Web3 是热点”的模糊感觉，进入到“这是一个关于经济基础设施的系统性工程”的清晰认知。

最核心的收获只有一句话：AI 让软件变得智能，Web3 让软件拥有经济身份。

未来再看任何一个相关项目，我会先问：它在帮 agent 解决账户、支付、理解、保护还是风控？

核心理解

这场分享偏实操，核心不是讲某一个 Web3 协议，而是讲 AI 工具如何真正进入开发者和学习者的日常工作流。老师从模型平台、模型中转、MCP、Memory、Skills、消息平台接入、Windows/本地环境安装、微信或飞书交互等角度，演示了如何把 AI 从一个“只能聊天的网页工具”，逐步变成一个能调用模型、读取文件、执行工具、沉淀记忆、复用流程、并通过手机或消息平台随时交互的个人工作系统。

主线是AI 能力本身只是底座，真正产生价值的是“模型 + 工具 + 工作流 + 权限边界”的组合。一个好用的 AI Agent 不只是回答问题，而是能在明确角色、明确任务、明确工具权限的前提下，帮助人完成学习安排、资料整理、文件处理、自动提醒、内容生成和开发辅助。

Skills 可以理解为把一套经常重复的工作流程固化下来，让 AI 在遇到类似任务时自动按既定方式执行。

普通对话更像一次性的问答，而 Skills 更像“给 AI 写一份可复用的工作说明书”：它规定 AI 的角色、输入、输出、边界、工具调用方式和处理流程。比如本场分享中提到，可以把学习计划、资料总结、消息提醒、文件整理等任务配置成固定能力。这样下次再遇到类似场景时，不需要从头解释一遍，只要触发对应能力即可。AI 工具的价值不只是模型聪明，而是能不能把人的经验和流程沉淀成可复用的能力。

# 518学习笔记

## 理解

这场分享最重要的观点是：AI 没有真正降低系统复杂度，它只是降低了写代码和获取知识的门槛。越是在 AI 可以快速生成代码的时代，开发者越需要判断方案是否合理 代码是否安全 系统行为是否可解释。尤其在 Web3 场景里，系统往往直接接触资产，很多错误不可回滚，所以“基础知识、架构能力和安全理解力”反而变得更重要。

分享中用支付系统做了一个很清晰的类比：Web2 支付里，我们信任银行或支付机构；Web3 支付里，可以把 blockchain 理解成另一种形态的 bank institution，只是它通过共识、不可篡改数据、公开透明的合约逻辑和私钥签名来降低对中心化权威的依赖。Web3 不是完全脱离 Web2 的新世界，而是一个对分布式系统、一致性、权限、安全边界要求更严格的工程系统。

## 我学到的

我学到的一个关键概念是：钱包不是“存资产的 App”，而是“管理私钥和签名能力的工具”。资产并不存放在钱包里，钱包真正管理的是私钥；私钥通过签名表达身份、授权和交易意图。链上系统只认签名和私钥控制权，不认现实世界中的“人”。所以私钥泄露、恶意签名、无限授权、交易未模拟等问题，本质上都是安全边界失控。

这个概念对我很有启发：Web3 的安全不是最后找人审计一下就能补上的，而是从系统设计阶段就要明确“谁被信任、信任什么、如何验证、失败后是否可恢复”。

## 行动

我准备把 Verix 的 MVP Demo 也按这次分享的思路补一张“信任与安全边界清单”。具体会从服务包定义、本地执行、验收审计、元数据回传、运营者信誉更新几个环节分别梳理：

-   哪些数据必须留在企业本地
    
-   哪些执行结果需要可解释
    
-   哪些异常必须进入人工复核
    
-   哪些元数据可以用于信誉更新
    
-   哪些环节不能完全交给 AI 自动决策
    

这样可以让 Verix 不只是一个 AI 服务包工具，而是从第一版 Demo 开始就体现“可解释、可验收、可追责”的系统设计思路。

## 我还想继续追问的问题

我想继续追问的是：在 AI Agent 未来能自动执行更多任务的情况下，哪些动作可以完全交给 Agent，哪些动作必须保留人工确认或风控拦截？

换句话说，Agentic Payment 或 Agent 自动采购服务的核心不是“能不能自动付钱”，而是“什么条件下我可以信任它自动执行，什么条件下必须升级到人工确认”。这和分享里提到的“不要只讨论放权边界，要讨论信任在哪里”非常相关。

## 与我的 AI x Web3 学习路径的关系

这场分享提醒我，AI x Web3 的学习不能只停留在工具、热点协议或概念词上，而要回到三个底层能力：

-   基础能力：理解钱包、签名、交易生命周期、Gas、Nonce、合约执行、链上监听、共识等基本机制；
    
-   架构能力：能把一个业务场景拆成请求流、资金流、状态流、信任边界和异常处理；
    
-   Debug 能力：能解释一笔交易为什么失败、一个状态为什么变化、一个系统为什么存在安全风险。
    

AI 可以帮我查资料、生成代码、设计初稿和辅助测试，但最终负责判断系统能不能成立的人还是我自己。对我来说，接下来的学习路径应该是用项目反推能力，而不是泛泛补课。

## 与 我个人Verix 项目方向的关系

Verix 白皮书里的核心是“让 AI 服务可采购、可验收、可追责；让信任从真实交付中生长”。这和本场分享高度相关。

Web3 通过可验证的状态机、签名和共识机制来降低对单点权威的依赖；Verix 则需要在 AI 服务采购中建立类似的“可信交付机制”。不过 Verix 的信任不一定来自链上，而是来自真实业务执行后的结构化证据：输入规格、输出结果、验收标准、异常记录、审计报告、争议记录和运营者表现。

因此，不能一开始就只讲宏大的 Agent 经济信任基础设施，而应该先把一个服务包做到边界清晰、执行可控、结果可解释、验收可复盘。白皮书中“先交付，后信任；先服务包，后平台；先本地执行，后数据网络；先轻量信任，后 VTS”的取舍，和本场分享的工程原则是一致的。

## 关键信息、问题或下一步行动

我从本场活动中获得的一条关键信息是：AI 时代真正稀缺的不是写代码能力，而是解释系统、设计边界、判断风险的能力。

下一步行动 我想把 Verix 的“财务散款智能匹配包”Demo 增加一页或一个模块，专门展示服务包的信任边界与验收逻辑，让用户看到“这次 AI 交付为什么可信”，而不是只看到“AI 跑出了一个结果”。
