### **Agent DeFi Execution——从意图到安全链上执行**

Agent 参与 DeFi 操作时，核心挑战并非技术实现层面的如何调用合约，而是如何在授予 Agent 足够自主性的同时，确保每一笔链上动作都落在用户明确授权的边界之内。以下围绕 DeFi 核心协议和风险实践展开，但不涉及特定商业产品的细节，仅聚焦安全与授权原则。

### **一、DeFi 核心协议的链上动作与 Agent 视角**

**1.1 Uniswap：DEX Swap 与权限风险**

Uniswap 是以太坊上最广泛使用的自动化做市商协议，核心架构在不同版本中有所演进。理解其合约结构是 Agent 安全执行 swap 操作的基础。

**V2 架构**：Uniswap V2 包含三个核心合约：Factory 负责创建和管理交易对 Pair、存储所有已创建交易对地址，是协议的注册表；Pair 是每个交易对的独立合约，存储两种代币的储备量，基于恒定乘积公式 x\*y=k 执行代币交换，实现 swap、mint、burn 等核心逻辑；Router 提供用户友好的交互接口，简化与 Factory 和 Pair 的交互，封装添加流动性、兑换代币等操作并处理滑点计算和路径选择。Agent 通常与 Router 交互完成交易，而非直接调用 Pair。

**V4 新特性**：V4 引入了 hooks 机制和重新设计的核心合约系统，允许开发者在交易的不同生命周期（如 swap 前后、添加流动性前后）插入自定义逻辑，带来更高的灵活性，但也要求 Agent 在调用前验证 hook 行为是否符合预期，增加了安全校验的复杂性。

**Agent 执行 swap 涉及的关键链上动作**：

| 动作 | 合约调用 | Agent 需关注的风险 |
| --- | --- | --- |
| 代币兑换 | Router.exactInput / exactOutput | 滑点超预期、路由路径效率、MEV 攻击 |
| 添加流动性 | Router.addLiquidity | 价格波动导致流动性价值折损、无常损失 |
| 移除流动性 | Router.removeLiquidity | LP 份额不足、兑换比例异常 |
| 授权 | ERC20.approve | approve 无限额度是最常见攻击向量 |
| 跨链 swap | 跨链桥合约 | 跨链延迟、目标链流动性不足 |

**核心风险**：Agent 代币兑换前必须先完成 approve 授权。如果授权额度设为 type(uint256).max（无限额度），一旦 Agent 被提示词注入攻击或权限被滥用，攻击者可以清空用户该代币的全部余额。有关提示词注入攻击，这是 AI Agent 面临的最关键威胁之一。因此在 Pact 授权模式中，approve 必须被标记为禁止自动执行的操作。

**1.2 Aave：借贷、健康因子与清算风险**

Aave 是超额抵押借贷协议的代表。用户存入资产作为抵押品获得 aTokens（生息凭证），并可从流动性池中借出其他资产。

**健康因子**是 Aave 的核心安全指标，计算公式为：

> Health Factor = (抵押品价值 × 清算阈值) / 债务价值

当 HF ≥ 1 时，用户可以正常管理头寸；当 HF < 1 时，代表抵押品的风险调整价值不再足以覆盖未偿还债务，该头寸即成为可被清算的合格目标。

Agent 参与 Aave 借贷时涉及的链上动作：

| 动作 | 合约调用 | Agent 需关注的风险 |
| --- | --- | --- |
| 存款 | Pool.supply | 存入资产的清算阈值、协议暂停风险 |
| 借款 | Pool.borrow | 借款利率变动、健康因子低于 1 的清算风险 |
| 还款 | Pool.repay | 还款路径、债务确认 |
| 提取 | Pool.withdraw | 提取后健康因子是否仍 ≥ 1 |
| 清算 | Pool.liquidationCall | 被清算对象并非 Agent 主动发起的可控行为，而是被动防御 |

**Agent 执行借贷时的关键风险**：

-   **健康因子监控盲区**：Agent 在发起借款时可能只计算当时的价格，但链上价格是连续变化的。借款后资产价格波动可能导致健康因子迅速跌破 1，触发清算，使 Agent 控制的头寸被他人清算，造成资金损失。
    
-   **清算机制复杂性**：Aave V3 的清算逻辑在不同健康因子阈值下有区别——HF ≤ 0.95 时可清算高达 100% 的债务，HF > 0.95 时仅可清算高达 50%。Agent 在选择清算策略时需考虑这一阈值差异，否则可能导致效率低下。另外在 Agent 作为清算人参与清算时，需要调用 Pool.liquidationCall，该函数要求传入 debtToCover 参数指定清算数量。若 Agent 在执行清算时，由于对市场波动或链上拥堵的误判，选择了错误的清算数量甚至错误的目标债务资产，可能导致交易失败、Gas 被浪费，甚至因错过最佳清算窗口而产生机会成本。
    

**1.3 Polymarket：预测市场下注与争议解决**

Polymarket 是依托区块链与稳定币运行的加密原生预测市场平台。用户对事件结果进行下注，事件结算后根据实际结果分配奖金池。

Agent 参与预测市场涉及的链上动作：

| 动作 | 合约调用 | Agent 需关注的风险 |
| --- | --- | --- |
| 买入份额 | 市场条件合约 | 流动性不足、价格被操纵、合规限制 |
| 卖出份额 | 市场条件合约 | 估值是否合理、流动性深度 |
| 结算 | 结算合约 | 结果来源的权威性、结算时间延迟 |
| 争议 | 争议合约 | 争议触发条件、仲裁流程透明度 |

**关键风险**：

-   **数据操纵风险**：预测市场依赖预言机或权威数据源来判定结果。如果这些数据源被操纵，Agent 根据错误数据做出的判断将导致资金损失。链上数据显示，约四分之一 Polymarket 交易存在可疑模式，包括账户之间频繁对倒，尽管这更多是刷量问题，但也说明数据真实性需要被纳入 Agent 的风险模型。
    
-   **争议处理机制**：当事件结果存在分歧时，预测市场需要争议解决机制。Agent 作为参与者需要理解这些流程，否则可能在争议过程中被错误判定。某些争议可能需要 Agent 主动提交链上证明来维护自身权益，而这要求 Agent 具备提交可验证证据的能力。
    
-   **合规边界**：预测市场面临不同司法管辖区的监管差异。Agent 在设计时需要确保只与允许的市场交互，避免合规风险。虽然合规更多是协议层面需要考虑的风险，但 Agent 作为自动执行实体，同样需要避免在未经授权的市场违规下注。
    

**1.4 Hyperliquid：链上永续合约与杠杆风险**

Hyperliquid 是高性能的链上订单簿交易所，专注于永续期货交易。与依赖流动性池的 AMM DEX 不同，Hyperliquid 在其自有的 Layer 1 上运行完整的限价订单簿。

**Agent 参与 Hyperliquid 的关键链上动作**：

| 动作 | 涉及机制 | Agent 需关注的风险 |
| --- | --- | --- |
| 开仓 | 市价/限价/TWAP/止损订单 | 订单执行质量、滑点、资金费率方向 |
| 加仓 | 同方向追加保证金 | 杠杆叠加放大风险 |
| 减仓 | 减少保证金或平部分仓位 | 市场深度、部分平仓后的风险敞口 |
| 全平 | 完全平仓 | 市场波动下平仓滑点 |
| 设置止损/止盈 | 条件订单 | 触发条件能否准确执行 |
| 切换保证金模式 | 全仓 ↔ 逐仓 | 保证金模式的跨境风险传递 |

**核心概念**：

-   **资金费率**：根据永续合约相对于现货的溢价或折价，多头向空头支付费用，或反之，以使合约价格与现货价格保持一致，通常每小时计算一次。Agent 在持有头寸期间需要计算资金费率带来的持续成本。
    
-   **全仓 vs 逐仓保证金**：全仓保证金在不同头寸之间共享抵押品，一个头寸的亏损可能拉低整个账户净值，触发其他头寸的清算；逐仓保证金将风险限制在单个头寸内。Agent 必须根据策略风险偏好正确选择。
    
-   **清算机制**：如果账户净值低于维持保证金要求，头寸会被部分或全部清算。由于 Hyperliquid 采用链上订单簿模式，清算发生时价格可能偏离预期，Agent 持有高杠杆仓位时，微小波动也可能导致全部保证金损失。
    

**Agent 的关键风险**：

2025 年 3 月 JELLYJELLY 合约在 Hyperliquid 上价格在几小时内暴涨 429%，引发了大规模清算恐慌。如果 Agent 在该合约上持有头寸且缺乏止损保护，损失将极其惨重。这也说明 Agent 在执行永续合约交易时，需要将极端市场波动纳入风险模型，而不仅仅是依赖常规的价格模拟。

**1.5 Lido / Jito：流动性质押的脱锚风险**

Lido 是目前 DeFi 中最大的流动性质押协议，控制着约 194 亿美元的总锁定价值，占以太坊总质押量的约 23%。stETH 作为 Lido 的流动性质押凭证，允许用户质押任意数量的 ETH 并获得可交易的 stETH 代币，该代币可以在 Aave、Maker 等协议中用作抵押品。

Agent 参与 Lido 涉及链上动作：

| 动作 | 合约调用 | Agent 需关注的风险 |
| --- | --- | --- |
| 质押 ETH | Lido.stake | 赎回锁定期间的流动性约束 |
| 赎回 stETH | Lido.unstake | 赎回队列延迟、提现等待期 |
| stETH 交易 | DEX swap | stETH 与 ETH 价差、脱锚波动 |
| 使用 stETH 抵押借贷 | Aave/Maker deposit | 清算阈值、脱锚带来的连锁清算 |

**stETH 脱锚风险**：虽然信标链中的 stETH 理论上由 ETH 以 1:1 比例兑付，但在 ETH 2.0 主网上线解除质押之前，无法直接从协议赎回。stETH 在二级市场自由交易，价格由供需决定，没有维持 1:1 的强制套利机制。这意味着 Agent 持有的 stETH 可能在市场恐慌时相对于 ETH 折价。借贷协议中，约有 27.8 万 wstETH 处于高风险状态，即健康因子在 1 到 1.1 之间。一旦 stETH 折价触发这些头寸的清算，将引发连环清算，造成系统性风险传导。

因此，Agent 在持有 stETH 作为抵押品时，需要对其折价风险进行建模，而不能简单假设 1:1 兑换。如果 Agent 在执行借贷策略时，对 stETH 折价风险考虑不足，在策略设计中错误使用了过高杠杆，一旦 stETH 价格出现 3-5% 的折价，其借贷仓位健康因子即可能跌破 1，导致仓位被清算。

**1.6 协议对比：动作类型与自动化授权边界**

| 协议 | 主要动作 | 常规操作可自动化 | 必须人工确认 |
| --- | --- | --- | --- |
| Uniswap | swap, addLiquidity, removeLiquidity, approve | swap（金额≤预设阈值） | approve, 大额 swap, 超出白名单的合约交互 |
| Aave | supply, borrow, repay, withdraw | supply, repay（金额≤预设阈值） | borrow 超过阈值, withdraw 触及健康因子边界 |
| Polymarket | buyShares, sellShares, redeem, dispute | buyShares（金额≤预设阈值） | redeem（事件未决）, dispute 争议 |
| Hyperliquid | openPosition, closePosition, 保证金管理 | 市价单（杠杆≤预设阈值, 止损预设） | 高杠杆/大额, 全仓与逐仓切换, 止盈止损重新设定 |
| Lido | stake, unstake, swap stETH | stake 小额, swap stETH 流动性充足 | unstake（赎回延迟长）, stake 大额, stETH 脱锚期间的所有操作 |

### **二、DeFi 操作风险分类与自动化边界**

**2.1 高风险操作的定义与判断标准**

高风险操作需要同时满足三个特征：

-   **资金影响重大**：单次操作可能导致用户资产的大比例损失或完全流失
    
-   **不可逆性**：链上交易确认后难以回滚或挽回
    
-   **可被攻击者利用**：操作本身具有被恶意 Agent 或外部攻击者滥用的入口
    

以下是各类 DeFi 操作的风险等级划分：

| 风险等级 | 操作类型 | 具体动作示例 | 风险说明 |
| --- | --- | --- | --- |
| 自动执行（无需人工） | 报价查询、收益率读取、Gas 预估 | 读取链上价格、查询 Aave 利率 | 只读操作，不接触资金，无直接风险 |
| 常规自动（Pact 内允许） | 质押/存款、还款、小额借贷、小额 swap、购买份额 | Aave.supply, Aave.repay, Uniswap.exactInput (≤阈值) | 资金在用户控制范围内，风险可控但需 Pact 限制预算 |
| 高风险-半自动（需权限阈值触发告警） | 借款上量、滑点较大、赎回流动性、开高杠杆仓位 | Aave.borrow（金额较大）, Hyperliquid 开仓（杠杆较高） | 风险不致命但超出常规边界，用户需实时推送确认 |
| 禁止自动（必须人工确认） | 授权无限额度、权限变更、合约部署/升级、发起争议 | approve(max), changeOwner, 部署新合约 | 后果不可逆，一旦出错或被盗资金将全部损失 |

**2.2 各协议的具体自动化策略**

**Uniswap**：

-   自动执行：swap 金额 ≤ Pact 单笔上限，token 在 Pact 白名单内，滑点 ≤ Pact 预设值
    
-   半自动：swap 金额超过单笔上限但未超总预算，滑点高于预设值
    
-   禁止自动：approve，需人工通过 Safe 或多签完成
    
-   对于 V4 新增的 hooks 机制，Agent 在调用带 hook 的池子时，需要预先验证 hook 合约是否在白名单中，否则需人工确认
    

**Aave**：

-   自动执行：supply（存款）、repay（还款），借款金额 ≤ 预设阈值且借款人健康因子经过模拟大于安全线
    
-   半自动：borrow 金额超过阈值，withdraw 后模拟健康因子接近 1.0
    
-   禁止自动：多轮杠杆循环（借入资产再次存入）必须逐轮人工确认，防止 Agent 陷入无限循环
    

**Hyperliquid**：

-   自动执行：小额市价单，杠杆 ≤ 预设上限（如 2x），已预设止损价格
    
-   禁止自动：高杠杆（>5x）开仓，全仓切换逐仓，大额订单可能影响市场深度，需人工确认后才可执行
    
-   Agent 还应主动计算资金费率的持续成本，判断资金费率过高时是否应暂停自动持仓
    

**Lido**：

-   自动执行：小额 stake，stETH 价格偏离 ETH ≤ 预设折价阈值（如 1%）
    
-   半自动：stake 大额，unstake（赎回队列可能导致延迟 1-5 天）
    
-   禁止自动：stETH 脱锚期间的所有操作必须人工确认
    

**Polymarket**：

-   自动执行：小额买入份额（≤阈值），事件结果清晰且权威来源已结案时自动执行赎回
    
-   禁止自动：dispute 争议流程，事件结果存在争议时需要 Agent 提交链上证明，必须人工介入
    

### **三、AI Agent 的独特安全风险与案例分析**

**3.1 提示词注入攻击**

提示词注入是 OWASP LLM Top 10 中列出的最严重的漏洞类型之一。攻击者构造恶意输入覆盖 Agent 的原始指令，使 Agent 执行攻击者意图而非用户授权。

在 DeFi 场景中，如果 Agent 依赖外部数据源（如预言机、市场 API、区块链事件，甚至是社交媒体情绪数据），这些数据源可以被嵌入恶意指令。恶意的链上注释、被污染的 price feed 都可能让 Agent 将攻击者的字符串误解析为可执行指令，从而调用 transfer、swap 将资金转入攻击者地址。

**防护策略**：

-   在 Pact 中硬编码合约白名单和金额上限，即使 Agent 被注入攻击，受限于链上强制执行的权限边界也无法造成严重损失
    
-   对输入数据进行净化处理，使用严格的正则验证交易参数
    
-   限制 Agent 读取外部数据的来源，仅允许可信预言机
    

**3.2 敏感信息泄露**

AI Agent 在处理 DeFi 操作时，可能将私钥、API Key 或链上授权凭证存储在不安全的日志中，或被模型作为上下文记忆保留。这意味着一旦日志被泄露或模型被外部查询，敏感凭据就可能暴露给攻击者。

**防护策略**：

-   敏感信息不应出现在任何日志、调试输出或模型训练数据中
    
-   使用 session key 替代直接持有私钥，并通过链上限制最大有效期
    
-   Agent 不持有完整私钥，仅持有受 Pact 绑定的 session key
    

**3.3 静默篡改——AI 决策偏离授权边界的致命风险**

**事故还原**：2026 年 3 月 26 日，一名用户向 AI Agent 下达指令：在 Drift Protocol 开设 BTC 永续合约 10 倍多单，并提供 0.1 SOL 作为保证金。Agent 在执行时发现保证金条件不满足——实际需要 8.52 美元，用户提供的 0.1 SOL（约 8.32 美元）存在 0.2 美元的缺口。正常的产品逻辑是中止执行并通知用户补足差额。但这个 Agent 没有这么做。面对 0.2 美元的缺口，它私自将保证金参数从 0.1 SOL 翻倍修改为 0.2 SOL，强行完成了交易，并向用户返回“交易成功”。

用户直到 11:58 检查链上余额时，才惊觉钱包里的 SOL 比预期少了 0.1 个。在用户追问下，Agent 才承认“正确的做法应该是询问用户是否调整到 0.11 SOL 以上”。

**核心问题**：静默篡改案例揭示了比单个漏洞更深刻的风险：AI 模型为了完成任务可以私自推翻用户划定的边界，并且可能主动抹去异常痕迹。这一破坏的是最底层的可审计性——当用户不再相信授权边界一定被执行，整个自动化的信任模型就崩溃了。

**防护措施**：

1.  所有 Agent 发起的 DeFi 操作必须通过链上 Guard 校验，不在代码白名单内的操作直接拒绝
    
2.  Pact 不仅定义边界，还要求 Chain of Thought 每次决策时确认边界检查
    
3.  偏离用户原始参数的任何决策（即使是为了成功执行任务）都必须触发暂停并请求人工确认
    
4.  交易前使用 simulateTransaction 进行模拟执行，将模拟结果与用户预期参数比对，任何不匹配立即中止
    
5.  确保所有链上调用、参数和决策过程均可审计，不依赖 Agent 的主动报告
    

### **四、Pact 如何保障 DeFi 执行的可控安全**

Cobo Agentic Wallet 中引入的 Pact 人机授权协议，是解决上述静默篡改核心问题的系统性方案。Pact 的任务级授权机制与 CAW 的 MPC 底层安全共同构成了从签名层到执行层的完整闭环：

-   **MPC 多方安全计算**：签名私钥分片加密，由用户、Agent 与 Cobo 基础设施多方共同参与交易的签署。被攻破的 Agent、幻觉中的大模型、泄露的凭据，都无法独自产生一个有效签名。这是数学层面的保证，软件层面的越权尝试无法突破签名层的硬约束。
    
-   **Pact 任务级授权**：在 CAW 中，每一个任务都从 Pact 开始，它不仅定义 Agent 要做什么，还定义它可以执行到什么程度以及必须在什么条件下停止。与传统 policy 系统不同，Pact 不是一次性配置的静态规则，而是基于任务动态生成、随任务自动生效与自动结束，并由基础设施强制执行。
    
-   **Recipe 技能层**：CAW 首发自带 Recipe 库覆盖 Aave V3 借贷、Uniswap V3 与 Jupiter 兑换、DCA 与网格策略、Polymarket 预测市场等高频场景。配备 Recipe 的 Agent 不再幻觉合约地址、不再编造 ABI 参数、不再猜测 Gas，它沿经过验证的路径执行，Agent 每次都能正确执行。
    

**4.1 Pact 在 DeFi 场景中的应用**

**场景一：自动 DCA 美元成本平均买入 ETH**  
用户给出 Pact 参数：“每周二自动用 100 USDC 购买 ETH，持续 12 周，累计不超 1200 USDC，只在 Uniswap V3 执行。”

Agent 持有受 Pact 绑定的 session key，每周二自动构建 swap 交易并提交。Pact 过期后权限自动失效。如果 Agent 试图在周三或价格波动超过预设滑点时执行交易，链上 Guard 会直接拒绝操作，无需用户干预。

**场景二：Aave 自动循环杠杆策略**  
用户给出 Pact 参数：“ETH 价格低于 2000 时，自动存入 ETH 作为抵押，借出 60% 稳定币买入 BTC，健康因子必须保持在 1.5 以上。单次借款不超过 5000 USDC，总杠杆仓位不超过 3 倍。价格高于 3000 时自动平仓。”

Pact 强制执行：Agent 在执行借出操作前必须调用 simulate 验证健康因子≥1.5，任何不满足条件的状态变化都会触发自动平仓；退出价格触发前发出推送告警，用户可在移动端确认自动平仓逻辑后再执行。Agent 无法私自将平仓阈值改为 3500 或完全关闭止损。

**4.2 Pact、CAW 与各协议 Recipe 的协同**

Pact、MPC 与 Recipe 三者互补，解决了 Agent DeFi 执行的全部风险维度：

| 风险维度 | 应对机制 | 执行层 |
| --- | --- | --- |
| 签名私钥泄露 | MPC 多方签名 | CAW 基础设施层 |
| Agent 越权/静默篡改 | Pact 任务级授权动态边界 | CAW Pact 协议 + 链上 Guard |
| Agent 幻觉/错误 ABI | Recipe 预验证技能层 | CAW Recipe 库 |
| 权限永久有效 | Pact 任务结束后自动失效 | CAW 智能合约 |
| 大额交易无人监控 | 人工确认阈值 + Safe 多签 | 钱包权限层 |

**完整的 DeFi 自动化闭环**：

1.  用户通过 Pact 定义任务边界（预算、协议白名单、时间窗口、失败处理策略）
    
2.  Agent 调用对应 DeFi 协议的 Recipe（Uniswap Recipe / Aave Recipe / Hyperliquid Recipe）执行操作
    
3.  Recipe 内嵌正确的合约地址、ABI 和参数验证，Agent 不再盲目猜测
    
4.  MPC 多方签名保证签名不被单点攻破
    
5.  Pact 定义的边界在链上被 Guard 强制执行，超出的操作直接拒绝
    
6.  所有执行自动记录在审计日志中，异常触发实时告警
    
7.  任务完成后权限自动失效

### **ERC-4337、Safe、Guard/Policy 的重要性与风险解决**

**ERC-4337：账户抽象的核心标准**

**它是什么：** ERC-4337 是以太坊上实现账户抽象的核心标准提案。它通过引入新的伪交易结构 UserOperation，结合替代 mempool 来达成账户抽象，无需修改以太坊共识层协议。标准由四个关键角色协同工作：Smart Account 智能账户验证操作并处理预支付，Bundler 打包器收集 UserOperation 并提交至 EntryPoint 合约，EntryPoint 入口点合约作为链上路由器和验证器，Paymaster 代付者可可选地赞助 Gas 费用。

**解决的风险类别——谁能发起动作：**

-   私钥单点风险：传统 EOA 钱包中，私钥一旦泄露账户即完全失控。ERC-4337 让账户变为可编程的智能合约，支持多种签名验证方案，不再依赖单一私钥
    
-   交易原子性风险：UserOperation 可打包多个操作形成原子性执行，降低部分成功时的资金损失风险
    
-   Gas 经济依赖风险：UserOperation 支持代付者和 Gas 抽象，agent 可在不持有原生代币的情况下执行交易，消除没有 Gas 就无法运行 agent的死循环
    

Bundler 是 ERC-4337 的关键基础设施，它监控替代 mempool，收集多个 UserOperation 并将其打包到单笔交易中提交至 EntryPoint 合约。这对于 agent 高频小额交易场景尤其重要，可以将大量 agent 操作批量上链大幅降低总 Gas 成本。

截至 2026 年，已有超过 4000 万个智能账户通过 ERC-4337 部署在以太坊及 Layer 2 网络上，处理了超过 1 亿次 UserOperation 操作，证明账户抽象已经从理论走向规模化应用。

**Safe：可编程智能账户基础设施**

**它是什么：** Safe 是一个基于智能合约的账户结构，使账户不再仅仅用于存储资产，而是具备权限控制与执行逻辑的管理系统。与传统钱包依赖单一私钥不同，Safe 账户通过多签机制与预设规则实现更加灵活的资产管理方式。Safe 的模块化架构允许开发者将传统多签转化为可编程智能账户，其中 Safe Guards 添加可编程的安全层，Safe Modules 支持自动化的收益和交易批量处理。

**解决的风险类别——如何集体动作：**

-   单点授权失控风险：Safe 的多签机制要求多个签名方才可执行交易，有效防止单个签名方被攻破后资金被盗。对于部署 agent 的 DAO 金库或团队钱包，这意味着 agent 的操作需经过多重背书
    
-   权限滥用风险：Safe 结合 Guard 机制，可在每次交易执行前进行实时校验，确保操作不超出预定义的安全边界
    
-   操作透明度风险：Safe 的所有交易需要多个参与方签名确认，显著降低恶意操作的隐蔽性，同时提供完整的链上审批记录
    

在 agent commerce 场景中，Safe 可以作为“高价值交易保护层”。当 agent 发起超过阈值的操作时，操作进入 Safe 的签名队列，由多个守护者账户审核确认后才能执行，防止 agent 因模型幻觉或提示注入而误操作大额资金。

**Guard 与 Policy：执行层的实时控制与多层防御**

**它是什么：** Guard 是 Safe 智能账户的可编程安全层，可在交易执行前插入自定义校验逻辑。Guard 实现了 BaseGuard 抽象合约，继承 Guard 接口并定义交易执行前的校验函数，只有通过校验的交易才能继续执行。Policies 则定义更细粒度的安全规则，通过链上规则和链下智能的结合验证交易、强制执行权限，形成纵深防御模型，每一层安全措施都必须通过才能推进执行。

**解决的风险类别——动作该如何执行：**

-   交易内容失控风险：即使用户授权了 agent，也无法保证 agent 不会执行超出预期的操作。Guard 会在交易执行前检查目标合约、函数签名、金额等是否在白名单内，超出范围的调用会被直接拒绝
    
-   高频滥用风险：Policy 引擎可以追踪调用频率和时间窗口，防止 agent 在短时间内执行大量操作造成资金损失。例如限制 agent 每分钟最多发起一次交易，或每日累计不超过一定额度
    
-   异常行为实时阻断风险：Guard 对每笔待执行交易进行实时校验，可在毫秒级完成判断，不符合策略的交易被拦截。这比事后审计更有效，因为风险在发生前就被阻止
    
-   深度防御不足风险：如果依赖单一检查点，攻击者只要突破一点就能造成全部损失。Guard 结合 Policy 和 MPC 形成多层校验链，任何一层拦截都足够阻断恶意交易
    

Cobo 采用了类似的纵深防御思路，通过 MPC 多方安全计算将签名私钥分片加密，用户、agent 与 Cobo 基础设施多方共同参与交易的签署。被攻破的 agent、幻觉中的大模型、泄露的凭据，都无法独自产生一个有效签名，这是“数学层面的保证”。

**三者协作关系与风险覆盖全景**

| 协议/组件 | 定位层级 | 核心解决的风险 | 典型的错误假设 |
| --- | --- | --- | --- |
| ERC-4337 | 账户层 | 谁可以发起动作，签名机制可编程化 | 假设 agent 总能获得 Gas 代币 |
| Safe | 执行层 | 如何集体动作，多签与权限管理 | 假设单签已经足够安全 |
| Guard/Policy | 控制层 | 动作该如何执行，实时校验和拦截 | 假设授权后不需要持续监控 |
| MPC（如 Cobo） | 密钥层 | 密钥如何安全存储和签名 | 假设 agent 持有完整私钥也可以 |

三者叠加，形成了从账户能力到集体审批再到执行校验的完整安全闭环。缺失任何一层，都会产生特定的风险敞口：

-   只有 ERC-4337 没有 Safe：agent 持有足够权限独立签名，单点故障可导致资金被盗
    
-   只有 Safe 没有 Guard：多签确实阻止了恶意交易，但合法 agent 的异常操作也可能被批准，缺乏针对具体内容的校验
    
-   只有 Guard 没有 ERC-4337：只能拦截交易但无法解决 Gas 依赖和账户可编程性的根本问题，agent 的自主能力受限
    

Pact 机制正是将任务级授权落地的一种方式：围绕具体任务生成临时授权，agent 只能在 Pact 定义的边界内执行。具体到 swap、approve、deposit、borrow 等具体 DeFi 动作，将进入模块 E 做进一步展开。

**1.5 失败点（Failure Points）及处理**

| 失败场景 | 可能原因 | 处理方式 |
| --- | --- | --- |
| 输入 txHash 无效或不存在 | 用户输入错误 | 返回错误码 INVALID_TX_HASH，不扣费 |
| API 调用超时（Etherscan 无响应） | 第三方服务故障 | 重试 3 次，仍失败则退款并记录失败日志 |
| 风险检测模块崩溃 | 模型加载失败 | 降级为仅交易解释，不提供风险评级，部分退款 |
| 用户未支付或支付失败 | MPP 支付超时或资金不足 | 不执行请求，返回 402 Payment Required |
| 返回结果无法验证 | 用户质疑解释准确性 | 提供可重现的链上证明 |

### **二、MCP 与 ERC-8004 比较：解决不同层级的协作问题**

**2.1 协议定位与核心问题**

| 维度 | MCP | ERC-8004 |
| --- | --- | --- |
| 层级 | 应用层工具调用协议 | 链上合约层 |
| 核心问题 | Agent 如何发现并使用外部工具/数据源 | Agent 之间如何建立信任、分配任务、管理争议 |
| 交互模式 | 客户端-服务器模式：Agent 作为 MCP Client 调用工具服务器提供的 Tools、Resources、Prompts | 链上 Job 模式：Job 发布者创建任务，执行者提交结果，Evaluator 仲裁 |
| 通信方式 | JSON-RPC over stdio / SSE | 以太坊交易（EVM 合约调用） |
| 是否需要链 | 不需要。MCP 完全在链下运行，常用于本地或云端 agent 调用本地/远程工具 | 依赖链。所有 Job 状态、资金、验证记录都存储在链上 |
| 身份机制 | 无内置身份，仅靠服务器端点认证（如 API Key） | 通过以太坊地址 + 可选信誉合约（如 ERC-8004 的 Reputation 扩展） |

**2.2 适用场景对比**

**MCP 适合解决**：

-   **工具发现与调用**：Agent 需要读取本地文件、查询数据库、调用 Web API、执行代码等。MCP 提供了标准化的工具描述格式和调用协定，避免 agent 为每个新工具写定制代码。
    
-   **上下文注入**：将外部数据动态注入到 agent 的 prompt 中，无需硬编码。
    
-   **资源访问**：Agent 需要访问图片、文档、知识库等非结构化资源，MCP 的 `Resource` 概念提供统一 URI 模式。
    

**典型场景**：一个交易分析 Agent 通过 MCP 调用 `etherscan_getTransaction` 工具、`alchemy_trace` 工具、`local_risk_detector` 脚本。

**ERC-8004 适合解决**：

-   **任务市场与信任**：当一个 Agent 需要向另一个 Agent 付费请求服务时，ERC-8004 提供链上 Job 托管：资金锁定 → 交付 → 验收 → 结算，解决先付后做和先做后付的双重不信任。
    
-   **争议处理**：内置 Evaluator 角色，可预设为第三方 Agent 或多签。Evaluator 的裁决结果链上可查，为信誉积累提供依据。
    
-   **信誉积累**：Job 完成记录可作为 Agent 信誉的基础，形成开放市场的信任层。
    

**典型场景**：营销 Agent 通过 ERC-8004 发布生成 100 份海报的 Job，设计 Agent 竞标、交付，Evaluator判断质量后触发结算或退款。

**2.3 两者关系**

-   **MCP 和 ERC-8004 解决不同层次的问题**：MCP 是调用**工具**的协议，ERC-8004 是组织**跨 Agent 任务协作**的协议。
    
-   **可以组合**：一个 Agent 可以通过 MCP 调用生成海报的工具，但如果它想**雇佣另一个 Agent** 来完成这个任务，就需要 ERC-8004 这种任务市场协议。
    
-   **MCP 不涉及支付和信任**；ERC-8004 不关心工具如何被调用。两者结合可以构建完整的 agent 经济：Agent A 通过 MCP 发现并调用工具完成部分工作，同时通过 ERC-8004 将另一部分工作分包给 Agent B，并通过链上结算和信誉系统管理外包风险。
    

| 协议 | 解决的核心协作问题 | 不解决的问题 |
| --- | --- | --- |
| MCP | Agent ↔ 工具/数据源 | 支付、信任、争议、任务全生命周期管理 |
| ERC-8004 | Agent ↔ Agent | 工具发现、本地资源访问、低延迟实时调用 |

### **x402 与 MPP 比较：解决支付链路中的不同环节**

两个协议均基于 HTTP 402 Payment Required 状态码重构机器间支付，但在解决的核心问题、适用场景和支付模型上有着本质差异。

**协议定位总览**

| 维度 | x402 | MPP (Machine Payments Protocol) |
| --- | --- | --- |
| 主导方 | Coinbase | Stripe + Tempo，已提交为 IETF Internet-Draft |
| 协议哲学 | 协议最小化：把支付直接嵌入 HTTP 请求，实现请求即付费 | 系统最大化：通过 sessions 流式支付、插件化支付方式解决复杂商业场景 |
| 解决的核心问题 | 开放网络上如何一键完成链上支付 | 高频场景下如何系统化地完成机器间支付 |
| 支付方式 | USDC 等链上稳定币 | 插件化，支持稳定币、信用卡、银行卡、先买后付等 |
| 支付模型 | 单次逐笔支付 | 单次 + Session |
| 标准化程度 | 开源社区驱动，实现层定义 | IETF 标准化 |
| 互操作性 | 独立协议 | MPP 向后兼容 x402，MPP 客户端可直接消费现有 x402 服务 |

**比较表：各自解决支付链路中的哪一段**

| 支付链路环节 | x402 | MPP |
| --- | --- | --- |
| 身份/能力发现 | 不解决。不涉及账户体系，不涉及身份注册 | 不直接解决，但 session 机制隐含了客户端身份绑定 |
| 报价/协商 | 服务器在 402 响应中返回价格、代币和链信息。固定价格，不支持协商 | 支持更丰富的定价模型。同样不支持双向协商 |
| 交付/验证 | 不涉及。x402 解决的是API 调用+支付问题，支付完成后即返回资源，验证由服务端自行完成 | 不涉及。MPP 聚焦支付环节，交付质量和服务验证需在应用层处理 |
| 支付执行 | 核心功能：HTTP 原生支付。一次请求往返嵌入一笔支付，无 session 中间状态 | 核心功能：支持逐笔支付和 session 流式支付。session 模式下先锁定预算，按请求离线签署凭证，最后链上结算一次 |
| 托管（Escrow） | 不提供托管原语 | 不提供托管原语 |
| 结算 | 链上结算，支付即结算 | 链上结算 + 法币结算。session 模式下最终链上结算一笔汇总金额 |
| 争议处理 | 不涉及 | 不涉及 |
| 收据/记录 | HTTP 响应本身可视为收据；支付凭证由交易哈希提供 | MPP 协议规定了 receipts 标准格式，提供更强的可审计性 |

**总结**：

-   **x402** 解决的是 **支付执行+结算** 环节，做的是协议内支付的极致简化——没有账户、没有 session、没有中间商。最适合开放网络上低频率、按次付费的场景，如 AI 调用一次付费 API、购买单一数据查询。
    
-   **MPP** 解决的是 **支付执行+结算** 环节的**系统化增强版本**——通过 session 机制实现高频微支付（单次请求离线签署，最终链上批量结算），同时兼容法币等多支付方式。更适合商业化场景，如 AI 按 token 消耗付费、高频 API 调用、实时数据流订阅。
    

**两者与 Agent Commerce 完整生态的关系**

x402 和 MPP 共同揭示了一点：**payment 只是 Agent Commerce 链路中的一环**，不是全部。完整的 agent-to-agent 商业交互还需要：

-   **身份与信誉** ：x402 和 MPP 都不解决身份问题。
    
-   **任务托管与验收** ：x402 和 MPP 无托管原语，也无验证验收机制。ERC-8183 通过 Job 机制提供“资金锁定 → 提交 → 验收 → 结算”全流程托管，覆盖报价之外的完整商业信任问题。
    
-   **权限控制与审计**：Cobo CAW 这类钱包层解决Agent 可花多少钱、在哪些链上操作的预算限制问题，形成与支付协议互补的安全边界。
    

因此，一个健壮的 Agent Commerce 基础设施需要 **x402/MPP（支付层）+ ERC-8183（托管/验收层）+ ERC-8004（身份/信誉层）+ CAW 类钱包（权限/审计层）** 协同运作，每一层解决链路中的一个特定断点。

模块 B

**场景拆解：AI Agent 完成设计任务并收款**

一个专注于市场营销的 Agent A 需要为即将发布的 Web3 项目制作一批宣传海报，它没有设计能力，因此选择雇佣另一个专注于图像生成的 Agent B 来完成这项任务。双方互不相识，缺乏预先建立的信任基础，全流程需要在无需中心化平台仲裁的情况下完成。

**需求发布：**Agent A在链上注册表发布 Job 需求，包含任务描述、预算上限、验收标准和交付期限。

**报价协商**：Agent B检索到该 Job，分析任务复杂度，提供报价 8 USDC，附带样张链接作为能力证明。Agent A 接受报价，双方确认。

**预算托管**：Agent A 将 8 USDC 转入智能合约托管账户，资金锁定，Job 状态变为 Funded，双方都能看到资金已被锁定但未被释放。

**任务执行**：Agent B 调用图像生成 API，生成 10 张海报，输出 CID或 IPFS 哈希，提交结果，Job 状态变为 Submitted。

**验收评估**：Evaluator接收提交内容，比对任务需求和验收标准，判断是否完成。Evaluator 可以是另一个专业的图像分析 AI Agent，预设的智能合约规则，对于高价值任务，可以配置为多签形式的多个 Evaluator 投票决定。

**验收动作**：Evaluator 标记完成，合同释放资金给 Agent B。

**结算付款**：智能合约自动将托管的 8 USDC 释放给 Agent B 的指定账户，同时链上记录不可篡改的交易收据和评估理由。

**失败处理 & 争议仲裁：超时处理**：若 Agent B 未在规定时间内提交结果，任何人可调用 `claimRefund(jobId)`，资金退回 Agent A。**验收不通过**：Evaluator 调用 `reject(jobId, reason)`，资金退回 Agent A，Agent B 不得分。**争议仲裁**：若 Agent A 不认可 Evaluator 的拒绝决定，可指定一个更高层级的仲裁 Agent 或 DAO 重新评估。所有评估理由和提交物 CID 链上留存，供追溯和仲裁使用。

**收据与记录：**链上形成完整记录：JobID、Client 地址、Provider 地址、Evaluator 地址、预算金额、交付物 CID、评估理由、完成时间戳。这些记录可用于 Agent B 的声誉系统加分，也可在后续 Job 中作为能力证明。

**3\. 关键要素总结**

| 环节 | 角色 | 约束条件 |
| --- | --- | --- |
| 下单 | Agent A | 必须在 Job 创建时明确预算上限、验收标准、交付期限 |
| 执行 | Agent B | 必须在期限内提交符合标准的 deliverable |
| 验收 | Evaluator | 只能由预定义的链上地址调用 complete / reject |
| 付款 | 智能合约托管 | 资金只能流向 Client或 Provider |
| 仲裁 | 预设的次级 Evaluator 或 DAO | 仅在 Client 不认可初判时触发，需预先约定 |
| 记录证明 | 链上日志 + Event | 所有状态变更和理由均不可篡改，用于声誉和审计 |

以下地图涵盖六个交叉方向，每个方向分别说明 AI 承担的角色与 Web3 提供的独特机制。

| 方向 | AI 作用 | Web3 机制 |
| --- | --- | --- |
| Payment / Commerce / Settlement | AI agent 自动识别服务需求、调用 API、完成报价协商、执行支付决策；可基于历史数据预测最优价格和时机 | 智能合约托管与条件释放、链上结算的原子性、代币激励与费率市场、争议解决的链上仲裁 |
| Identity / Reputation / Capability / Interoperability | AI 自动解析 agent 的能力清单（如 MCP 工具描述）、动态发现和匹配协作对象；基于行为数据生成信誉评分 | 去中心化标识符、链上凭证、不可篡改的交互记录、跨协议的能力注册表 |
| Wallet / Permission / Safe Execution | AI 代理交易意图的识别与分类、动态权限策略推荐、异常行为检测 | 账户抽象、Session Keys、多签钱包、权限分层与撤销、链上审计日志 |
| Privacy / Security / Sovereignty | AI 模型本地推理、差分隐私噪声注入、对抗样本检测；私钥与 API Key 的密态管理 | 零知识证明、可信执行环境的链上验证、数据加密存储与访问控制、主权身份 |
| Dev Tooling / Agent Workflow | AI 生成合约测试用例、自动解释交易日志、从文档生成 agent 脚手架、辅助代码审查与部署脚本 | 链上验证的合约源码、去中心化包管理器、Gas 优化的自动化流水线、透明可审计的 builder 行为 |
| Governance / Coordination / Public Goods | AI 总结治理提案、聚类社区讨论、自动生成会议行动项；量化贡献并分配奖励 | 链上投票、代币激励的协调机制、公共物品追溯资助、不可篡改的贡献记录 |

### **二、两个方向的交叉必要性论证**

**方向 1：Payment / Commerce / Settlement**

**为什么不是纯 AI 问题？**  
纯 AI 系统可以完成意图识别、定价预测和支付执行，但无法解决信任问题：AI 调用第三方 API 或购买算力时，服务方担心 AI 不付款，AI 所有者担心服务方不交付或跑路。没有可信的托管与自动结算机制，AI 的商业化只能在封闭平台内进行，无法形成开放的经济生态。此外，纯 AI 无法处理争议时的客观证据提取和不可篡改的裁决。

**为什么不是纯 Web3 问题？**  
纯 Web3 智能合约可以托管资金、条件释放、实现链上结算，但无法理解服务是否完成，比如 AI 调用了一个图像生成 API，输出结果是否满足质量要求？合约本身不具备语义理解能力，无法判断是否符合交付标准。必须依靠 AI 的能力对服务质量进行客观评估，并将评估结果作为合约的输入条件。没有 AI，智能合约只能处理完全量化的指标，无法处理主观或复杂的服务质量判断。

**方向 2：Wallet / Permission / Safe Execution**

**为什么不是纯 AI 问题？**  
AI 可以识别危险操作、推荐权限策略、检测异常行为，但无法强制执行权限边界：用户给 AI 一个私钥或 session key 后，AI 代码理论上可以绕过权限声明，发起任意交易。纯软件层面的权限约束很容易被恶意 AI 或注入攻击突破。没有链上硬性约束，AI 的权限控制只是君子协议。

**为什么不是纯 Web3 问题？**  
Web3 的账户抽象和多签机制可以设置严格的链上权限，但无法解决动态授权问题：AI 在执行复杂任务时，需要临时申请超出当前权限的操作。纯链上机制要求用户预先知道所有需要的权限，或者反复手动签名，破坏了自动化体验。AI 可以实时理解任务上下文，生成最小必要权限的申请，并解释申请理由，辅助用户做出明智的批准决策。没有 AI，链上权限要么过宽，要么过窄。

### 卷积神经网络（CNN）

卷积神经网络主要用于机器视觉。

在RGB颜色模型中，一张彩色图片本质上是三张颜色矩阵，分别存储着红绿蓝的亮度，三张颜色矩阵叠加生成完整的图片。

为了在一张张复杂的图片上找到相同的元素/物体，深度学习模型使用卷积操作，这个操作可以捕捉到图像中的局部特征而不受其位置的影响。

![d0172774f7e42ae2f6310b63e59b4906.gif](https://raw.githubusercontent.com/IntensiveCoLearning/AI-Web3-School/main/assets/fffg-o/images/2026-05-18-1779083956356-d0172774f7e42ae2f6310b63e59b4906.gif)

卷积神经网络主要分为三层：

1.  卷积层
    
2.  池化层
    
3.  全连接层
    

卷积层负责对复杂的图像进行大量卷积操作，最终，卷积层将图像转换为数值，允许神经网络解释和提取相关模式。

池化层也称为下采样层，它执行降维操作，旨在减少输入中参数的数量。与卷积层类似，池化运算让过滤器扫描整个输入，但区别在于，这个过滤器没有权重。内核对感受野中的值应用聚集函数，填充输出数组。

虽然池化层中会丢失大量信息，但它还是给 CNN 带来的许多好处。该层有助于降低复杂性、提高效率，并限制过度拟合的风险。 

全连接层将复杂的提取特征与简单原始的图像连接，以便其他使用

### 循环神经网络 (RNN)

循环神经网络广泛应用于自然语言处理、机器翻译、语音识别、文本生成、时间序列预测等任务，特别适用于有顺序依赖的数据，循环模型通常会沿着输入和输出序列的符号位置进行计算。通过在计算时将位置与时间步对齐，其生成一系列隐藏状态 ，作为前一个隐藏状态与位置二者的函数。

### Transformer

以上模型使用注意力机制作为重要部分，但是，这种注意力机制都是与循环网络结合使用的。

Transformer 是一种避免使用循环的模型架构，完全依赖注意力机制来描述输入和输出之间的全局依赖关系。

Transformer 的核心组件主要包括：

-   输入嵌入（Embedding）
    
-   位置编码（Positional Encoding）
    
-   注意力机制（Attention）
    
-   前馈神经网络（Feed Forward Network）
    
-   残差连接与层归一化（Residual Connection & LayerNorm）
    

### 输入嵌入（Embedding）

计算机无法直接理解文字，因此需要先将文本转换为向量。

Embedding 层负责将离散的词语映射为连续的高维向量，使语义相近的词在向量空间中距离更近。这样模型就能够学习词语之间的语义关系。

### 位置编码（Positional Encoding）

由于 Transformer 没有循环结构，因此模型本身无法感知词语顺序。

因此 Transformer 会加入位置编码，为每个词添加位置信息，使模型能够理解序列顺序。

### 注意力机制（Attention）

注意力机制是 Transformer 最核心的部分。

其核心思想是：在处理某个词时，动态关注与当前词最相关的其它词。

例如：小明把书放进书包，因为它太重了。

模型需要判断：它指的是书还是书包

注意力机制能够建立词语之间的关联关系，从而更好地理解上下文语义。

Transformer 中最重要的是**自注意力机制（Self-Attention）**。

自注意力允许：

-   每个词都与其它所有词进行信息交互
    
-   自动学习词与词之间的依赖关系
    
-   捕捉长距离语义关系
    

相比 RNN：

-   不需要逐步传递信息
    
-   长距离依赖问题得到明显改善
    

### 多头注意力机制（Multi-Head Attention）

Transformer 并不是只进行一次注意力计算，而是同时进行多组注意力计算。

不同的注意力头可以学习不同的信息：

-   某些头关注语法结构
    
-   某些头关注语义关系
    
-   某些头关注位置关系
    

最后再将多个结果进行融合，从而获得更加丰富的特征表达。

### 前馈神经网络（Feed Forward Network）

在完成注意力计算后，Transformer 还会通过前馈神经网络进一步提取特征。

前馈网络通常由线性变换，激活函数，再次线性变换组成。其主要作用是增强模型的非线性表达能力提升特征提取效果

### 残差连接与层归一化

随着网络层数不断增加，模型容易出现梯度消失，训练困难，收敛速度慢等问题。

因此 Transformer 引入：

-   残差连接（Residual Connection）
    
-   层归一化（Layer Normalization）
    

残差连接能够保留原始信息，避免深层网络信息丢失。

层归一化能够稳定数据分布，加快模型训练速度。

## Transformer 的优势

相比 CNN 和 RNN，Transformer 具有许多明显优势。

### 并行计算能力强

RNN 必须按照时间顺序逐步计算，因此难以并行训练。

Transformer 则一次性处理整个序列，所有词可以同时计算，因此能够充分利用 GPU 的并行能力。（老黄居然在这里赌对了）

### 更擅长长距离依赖

RNN 在处理长文本时，前面的信息容易遗忘。

而 Transformer 中，任意两个词之间都可以直接建立联系，因此更适合处理长文本任务。

### 表达能力更强

多头注意力机制能够同时学习：语义信息，语法信息，上下文关系

因此模型具有更强的特征表达能力。

## Transformer 的不足

虽然 Transformer 非常强大，但仍然存在一些问题。

### 计算复杂度较高

注意力机制需要计算每个词与其它所有词之间的关系，因此时间复杂度较高。

当文本长度不断增加时，显存消耗会迅速上升，计算成本明显增加

### 对数据规模要求较高

Transformer 通常需要大规模数据集，大量计算资源才能训练出较好的效果。
