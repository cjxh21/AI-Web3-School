# 2026-06-03
<!-- DAILY_CHECKIN_2026-06-03_START -->
在学习了 Agent 的规划能力（Planning）之后，接下来需要理解一个更接近真实世界应用的概念——Tool Use（工具使用能力）。很多人认为 Agent 最重要的是大模型本身，但实际上一个只能聊天、不能操作外部世界的模型，其价值是非常有限的。人类之所以能够完成复杂任务，并不仅仅依靠大脑思考，还需要使用各种工具，例如计算器帮助计算、搜索引擎帮助查找资料、电脑帮助处理数据、手机帮助沟通交流。同样，对于 Agent 而言，大模型负责理解和决策，而工具则负责执行具体动作。没有工具的大模型只能停留在“告诉你怎么做”的阶段，而拥有工具能力的 Agent 则可以进一步发展到“帮你做完”。例如当用户要求分析某个代币时，Agent 可以调用链上数据查询工具获取实时数据；当用户要求总结最新行业动态时，Agent 可以调用搜索工具访问互联网；当用户要求整理一份报告时，Agent 可以调用文档处理工具自动完成排版和汇总。工具调用的本质是让 Agent 获得与外部世界交互的能力，从封闭系统变成开放系统。现代 Agent 系统中的工具通常以 API 的形式存在，每个工具都有明确的输入和输出规范，Agent 通过理解任务需求来判断应该调用哪个工具、以什么参数调用以及如何处理返回结果。例如在 Web3 场景中，一个完整的 Agent 可能接入钱包工具、区块链浏览器工具、价格查询工具、链上分析工具以及治理提案查询工具。当用户提出一个复杂问题时，Agent 会自动选择多个工具协同工作，然后整合结果生成最终答案。更重要的是，工具使用能力并不是简单地调用函数，而是一种决策能力。Agent 需要判断什么时候应该自己回答，什么时候应该调用工具，什么时候需要调用多个工具，以及当工具返回异常结果时应该如何处理。因此，工具使用能力实际上是模型推理能力与执行能力结合的体现。从当前行业发展来看，Agent 的竞争已经逐渐从“谁的模型更大”转向“谁的工具生态更完善”。因为模型决定的是理论上能思考什么，而工具决定的是实际上能做到什么。对于未来的 AI Agent 和 Web3 Agent 而言，一个优秀的系统往往不是拥有最多参数，而是拥有最丰富且最稳定的工具链。到目前为止，我们已经逐步建立起 Agent 的核心认知框架：Memory 负责记住过去，RAG 负责获取知识，Planning 负责制定行动方案，Tools 负责执行具体动作。下一步，我们将进入更高级的 Agent 架构学习，理解多个 Agent 如何协同工作，以及为什么未来很多复杂任务将不再由单个 Agent 完成，而是由多个专业 Agent 组成团队共同解决问题。

# 2026-06-02
<!-- DAILY_CHECKIN_2026-06-02_START -->

在理解了 RAG 如何帮助 Agent 获取外部知识之后，下一步最值得学习的内容就是 Planning（规划能力），因为这决定了 Agent 为什么能够完成复杂任务，而普通聊天机器人往往只能完成简单问答。很多人认为 Agent 的核心是工具调用，但实际上工具只是执行手段，真正决定 Agent 水平高低的是它是否具备规划能力。举个简单例子，如果用户说“帮我分析一个 Web3 项目是否值得投资”，普通大模型可能直接给出一段分析，但高级 Agent 不会立刻回答，而是先思考应该如何完成这项任务。它可能会将目标拆分为多个步骤：首先收集项目背景信息，其次分析团队情况，然后查看融资记录，再研究代币经济模型，接着获取链上数据和社区活跃度，最后综合所有信息形成投资分析报告。这种先思考、后执行的过程就是 Planning。规划能力的重要性在于现实世界的大多数任务都不是一步完成的，而是由多个相互关联的子任务组成。如果没有规划机制，Agent 每次都只能根据当前信息做出局部决策，很容易遗漏关键环节或者陷入低效率循环。为了解决这个问题，研究人员提出了许多经典 Agent 架构，其中最著名的是 ReAct（Reason + Act）模式。ReAct 的核心思想是让 Agent 在思考和行动之间不断循环：先推理当前应该做什么，再调用工具执行，获得结果后继续推理下一步行动，直到完成目标。这种模式让 Agent 不再是一次性输出答案，而是能够动态调整执行策略。随后又出现了 Plan-and-Execute 架构，它将规划与执行彻底分离，先制定完整计划，再逐步执行各个步骤，这种方式在长任务和复杂任务中更加稳定。随着技术发展，还出现了多层规划系统，高层 Agent 负责制定整体战略，低层 Agent 负责具体执行，类似于公司中的管理层与员工协同工作。从本质上看，规划能力就是把一个模糊目标转化为一系列可执行步骤的能力，它让 Agent 从“知道答案”升级为“知道如何完成任务”。对于未来的 AI Agent 和 Web3 Agent 而言，规划能力的重要性甚至可能超过模型本身，因为当模型能力逐渐接近时，真正拉开差距的是谁能更好地组织任务、协调工具、管理状态以及调整策略。可以把 Agent 的核心能力简单概括为：RAG 负责获取知识，Memory 负责保存经验，Tools 负责执行动作，而 Planning 则负责指挥整个系统如何行动。没有规划，其他能力就像散落的零件；有了规划，这些能力才能被组织起来，形成真正能够自主完成任务的智能体。

# 2026-06-01
<!-- DAILY_CHECKIN_2026-06-01_START -->


在学习完 Agent 的 Skills、Workflow 和 Memory 之后，接下来必须理解 RAG（Retrieval-Augmented Generation，检索增强生成）这一核心概念，因为它几乎是现代 AI Agent 系统的标准配置。很多人会认为，大语言模型已经学习了海量知识，似乎不需要额外获取信息就能回答问题，但实际上模型的知识来源于训练数据，它的认知范围被训练时间所限制，无法自动获取训练之后发生的新信息，更无法实时访问企业内部文档、数据库、区块链数据或最新的互联网内容。例如询问某个 DeFi 协议今天的 TVL、某个治理提案的最新投票情况，或者某家公司的最新财务数据时，仅依靠模型参数中的知识往往会得到过时甚至错误的答案。RAG 的出现正是为了解决这一问题，它的核心思想非常简单：先查资料，再回答问题。具体来说，当用户提出问题时，系统不会立刻让大模型生成答案，而是先将问题转换为向量，在向量数据库中检索与问题语义最相关的知识片段，然后将这些检索到的内容与用户问题一起发送给大模型进行推理和生成。这样模型在回答时依据的不仅仅是训练时期掌握的知识，还包括实时获取的外部信息，因此结果会更加准确、可靠和具有时效性。对于 Agent 而言，RAG 的价值远远超过一个简单的搜索功能，因为 Agent 的目标不是聊天，而是完成任务。完成任务意味着需要接触真实世界，而真实世界的信息分散在网页、API、PDF、数据库、企业知识库以及区块链网络之中。RAG 相当于 Agent 的“信息获取系统”，帮助 Agent 从外部世界持续吸收知识。与此同时，RAG 与 Memory 之间也存在明显区别：Memory 负责保存过去的经验和历史信息，例如用户偏好、任务记录和执行经验；而 RAG 负责获取当前需要的新知识和外部事实。可以简单理解为，Memory 让 Agent 记住过去，RAG 让 Agent 感知现在，而大语言模型则负责分析和决策。三者结合后，Agent 才真正具备了长期服务用户的能力。特别是在 Web3 领域，由于协议升级、治理提案、链上数据和市场环境每天都在变化，任何依赖静态知识的系统都会迅速落后，因此绝大多数优秀的 Web3 Agent 都会将 RAG 作为核心基础设施，通过实时检索链上和链下信息来增强决策能力。从某种意义上说，RAG 的出现标志着 AI 从“记忆知识”走向“获取知识”，也是 Agent 从单纯问答工具进化为智能执行系统的重要一步。

# 2026-05-31
<!-- DAILY_CHECKIN_2026-05-31_START -->



学习 Agent 系统时，Memory（记忆系统）是一个绕不开的核心概念。如果说大语言模型是 Agent 的大脑，Skills 是能力，Tools 是双手，那么 Memory 就是 Agent 的记忆力。没有记忆的 Agent，每次对话都像第一次见到用户；而拥有记忆的 Agent，才能持续学习、积累经验并完成复杂任务。

首先要理解，大语言模型本身并没有真正意义上的长期记忆。它只能看到当前输入的上下文内容，并根据这些内容进行推理。当一次对话结束后，模型并不会像人类一样自动记住经历过的事情。因此，如果没有额外的记忆系统支持，Agent 每次执行任务都需要重新开始。

在 Agent 领域，记忆通常分为短期记忆和长期记忆两种。

短期记忆主要负责保存当前任务中的上下文信息。例如，当用户要求 Agent 帮助分析一个项目时，项目名称、分析进度、中间结果以及之前的讨论内容都会被保存在短期记忆中。短期记忆的作用类似于人脑中的“工作记忆”，帮助 Agent 在执行当前任务时保持连贯性。当任务结束后，这部分内容可能会被删除或者压缩存储。

长期记忆则用于保存具有持续价值的信息。例如用户的偏好、历史任务记录、常用工作流程、过往经验总结等。这些信息可能在未来数周、数月甚至更长时间后仍然会被使用。长期记忆让 Agent 不再只是一个工具，而逐渐具备了持续成长和个性化服务的能力。

那么，Agent 如何存储这些记忆呢？目前最常见的方法是利用向量数据库（Vector Database）。

传统数据库依靠关键词进行匹配。例如搜索“比特币”，系统寻找包含“比特币”这几个字的内容。而向量数据库则存储文本的语义特征。当用户提出问题时，系统会将问题转换为向量，然后寻找语义最接近的历史记录。

例如用户曾经记录过“我未来想成为一名顶级量化交易员”，后来又询问“我应该重点学习哪些技能”。即使问题中没有出现“量化交易员”几个字，向量数据库依然能够根据语义关联找到相关记忆，并提供更符合用户目标的回答。

这种机制被称为 Retrieval（检索）。Agent 在回答问题之前，会先从长期记忆中检索相关内容，然后将其作为上下文提供给大语言模型。模型再结合当前问题和历史记忆进行推理，最终生成更准确、更个性化的结果。

从系统运行角度来看，一个完整的记忆流程通常包括四个步骤：信息产生、记忆存储、记忆检索和记忆更新。用户与 Agent 的每次互动都会产生信息，其中有价值的部分被存储下来；当新的任务出现时，系统检索相关记忆；任务完成后，再将新的经验写回记忆库。这就形成了一个持续循环的学习过程。

对于未来的 Autonomous Agent（自主智能体）而言，记忆系统的重要性甚至不亚于大语言模型本身。因为模型决定了 Agent 的思考能力，而记忆系统决定了 Agent 能否积累经验、形成个性、持续成长。一个没有记忆的 Agent 只能不断重复昨天，而一个拥有完善记忆系统的 Agent，才能真正朝着长期自主执行任务的方向发展。

# 2026-05-30
<!-- DAILY_CHECKIN_2026-05-30_START -->




在学习 Agent 系统的过程中，一个非常重要的问题是：为什么大语言模型已经能够进行高质量对话，却仍然需要 Agent、Skills（技能）、工具调用以及状态管理等机制？理解这个问题，有助于从根本上认识 Agent 系统的价值。

传统的大语言模型本质上是一种“生成答案”的系统。当用户提出问题时，模型根据训练数据和上下文生成最可能的文本回复。例如询问某个知识点、要求写一篇文章或者分析一个案例时，模型能够给出相当优秀的答案。但这种能力更多停留在“会回答”的层面。它能够告诉你应该怎么做，却未必能够真正帮你完成任务。

而 Agent 系统的目标则更进一步，它不仅需要理解问题和生成内容，还需要能够完成一类任务。这里的关键桥梁就是 Skills，也就是技能模块。

Skills 可以理解为 Agent 的能力单元。一个技能往往封装了某种特定任务的执行逻辑。例如网页搜索技能、数据分析技能、邮件发送技能、链上查询技能、智能合约部署技能等。对于 Agent 来说，它不需要每次都从零开始思考如何完成任务，而是可以直接调用已经构建好的技能模块。这样一来，技能便成为一种可复用的能力资产。

可复用技能的重要意义在于将复杂任务标准化和模块化。如果没有 Skills，Agent 每次面对类似问题时都需要重新规划和推理，效率低且稳定性不足。而拥有 Skills 后，Agent 可以像程序调用函数一样调用现成能力。例如用户要求分析某个代币数据时，Agent 可以直接调用链上查询技能、数据整理技能和分析报告生成技能，而不是依靠语言模型凭空推测结果。这样不仅提升效率，也提高了结果的准确性和可靠性。

工具调用（Tool Calling）则是 Skills 真正发挥作用的执行机制。大语言模型本身无法直接访问互联网、数据库或者区块链网络，它只能生成文本。但当 Agent 具备工具调用能力后，它便能够连接外部世界。例如调用搜索引擎获取最新资讯，调用数据库查询用户信息，调用 API 获取实时价格，调用区块链节点读取链上数据等。

工具调用让 Agent 从静态知识库变成动态执行系统。模型负责决策和规划，工具负责实际执行，两者结合后形成完整的任务闭环。此时 Agent 不再只是告诉用户答案，而是能够主动获取信息、处理信息并返回结果。

除了技能和工具，状态管理（State Management）同样是 Agent 系统的重要组成部分。许多真实任务并非一步完成，而是包含多个阶段。例如开发一个 Web3 项目，可能需要需求分析、技术选型、代码编写、测试部署和上线运营等多个步骤。如果 Agent 每次交互都忘记之前发生的事情，那么任务就无法持续推进。

状态管理解决的正是这个问题。它负责记录任务进展、保存中间结果、维护上下文信息以及追踪执行历史。借助状态管理，Agent 能够记住当前处于哪个阶段、已经完成哪些步骤、接下来需要执行什么操作。这样，多轮任务就能够持续进行，而不会因为一次对话结束而丢失关键信息。

从系统设计角度来看，大语言模型相当于 Agent 的“大脑”，负责理解、推理和决策；Skills 相当于“专业能力”；工具调用相当于“手和脚”，负责与外部环境交互；状态管理则相当于“记忆系统”，负责保存经验和上下文。只有这几个部分协同工作，才能构成真正意义上的 Agent。

因此，Agent 系统与 Skills 的关系并不仅仅是简单的功能扩展，而是一种能力层级的跃迁。可复用技能提供标准化能力，工具调用赋予执行能力，状态管理保障持续执行能力。当这些机制结合在一起时，系统便从单纯“会回答问题”的聊天机器人，逐渐演化为“能够完成一类任务”的智能执行体。这也是当前 AI Agent 迅速发展的核心原因之一。未来随着技能库不断丰富、工具生态不断完善以及状态管理机制不断增强，Agent 将具备越来越强的自主执行能力，成为连接人工智能与现实世界的重要桥梁。

# 2026-05-28
<!-- DAILY_CHECKIN_2026-05-28_START -->






今天研究了一条线：Prompt → Workflow → Agent，这不只是AI的进化路径，是人类把决策权逐步交出去的过程。

✅ Prompt：你的思维质量决定AI的输出质量

Prompt就是给AI下的指令，但本质上它是一面镜子。

你的问题模糊，AI的回答就废。你的问题清晰，AI的回答就精准。

很多人觉得AI不好用，根本原因不是AI太笨，是自己没想清楚要什么。

Prompt能力的核心不是学话术，是把复杂需求结构化表达的能力。这个能力放在任何场景都值钱，AI只是让它变得更可见了。

✅ Workflow：把重复的判断变成流水线

单个Prompt解决单个问题，Workflow解决的是流程问题。

它把多个步骤串联起来自动运行。比如监控链上某个协议的TVL数据，发现异常自动生成报告，报告自动发到你手机。这条链路一旦搭好，永远替你跑，不需要人盯着。

Workflow的价值不是效率，是把你的注意力从重复性工作里解放出来，专注真正需要判断的事。

重复的事情交给机器，判断的事情留给人，这是正确的分工。

✅ Agent：你只给目标，它自己找路

Workflow是按你设计的路线走，Agent是你给它终点，它自己决定怎么走。

一个Web3场景：你告诉Agent，帮我找收益最高且风险可控的流动性池，自动存进去，每天监控，发现更好的机会自动迁移。

它自己搜索数据，自己分析风险，自己执行链上操作，全程不需要你参与。

这在以前不可能，因为链上操作需要人工确认。现在AI结合Web3钱包，Agent可以直接操作链上资产，真正意义上的全自动。

✅ 这条线的本质是什么

Prompt → Workflow → Agent，这三个层次走下来，人在链上扮演的角色越来越轻。

但有一件事始终替代不了，就是你对风险边界的判断，对目标本身的定义。

AI可以帮你执行，但它不知道你能承受多大的损失，不知道你的钱有多重要，不知道什么时候该停下来。

所以未来真正有价值的人，不是最懂技术的人，是那个既理解Web3底层逻辑，又知道怎么用AI放大自己判断力的人。

<!-- DAILY_CHECKIN_2026-05-25_START -->









LLM（Large Language Model，大语言模型）是现代人工智能最核心的技术之一，本质上是一种基于海量数据训练出来的深度学习模型，其核心任务是“根据上下文预测下一个最可能出现的 token（词元）”。所谓 token，可以理解为模型处理语言时的最小单位，它可能是一个字、一个词，甚至一个符号。例如当用户输入“今天天气很好，我想去”，模型会根据训练过程中学到的大量语言规律，预测“公园”“散步”等词出现的概率，并选择最合理的结果继续生成。这种看似“理解语言”的能力，本质上是一种极其复杂的概率统计与模式学习。

现代 LLM 大多建立在 Transformer 架构之上，而 Transformer 的核心突破来自 Attention（注意力机制）。传统神经网络在处理长文本时容易遗忘前面的内容，而 Attention 可以让模型动态地判断“当前词应该关注哪些词”。例如一句话中“她”指代谁，模型会通过注意力机制分析上下文关系，从而得到更合理的理解。Transformer 的另一个巨大优势是支持并行训练，这使得模型可以扩展到数十亿甚至上万亿参数规模。参数可以理解为模型内部用于存储“知识”和“规律”的数字，参数越多、数据越大、算力越强，模型能力通常越强，这种现象被称为 Scaling Law（规模法则）。

LLM 的训练过程本质上是一个不断“预测—纠错—优化”的循环。模型先根据输入预测结果，然后计算预测与真实答案之间的误差，再通过反向传播算法利用微积分自动调整内部参数。经过海量数据和长时间训练后，模型逐渐学会语言、知识、逻辑关系以及大量隐含模式。如今的 LLM 不仅能完成文本生成、翻译、总结等基础任务，还表现出代码生成、数学推理、多轮对话、多模态理解等“涌现能力”。例如 ChatGPT、Claude、Gemini 等产品，都建立在 LLM 的能力之上。

然而，LLM 并不真正拥有意识、情感或真正意义上的“理解”。它并不知道自己在说什么，也无法像人类一样感知世界。它的本质仍然是高维向量空间中的函数逼近系统，通过统计规律生成最可能的输出。因此，LLM 会出现“幻觉”问题，也就是生成看似合理但实际上错误的信息，因为它追求的是“语言概率正确”，而不一定是“事实正确”。此外，LLM 的训练成本极高，需要大量 GPU、数据、电力和工程系统支持，因此现代 AI 的竞争不仅仅是模型算法的竞争，更是数据、算力和系统工程能力的竞争。

从产业角度看，LLM 已经成为现代 AI 的基础设施层。它不仅改变了搜索、写作、编程、设计和教育等行业，还推动 AI 从“聊天工具”向“自动执行系统”演化。随着 Agent（智能代理）技术的发展，未来的 LLM 将不仅负责生成文本，还会调用工具、操作软件、管理工作流，逐渐成为数字世界中的“智能操作系统”。因此，理解 LLM，不仅是在学习一个模型，更是在理解未来 AI 时代最重要的基础能力。

Hermes 学习笔记

最近在了解 Hermes 这个 AI Agent 框架，感觉它和传统聊天机器人最大的区别，就是它不只是“会聊天”，而是真正开始往“自主执行任务”方向发展。

以前的大模型更多像一个知识问答工具，你问一句，它答一句。但 Hermes 更像一个“智能代理人（Agent）”，它能够理解目标、拆解任务、调用工具，然后一步一步完成事情。比如搜索资料、读取文件、调用 API、分析数据，甚至还能根据结果继续调整下一步动作。这种感觉已经有点像“数字员工”了。

我觉得 Hermes 的核心思路，其实就是把大模型从“语言输出”升级成“行动系统”。因为现实世界很多问题，并不是一句回答就能解决的，而是需要连续决策。比如“帮我做一份行业研究”，正常情况下需要搜索信息、筛选内容、整理逻辑、输出文档，这里面其实包含很多步骤。Hermes 的价值就在于，它能把这些流程串起来。

另外，我发现 AI Agent 和普通 AI 最大的差别，是它有“工具调用能力”。以前模型知道很多知识，但没法真正操作外部世界；而 Agent 可以连接搜索引擎、数据库、浏览器、代码环境等工具，相当于给 AI 装上了“手”和“脚”。这样一来，它就不仅是回答问题，而是真能干活。

不过目前 Hermes 这类 Agent 也有明显问题。第一是稳定性还不够，有时候任务链一长就容易跑偏；第二是成本比较高，因为每一步思考和调用工具都会消耗 token；第三是“自主性”和“可控性”之间很难平衡，放得太开可能乱操作，限制太多又发挥不出 Agent 的能力。

但整体来看，我觉得 AI Agent 很可能是未来几年的重要方向。因为大模型本身已经越来越像“大脑”，接下来拼的，就是谁能把“大脑”和“执行能力”结合得更好。Hermes 这种框架，本质上是在探索“让 AI 从会说话，变成会做事”。

学习 Hermes 之后，我最大的感受是：未来 AI 的竞争，可能不只是模型参数大小，而是谁能构建更完整、更稳定的 Agent 生态。
