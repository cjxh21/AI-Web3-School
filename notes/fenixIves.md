# AI基础：从LLM到Agent Workflow

## 一、引言：为什么要从LLM学起再理解Agent

很多人直接上手Agent开发却效果不佳，根本原因是**不理解LLM的能力边界**。Agent不是"更聪明的LLM"，而是**以LLM为大脑，通过模块化设计弥补LLM固有缺陷的系统工程**。

本笔记遵循"底层原理→能力边界→系统演进→运行机制→实践落地"的逻辑，帮你建立从"模型是什么"到"Agent怎么运行"的完整认知体系。

## 二、LLM基础：模型到底是什么

### 2.1 LLM的本质：不是"理解"，是"预测"

-   **核心定义**：大语言模型是基于海量文本训练的**统计概率模型**，它的唯一核心能力是"根据上文预测下一个最可能出现的token"
    
-   **工作原理**：
    

1.  将输入文本转换为数字向量（tokenization）
    
2.  通过Transformer架构的自注意力机制计算每个token与其他token的关联权重
    
3.  基于训练学到的统计规律，输出概率最高的下一个token
    
4.  重复这个过程直到生成结束符或达到最大长度
    

-   **关键认知**：LLM没有真正的"理解"、"思考"或"意识"，它所有的输出都是对训练数据中语言模式的模仿和重组
    

### 2.2 LLM的三大核心能力

1.  **语言处理能力**：文本生成、翻译、摘要、改写、分类、情感分析等
    
2.  **推理能力**：通过思维链（Chain of Thought）提示，能够进行逻辑推理、数学计算、问题解决
    
3.  **泛化能力**：能够处理训练数据中没有见过的新任务，只需要通过自然语言描述任务要求
    

### 2.3 LLM的五大致命局限性（这正是Agent要解决的问题）

| 局限性 | 具体表现 | 后果 |
| 上下文窗口有限 | 无法处理过长的文本，超过窗口长度的信息会被遗忘 | 无法完成需要大量背景知识的复杂任务 |
| 幻觉问题 | 会编造看似合理但实际错误的信息 | 输出不可靠，无法用于高风险场景 |
| 知识截止 | 训练数据有截止日期，无法获取实时信息 | 无法回答最新的新闻、天气、价格等问题 |
| 无法执行外部动作 | 只能生成文本，不能直接操作文件、调用API、控制设备 | 无法完成需要与外部世界交互的任务 |
| 缺乏长期记忆 | 每次对话都是独立的，无法记住之前对话的所有细节 | 无法进行长期的、多轮的复杂任务 |

## 三、从LLM到Agent：为什么需要Agent

### 3.1 什么是Agent

-   **标准定义**：能够**自主感知环境、做出决策、执行动作**，并通过反馈循环不断优化，以达成特定目标的智能系统
    
-   **通俗理解**：如果说LLM是一个"只会说话的大脑"，那么Agent就是给这个大脑装上了"眼睛（感知）"、"手脚（工具）"和"记忆（存储）"，让它能够真正"做事"
    

### 3.2 Agent与普通LLM应用的本质区别

| 类型 | 核心逻辑 | 能力边界 | 适用场景 |
| 纯提示词工程 | 输入→LLM→输出，单轮交互 | 只能利用LLM自身的知识和能力 | 简单的文本处理、问答、创意生成 |
| RAG应用 | 输入→检索相关知识→增强提示→LLM→输出 | 可以利用外部知识库，但仍然只能生成文本 | 文档问答、知识库客服 |
| Agent | 输入→规划→工具调用→执行→反馈→迭代→输出，多轮循环 | 可以自主调用工具、执行动作、长期记忆、动态规划 | 复杂任务自动化、智能助手、自主系统 |

### 3.3 Agent的核心价值

-   **突破LLM的能力边界**：解决LLM无法获取实时信息、无法执行外部动作、缺乏长期记忆等问题
    
-   **实现复杂任务自动化**：将需要多步骤、多工具、多轮交互的复杂任务交给Agent自主完成
    
-   **降低使用门槛**：用户只需要用自然语言描述目标，不需要关心具体的执行步骤
    

## 四、Agent的四大核心组件

一个完整的Agent系统由四个不可缺少的核心组件组成，它们协同工作，实现Agent的自主运行。

### 4.1 规划模块（大脑的"思考中枢"）

-   **核心作用**：将用户的复杂目标分解为可执行的子任务，制定执行计划，并根据执行结果动态调整
    
-   **核心技术**：
    

1.  **任务分解**：
    

-   思维链（CoT）：将复杂问题分解为一系列简单的推理步骤
    
-   思维树（ToT）：同时探索多个可能的推理路径，选择最优解
    
-   思维图（GoT）：将任务分解为有向无环图，支持并行执行和依赖管理
    

2.  **反思与优化**：
    

-   自我反思（Self-Reflection）：Agent对自己的执行结果进行评估，找出错误和不足
    
-   自我批评（Self-Criticism）：Agent对自己的计划和决策进行批判性分析
    
-   迭代优化：根据反思结果调整计划，重新执行
    

3.  **目标管理**：
    

-   长期目标与短期目标的分解
    
-   任务优先级排序
    
-   进度跟踪与状态更新
    

### 4.2 记忆模块（大脑的"存储系统"）

-   **核心作用**：存储Agent的历史对话、执行过程、学习到的知识和经验，让Agent能够"记住过去"
    
-   **记忆的三个层次**：
    

1.  **短期记忆（工作记忆）**：
    

-   存储当前对话和执行过程中的临时信息
    
-   对应LLM的上下文窗口
    
-   特点：速度快、容量小、临时存储
    

2.  **长期记忆**：
    

-   存储Agent的历史经验、知识和用户偏好
    
-   通常使用向量数据库（Vector DB）实现
    
-   特点：容量大、持久化存储、需要检索才能访问
    

3.  **程序记忆**：
    

-   存储Agent的技能、规则和工作流程
    
-   例如：如何调用某个工具、如何完成某个特定任务
    
-   通常通过提示词工程或微调实现
    

-   **记忆管理技术**：
    

-   向量检索：将记忆转换为向量，通过相似度搜索快速找到相关信息
    
-   记忆摘要：对长期记忆进行摘要，减少检索结果的长度
    
-   重要性评分：根据记忆的重要性进行排序，优先保留重要信息
    
-   遗忘机制：自动删除不重要或过时的记忆
    

### 4.3 工具使用模块（大脑的"手脚"）

-   **核心作用**：让Agent能够与外部世界交互，获取信息和执行动作
    
-   **工具调用原理**：
    

1.  工具注册：开发者将工具的名称、描述、参数和返回值格式告诉LLM
    
2.  工具选择：LLM根据当前任务需要，选择合适的工具
    
3.  参数生成：LLM生成调用工具所需的参数
    
4.  工具执行：Agent调用工具，获取返回结果
    
5.  结果处理：LLM根据工具返回的结果，继续下一步操作
    

-   **常见工具类型**：
    

1.  **信息检索工具**：搜索引擎、RAG、数据库查询
    
2.  **计算工具**：计算器、Python解释器
    
3.  **文件操作工具**：读取、写入、修改文件
    
4.  **API调用工具**：调用第三方服务（天气、地图、邮件等）
    
5.  **代码执行工具**：运行代码、调试程序
    
6.  **控制工具**：控制硬件设备、自动化操作
    

### 4.4 执行模块（系统的"调度中心"）

-   **核心作用**：协调规划、记忆和工具模块，按照计划执行任务，并处理执行过程中的错误和异常
    
-   **核心功能**：
    

1.  状态管理：跟踪任务的执行状态和进度
    
2.  动作调度：按照计划调用相应的工具和模块
    
3.  错误处理：处理工具调用失败、LLM输出异常等情况
    
4.  反馈循环：将执行结果反馈给规划模块，用于调整计划
    
5.  终止判断：判断任务是否完成，是否需要继续执行
    

## 五、Agent的完整工作流：Agent到底是怎么运行的

我们用一个具体的例子来拆解Agent的完整运行过程：**用户请求："帮我预订下周三从北京到上海的机票，预算1000元以内，优先上午的航班"**

### 5.1 步骤1：任务接收与理解

-   Agent接收用户的自然语言请求
    
-   LLM解析用户的核心目标、约束条件和偏好：
    

-   核心目标：预订机票
    
-   出发地：北京
    
-   目的地：上海
    
-   日期：下周三
    
-   预算：1000元以内
    
-   偏好：优先上午的航班
    

### 5.2 步骤2：任务分解与规划

-   规划模块将复杂目标分解为可执行的子任务：
    

1.  查询下周三北京到上海的所有航班信息
    
2.  筛选出价格在1000元以内且上午起飞的航班
    
3.  选择最合适的航班（综合考虑价格、时间、航空公司等）
    
4.  获取用户的个人信息（姓名、身份证号、联系方式）
    
5.  调用机票预订API完成预订
    
6.  向用户发送预订成功的通知
    

-   制定执行计划，明确每个子任务的顺序和依赖关系
    

### 5.3 步骤3：信息检索与工具调用

-   Agent判断需要调用"航班查询工具"来获取实时航班信息
    
-   LLM生成工具调用参数：
    

```
{
  "tool_name": "flight_search",
  "parameters": {
    "departure": "北京",
    "destination": "上海",
    "date": "2026-06-03",
    "max_price": 1000
  }
}
```

-   执行模块调用航班查询工具，获取返回结果
    
-   将工具返回的结果存入短期记忆
    

### 5.4 步骤4：结果整合与推理

-   LLM根据工具返回的航班信息，筛选出符合条件的上午航班
    
-   对筛选后的航班进行排序，选择最合适的3个选项
    
-   生成航班对比信息，包括：航班号、起飞时间、到达时间、价格、航空公司
    

### 5.5 步骤5：人工确认与反馈

-   Agent将3个航班选项展示给用户，请求用户选择
    
-   用户选择其中一个航班，并提供个人信息
    
-   Agent将用户的选择和个人信息存入短期记忆
    

### 5.6 步骤6：继续执行与完成

-   Agent调用"机票预订工具"，传入用户选择的航班和个人信息
    
-   工具返回预订成功的结果，包括订单号、支付链接
    
-   Agent将订单信息存入长期记忆，方便用户后续查询
    
-   向用户发送预订成功的通知，并提供支付链接和注意事项
    

### 5.7 步骤7：任务结束与复盘

-   Agent判断任务已经完成，结束执行
    
-   对本次任务的执行过程进行复盘，记录经验教训
    
-   更新长期记忆，优化未来的任务执行
    

## 六、主流Agent架构对比

### 6.1 ReAct架构（最基础、最常用）

-   **核心思想**：将**推理（Reasoning）**和**行动（Acting）**结合起来，让Agent在推理的过程中穿插行动，通过行动获取更多信息来辅助推理
    
-   **工作流程**：思考→行动→观察→思考→行动→观察→...→完成
    
-   **优点**：简单易懂、易于实现、效果稳定
    
-   **缺点**：规划能力有限，不适合非常复杂的任务
    
-   **适用场景**：大多数简单到中等复杂度的Agent任务
    

### 6.2 AutoGPT架构（自主Agent）

-   **核心思想**：让Agent完全自主地分解目标、制定计划、执行任务、反思优化，不需要人类干预
    
-   **工作流程**：设定目标→生成任务列表→执行任务→评估结果→调整计划→...→达成目标
    
-   **优点**：自主性强，能够处理复杂的长期任务
    
-   **缺点**：容易陷入无限循环、容易偏离目标、资源消耗大
    
-   **适用场景**：需要长期自主运行的任务，如研究、写作、项目管理
    

### 6.3 BabyAGI架构（任务驱动型Agent）

-   **核心思想**：基于任务优先级的自主Agent，能够动态生成和管理任务列表
    
-   **工作流程**：接收目标→生成任务→优先级排序→执行最高优先级任务→生成新任务→重新排序→...→完成
    
-   **优点**：任务管理能力强，能够高效处理多个并行任务
    
-   **缺点**：规划能力较弱，容易产生无关任务
    
-   **适用场景**：任务密集型工作，如内容创作、数据处理
    

### 6.4 CrewAI架构（多Agent协作）

-   **核心思想**：将复杂任务分解给多个不同角色的Agent，通过Agent之间的协作来完成任务
    
-   **核心概念**：
    

-   Agent：具有特定角色、目标和工具的智能体
    
-   Task：分配给Agent的具体任务
    
-   Crew：由多个Agent和Task组成的团队
    

-   **优点**：能够处理非常复杂的任务，发挥不同Agent的专长
    
-   **缺点**：架构复杂，需要精心设计角色和任务分配
    
-   **适用场景**：大型复杂项目，如软件开发、市场营销、研究报告撰写
    

### 6.5 LangGraph架构（状态机式工作流）

-   **核心思想**：将Agent的工作流建模为状态机，通过节点和边来定义任务的执行流程和状态转换
    
-   **优点**：可控性强、可观测性好、支持复杂的分支和循环逻辑
    
-   **缺点**：学习曲线较陡，需要理解状态机的概念
    
-   **适用场景**：需要精确控制执行流程的生产级Agent应用
    

## 七、Agent开发的关键挑战与解决方案

### 7.1 上下文窗口限制

-   **问题**：LLM的上下文窗口有限，无法处理过长的对话和大量的信息
    
-   **解决方案**：
    

-   使用RAG技术，将外部知识存储在向量数据库中，只检索相关信息
    
-   对长文本进行摘要，只保留关键信息
    
-   使用分块处理，将长文本分成多个小块分别处理
    
-   选择上下文窗口更大的模型（如GPT-4o、Claude 3 Opus）
    

### 7.2 幻觉问题

-   **问题**：LLM会编造错误信息，导致Agent的输出不可靠
    
-   **解决方案**：
    

-   强制Agent使用工具获取事实信息，不要依赖自身知识
    
-   对Agent的输出进行事实核查，使用多个来源交叉验证
    
-   要求Agent提供信息来源和证据
    
-   使用低温度参数（temperature<0.3），减少随机性
    

### 7.3 工具调用准确性

-   **问题**：LLM经常会生成错误的工具参数，或者选择错误的工具
    
-   **解决方案**：
    

-   提供清晰、详细的工具描述和参数说明
    
-   使用结构化输出（如JSON），强制LLM按照指定格式生成参数
    
-   对工具参数进行验证，发现错误时要求LLM重新生成
    
-   实现错误重试机制，最多重试3-5次
    

### 7.4 长期记忆管理

-   **问题**：长期记忆过多会导致检索效率低下，并且容易引入无关信息
    
-   **解决方案**：
    

-   对记忆进行重要性评分，只保留重要的记忆
    
-   定期对长期记忆进行摘要和整理
    
-   使用时间衰减机制，降低旧记忆的权重
    
-   实现记忆的分类管理，将不同类型的记忆分开存储
    

### 7.5 多Agent协作

-   **问题**：多个Agent之间容易出现沟通不畅、任务重复、责任不清等问题
    
-   **解决方案**：
    

-   明确每个Agent的角色、职责和权限
    
-   设计清晰的通信协议和信息共享机制
    
-   设立一个"协调者Agent"，负责任务分配和进度管理
    
-   建立冲突解决机制，处理Agent之间的意见分歧
    

## 八、入门实践指南：5分钟写出你的第一个Agent

### 8.1 必备工具

-   Python 3.10+
    
-   LangChain（最流行的Agent开发框架）
    
-   OpenAI API密钥（或其他LLM API）
    
-   Tavily API密钥（搜索引擎工具）
    

### 8.2 安装依赖

```
pip install langchain langchain-openai langchain-community tavily-python
```

### 8.3 第一个简单Agent（能搜索和计算）

```
from langchain_openai import ChatOpenAI
from langchain_community.tools.tavily_search import TavilySearchResults
from langchain.tools import tool
from langchain.agents import create_react_agent, AgentExecutor
from langchain import hub

# 初始化LLM
llm = ChatOpenAI(model="gpt-3.5-turbo", temperature=0)

# 定义工具
search = TavilySearchResults(max_results=2)

@tool
def calculator(expression: str) -> float:
    """执行数学计算，输入是一个数学表达式字符串，如"2+3*4" """
    return eval(expression)

tools = [search, calculator]

# 获取ReAct提示词
prompt = hub.pull("hwchase17/react")

# 创建Agent
agent = create_react_agent(llm, tools, prompt)

# 创建Agent执行器
agent_executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# 运行Agent
result = agent_executor.invoke({
    "input": "2026年世界杯的举办地是哪里？举办时间是什么时候？从现在到开幕还有多少天？"
})

print(result["output"])
```

1.  学习LangGraph，掌握状态机式Agent开发
    
2.  添加记忆模块，实现有记忆的Agent
    
3.  开发自定义工具，扩展Agent的能力
    
4.  学习CrewAI，尝试多Agent协作
    
5.  部署Agent为API或Web应用

# AI系统六大核心安全与治理意识详细笔记

## 一、引言：为什么这六大意识是AI系统的生命线

随着AI系统从实验环境走向生产应用，特别是在金融、医疗、政务、工业控制等高风险领域，单纯的功能正确性已经远远不够。**权限、安全、人工确认、日志、验证材料和失败恢复**这六大意识，构成了AI系统可靠运行的基础防线，也是区分"玩具级AI"和"生产级AI"的关键标志。

缺乏这些意识的AI系统，即使在测试环境中表现完美，在生产环境中也必然会出现灾难性故障，可能导致数据泄露、财产损失、人身伤害甚至法律责任。

## 二、权限意识：最小权限原则的全面贯彻

### 2.1 核心原则

-   **最小权限原则**：AI系统及其组件只能拥有完成其指定任务所必需的最小权限，绝不多给
    
-   **权限分离原则**：将关键操作的权限分配给不同的角色，避免单一角色拥有过大权力
    
-   **按需授权原则**：权限应根据任务需要临时授予，任务完成后立即收回
    
-   **零信任原则**：默认不信任任何内部或外部请求，所有访问都必须经过验证和授权
    

### 2.2 具体实践

1.  **AI模型权限控制**
    

-   模型只能访问其训练和推理所需的特定数据集
    
-   禁止模型拥有写入数据库、修改系统配置或执行系统命令的权限
    
-   不同模型实例应使用不同的服务账号，避免权限共享
    

2.  **API接口权限管理**
    

-   为每个API端点定义明确的访问角色和权限级别
    
-   使用API密钥、OAuth2.0或JWT进行身份验证
    
-   对API调用频率和并发数进行限制，防止滥用
    

3.  **用户权限分级**
    

-   建立清晰的用户角色体系：管理员、开发者、普通用户、访客等
    
-   管理员权限应严格限制人数，并采用多因素认证
    
-   定期审计用户权限，及时撤销不再需要的权限
    

### 2.3 常见误区

-   为了开发方便给AI系统"root"或"管理员"权限
    
-   使用单一账号运行所有AI服务
    
-   权限授予后长期不审计、不回收
    
-   忽视第三方依赖库和插件的权限风险
    

## 三、安全意识：构建多层次防御体系

### 3.1 数据安全

-   **训练数据安全**：对训练数据进行脱敏处理，去除个人隐私信息；加密存储训练数据；建立数据访问审计机制
    
-   **推理数据安全**：对用户输入和模型输出进行加密传输；避免在推理过程中缓存敏感数据
    
-   **模型权重安全**：加密存储模型权重文件；防止模型被窃取或逆向工程；对模型部署环境进行隔离
    

### 3.2 模型安全

-   **对抗攻击防护**：训练模型时加入对抗样本，提高模型鲁棒性；在推理阶段检测并拦截对抗性输入
    
-   **投毒攻击防护**：对训练数据进行清洗和验证；监控训练过程中的异常数据分布
    
-   **模型窃取防护**：使用模型水印技术；限制模型API的输出格式和内容；对异常查询模式进行检测
    

### 3.3 运行安全

-   **容器化部署**：使用Docker等容器技术隔离AI运行环境
    
-   **网络隔离**：将AI系统部署在独立的网络区域，与其他系统进行隔离
    
-   **漏洞管理**：定期扫描AI系统及其依赖的漏洞；及时更新安全补丁
    
-   **入侵检测**：部署入侵检测系统，监控异常网络流量和系统行为
    

### 3.4 供应链安全

-   只使用经过安全验证的第三方库和模型
    
-   建立软件物料清单(SBOM)，跟踪所有依赖组件
    
-   对开源模型进行安全审计，避免引入恶意代码
    

## 四、人工确认意识：明确人机协作的边界

### 4.1 核心原则

-   **关键决策必须人工确认**：任何可能产生重大影响的决策，都不能由AI单独做出
    
-   **人类拥有最终决定权**：AI只能提供建议，最终决策权必须掌握在人类手中
    
-   **明确的人工介入触发条件**：预先定义哪些情况必须触发人工审核
    
-   **可解释性优先**：AI的决策过程必须能够向人类解释清楚
    

### 4.2 必须人工确认的场景

1.  **高风险决策**
    

-   金融领域：大额交易审批、贷款发放、投资建议执行
    
-   医疗领域：疾病诊断、治疗方案制定、手术决策
    
-   工业领域：设备启停、工艺流程调整、安全联锁操作
    
-   法律领域：合同审核、法律意见出具、案件判决建议
    

2.  **异常情况**
    

-   AI输出结果与历史数据偏差超过预设阈值
    
-   AI对同一输入产生相互矛盾的输出
    
-   AI的置信度低于预设值
    
-   检测到可能的对抗性输入
    

3.  **首次操作**
    

-   AI系统首次执行某项新任务
    
-   AI系统在新环境中首次运行
    
-   AI模型更新后首次投入使用
    

### 4.3 人工确认流程设计

-   明确人工确认的责任人、职责和响应时间要求
    
-   提供清晰的AI决策依据和解释信息
    
-   允许人工审核者拒绝AI决策并给出理由
    
-   记录人工确认的全过程，包括审核者、时间、结果和理由
    

### 4.4 常见误区

-   过度信任AI，将所有决策权交给AI
    
-   人工确认流于形式，审核者不认真查看AI输出
    
-   没有明确的人工介入触发条件，导致要么过度干预要么干预不足
    
-   忽视人工审核者的培训和能力建设
    

## 五、日志意识：建立可追溯的审计体系

### 5.1 核心原则

-   **全面性原则**：记录所有关键操作和事件，确保没有遗漏
    
-   **不可篡改原则**：日志一旦生成，就不能被修改或删除
    
-   **可审计原则**：日志内容应清晰易懂，便于审计和调查
    
-   **长期保存原则**：日志应保存足够长的时间，满足合规和调查需求
    

### 5.2 必须记录的日志内容

1.  **访问日志**
    

-   所有用户和系统的登录、登出记录
    
-   对AI系统资源的所有访问请求
    
-   权限变更记录
    

2.  **操作日志**
    

-   AI模型的训练、微调、部署和更新记录
    
-   所有API调用记录，包括调用者、时间、输入参数和输出结果
    
-   人工确认操作记录
    
-   系统配置变更记录
    

3.  **错误和异常日志**
    

-   所有系统错误和异常信息
    
-   AI模型推理失败记录
    
-   安全事件记录
    
-   性能异常记录
    

4.  **数据处理日志**
    

-   数据导入、导出和处理记录
    
-   敏感数据访问记录
    
-   数据备份和恢复记录
    

### 5.3 日志管理最佳实践

-   使用集中式日志管理系统，如ELK Stack、Splunk等
    
-   对日志进行加密存储和传输
    
-   定期备份日志，防止日志丢失
    
-   建立日志审计机制，定期检查日志中的异常行为
    
-   确保日志符合相关法律法规的要求，如GDPR、等保2.0等
    

### 5.4 常见误区

-   只记录错误日志，不记录正常操作日志
    
-   日志内容过于简单，无法追溯问题根源
    
-   日志可以被随意修改或删除
    
-   日志保存时间过短，无法满足长期调查需求
    

## 六、验证材料意识：确保输入输出的真实性和有效性

### 6.1 核心原则

-   **输入验证原则**：对所有输入数据进行严格验证，拒绝无效和恶意输入
    
-   **输出验证原则**：对AI的输出结果进行验证，确保其准确性和合理性
    
-   **材料真实性原则**：验证所有提交给AI的材料的真实性和完整性
    
-   **可追溯原则**：所有验证过程和结果都应可追溯
    

### 6.2 输入验证

-   **格式验证**：检查输入数据的格式是否符合要求
    
-   **范围验证**：检查输入数据的取值是否在合理范围内
    
-   **类型验证**：检查输入数据的类型是否正确
    
-   **长度验证**：检查输入数据的长度是否在允许范围内
    
-   **内容验证**：检查输入数据是否包含恶意内容，如SQL注入、XSS攻击等
    

### 6.3 输出验证

-   **合理性验证**：检查AI输出结果是否符合常识和业务逻辑
    
-   **一致性验证**：检查AI对同一输入的多次输出是否一致
    
-   **准确性验证**：将AI输出结果与已知正确答案进行比对
    
-   **安全性验证**：检查AI输出是否包含敏感信息或有害内容
    

### 6.4 材料真实性验证

-   **文档验证**：验证提交的文档的真实性，如检查水印、印章、签名等
    
-   **身份验证**：验证用户的身份信息，如身份证、人脸识别等
    
-   **数据来源验证**：验证数据的来源是否可靠
    
-   **时间戳验证**：验证材料的生成时间是否合理
    

### 6.5 常见误区

-   信任用户输入，不进行任何验证
    
-   只在客户端进行验证，不在服务端进行验证
    
-   过度信任AI的输出，不进行任何验证
    
-   忽视材料真实性验证，导致AI基于虚假材料做出错误决策
    

## 七、失败恢复意识：构建韧性系统

### 7.1 核心原则

-   **预防为主原则**：尽可能预防故障的发生
    
-   **快速恢复原则**：当故障发生时，能够快速恢复系统正常运行
    
-   **数据不丢失原则**：确保在故障发生时，重要数据不会丢失
    
-   **业务连续性原则**：将故障对业务的影响降到最低
    

### 7.2 故障预防

-   进行充分的测试，包括单元测试、集成测试、系统测试和压力测试
    
-   建立监控系统，实时监控AI系统的运行状态
    
-   定期进行健康检查，及时发现潜在问题
    
-   建立冗余系统，避免单点故障
    

### 7.3 故障检测

-   定义明确的故障指标和阈值
    
-   建立自动告警机制，当故障发生时及时通知相关人员
    
-   使用AI技术进行异常检测，提前发现潜在故障
    
-   定期进行故障演练，检验故障检测机制的有效性
    

### 7.4 故障恢复

-   **回滚机制**：当AI模型更新后出现问题时，能够快速回滚到之前的稳定版本
    
-   **备份恢复**：定期备份重要数据和系统配置，当数据丢失或系统损坏时能够快速恢复
    
-   **降级机制**：当系统负载过高或部分组件故障时，能够自动降级服务，保证核心功能正常运行
    
-   **灾难恢复**：建立异地灾备中心，应对重大灾难事件
    

### 7.5 故障复盘

-   每次故障发生后，都要进行详细的复盘分析
    
-   找出故障的根本原因，而不仅仅是表面原因
    
-   制定改进措施，防止类似故障再次发生
    
-   记录故障处理过程和经验教训，形成知识库
    

### 7.6 常见误区

-   只关注功能开发，忽视故障预防和恢复
    
-   没有备份或备份不及时、不完整
    
-   没有回滚机制，一旦出现问题只能从头开始
    
-   故障发生后不进行复盘，导致同样的问题反复出现
    

## 八、综合实践：如何在实际项目中培养和落实这六大意识

### 8.1 团队建设

-   对所有团队成员进行AI安全和治理培训
    
-   建立专门的AI安全团队，负责监督和落实安全措施
    
-   将安全指标纳入团队绩效考核
    
-   鼓励团队成员发现和报告安全问题
    

### 8.2 流程建设

-   将六大意识融入AI系统开发的全生命周期
    
-   建立安全评审机制，在项目的关键节点进行安全评审
    
-   制定详细的操作手册和应急预案
    
-   定期进行安全审计和漏洞扫描
    

### 8.3 技术建设

-   采用成熟的AI安全框架和工具
    
-   建立统一的权限管理、日志管理和监控系统
    
-   开发自动化的验证和测试工具
    
-   持续跟踪AI安全领域的最新进展，及时采用新技术
    

## 九、总结与行动清单

### 9.1 总结

权限、安全、人工确认、日志、验证材料和失败恢复这六大意识，是AI系统可靠运行的基石。它们不是可有可无的"锦上添花"，而是必须严格落实的"生命线"。只有将这六大意识深入到每个团队成员的心中，融入到AI系统开发和运行的每一个环节，才能构建出真正安全、可靠、负责任的AI系统。

### 9.2 立即行动清单

1.  审核当前AI系统的权限设置，确保符合最小权限原则
    
2.  检查AI系统的日志记录情况，确保所有关键操作都被记录
    
3.  列出所有必须人工确认的场景，制定明确的人工确认流程
    
4.  建立输入输出验证机制，确保数据的真实性和有效性
    
5.  制定故障应急预案，并进行一次故障演练
    
6.  对团队成员进行一次AI安全和治理培训

# 区块链链上操作链完整学习笔记

版本：v1.0 | 更新时间：2026-05-22 | 适用人群：Web3初学者、区块链开发者、加密货币投资者

## 前言

区块链的所有操作本质上都是**状态变更**。账户、钱包、签名、交易、Gas、合约、测试网和区块浏览器这八个核心概念，构成了一条完整且不可分割的**链上操作流水线**。任何链上行为（转账、NFT铸造、DeFi交互）都必须严格按照这条流水线执行，缺一不可。

本笔记将以\*\*"一次完整的ETH转账"\*\*为主线，串联所有概念，让你不仅理解每个概念的定义，更能明白它们如何协同工作。

* * *

## 一、核心概念详解（按操作顺序排列）

### 1\. 账户（Account）：链上身份的唯一标识

核心定义

账户是区块链上的**数字身份**，是所有资产和操作的归属主体。每个账户都有一个全球唯一的地址，类似于银行账号。

关键分类（以太坊模型，绝大多数公链沿用）

| 类型 | 全称 | 核心特征 | 能否主动发起交易 | 存储内容 |
| EOA账户 | Externally Owned Account（外部拥有账户） | 由人类用户控制，通过私钥管理 | ✅ 可以 | 以太币余额、nonce值 |
| 合约账户 | Contract Account | 由代码控制，没有私钥 | ❌ 只能被动响应调用 | 以太币余额、合约代码、存储数据 |

在操作链中的作用

-   所有资产都归属在账户地址下，而非个人
    
-   所有交易都必须从一个EOA账户发起
    
-   合约账户是所有智能合约应用（DeFi、NFT、GameFi）的载体
    

常见误区

-   ❌ 账户里"存着"加密货币：实际上账户只记录余额，加密货币是区块链账本上的状态
    
-   ❌ 地址可以修改：地址由公钥哈希生成，一旦创建永久不变
    

* * *

### 2\. 钱包（Wallet）：管理账户的工具

核心定义

钱包**不存储加密货币**，它是用于**管理私钥、发起交易、查看余额**的客户端工具。它就像你管理银行账户的手机银行APP。

核心功能

1.  生成和存储私钥/助记词
    
2.  从私钥推导公钥和账户地址
    
3.  对交易进行签名
    
4.  广播交易到区块链网络
    
5.  查询账户余额和交易历史
    

主流钱包类型

-   **热钱包**：联网钱包，如MetaMask、Trust Wallet、Coinbase Wallet（方便但有被黑客攻击风险）
    
-   **冷钱包**：离线硬件钱包，如Ledger、Trezor（安全但操作复杂）
    
-   **托管钱包**：由交易所托管私钥，如Binance、Coinbase（方便但失去资产控制权）
    

在操作链中的作用

-   是用户与区块链交互的唯一入口
    
-   负责将用户的操作意图转化为区块链可识别的交易
    
-   保护用户的私钥安全（私钥=资产所有权）
    

常见误区

-   ❌ 钱包丢了资产就没了：只要私钥/助记词还在，就能在任何钱包中恢复资产
    
-   ❌ 不同钱包之间不能互转：只要地址正确，任何钱包都可以向任何地址转账
    

* * *

### 3\. 签名（Signature）：链上操作的身份验证

核心定义

签名是使用**私钥**对交易数据进行加密的过程，用于证明"这笔交易确实是账户所有者发起的"。它是区块链上唯一的身份验证方式，没有密码、没有验证码。

核心原理（非对称加密）

1.  私钥：只有你知道，用于签名
    
2.  公钥：可以公开，用于验证签名
    
3.  地址：公钥的哈希值，用于接收资产
    

**特性**：用私钥签名的交易，只能用对应的公钥验证；任何人都无法从公钥反推出私钥。

在操作链中的作用

-   验证交易发起者的身份
    
-   防止交易被篡改（一旦签名，交易数据任何修改都会导致签名无效）
    
-   防止交易被抵赖（签名者无法否认自己发起的交易）
    

常见误区

-   ❌ 签名会泄露私钥：签名过程在本地完成，私钥永远不会离开你的钱包
    
-   ❌ 交易可以取消：一旦签名并广播，交易无法取消或撤回
    

* * *

### 4\. 交易（Transaction）：链上状态变更的载体

核心定义

交易是一条**签名后的指令**，用于请求区块链网络改变状态。所有链上状态变更（转账、合约调用、NFT铸造）都必须通过交易来实现。

交易的核心字段（以太坊为例）

| 字段 | 含义 |
| from | 发送方地址（必须是EOA账户） |
| to | 接收方地址（可以是EOA或合约账户） |
| value | 转账的ETH数量（单位：wei，1 ETH=10^18 wei） |
| gasLimit | 愿意为这笔交易支付的最大Gas量 |
| gasPrice | 愿意为每单位Gas支付的价格（单位：gwei） |
| nonce | 发送方账户的交易序号（防止重放攻击） |
| data | 附加数据（转账时为空，调用合约时为合约方法和参数） |
| signature | 发送方的签名 |

交易的生命周期

1.  **创建**：用户在钱包中填写交易信息
    
2.  **签名**：钱包用私钥对交易进行签名
    
3.  **广播**：钱包将签名后的交易发送到区块链网络
    
4.  **验证**：节点验证交易的签名和有效性
    
5.  **打包**：矿工将交易打包进一个新的区块
    
6.  **确认**：区块被添加到区块链上，交易生效
    
7.  **最终确认**：后续多个区块被添加，交易不可逆转
    

在操作链中的作用

-   是用户向区块链网络发送指令的唯一方式
    
-   包含了所有需要执行的操作信息
    
-   是区块链状态变更的唯一触发条件
    

* * *

### 5\. Gas：链上操作的燃料费

核心定义

Gas是区块链网络的**运行燃料**，用户需要支付Gas费来激励矿工处理交易并维护网络安全。它就像汽车的汽油，没有Gas，任何交易都无法执行。

核心概念

-   **Gas Price**：每单位Gas的价格，由用户自行设定（单位：gwei）。价格越高，交易被矿工优先处理的概率越大
    
-   **Gas Limit**：用户愿意为这笔交易支付的最大Gas量。如果交易执行消耗的Gas超过Gas Limit，交易失败，但已消耗的Gas不会退还
    
-   **Gas Used**：交易实际消耗的Gas量
    
-   **交易手续费** = Gas Used × Gas Price
    

Gas费的动态调整

-   Gas价格由市场供需决定：网络拥堵时，Gas价格上涨；网络空闲时，Gas价格下跌
    
-   EIP-1559后，以太坊引入了基础费（Base Fee）和优先费（Priority Fee）：
    

-   基础费：由网络自动计算，会被销毁
    
-   优先费：给矿工的小费，用于提高交易优先级
    

在操作链中的作用

-   防止网络被垃圾交易攻击
    
-   激励矿工维护网络安全
    
-   调节网络拥堵状况
    

常见误区

-   ❌ Gas费是固定的：Gas费随网络拥堵情况实时变化
    
-   ❌ 交易失败不用付Gas费：只要交易被矿工执行，无论成功与否，都需要支付已消耗的Gas费
    

* * *

### 6\. 智能合约（Smart Contract）：链上的自动化程序

核心定义

智能合约是部署在区块链上的**一段可执行代码**，它按照预先设定的规则自动执行，无需第三方干预。它是所有区块链应用的基础。

核心特性

-   **去中心化**：部署在区块链上，没有单一控制者
    
-   **透明性**：代码公开可见，任何人都可以查看
    
-   **不可篡改**：一旦部署，代码无法修改
    
-   **自动执行**：满足条件时自动执行，无需人工干预
    

与普通交易的区别

-   普通转账交易：`to`字段是EOA地址，`data`字段为空，只改变账户余额
    
-   合约调用交易：`to`字段是合约地址，`data`字段包含要调用的方法和参数，会执行合约代码并改变合约状态
    

在操作链中的作用

-   实现复杂的链上逻辑（如DeFi的借贷、交易、挖矿）
    
-   管理链上资产（如ERC-20代币、ERC-721 NFT）
    
-   构建去中心化应用（DApp）
    

* * *

### 7\. 测试网（Testnet）：安全的实验环境

核心定义

测试网是与主网（Mainnet）功能完全相同但**资产没有实际价值**的区块链网络。它是开发者和用户进行实验、测试和学习的安全环境。

主流测试网（以太坊生态）

| 测试网 | 状态 | 特点 |
| Sepolia | 活跃 | 目前最常用的测试网，适合大多数测试场景 |
| Holesky | 活跃 | 大型测试网，适合测试高负载场景 |
| Goerli | 已弃用 | 逐步被Sepolia取代 |

测试网水龙头（Faucet）

-   用于免费获取测试网代币（如Sepolia ETH）
    
-   测试网代币没有实际价值，只能在测试网上使用
    
-   每个水龙头都有领取限制，防止滥用
    

在操作链中的作用

-   开发者在部署主网合约前，先在测试网上测试代码
    
-   用户在使用新的DApp前，先在测试网上熟悉操作流程
    
-   学习区块链操作时，在测试网上练习，避免损失真实资产
    

* * *

### 8\. 区块浏览器（Block Explorer）：链上数据的查询窗口

核心定义

区块浏览器是一个**可视化的区块链数据查询工具**，它允许用户查看区块链上的所有公开数据，包括区块、交易、账户、合约等。它就像区块链的"搜索引擎"。

核心功能

1.  查询账户余额和交易历史
    
2.  查询交易状态和详情
    
3.  查询区块信息和区块高度
    
4.  查看智能合约代码和状态
    
5.  追踪资产流向
    
6.  查看网络统计数据（Gas价格、区块时间、算力等）
    

主流区块浏览器

-   以太坊：Etherscan（[https://etherscan.io/](https://etherscan.io/)[）](https://etherscan.io/%EF%BC%89)
    
-   比特币：[Blockchain.com](http://Blockchain.com)（[https://www.blockchain.com/](https://www.blockchain.com/)[）](https://www.blockchain.com/%EF%BC%89)
    
-   Solana：Solscan（[https://solscan.io/](https://solscan.io/)[）](https://solscan.io/%EF%BC%89)
    
-   测试网：Sepolia Etherscan（[https://sepolia.etherscan.io/](https://sepolia.etherscan.io/)[）](https://sepolia.etherscan.io/%EF%BC%89)
    

在操作链中的作用

-   验证交易是否成功
    
-   追踪资产的来源和去向
    
-   审计智能合约代码
    
-   监控网络状态

## 二、完整链上操作流程示例（小明给小红转1 ETH）

让我们用一个最常见的ETH转账例子，把所有概念串联起来，看看每一步到底发生了什么：

### 步骤1：准备工作

-   小明和小红都有自己的EOA账户
    
-   小明使用MetaMask钱包管理自己的账户
    
-   小明的账户里有足够的ETH（至少1 ETH + Gas费）
    

### 步骤2：创建交易

1.  小明打开MetaMask钱包，点击"发送"
    
2.  输入小红的账户地址（`to`字段）
    
3.  输入转账金额：1 ETH（`value`字段）
    
4.  MetaMask自动获取小明账户的当前nonce值
    
5.  MetaMask根据当前网络状况推荐Gas Price和Gas Limit
    

### 步骤3：签名交易

1.  小明确认交易信息无误后，点击"确认"
    
2.  MetaMask在**本地**用小明的私钥对交易数据进行签名
    
3.  生成一个签名后的交易对象（包含所有交易字段和签名）
    

### 步骤4：广播交易

1.  MetaMask将签名后的交易发送到以太坊网络的一个节点
    
2.  该节点验证交易的签名是否有效
    
3.  如果有效，节点将交易广播到整个网络
    
4.  交易进入交易池（Mempool），等待矿工打包
    

### 步骤5：支付Gas费

-   小明设置的Gas Price为20 gwei，Gas Limit为21000（ETH转账的标准Gas消耗）
    
-   最大可能手续费：21000 × 20 gwei = 420000 gwei = 0.00042 ETH
    
-   小明的账户会被暂时锁定这部分金额，直到交易被处理
    

### 步骤6：矿工打包交易

1.  矿工从交易池中选择Gas Price高的交易
    
2.  矿工将小明的交易和其他交易一起打包进一个新的区块
    
3.  矿工执行交易：
    

-   从小明的账户扣除1 ETH + 实际消耗的Gas费
    
-   向小红的账户增加1 ETH
    
-   小明的nonce值加1
    

4.  矿工将新区块广播到网络
    

### 步骤7：交易确认

1.  其他节点验证新区块的有效性
    
2.  当新区块被添加到区块链上时，交易正式生效
    
3.  小红的钱包会显示收到1 ETH
    
4.  小明的钱包会显示交易成功
    

### 步骤8：查询交易

1.  小明在MetaMask中点击交易记录，复制交易哈希（Transaction Hash）
    
2.  打开Etherscan，粘贴交易哈希进行查询
    
3.  可以看到交易的所有详情：发送方、接收方、金额、Gas费、区块高度、确认数等

## 三、概念关系与依赖图

```
用户
  ↓
钱包（管理私钥）→ 签名（用私钥签名）→ 交易（包含操作指令）
  ↓                                      ↓
测试网（实验环境）/主网（真实环境）← 广播交易
  ↓
节点验证交易 → 矿工打包交易 → 支付Gas费给矿工
  ↓
区块链状态变更（账户余额、合约状态）
  ↓
区块浏览器（查询所有链上数据）
```

**核心依赖关系**：

-   没有账户 → 无法拥有资产和发起交易
    
-   没有钱包 → 无法管理私钥和签名交易
    
-   没有签名 → 交易无效，无法被网络接受
    
-   没有交易 → 无法改变区块链状态
    
-   没有Gas → 交易无法被矿工处理
    
-   没有合约 → 只能进行简单转账，无法实现复杂应用
    
-   没有测试网 → 学习和测试会有资产损失风险
    
-   没有区块浏览器 → 无法验证交易和查询链上数据

## 四、常见误区与关键注意事项

1.  **私钥安全第一**：私钥=资产所有权，永远不要泄露给任何人，不要存储在联网设备上
    
2.  **交易不可撤回**：一旦交易被打包进区块，就无法取消或修改，转错地址只能靠对方自愿退回
    
3.  **Gas费不是浪费**：Gas费是网络安全的保障，过低的Gas费会导致交易长时间不被处理
    
4.  **合约风险**：智能合约代码可能存在漏洞，使用前一定要审计代码，不要和未知合约交互
    
5.  **测试网≠主网**：测试网资产没有实际价值，操作前一定要确认自己在正确的网络上
    
6.  **区块浏览器是真相来源**：钱包显示的信息可能有延迟，以区块浏览器的数据为准

AI大模型核心概念详尽笔记

## 一、大语言模型（LLM, Large Language Model）

### 1\. 核心定义

大语言模型是基于海量文本数据训练的深度学习模型，通过学习语言的统计规律和语义结构，能够生成人类可理解的自然语言文本，并执行各种语言理解和生成任务。它本质上是一个"下一个词预测器"，根据输入的上下文，预测最可能出现的下一个词/子词。

### 2\. 关键特性

-   **规模巨大**：拥有数十亿甚至数万亿个参数（如GPT-4o、Claude 3 Opus、Gemini Ultra）
    
-   **涌现能力**：当模型规模达到一定阈值后，会突然具备一些小规模模型没有的能力（如逻辑推理、数学计算、代码生成）
    
-   **上下文窗口**：模型能够处理的最大输入文本长度，单位为token（1个token≈0.75个英文单词或0.5个中文汉字）
    
-   **多模态能力**：现代LLM不仅能处理文本，还能理解图像、音频、视频等多种模态信息
    
-   **通用能力**：无需针对特定任务重新训练，通过提示即可完成翻译、写作、问答、推理等多种任务
    

### 3\. 核心技术原理

-   **Transformer架构**：所有现代LLM的基础，核心是自注意力机制（Self-Attention），能够捕捉文本中不同位置词之间的依赖关系
    
-   **预训练+微调范式**：
    

-   预训练：在海量无标注文本上进行自监督学习，学习通用语言知识
    
-   微调：在少量标注数据上进行监督学习，优化模型在特定任务上的表现
    

-   **对齐技术**：通过RLHF（人类反馈强化学习）、DPO（直接偏好优化）等方法，使模型输出符合人类价值观和偏好
    

### 4\. 典型应用场景

-   自然语言理解：文本分类、情感分析、实体识别、关系抽取
    
-   自然语言生成：文章写作、邮件撰写、代码生成、翻译
    
-   对话系统：智能客服、个人助理、聊天机器人
    
-   知识问答：知识库问答、搜索引擎增强
    
-   推理与决策：逻辑推理、数学计算、问题解决
    

### 5\. 主流模型与发展趋势

-   **闭源模型**：OpenAI GPT系列、Anthropic Claude系列、Google Gemini系列、字节跳动豆包系列
    
-   **开源模型**：Meta Llama系列、Mistral系列、Qwen（通义千问）系列、GLM（智谱清言）系列
    
-   **发展趋势**：模型规模持续增长、多模态融合、推理能力增强、效率提升（量化、蒸馏）、专用模型涌现
    

## 二、提示词工程（Prompt Engineering）

### 1\. 核心定义

提示词工程是设计和优化输入文本（提示词），以引导大语言模型生成高质量、符合预期输出的技术和实践。它是与LLM交互的最基本方式，也是充分发挥LLM能力的关键。

### 2\. 提示词的基本结构

一个完整的提示词通常包含以下部分：

-   **角色设定**：明确模型需要扮演的角色（如"你是一名资深软件工程师"）
    
-   **任务描述**：清晰说明需要完成的具体任务
    
-   **输入数据**：提供任务所需的原始数据或上下文
    
-   **输出要求**：指定输出的格式、长度、风格、语言等
    
-   **约束条件**：明确模型需要遵守的规则和限制
    

### 3\. 核心提示词技巧

-   **清晰明确**：使用具体、无歧义的语言，避免模糊表述
    
-   **分步指令**：将复杂任务分解为多个简单步骤，引导模型逐步完成
    
-   **少样本学习（Few-shot）**：在提示词中提供几个输入输出示例，让模型学习任务模式
    
-   **思维链（Chain of Thought, CoT）**：引导模型先展示推理过程，再给出最终答案，显著提升推理能力
    
-   **自我一致性（Self-Consistency）**：让模型生成多个推理路径，然后选择最一致的答案
    
-   **结构化输出**：要求模型以JSON、Markdown、表格等结构化格式输出，便于后续处理
    

### 4\. 高级提示词技术

-   **提示词模板**：将提示词的固定部分和可变部分分离，便于复用和批量处理
    
-   **提示词链**：将一个复杂任务拆分为多个提示词，前一个提示词的输出作为后一个提示词的输入
    
-   **检索增强生成（RAG）**：将外部知识库检索到的相关信息加入提示词，解决模型知识过时和幻觉问题
    
-   **自动提示词优化**：使用LLM本身或专门工具来生成和优化提示词
    

### 5\. 常见工具与框架

-   提示词管理：PromptPerfect、PromptBase、LangChain Prompt Hub
    
-   提示词测试：PromptLayer、Helicone、LangSmith
    
-   提示词工程平台：Cursor、Copilot Chat、Claude Projects
    

## 三、工作流（Workflow）

### 1\. 核心定义

AI工作流是将多个LLM调用、工具使用、人工审核等步骤按照一定的逻辑顺序和条件组合起来，形成的自动化流程。它用于解决需要多步操作、多角色协作或复杂决策的任务，比单个提示词更强大、更可靠。

### 2\. 工作流的核心组成部分

-   **节点（Node）**：工作流中的基本执行单元，可以是LLM调用、工具调用、条件判断、人工审核等
    
-   **边（Edge）**：连接不同节点的连线，表示执行顺序和数据流向
    
-   **条件分支**：根据某个条件的结果，选择不同的执行路径
    
-   **循环**：重复执行某个步骤直到满足特定条件
    
-   **变量**：在工作流中传递和存储数据的容器
    

### 3\. 工作流的类型

-   **线性工作流**：步骤按照固定顺序依次执行，没有分支和循环
    
-   **分支工作流**：根据条件判断选择不同的执行路径
    
-   **循环工作流**：重复执行某个步骤直到满足终止条件
    
-   **并行工作流**：多个步骤同时执行，提高效率
    
-   **混合工作流**：结合以上多种类型的复杂工作流
    

### 4\. 典型应用场景

-   内容创作：文章大纲生成→内容撰写→润色→排版→审核
    
-   客户服务：问题分类→知识库检索→回答生成→人工审核→发送
    
-   数据处理：数据提取→数据清洗→数据分析→报告生成
    
-   软件开发：需求分析→代码生成→代码审查→测试→部署
    
-   研究助理：文献检索→文献摘要→文献综述→论文撰写
    

### 5\. 主流工作流工具与平台

-   **低代码/无代码平台**：[Make.com](http://Make.com)、n8n、Zapier、Dify、Flowise
    
-   **代码优先框架**：LangChain、LlamaIndex、LangGraph、AutoGen
    
-   **大模型内置工作流**：GPTs、Claude Projects、Gemini Advanced Workflows
    

## 四、智能体（Agent）

### 1\. 核心定义

AI智能体是能够自主感知环境、制定计划、执行动作并根据反馈调整行为的人工智能系统。它比工作流更高级，具备自主决策和学习能力，能够在没有人类持续干预的情况下完成复杂任务。

### 2\. 智能体的核心组件

-   **感知模块**：接收和处理来自环境的信息（文本、图像、音频等）
    
-   **记忆模块**：存储智能体的知识、经验和历史交互信息
    

-   短期记忆：当前对话上下文
    
-   长期记忆：知识库、过往经验
    

-   **规划模块**：将复杂目标分解为子目标，制定执行计划
    
-   **决策模块**：根据当前状态和目标，选择下一步要执行的动作
    
-   **执行模块**：调用工具或直接生成输出，执行决策
    
-   **反思模块**：评估执行结果，总结经验教训，优化未来行为
    

### 3\. 智能体的工作流程

1.  **目标接收**：接收用户指定的目标或任务
    
2.  **任务分解**：将复杂目标分解为多个可执行的子任务
    
3.  **计划制定**：制定完成子任务的具体步骤和顺序
    
4.  **动作执行**：调用工具或生成输出，执行计划中的步骤
    
5.  **结果观察**：观察动作执行后的环境变化和结果
    
6.  **反思调整**：评估结果是否符合预期，如有偏差则调整计划
    
7.  **循环迭代**：重复步骤4-6，直到目标完成
    

### 4\. 智能体的类型

-   **单智能体**：单个智能体独立完成任务
    
-   **多智能体**：多个具有不同角色和能力的智能体协作完成任务
    

-   角色分工：如产品经理、工程师、设计师、测试员
    
-   通信机制：智能体之间通过消息传递进行沟通
    
-   协调机制：解决冲突、分配任务、同步进度
    

-   **专用智能体**：针对特定领域或任务优化的智能体（如代码智能体、研究智能体）
    
-   **通用智能体**：具备广泛能力，能够完成各种不同任务的智能体
    

### 5\. 主流智能体框架与项目

-   **框架**：LangGraph、AutoGen、CrewAI、BabyAGI、AutoGPT
    
-   **知名项目**：Devin（AI软件工程师）、SWE-agent、OpenAI o1、Google Gemini Advanced Agent
    

## 五、工具使用（Tool Use）

### 1\. 核心定义

工具使用是指大语言模型能够调用外部工具来扩展自身能力的技术。LLM本身存在知识过时、计算能力有限、无法与物理世界交互等局限性，通过调用工具可以弥补这些不足，实现更强大的功能。

### 2\. 工具使用的核心原理

-   **工具描述**：用自然语言或结构化格式（如JSON Schema）描述工具的功能、输入参数和输出格式
    
-   **工具选择**：模型根据当前任务需求，判断是否需要调用工具以及调用哪个工具
    
-   **参数生成**：模型生成调用工具所需的参数
    
-   **工具执行**：实际调用外部工具，执行相应的操作
    
-   **结果处理**：模型接收工具返回的结果，并将其整合到回答中
    

### 3\. 常见的工具类型

-   **信息检索工具**：搜索引擎（Google、Bing）、知识库查询、数据库查询
    
-   **计算工具**：计算器、数学计算库（Wolfram Alpha）、统计分析工具
    
-   **代码执行工具**：Python解释器、代码沙箱、终端命令执行
    
-   **文件操作工具**：文件读写、格式转换、压缩解压
    
-   **API调用工具**：调用第三方服务API（如天气、地图、邮件、支付）
    
-   **多模态工具**：图像生成（DALL-E、Midjourney）、语音识别、视频处理
    

### 4\. 工具使用的进阶技术

-   **并行工具调用**：同时调用多个工具，提高执行效率
    
-   **链式工具调用**：前一个工具的输出作为后一个工具的输入
    
-   **工具选择优化**：使用更智能的算法来选择最合适的工具
    
-   **工具错误处理**：当工具调用失败时，能够自动重试或选择替代方案
    
-   **自定义工具**：用户可以根据自己的需求创建和集成自定义工具
    

### 5\. 工具使用的应用场景

-   实时信息查询：天气、新闻、股票价格
    
-   复杂计算：数学题、科学计算、数据分析
    
-   代码开发：代码运行、调试、测试
    
-   自动化任务：邮件发送、日程安排、文件处理
    
-   多模态生成：根据文本描述生成图像、视频
    

## 六、AI编程（AI Coding）

### 1\. 核心定义

AI编程是指利用人工智能技术辅助或自动完成软件开发过程中的各种任务，包括代码生成、代码补全、代码审查、调试、测试、文档生成等。它是LLM最成功的应用领域之一，极大地提高了软件开发效率。

### 2\. AI编程的核心能力

-   **代码补全**：根据上下文自动补全代码片段
    
-   **代码生成**：根据自然语言描述生成完整的函数、类甚至整个项目
    
-   **代码解释**：解释现有代码的功能和工作原理
    
-   **代码审查**：检查代码中的bug、安全漏洞、性能问题和代码风格问题
    
-   **代码重构**：优化代码结构，提高代码的可读性和可维护性
    
-   **调试**：分析错误信息，定位并修复bug
    
-   **测试生成**：自动生成单元测试、集成测试用例
    
-   **文档生成**：为代码自动生成注释和文档
    

### 3\. 主流AI编程工具

-   **代码编辑器插件**：GitHub Copilot、GitHub Copilot Chat、Cursor、CodeLlama、Tabnine
    
-   **独立IDE**：Cursor、GitHub Copilot X、JetBrains AI Assistant
    
-   **代码智能体**：Devin、SWE-agent、CodeLlama Agent、OpenAI o1
    
-   **代码审查工具**：CodeGuru、DeepCode、Sourcery
    
-   **测试生成工具**：Testim、Applitools、Katalon
    

### 4\. AI编程的工作流程

1.  **需求描述**：用自然语言清晰描述需要实现的功能
    
2.  **代码生成**：AI根据需求描述生成初始代码
    
3.  **代码审查**：AI或人类开发者审查代码，发现问题
    
4.  **代码修改**：根据审查意见修改和优化代码
    
5.  **测试运行**：运行代码和测试用例，验证功能正确性
    
6.  **调试修复**：如果发现bug，进行调试和修复
    
7.  **文档生成**：为最终代码生成注释和文档
    

### 5\. AI编程的优势与挑战

-   **优势**：
    

-   显著提高开发效率，减少重复劳动
    
-   帮助开发者学习新技术和最佳实践
    
-   降低编程门槛，让非专业开发者也能创建简单应用
    
-   提高代码质量，减少bug和安全漏洞
    

-   **挑战**：
    

-   生成的代码可能存在bug、安全漏洞或性能问题
    
-   难以理解复杂的业务逻辑和系统架构
    
-   可能生成过时或不兼容的代码
    
-   知识产权和版权问题
    
-   过度依赖AI可能导致开发者能力下降

## 五、网络与安全配置（关键步骤）

### 5.1 云平台安全组配置

必须在云服务器控制台的安全组中开放以下端口：

| 端口 | 用途 | 协议 | 来源IP |
| 22 | SSH远程登录 | TCP | 你的Mac公网IP（推荐）或0.0.0.0/0 |
| 80 | HTTP | TCP | 0.0.0.0/0 |
| 443 | HTTPS | TCP | 0.0.0.0/0 |
| 8080 | Hermes Dashboard（临时） | TCP | 你的Mac公网IP |

**安全提示**：不要将22端口和8080端口开放给0.0.0.0/0，只允许你的Mac IP访问。

### 5.2 配置Nginx反向代理与HTTPS

为了安全地从公网访问Hermes Dashboard，需要配置Nginx反向代理和Let's Encrypt免费SSL证书：

```
# 安装Nginx和Certbot
apt install -y nginx certbot python3-certbot-nginx
```

创建Nginx配置文件：

```
nano /etc/nginx/sites-available/hermes
```

粘贴以下内容（替换`hermes.yourdomain.com`为你的域名）：

```
server {
    listen 80;
    server_name hermes.yourdomain.com;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
```

启用配置并申请SSL证书：

```
# 启用站点
ln -s /etc/nginx/sites-available/hermes /etc/nginx/sites-enabled/

# 验证Nginx配置
nginx -t

# 重启Nginx
systemctl restart nginx

# 申请并自动配置SSL证书
certbot --nginx -d hermes.yourdomain.com
```

现在你可以通过`https://hermes.yourdomain.com`安全地访问Hermes Dashboard了。

# **TC老师Web3+AI时代开发者能力分享会详细总结**

本次分享会由安全领域技术专家TC老师主讲，聚焦**AI时代Web3开发者核心能力**，以「理论答疑+支付系统实战案例+钱包核心原理+行业就业答疑」为核心，深度拆解Web2与Web3的技术关联、AI与开发者的协作边界、Web3核心技术底层逻辑，同时解答大量学员实操、学习、就业相关问题，核心知识点整理如下：

## 一、核心认知答疑：破除Web3与AI开发常见误区

本部分为分享开篇核心，解答开发者最关心的两大基础问题，奠定整体技术认知基调。

### 1\. 有AI加持，开发者是否还需要学习技术基础知识？

**核心结论：AI不会降低开发复杂度，反而对开发者能力要求更高，基础知识和架构能力必不可少**。

-   **人机明确协作边界**：人类开发者负责顶层方案设计、方案合理性评估、AI代码成果验收、问题排查与复盘；AI仅作为辅助工具，负责细化技术方案、梳理数据流、落地编码、补充开发者遗漏的细节逻辑。
    
-   **AI高质量编码的核心前提**：AI编码质量的关键是**自我验证能力**，通过反复编写、校验迭代，输出合规代码，但该过程需要人类开发者把控整体方向，否则AI会出现逻辑偏差、代码漏洞。
    
-   **基础能力的不可替代性**：若开发者底层知识、架构能力薄弱，无法辨别AI输出内容的对错、逻辑漏洞、方向偏差，无法规避AI编码带来的安全风险，最终导致项目隐患。
    

### 2\. Web2与Web3技术体系是否完全割裂？

**核心结论：二者并非独立体系，高度互通，Web3大量复用Web2技术能力**。

-   **技术复用现状**：主流Web3项目（交易所、链上衍生品平台等）中，Web2技术占比极高，后端服务、状态管理、接口交互等核心能力均依赖成熟的Web2技术栈，Web3并非仅局限于智能合约开发。
    
-   **核心差异本质**：Web2完全依赖中心化机构的信用背书（信任权威平台）；Web3通过算法共识、代码透明性，解决中心化信任弊端。
    
-   **Web3安全核心准则**：安全无法事后补救，必须**源于设计、贯穿项目全生命周期**。审计仅为锦上添花，无法兜底安全；Web3安全是0和1的绝对概念，不存在折中，基础架构薄弱则所有安全优化都是空中楼阁。
    

## 二、实战案例拆解：Web2与Web3支付系统技术对比

本次分享以**支付系统**为落地案例，直观展示Web2到Web3的技术升级逻辑，是理解Web3架构的核心切入点。

### 1\. Web2传统支付系统模型

-   **参与主体**：普通用户、电商平台、支付服务、银行金融机构（中心化核心）。
    
-   **完整流程**：用户电商下单→支付服务向银行获取支付单据→用户法币转账付款→银行确认到账并回调支付服务→支付服务同步状态至电商平台→平台完成发货履约。
    
-   **资金流转路径**：用户银行卡 → 支付服务对公银行卡 → 定时清算至电商平台银行卡。
    
-   **安全与信任机制**：服务间通过API Key、Secret Key完成身份校验；用户侧依赖支付密码、MFA多重风控；全程高度信任银行、支付平台等中心化机构，默认机构不会造假、不会作恶。
    
-   **核心痛点**：完全依赖中心化权威，存在机构跑路、数据篡改、规则变更等未知风险。
    

### 2\. Web3链上支付系统升级逻辑

-   **核心改造**：仅将Web2的中心化银行金融机构，替换为**区块链分布式网络**，整体业务架构、资金流转逻辑完全复用Web2体系，学习门槛大幅降低。
    
-   **新增核心组件**：链上监听服务。区块链无法主动对外网服务发起请求，需通过监听、轮询的方式拉取链上交易数据，完成交易解析、状态回调。
    
-   **信任问题解决方案**：① 区块链共识算法保障数据**不可篡改**，杜绝中心化造假；② 智能合约代码公开透明，执行逻辑可全员校验、可追溯。
    
-   **支付服务层变化**：对前端用户、商户而言，使用体验与Web2几乎无差异，底层信任逻辑彻底重构，无需依赖第三方权威机构。
    

## 三、Web3核心模块：钱包体系全维度详解

钱包是Web3生态的核心基础设施，是身份认证、资产管控的核心载体，也是安全风险最高的模块，本次分享做了全方位拆解。

### 1\. 钱包核心本质与签名原理

-   **两大核心功能**：身份认证（证明账户归属权）、资产确权（掌控链上资产交易、操作权限）。
    
-   **签名核心逻辑**：私钥为固定不变的唯一标识，通过「私钥+自定义消息」生成唯一签名；第三方可验证签名合法性，全程无需泄露私钥，依托数学算法绝对安全。
    
-   **关键安全特性**：私钥永久不变，但消息不同则签名不同；**私钥一旦泄露、丢失，资产将完全失控，无任何找回途径**，链上仅识别私钥身份，不识别用户本人。
    

### 2\. 钱包全维度分类体系

（1）按底层技术形态分类

-   **EOA私钥钱包**：传统外部账户，单一私钥掌控全部资产，使用简单但风险集中，私钥泄露即资产归零。
    
-   **合约钱包**：基于链上智能合约实现的钱包，天然支持多签权限，适用于企业、DAO团队、机构资产管理。
    
-   **AA账户抽象钱包**：Web3未来核心趋势，旨在融合EOA钱包与合约钱包的优势，统一账户体系；目前技术尚未完全成熟（EIP-4337、3074、17702均为迭代方案），生态落地较少。
    

（2）按多签技术方案分类

-   **MPC多方签名钱包**：将单一私钥拆分为多份分片，由多方分别持有，需达到预设签名阈值（如2/3、3/5）方可完成交易，无需链上合约支撑，适配多链生态。
    
-   **合约多签钱包**：通过智能合约定义多签权限规则，是目前机构、团队资产管理的主流方案，安全性高、规则透明。
    

（3）按私钥托管模式分类

-   **自托管钱包**：私钥由用户自主持有，安全性最高，资产风险完全由用户承担。
    
-   **全托管钱包**：私钥由平台托管（如主流交易所），用户无需管理私钥、操作便捷，但存在平台跑路、风控冻结风险。
    
-   **混合托管钱包**：结合MPC多签技术，用户与服务商各持有部分私钥分片，单方无法独立完成交易，平衡了安全性与便捷性，是目前主流优化方案。
    

（4）其他细分钱包类型

-   硬件钱包：离线物理设备存储私钥，抵御网络攻击，安全性顶级；
    
-   隐私钱包：基于ZK零知识证明技术，隐匿链上交易信息，保护用户隐私；
    
-   分层钱包：通过路径算法批量管理多账户私钥，适用于企业批量账户管理、BTC生态场景。
    

### 3\. 钱包核心安全风险与行业解决方案

-   **核心风险1**：私钥泄露、丢失，造成不可逆资产损失，无任何补救方式；
    
-   **核心风险2**：签名钓鱼欺骗，恶意平台伪造交易消息，用户误签名后资产被盗；
    
-   **行业防护方案**：通过EIP721签名可视化迭代，优化交易信息展示，降低钓鱼风险；官方废弃高危eth\_sign方法，杜绝恶意交易签名漏洞；
    
-   **核心原则**：开发者必须吃透钱包底层原理，否则所有安全防护都是形式化操作，无法规避系统性风险。
    

## 四、链上交易全生命周期与核心技术参数

### 1\. 链上交易完整流程

构造交易（确定转账地址、金额、自定义数据）→ 私钥签名确权 → 广播交易至区块链节点 → 节点共识、区块打包 → 链上账本状态更新，交易正式生效。

### 2\. 交易核心关键参数

-   **Gas机制**：决定交易上链优先级与手续费成本，核心包含Gas Limit（交易资源消耗上限，固定值）、Gas Price（单位资源手续费，自定义）；
    
-   **EIP1559升级优化**：替代传统Gas定价模型，新增Base Fee（基础手续费）、Tip Fee（矿工小费）、Max Fee（最大手续费上限），解决链上交易拥堵、手续费混乱问题；
    
-   **Data字段（核心灵魂）**：区块链若仅支持转账无任何生态价值，Data字段支持自定义指令、合约交互、数据写入，是所有链上DApp、智能合约生态百花齐放的核心基础，可类比编程语言解析器，执行各类自定义逻辑。
    

### 3\. 链上交易监听实现方案

主流两种技术实现方式，适配不同业务场景：

-   **HTTP轮询**：主动批量查询区块交易数据，实现简单、稳定性高，适合常规业务场景；
    
-   **WebSocket长连接**：订阅区块事件，实时推送交易信息，底层仍为轮询逻辑封装，实时性更强；
    
-   **实操落地建议**：多链监听索引项目，可采用「Go协程+Redis」架构搭建，参考开源项目**ep-usdt**（GM WAS团队开发）的多链收款监听逻辑，适配高并发场景。
    

## 五、学员问答核心干货（学习、实操、就业、赛道前景）

### 1\. 技术实操问题解答

-   **链上小额免密支付**：纯支付层无可行落地方案，核心瓶颈是信任风险；可通过**账户层优化**解决，采用MPC多签拆分权限，设置单日交易限额、高频拦截、时段风控等规则，平衡便捷与安全；
    
-   **大数据在Web3的应用**：需求远低于Web2，非刚需技术，仅在量化交易策略、链上数据分析场景少量落地，普通开发者无需重点深耕；
    
-   **跨链赛道技术现状**：目前行业无完全去中心化的跨链方案，中心化跨链仍是主流；熊市赛道需求低迷，牛市随多链生态繁荣会小幅增长，岗位核心考察风险把控、跨链漏洞识别能力。
    

### 2\. 技术学习与基础搭建

-   **密码学学习深度**：无需深挖复杂理论，以落地为导向，重点掌握MPC算法、椭圆算法、ZK零知识证明的**运行逻辑、交互流程、漏洞风险**，能落地方案、清晰讲解原理即可；
    
-   **文科生入行Web3**：AI时代文理边界大幅弱化，无需深厚理科基础，核心培养逻辑拆解、架构分层、问题分析能力；
    
-   **Debug能力提升**：核心是逆向思维，从运行结果倒推代码逻辑，多拆解字节码、汇编底层逻辑，通过大量实操积累问题排查经验。
    

### 3\. 行业就业与求职技巧

-   **Web3面试考察重点**：一面侧重底层基础知识、技术八股；二三面侧重架构设计、场景解题、风险分析；新增**AI工具使用能力**考察，关注AI落地开发的实际场景；
    
-   **高效求职方法**：对标目标企业岗位JD，让AI拆解能力要求，针对性查漏补缺；优先直接投递项目方官网，海外非华人团队门槛相对宽松；
    
-   **智能合约审计岗**：看重实操能力与项目背书，无捷径，完全对标岗位JD补齐技术栈、积累实战案例；
    
-   **风控赛道前景**：行业合规化是必然趋势，牌照相关风控需求持续增长，岗位需求稳步提升。
    

### 4\. 主流赛道与技术前景判断

-   **AA钱包**：长期必然是行业趋势，以太坊官方主推，适配AI生态、权限管控更灵活，未来生态会持续成熟；
    
-   **Uniswap V4、402协议、Agent支付**：目前落地场景有限，行业普及度低，短期就业和落地前景一般；
    
-   **链上监听服务**：企业主流方案为「付费订阅RPC服务+多源校验」，自研成本高、性价比低，不建议中小企业自研；
    
-   **整体行业现状**：当前熊市整体岗位需求低迷，多数赛道增量有限，等待牛市生态爆发后，多链交互、链上应用、安全风控赛道会迎来需求增长。
    

## 六、分享核心总结

1.  AI是开发者的辅助工具，无法替代底层基础与架构能力，AI时代对Web3开发者的专业门槛更高；
    
2.  Web3并非全新技术体系，高度复用Web2成熟能力，核心革新是**去中心化信任机制**；
    
3.  钱包、签名、链上交易是Web3开发的核心底层，所有上层生态都基于此搭建，安全必须前置设计；
    
4.  入行学习以「落地场景+岗位需求」为导向，拒绝盲目学理论，依托AI高效补齐技术短板，聚焦安全、架构、实操核心能力。
