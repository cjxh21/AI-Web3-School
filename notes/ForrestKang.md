模块 A｜AI 基础：从 LLM 到 Agent

这一模块的核心逻辑是：**理解模型局限，掌控工作流，构建可控 Agent。**

-   **LLM 的本质与局限：** LLM 是基于上下文预测下一个 Token 的概率模型。它擅长推理和生成，但不擅长精确计算和事实记忆。
    
-   **控制的四个维度：**
    
    -   **上下文窗口：** 决定模型能“看”到多少短期信息。
        
    -   **系统指令：** 设定角色、语气和全局边界。
        
    -   **提示词：** 下达具体任务意图。
        
    -   **工具调用：** 让模型从“单向输出文字”跃升为“双向执行动作”。
        
-   **从 Prompt 到 Agent 的演进边界：**
    
    -   **Prompt：** 单次问答，人类做决策。
        
    -   **Workflow：** 预设SOP，AI 只是流水线上的节点，路径固定可控。
        
    -   **Agent：** 目标导向，AI 自主规划路径、动态调用工具。**复杂度最高，失控风险最大。**
        
-   **安全与验证：** AI 的输出必须被质疑。应对事实错误、引用编造、逻辑漂移、工具误用和执行越权，必须引入外部验证、执行追踪以及人工介入机制。
    

## 模块 B｜Web3 基础：账户、签名与执行

这一模块的核心逻辑是：**建立对“所有权与链上执行”的敬畏心。**

-   **账户安全的底线：** 钱包不仅仅是一个账户，更是私钥和资产安全责任的载体。**AI Agent 绝不能直接接触用户的私钥或助记词。**
    
-   **签名的本质：** 签名不等于普通 App 的“点击登录”，而是对一个具体链上动作的**不可逆授权**。
    
-   **链上执行机制：** 区块链（特别是 L1）的执行需要消耗 Gas，且状态和代码完全公开。智能合约具有不可更改性，任何实验都应先在测试网进行。
    
-   **AI × Web3 的基石：账户抽象（Account Abstraction）**
    
    -   传统的外部拥有账户（EOA，即普通钱包）无法安全地交给 AI 使用。
        
    -   通过智能账户（Smart Accounts）、ERC-4337、多签（Safe）等技术，可以实现权限的精细化分割。
        
    -   **Session Key：** 允许 AI 在特定时间、特定限额内自动执行特定合约操作，而无需每次都让用户签名，同时在涉及高风险操作（如大额转账）时触发人工确认拦截，这是实现 AI-native 体验的关键。
        

结合 AI 的自主规划能力与 Web3 的去中心化执行层，构建 AI Agent 的核心在于“能力释放与安全边界”的平衡。AI 正从单纯的概率生成模型进化为具备状态管理、工具调用和长期记忆的 Agent，能够自主处理复杂开放式任务；但由于其固有的幻觉、逻辑漂移与越权风险，当其步入涉及资产、签名且不可篡改的 Web3 世界时，传统的助记词钱包已无法满足需求。必须通过账户抽象、智能账户和会话密钥等 Web3 基础设施，为 AI 构建专属的 AI-native Account，在赋予其链上执行能力的同时，设立严格的权限隔离、限额管理与“人在回路”确认机制，从而实现高效与安全的完美闭环。
